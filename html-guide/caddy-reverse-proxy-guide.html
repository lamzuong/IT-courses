One Server, Many Websites Một Server, Nhiều Website

You have several sites on one server, told apart only by their domain. This guide explains the "receptionist" that reads each domain and sends it to the right place — line by line — and shows how to add new sites whether they run in Docker or not. We use Caddy for a fresh setup; if your server already runs nginx, jump to the nginx section. Bạn có nhiều website trên cùng một server, chỉ phân biệt nhau bằng tên miền. Hướng dẫn này giải thích anh "lễ tân" đọc từng tên miền rồi đưa khách tới đúng nơi — từng dòng một — và chỉ bạn cách thêm site mới dù nó chạy bằng Docker hay không. Mình dùng Caddy cho server làm mới; nếu server của bạn đã chạy nginx, hãy nhảy tới mục nginx.

Why one server needs Caddy Vì sao một server cần Caddy

Here's the core problem. On any server, port 443 (and port 80) can be held by only one program at a time. You can't give five containers -p 443:... each — only the first one grabs the port; the rest fail with "address already in use".

But when a visitor types a.com and another types b.com, both requests arrive at the same server IP, on the same port 443. So how does the server know which website each request wants? It reads the domain name carried inside the request (the Host header). Your app container can't sort that out — it only knows how to serve itself.

So you need one program that owns port 443, reads the domain on every request, and forwards it to the right website. That program is a reverse proxy, and we'll use Caddy.

☕ Café analogy
Your server is a building with one front door (port 443). Inside are several shops (your websites). Without a receptionist, one door can only lead to one shop. Caddy is the receptionist: a guest walks in and says "I'm here for shop B" (that's the domain), and Caddy walks them to the right room. Cloudflare is the security gate on the street that also whispers to the receptionist which shop the guest asked for.
💡 Why Caddy and not Nginx?
Both work. Caddy's config is far shorter, and it can fetch and renew HTTPS certificates automatically. For a handful of sites told apart by domain, Caddy is the friendliest choice.

Đây là vấn đề cốt lõi. Trên bất kỳ server nào, cổng 443 (và cổng 80) tại một thời điểm chỉ một chương trình giữ được. Bạn không thể cho năm container mỗi cái một -p 443:... — chỉ cái đầu tiên chiếm được cổng; các cái sau báo lỗi "address already in use" (cổng đang bị dùng).

Nhưng khi một khách gõ a.com và khách khác gõ b.com, cả hai request đều tới cùng một IP server, cùng cổng 443. Vậy server làm sao biết mỗi request muốn website nào? Nó đọc tên miền nằm bên trong request (gọi là Host header). Container app của bạn không phân loại được chuyện đó — nó chỉ biết phục vụ chính nó.

Vậy nên bạn cần một chương trình giữ cổng 443, đọc tên miền trên từng request, rồi chuyển tiếp tới đúng website. Chương trình đó gọi là reverse proxy, và mình sẽ dùng Caddy.

☕ Ví dụ quán cà phê
Server của bạn là một tòa nhà có một cửa chính duy nhất (cổng 443). Bên trong có nhiều quán (các website của bạn). Không có lễ tân thì một cửa chỉ dẫn vào được một quán. Caddy chính là anh lễ tân: khách bước vào nói "tôi tìm quán B" (đó là tên miền), Caddy dẫn khách tới đúng phòng. Cloudflare là cổng bảo vệ ngoài đường, đồng thời nói nhỏ cho lễ tân biết khách hỏi quán nào.
💡 Sao lại Caddy mà không phải Nginx?
Cả hai đều được. Nhưng cấu hình Caddy ngắn hơn nhiều, và nó tự xin cùng tự gia hạn chứng chỉ HTTPS. Với vài site phân biệt bằng tên miền, Caddy là lựa chọn dễ thở nhất.

How a request flows Một request đi như thế nào

Keep this picture in your head — every later step is just filling in one of these arrows. Caddy sits on port 443, reads the domain, and routes to whichever site matches, no matter how that site happens to run.

                                       Server (one IP address)
                                  ┌───────────────────────────────────────┐
  Visitor                         │                                         │
  types a   ──►  Cloudflare  ──►  │  Caddy  ──┬──►  localhost:8888  (Docker) │
  domain         (the gate,  :443 │  reads    ├──►  localhost:3000  (Node app,│
                  HTTPS)          │  the      │                   no Docker)  │
                                  │  domain   └──►  /var/www/landing (plain   │
                                  │                                  files)   │
                                  └───────────────────────────────────────┘

Notice the three destinations are all "localhost:something" or a folder. That's the trick that makes mixing Docker and non-Docker sites easy: from Caddy's point of view they all look the same — a port on the same machine, or a folder of files.

Giữ bức hình này trong đầu — mọi bước sau chỉ là điền vào một trong các mũi tên này. Caddy ngồi ở cổng 443, đọc tên miền, rồi định tuyến tới site nào khớp, bất kể site đó chạy kiểu gì.

                                       Server (một địa chỉ IP)
                                  ┌───────────────────────────────────────┐
  Khách                           │                                         │
  gõ một   ──►  Cloudflare  ──►   │  Caddy  ──┬──►  localhost:8888  (Docker) │
  tên miền      (cổng bảo   :443  │  đọc      ├──►  localhost:3000  (app Node,│
                vệ, HTTPS)        │  tên      │                   không Docker)│
                                  │  miền     └──►  /var/www/landing (file    │
                                  │                                  tĩnh)    │
                                  └───────────────────────────────────────┘

Để ý cả ba đích đến đều là "localhost:cổng nào đó" hoặc một thư mục. Đó chính là mẹo khiến việc trộn site Docker và không-Docker trở nên dễ: dưới con mắt của Caddy chúng đều giống nhau — một cổng trên cùng máy, hoặc một thư mục chứa file.

1

Install Caddy on the server Cài Caddy trên server

We install Caddy directly on the server (not in a container). Why? Because it makes the receptionist able to reach everything uniformly with localhost — your Docker sites, your non-Docker apps, and plain files all look the same to it. (Running Caddy in Docker is possible but makes the "without Docker" case fiddly.)

On Ubuntu/Debian, the official install is a copy-paste block:

sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install -y caddy

That installs Caddy and registers it as a background service that starts on boot. Check it:

caddy version
systemctl status caddy
✓ Check it worked
caddy version prints a version like v2.x.x, and systemctl status caddy shows active (running). Press q to exit the status view.
⚠️ Free up port 80 and 443 first
If a container is currently holding port 80 (from the Docker guide's -p 80:8888), Caddy can't start. We fix that in step 6 by re-binding the app to 127.0.0.1. Do that before expecting Caddy to take over the public ports.

Mình cài Caddy thẳng trên server (không bỏ trong container). Tại sao? Vì như vậy anh lễ tân với tới được mọi thứ một cách đồng nhất qua localhost — site Docker, app không-Docker, và file tĩnh đều giống nhau dưới mắt nó. (Chạy Caddy trong Docker cũng được, nhưng làm trường hợp "không Docker" rối hơn.)

Trên Ubuntu/Debian, lệnh cài chính thức là một khối copy-paste:

sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install -y caddy

Lệnh đó cài Caddy và đăng ký nó thành một dịch vụ nền tự khởi động khi server bật. Kiểm tra:

caddy version
systemctl status caddy
✓ Kiểm tra đã chạy chưa
caddy version in ra phiên bản kiểu v2.x.x, và systemctl status caddy hiện active (running). Bấm q để thoát màn hình trạng thái.
⚠️ Giải phóng cổng 80 và 443 trước
Nếu một container đang giữ cổng 80 (do -p 80:8888 ở hướng dẫn Docker), Caddy sẽ không khởi động được. Mình sửa ở bước 6 bằng cách buộc app về 127.0.0.1. Làm việc đó trước khi mong Caddy nắm giữ các cổng công khai.
2

Your first Caddyfile Caddyfile đầu tiên của bạn

The whole brain of the receptionist lives in one text file at /etc/caddy/Caddyfile. Open it with a simple editor:

sudo nano /etc/caddy/Caddyfile

Delete what's there and paste this example — three sites of three different kinds, so you can see the pattern at a glance. Replace the domains with your own.

# === Site 1: IT-courses — runs in Docker on port 8888 ===
it-courses.com, www.it-courses.com {
    reverse_proxy localhost:8888
}

# === Site 2: A blog — a Node app (no Docker) on port 3000 ===
blog.example.com {
    reverse_proxy localhost:3000
}

# === Site 3: A landing page — just static HTML files ===
landing.example.com {
    root * /var/www/landing
    file_server
}

Save in nano with Ctrl + O, Enter, then exit with Ctrl + X. Don't apply it yet — first let's understand every line (next step), then we'll reload safely (step 9).

💡 You only keep the blocks you need
This file shows all three styles for teaching. In real life you might have only the first block today, and add more over time. Each { ... } block is one independent website.

Toàn bộ "bộ não" của anh lễ tân nằm trong một file văn bản duy nhất tại /etc/caddy/Caddyfile. Mở nó bằng một trình soạn đơn giản:

sudo nano /etc/caddy/Caddyfile

Xóa nội dung sẵn có và dán ví dụ này — ba site thuộc ba kiểu khác nhau, để bạn nhìn một phát là thấy quy luật. Thay tên miền bằng của bạn.

# === Site 1: IT-courses — chạy trong Docker ở cổng 8888 ===
it-courses.com, www.it-courses.com {
    reverse_proxy localhost:8888
}

# === Site 2: Một blog — app Node (không Docker) ở cổng 3000 ===
blog.example.com {
    reverse_proxy localhost:3000
}

# === Site 3: Một trang landing — chỉ là file HTML tĩnh ===
landing.example.com {
    root * /var/www/landing
    file_server
}

Trong nano, lưu bằng Ctrl + O, Enter, rồi thoát bằng Ctrl + X. Khoan áp dụng vội — mình hiểu từng dòng đã (bước kế), rồi reload an toàn sau (bước 9).

💡 Bạn chỉ giữ những khối mình cần
File này hiện cả ba kiểu để dạy thôi. Thực tế hôm nay bạn có thể chỉ có khối đầu, rồi thêm dần về sau. Mỗi khối { ... } là một website độc lập.
3

The Caddyfile, line by line Caddyfile, từng dòng một

Now the promised part — every line, what it does and why it's there. Caddy's syntax is just: a site address, then a { } block of directives telling Caddy what to do for that site.

LineWhat it does
# === Site 1: ... ===A comment. Any line starting with # is ignored by Caddy — it's only a note to your future self. Label each block so you remember what it is.
it-courses.com, www.it-courses.com {The site address — the domain(s) this block answers for. A comma lets one block serve several names (here the bare domain and the www version). The { opens the block of rules. Caddy automatically gets a free HTTPS certificate for every name listed here.
reverse_proxy localhost:8888The actual job: forward every incoming request to a program listening on port 8888 on this same machine, then pass its reply back to the visitor. That program is your IT-courses container.
}Closes the block. Everything between { and } applies only to that site.
reverse_proxy localhost:3000Same idea, different port. Site 2's app happens to listen on 3000. Caddy doesn't care whether that app is in Docker or not — it's just "a port on localhost".
root * /var/www/landingFor a static site there's no app to forward to — Caddy serves files itself. root sets the folder. The * means "for all request paths, use this folder."
file_serverTurns on the actual file-serving. Together with root, this makes Caddy hand out the HTML/CSS/images in that folder directly. No reverse_proxy needed here.
💡 The mental shortcut
domain → { → either reverse_proxy localhost:PORT (an app) or root + file_server (plain files) → }. That's 95% of everything you'll ever write in a Caddyfile.

Giờ tới phần đã hứa — từng dòng, làm gì và vì sao có mặt. Cú pháp Caddy chỉ gồm: một địa chỉ site, rồi một khối { } chứa các chỉ thị bảo Caddy làm gì cho site đó.

DòngLàm gì
# === Site 1: ... ===Một dòng chú thích. Bất kỳ dòng nào bắt đầu bằng # đều bị Caddy bỏ qua — chỉ là ghi chú cho chính bạn sau này. Đặt nhãn cho mỗi khối để nhớ nó là gì.
it-courses.com, www.it-courses.com {Địa chỉ site — (các) tên miền mà khối này phụ trách. Dấu phẩy cho phép một khối phục vụ nhiều tên (ở đây là tên miền trần và bản www). Dấu { mở khối luật. Caddy tự động xin chứng chỉ HTTPS miễn phí cho mọi tên liệt kê ở đây.
reverse_proxy localhost:8888Việc chính: chuyển tiếp mọi request đến cho chương trình đang lắng nghe ở cổng 8888 trên cùng máy này, rồi trả phản hồi của nó về cho khách. Chương trình đó là container IT-courses của bạn.
}Đóng khối. Mọi thứ giữa { và } chỉ áp dụng cho site đó.
reverse_proxy localhost:3000Cùng ý tưởng, khác cổng. App của Site 2 tình cờ lắng nghe ở 3000. Caddy không quan tâm app đó trong Docker hay không — với nó chỉ là "một cổng trên localhost".
root * /var/www/landingVới site tĩnh thì không có app để chuyển tiếp — Caddy tự phục vụ file. root đặt thư mục. Dấu * nghĩa là "với mọi đường dẫn request, dùng thư mục này".
file_serverBật chức năng phục vụ file thật sự. Cùng với root, nó khiến Caddy đưa trực tiếp HTML/CSS/ảnh trong thư mục đó ra. Ở đây không cần reverse_proxy.
💡 Mẹo ghi nhớ
tên miền → { → hoặc reverse_proxy localhost:CỔNG (một app) hoặc root + file_server (file tĩnh) → }. Đó là 95% mọi thứ bạn sẽ viết trong một Caddyfile.
4

Make your Docker site work behind Caddy Cho site Docker chạy sau Caddy

In the Docker guide you ran the container with -p 80:8888 — that gave the container the building's front door directly. Now Caddy owns the front door, so the container should step back and only be reachable from inside the server. We change the port mapping to bind to 127.0.0.1.

Stop the old container and start it bound to localhost only:

docker rm -f it-courses
docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 127.0.0.1:8888:8888 \
  it-courses

The only change is -p 127.0.0.1:8888:8888 instead of -p 80:8888. Reading it: "publish container port 8888, but only on the server's internal address 127.0.0.1, on port 8888." The outside world can no longer reach 8888 directly — only Caddy can, via localhost:8888, exactly as written in the Caddyfile.

💡 Why 127.0.0.1 and not just leave it open?
127.0.0.1 means "this machine only". It keeps visitors from bypassing Caddy (and your HTTPS) by hitting http://your-ip:8888 directly. The receptionist becomes the only way in — which is exactly what you want.
✓ Check it worked
curl localhost:8888 on the server returns your site's HTML. From the outside, http://your-ip:8888 should now refuse to connect. Good — that means only Caddy can reach it.

Ở hướng dẫn Docker, bạn chạy container với -p 80:8888 — tức là đưa cho container cửa chính của tòa nhà luôn. Giờ Caddy giữ cửa chính, nên container nên lùi lại và chỉ cho phép truy cập từ bên trong server. Mình đổi ánh xạ cổng để gắn vào 127.0.0.1.

Dừng container cũ và chạy lại, gắn chỉ vào localhost:

docker rm -f it-courses
docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 127.0.0.1:8888:8888 \
  it-courses

Thay đổi duy nhất là -p 127.0.0.1:8888:8888 thay cho -p 80:8888. Đọc cho hiểu: "công bố cổng 8888 của container, nhưng chỉ trên địa chỉ nội bộ 127.0.0.1 của server, ở cổng 8888". Thế giới bên ngoài không còn với tới 8888 trực tiếp được nữa — chỉ Caddy với được, qua localhost:8888, đúng như ghi trong Caddyfile.

💡 Sao lại 127.0.0.1 chứ không để mở luôn?
127.0.0.1 nghĩa là "chỉ máy này". Nó ngăn khách lách qua Caddy (và HTTPS của bạn) bằng cách gõ thẳng http://ip-cua-ban:8888. Anh lễ tân trở thành lối vào duy nhất — đúng điều bạn muốn.
✓ Kiểm tra đã chạy chưa
Chạy curl localhost:8888 trên server sẽ trả về HTML của web. Từ bên ngoài, http://ip-cua-ban:8888 giờ phải từ chối kết nối. Vậy là tốt — chỉ Caddy với tới được.
5

Add another site — with Docker Thêm site khác — có Docker

Say you want to launch a second Dockerised site, shop.example.com. The rule of thumb: give each site its own internal port so they don't collide. IT-courses uses 8888, so let's give the shop 8890.

1. Run its container, bound to localhost on its own port:

docker run -d \
  --name shop \
  --restart unless-stopped \
  -p 127.0.0.1:8890:8888 \
  shop-image

Note: the container still listens on 8888 inside (that's baked into the image), but we publish it on the server as 8890. Format is host:container, so 127.0.0.1:8890:8888 = "server's 8890 → container's 8888".

2. Add a block to the Caddyfile pointing at that port:

shop.example.com {
    reverse_proxy localhost:8890
}

3. Reload Caddy (step 9). Done — a new live site in three small moves.

⚠️ Two containers can't share one host port
If you try 127.0.0.1:8888:8888 for the shop too, Docker refuses — 8888 is taken by IT-courses. Always pick a fresh host port (8890, 8891, …) for each new site.

Giả sử bạn muốn ra mắt một site Docker thứ hai, shop.example.com. Quy tắc bỏ túi: mỗi site một cổng nội bộ riêng để chúng không đụng nhau. IT-courses dùng 8888, vậy cho shop dùng 8890.

1. Chạy container của nó, gắn vào localhost ở cổng riêng:

docker run -d \
  --name shop \
  --restart unless-stopped \
  -p 127.0.0.1:8890:8888 \
  shop-image

Lưu ý: bên trong container vẫn lắng nghe ở 8888 (điều này nằm sẵn trong image), nhưng mình công bố ra server ở cổng 8890. Định dạng là host:container, nên 127.0.0.1:8890:8888 = "cổng 8890 của server → cổng 8888 của container".

2. Thêm một khối vào Caddyfile trỏ tới cổng đó:

shop.example.com {
    reverse_proxy localhost:8890
}

3. Reload Caddy (bước 9). Xong — một site mới chạy thật chỉ trong ba động tác nhỏ.

⚠️ Hai container không thể chung một cổng host
Nếu bạn cũng thử 127.0.0.1:8888:8888 cho shop, Docker sẽ từ chối — 8888 đã bị IT-courses chiếm. Luôn chọn một cổng host mới (8890, 8891, …) cho mỗi site mới.
6

Add another site — without Docker Thêm site khác — không Docker

Not every site needs Docker. There are two common no-Docker shapes, and Caddy handles both. Pick the one that matches your site.

A — An app running directly (Node, Python, PHP…)

Say you run a Node app straight on the server and it listens on port 3000 (for example started with npm start, or kept alive by pm2). There's nothing Docker-specific to do — it's just a program on a port. Point Caddy at it:

blog.example.com {
    reverse_proxy localhost:3000
}

That's the same reverse_proxy line as a Docker site. To Caddy, "an app on port 3000" is identical whether Docker started it or you did by hand.

💡 Keep a non-Docker app alive
A bare npm start dies when you close SSH. Tools like pm2 (npm i -g pm2, then pm2 start npm -- start) or a small systemd service keep it running 24/7 and restart it on reboot — the job Docker's --restart did for you before.

B — Plain static files (HTML/CSS/JS, no app at all)

If the site is just files — a landing page, a built React/Vite export, a portfolio — there's no app to proxy to. Caddy serves the folder itself. First put your files somewhere, e.g.:

sudo mkdir -p /var/www/landing
# copy your index.html and assets into that folder

Then the block is just two lines:

landing.example.com {
    root * /var/www/landing
    file_server
}

root says where the files live; file_server says "hand them out." No port, no app, no Docker. Caddy even sets the right cache and content-type headers for you.

💡 Which shape is mine?
If you run a command to "start" the site → it's an app, use A (reverse_proxy). If the site is a folder of files you just need to display → use B (root + file_server).

Không phải site nào cũng cần Docker. Có hai kiểu không-Docker thường gặp, và Caddy lo được cả hai. Chọn kiểu khớp với site của bạn.

A — Một app chạy trực tiếp (Node, Python, PHP…)

Giả sử bạn chạy một app Node thẳng trên server và nó lắng nghe ở cổng 3000 (ví dụ khởi động bằng npm start, hoặc được pm2 giữ sống). Không có gì riêng của Docker phải làm cả — nó chỉ là một chương trình trên một cổng. Trỏ Caddy tới nó:

blog.example.com {
    reverse_proxy localhost:3000
}

Đó là đúng dòng reverse_proxy giống như site Docker. Với Caddy, "một app ở cổng 3000" là như nhau, bất kể Docker khởi động nó hay bạn tự tay chạy.

💡 Giữ app không-Docker sống
Một lệnh npm start trơ trọi sẽ chết khi bạn đóng SSH. Công cụ như pm2 (npm i -g pm2, rồi pm2 start npm -- start) hoặc một dịch vụ systemd nhỏ sẽ giữ nó chạy 24/7 và tự bật lại khi reboot — đúng việc mà --restart của Docker đã làm hộ bạn trước đây.

B — File tĩnh thuần (HTML/CSS/JS, không có app)

Nếu site chỉ là file — một trang landing, bản export React/Vite đã build, một portfolio — thì không có app để proxy tới. Caddy tự phục vụ thư mục. Trước hết đặt file vào đâu đó, ví dụ:

sudo mkdir -p /var/www/landing
# copy index.html và các file của bạn vào thư mục đó

Rồi khối cấu hình chỉ gồm hai dòng:

landing.example.com {
    root * /var/www/landing
    file_server
}

root nói file nằm ở đâu; file_server nói "đưa chúng ra". Không cổng, không app, không Docker. Caddy còn tự đặt header cache và content-type đúng giúp bạn.

💡 Site của mình thuộc kiểu nào?
Nếu bạn phải chạy một lệnh để "khởi động" site → đó là app, dùng A (reverse_proxy). Nếu site là một thư mục file chỉ cần hiển thị → dùng B (root + file_server).
7

Apply your changes safely Áp dụng thay đổi an toàn

After every edit to the Caddyfile, do these two things — in this order. First, check the file for mistakes before applying it, so a typo can't take all your sites down:

sudo caddy validate --config /etc/caddy/Caddyfile

If it says Valid configuration, apply it with a reload — this swaps in the new config with zero downtime; visitors don't notice:

sudo systemctl reload caddy
✓ Check it worked
Visit each domain in a browser. They should load over https:// with a padlock. systemctl status caddy stays active (running).
💡 reload vs restart
reload = re-read the config without dropping connections (use this normally). restart = fully stop and start (a brief blip; only needed if Caddy is truly stuck). Prefer reload.
⚠️ See the real reason it failed
If a reload doesn't behave, read the log: sudo journalctl -u caddy --no-pager -n 50. It tells you exactly which line or which certificate caused trouble.

Sau mỗi lần sửa Caddyfile, làm hai việc này — theo đúng thứ tự. Trước hết, kiểm tra file có lỗi không trước khi áp dụng, để một lỗi gõ nhầm không kéo sập toàn bộ site của bạn:

sudo caddy validate --config /etc/caddy/Caddyfile

Nếu nó báo Valid configuration, áp dụng bằng lệnh reload — lệnh này thay cấu hình mới vào mà không gián đoạn; khách không hề hay biết:

sudo systemctl reload caddy
✓ Kiểm tra đã chạy chưa
Vào từng tên miền trên trình duyệt. Chúng phải lên qua https:// kèm ổ khóa. systemctl status caddy vẫn active (running).
💡 reload và restart khác gì
reload = đọc lại cấu hình mà không ngắt kết nối (dùng cái này bình thường). restart = dừng hẳn rồi bật lại (gián đoạn một thoáng; chỉ cần khi Caddy thật sự kẹt). Ưu tiên reload.
⚠️ Xem lý do thật khi hỏng
Nếu reload không như ý, đọc log: sudo journalctl -u caddy --no-pager -n 50. Nó nói chính xác dòng nào hoặc chứng chỉ nào gây trục trặc.
8

HTTPS with Cloudflare HTTPS cùng Cloudflare

Caddy gets HTTPS certificates by itself — normally you write nothing. But because your domains sit behind Cloudflare's orange cloud, there's one detail to get right, or you'll see redirect loops or certificate warnings. Pick the mode that fits.

Mode 1 — Grey cloud / no Cloudflare proxy

If a domain's cloud in Cloudflare is grey (DNS only), visitors reach your server directly. Caddy then fetches a real Let's Encrypt certificate automatically — your plain blocks already do this, nothing to add. Easiest, but you lose Cloudflare's caching and IP hiding.

Mode 2 — Orange cloud (recommended): Origin Certificate

With the orange cloud on, Cloudflare wants the server to present a certificate it trusts. The clean way is a free Cloudflare Origin Certificate (valid ~15 years) that you hand to Caddy.

  • In Cloudflare: SSL/TLS → Origin Server → Create Certificate. Save the certificate and private key to the server, e.g. /etc/caddy/origin.pem and /etc/caddy/origin.key.
  • Set SSL/TLS → Overview → Full (strict).
  • Tell Caddy to use that certificate with one extra line per site:
it-courses.com, www.it-courses.com {
    tls /etc/caddy/origin.pem /etc/caddy/origin.key
    reverse_proxy localhost:8888
}

The new tls line says "use this exact certificate instead of fetching one." Everything else is unchanged. Reload Caddy and you're encrypted end to end.

⚠️ ERR_TOO_MANY_REDIRECTS
Almost always Cloudflare set to Flexible while Caddy also redirects to HTTPS — they bounce forever. Use Full (strict) with the Origin Certificate above and the loop disappears.
💡 The Origin Certificate covers all your sites
When you create it, list every domain and a wildcard (*.example.com) so the one certificate works for every block. Reuse the same tls line in each.

Caddy tự lấy chứng chỉ HTTPS — bình thường bạn không phải viết gì. Nhưng vì tên miền của bạn nằm sau đám mây cam của Cloudflare, có một chi tiết phải làm đúng, nếu không bạn sẽ gặp lặp chuyển hướng hoặc cảnh báo chứng chỉ. Chọn chế độ phù hợp.

Chế độ 1 — Mây xám / không proxy qua Cloudflare

Nếu đám mây của một tên miền trong Cloudflare là màu xám (chỉ DNS), khách vào thẳng server. Khi đó Caddy tự xin một chứng chỉ Let's Encrypt thật — các khối thường của bạn đã làm điều này rồi, không thêm gì cả. Dễ nhất, nhưng mất cache và giấu IP của Cloudflare.

Chế độ 2 — Mây cam (khuyên dùng): Origin Certificate

Khi bật mây cam, Cloudflare muốn server xuất trình một chứng chỉ mà nó tin tưởng. Cách gọn là một Cloudflare Origin Certificate miễn phí (hạn ~15 năm) mà bạn giao cho Caddy.

  • Trong Cloudflare: SSL/TLS → Origin Server → Create Certificate. Lưu chứng chỉ và khóa riêng lên server, ví dụ /etc/caddy/origin.pem và /etc/caddy/origin.key.
  • Đặt SSL/TLS → Overview → Full (strict).
  • Bảo Caddy dùng chứng chỉ đó bằng một dòng thêm cho mỗi site:
it-courses.com, www.it-courses.com {
    tls /etc/caddy/origin.pem /etc/caddy/origin.key
    reverse_proxy localhost:8888
}

Dòng tls mới nói "dùng đúng chứng chỉ này thay vì tự đi xin". Mọi thứ còn lại giữ nguyên. Reload Caddy là bạn được mã hóa từ đầu tới cuối.

⚠️ ERR_TOO_MANY_REDIRECTS
Gần như luôn do Cloudflare để chế độ Flexible trong khi Caddy cũng chuyển hướng sang HTTPS — chúng nảy qua nảy lại mãi. Dùng Full (strict) cùng Origin Certificate ở trên là hết lặp.
💡 Một Origin Certificate dùng cho mọi site
Khi tạo, hãy liệt kê mọi tên miền và một wildcard (*.example.com) để một chứng chỉ chạy cho mọi khối. Dùng lại đúng dòng tls trong từng khối.
🔀

Already running nginx? Add your site to it Đã có nginx sẵn? Thêm site vào đó

If you tried to start Caddy and saw bind: address already in use on port 80, check who holds it:

sudo ss -tlnp | grep -E ':80 |:443 '

If that shows nginx, then nginx is already the receptionist serving your other sites. Do not run Caddy too — two receptionists can't share one door, and starting a second one risks knocking your other sites offline. Instead, just hand nginx one more name card: "when someone asks for the IT-courses domain, send them to port 8888."

☕ Café analogy
Nginx is the receptionist who's already on duty. You're not hiring a second one — you're adding a line to her book: "IT-courses → room 8888."

1. Stop Caddy (nginx wins this server)

sudo systemctl stop caddy
sudo systemctl disable caddy

2. Bind the container to localhost:8888

So nginx can reach it and outsiders can't bypass it:

docker rm -f it-courses
docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 127.0.0.1:8888:8888 \
  it-courses

Check it's alive: curl localhost:8888 should return your site's HTML.

3. Give nginx a config for this site

Create a new file (replace the domain with your real subdomain):

sudo nano /etc/nginx/sites-available/it-courses.conf
server {
    listen 80;
    server_name it-courses.yourdomain.com;

    location / {
        proxy_pass http://127.0.0.1:8888;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade           $http_upgrade;
        proxy_set_header Connection        "upgrade";
    }
}
LineWhat it does
listen 80;This block handles visitors arriving on port 80 (nginx keeps owning that port for every site).
server_name it-courses...;The key line — nginx only uses this block when the visitor's domain matches. Your other sites have their own server_name, so they're untouched.
location / { … }Applies to every path of this site.
proxy_pass http://127.0.0.1:8888;Forwards the request into the IT-courses container on port 8888.
proxy_set_header Host $host;Keeps the original domain when passing inwards (Next.js needs to know which domain was asked for).
X-Forwarded-Proto $scheme;Tells the app whether the visitor arrived via http or https.
Upgrade + Connection "upgrade"Lets WebSockets work (Next.js uses them for some live features).

4. Enable it, test, reload

sudo ln -s /etc/nginx/sites-available/it-courses.conf /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

nginx -t checks the syntax first. Only reload if it says syntax is ok and test is successful — that way a typo can't take your other sites down.

5. Point the subdomain in Cloudflare

In Cloudflare → DNS → Add record: an A record, Name = your subdomain (e.g. it-courses), Content = your server's IP, orange cloud on. For HTTPS, match whatever mode your other nginx sites already use.

💡 How are your other sites doing HTTPS?
Peek at an existing site's config: ls /etc/nginx/sites-available/, then open one and look for listen 443 ssl; and ssl_certificate. If they use a Cloudflare Origin Certificate or certbot, copy the same listen 443 block into your new file so IT-courses gets HTTPS the same way.

Nếu bạn thử khởi động Caddy và thấy lỗi bind: address already in use ở cổng 80, hãy xem ai đang giữ nó:

sudo ss -tlnp | grep -E ':80 |:443 '

Nếu kết quả hiện nginx, thì nginx đã là anh lễ tân phục vụ các site khác của bạn. Đừng chạy thêm Caddy — hai lễ tân không thể chung một cửa, và bật cái thứ hai lên có thể làm sập các site đang chạy. Thay vào đó, chỉ cần đưa cho nginx thêm một tấm danh thiếp: "khi ai hỏi tên miền IT-courses, dẫn họ tới cổng 8888".

☕ Ví dụ quán cà phê
Nginx là anh lễ tân đang trực sẵn. Bạn không thuê thêm người thứ hai — bạn chỉ ghi thêm vào sổ của anh ấy một dòng: "IT-courses → phòng 8888".

1. Tắt Caddy (server này thuộc về nginx)

sudo systemctl stop caddy
sudo systemctl disable caddy

2. Gắn container vào localhost:8888

Để nginx với tới được, còn người ngoài không lách qua mặt được:

docker rm -f it-courses
docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 127.0.0.1:8888:8888 \
  it-courses

Kiểm tra nó sống: curl localhost:8888 phải trả về HTML của web.

3. Tạo cấu hình nginx cho site này

Tạo một file mới (thay tên miền bằng subdomain thật của bạn):

sudo nano /etc/nginx/sites-available/it-courses.conf
server {
    listen 80;
    server_name it-courses.yourdomain.com;

    location / {
        proxy_pass http://127.0.0.1:8888;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade           $http_upgrade;
        proxy_set_header Connection        "upgrade";
    }
}
DòngLàm gì
listen 80;Khối này xử lý khách đến ở cổng 80 (nginx vẫn giữ cổng đó cho mọi site).
server_name it-courses...;Dòng mấu chốt — nginx chỉ dùng khối này khi tên miền khách gõ khớp. Các site khác có server_name riêng nên không bị ảnh hưởng.
location / { … }Áp dụng cho mọi đường dẫn của site này.
proxy_pass http://127.0.0.1:8888;Chuyển tiếp request vào container IT-courses ở cổng 8888.
proxy_set_header Host $host;Giữ nguyên tên miền gốc khi chuyển vào trong (Next.js cần biết khách gõ tên miền nào).
X-Forwarded-Proto $scheme;Báo cho app biết khách vào bằng http hay https.
Upgrade + Connection "upgrade"Cho phép WebSocket hoạt động (Next.js dùng cho một số tính năng động).

4. Bật site, kiểm tra, reload

sudo ln -s /etc/nginx/sites-available/it-courses.conf /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

nginx -t kiểm tra cú pháp trước. Chỉ reload khi nó báo syntax is ok và test is successful — như vậy một lỗi gõ nhầm không làm sập các site khác.

5. Trỏ subdomain trong Cloudflare

Vào Cloudflare → DNS → Add record: một bản ghi A, Name = subdomain của bạn (ví dụ it-courses), Content = IP server, bật mây cam. Phần HTTPS thì làm theo đúng chế độ mà các site nginx khác của bạn đang dùng.

💡 Các site khác đang chạy HTTPS kiểu gì?
Ngó qua cấu hình một site có sẵn: ls /etc/nginx/sites-available/, rồi mở một file và tìm dòng listen 443 ssl; và ssl_certificate. Nếu chúng dùng Origin Certificate của Cloudflare hoặc certbot, hãy chép đúng khối listen 443 đó vào file mới để IT-courses có HTTPS theo cùng cách.
