Why Lightsail (and what's actually flat-rate)

Most AWS services bill per-request. Lightsail is the exception — you pick a plan, you pay that amount on the 1st of every month, full stop. The bill arrives the same shape every time.

Lightsail Linux instance plans (Singapore region, 2026 pricing)

Nano

512 MB · 2 vCPU · 20 GB SSD · 1 TB
Micro
$5/mo
1 GB · 2 vCPU · 40 GB SSD · 2 TB
Small
$10/mo
2 GB · 2 vCPU · 60 GB SSD · 3 TB

$20/mo

The "Small" plan ($10/mo) is the one to pick — Next.js builds need ~1.5 GB of RAM, the Micro at 1 GB will swap and time out during next build. The Small leaves comfortable headroom for the app to run plus rebuild at the same time.

What's included (no per-request fees)

  • The instance itself, running 24/7
  • One free static IP (so your IP doesn't change on reboots)
  • 3 free DNS hosted zones (you can manage your domain right from Lightsail)
  • 3 TB outbound data transfer per month (huge — most small sites use under 50 GB)
  • Built-in firewall and metrics
★

You can stop paying any month by deleting the instance. Unlike a yearly contract, Lightsail is month-to-month. Hourly billing if you delete mid-month means you only pay for what you used.

Prep checklist

Before clicking anything in AWS, make sure these are true:

  • Your project builds locally — run npm run build from PowerShell, no errors.
  • You have a GitHub account at github.com.
  • Git is installed on Windows — git --version in PowerShell should print a version. If not, install from git-scm.com.
  • You have a credit/debit card for AWS sign-up (won't be charged until you create the instance).
  • You have your domain registrar login ready (Namecheap, GoDaddy, Hostinger, wherever you bought it).
  • Modern Windows 10/11 — your SSH client is built-in (no PuTTY needed). Confirm with ssh -V.
3

Create AWS account & set the billing alarm

3.1 — Sign up (skip if done)

Go to aws.amazon.com → Create an AWS Account. Email, password, card, phone for SMS, pick the Basic Support — Free plan.

3.2 — Turn on MFA on the root account

Top-right account name → Security credentials → Assign MFA device. Use Google Authenticator, Authy, or 1Password. Do this before anything else.

3.3 — Set a $15 budget alarm (your safety net)

Console → search Billing and Cost Management → Budgets → Create budget:

  • Choose Customize (advanced) → Cost budget
  • Period: Monthly
  • Budget amount: $15
  • Alert at 80% threshold ($12) — email yourself
  • Alert at 100% threshold ($15) — email yourself again

Save. AWS now emails you the moment your bill is on track to exceed $15. With Lightsail-only at $10/mo, this should never fire — but the alarm is the proof that it can't.

Push your project to GitHub

You'll git clone from GitHub onto the server. Make sure the project is up there first.

4.1 — Create a new GitHub repo

Go to github.com/new. Name it it-courses. Pick Private (you'll set up a deploy key later for SSH). Do not check any init options.

4.2 — Push from PowerShell

# From your project folder
cd C:\Users\ADMIN\Desktop\claude-demo\IT-courses

git status
git add .
git commit -m "prepare for Lightsail deploy"

# Paste your repo's URL after `origin`
git remote add origin https://github.com/YOUR-USERNAME/it-courses.git
git branch -M main
git push -u origin main
i

If GitHub asks you to log in, the modern flow opens a browser window. Approve there. The old "type your password" flow was deprecated in 2021.

5

Create the Lightsail instance

5.1 — Open Lightsail

Go to lightsail.aws.amazon.com (it's a separate console from the main AWS one). Top-right region picker: pick the one closest to your visitors. Singapore (ap-southeast-1) for SE Asia, Mumbai for South Asia, N. Virginia for US/global.

5.2 — Click "Create instance"

Choose:

  • Instance location — the region you just picked. Pick any availability zone (a, b, or c — doesn't matter for one instance).
  • Platform — Linux/Unix.
  • Blueprint — OS Only → Ubuntu 24.04 LTS. Don't pick the "Node.js" pre-built blueprint — it ships with old Apache config that fights Caddy.
  • Instance plan — the $10/month — 2 GB RAM Linux plan (the "Small" tier you saw earlier).
  • Identify your instance — name it it-courses-prod.

Click Create instance. After ~30 seconds the instance state goes from Pending to Running.

★

Why Ubuntu 24.04? It's the latest LTS — supported until 2029. Modern Node.js installers, modern Caddy package, less troubleshooting than older Ubuntu LTS releases.

6

Attach a free static IP

Without this, your instance's public IP changes every time you reboot — which means your domain stops working until you update DNS again. The static IP pins the address forever (and it's free as long as it's attached to a running instance).

  • Lightsail console → Networking tab → Create static IP
  • Region: same as your instance
  • Attach to: it-courses-prod
  • Name it: it-courses-ip
  • Create

You'll see something like 13.250.xxx.xxx — copy this address, you'll need it for DNS later. Paste it in a sticky note or wherever you keep these.

!

If you ever detach a static IP from a running instance, Lightsail starts charging $0.005/hour (~$3.65/mo) for it as an unused IP. Either keep it attached or delete it — never just unlink and leave it floating.

7

Open ports 80 and 443

By default Lightsail only opens port 22 (SSH). Web traffic uses 80 (HTTP) and 443 (HTTPS), so you need to add those.

  • Lightsail console → click your instance → Networking tab
  • Scroll to IPv4 Firewall
  • Click Add rule → Application: HTTP (port 80) → Create
  • Click Add rule again → Application: HTTPS (port 443) → Create

Your firewall now has three rules: SSH (22), HTTP (80), HTTPS (443). That's all you need.

8

SSH into the server

8.1 — Download the SSH key

Lightsail console → Account (top right) → SSH keys tab → find the default key for your region → Download. You get a file like LightsailDefaultKey-ap-southeast-1.pem.

Move it somewhere safe — the standard place on Windows is C:\Users\ADMIN\.ssh\ (create the folder if it doesn't exist):

# In PowerShell, from your Downloads folder
mkdir $env:USERPROFILE\.ssh -Force
Move-Item LightsailDefaultKey-ap-southeast-1.pem $env:USERPROFILE\.ssh\

# Lock down permissions (Windows will refuse to use a key that's too open)
icacls "$env:USERPROFILE\.ssh\LightsailDefaultKey-ap-southeast-1.pem" /inheritance:r
icacls "$env:USERPROFILE\.ssh\LightsailDefaultKey-ap-southeast-1.pem" /grant:r "$env:USERNAME:R"

8.2 — Connect

Replace YOUR.STATIC.IP.HERE with the static IP you copied:

ssh -i "$env:USERPROFILE\.ssh\LightsailDefaultKey-ap-southeast-1.pem" ubuntu@YOUR.STATIC.IP.HERE

First time you connect, type yes when asked about the host fingerprint. You should land at a prompt like ubuntu@ip-172-26-x-x:~$ — that's the server. You're inside the box now. Every command from here until step 12 runs there.

★

The browser-SSH shortcut. Lightsail console → click your instance → orange Connect using SSH button at the top opens an in-browser terminal. No keys, no setup. Use it for one-off commands; use real SSH from PowerShell for actual work because copy-paste is friendlier.

9

Install Node 22, Git, & build tools

You're SSH'd into the Ubuntu instance. Run these in order. Lines starting with # are comments you can skip — paste the actual commands.

# Update the package index and upgrade
sudo apt update
sudo apt upgrade -y

# Install Node.js 22 LTS via NodeSource (official Node packages)
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs

# Verify (you should see v22.x and a matching npm)
node --version
npm --version

# Install git and basic build tools (some npm packages need them)
sudo apt install -y git build-essential

# Install PM2 globally (process manager — keeps Next.js running)
sudo npm install -g pm2

Why Node 22? Next.js 16 + React 19 needs Node 18.18 or newer. Node 22 LTS is supported through April 2027 — install it once and stop thinking about it.

10

Clone & build the app

10.1 — Generate an SSH key for the server-to-GitHub connection

Your repo is private, so the server needs to authenticate with GitHub. Easiest path: deploy keys.

# Still on the server. Generate a key pair.
ssh-keygen -t ed25519 -C "lightsail-it-courses" -f ~/.ssh/github -N ""

# Print the public key so you can copy it
cat ~/.ssh/github.pub

Copy the entire ssh-ed25519 AAAA… line that prints. Then in your browser:

  • GitHub → your it-courses repo → Settings → Deploy keys → Add deploy key
  • Title: lightsail · Key: paste · Allow write access: NO (read-only is what we want)
  • Add key

Now tell SSH on the server to use this key for GitHub:

cat >> ~/.ssh/config <<'EOF'
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/github
  IdentitiesOnly yes
EOF
chmod 600 ~/.ssh/config

# Test it
ssh -T git@github.com
# You should see: "Hi YOUR-USERNAME/it-courses! You've successfully authenticated..."

10.2 — Clone, install, build

# Clone (use the SSH URL, not HTTPS)
cd ~
git clone git@github.com:YOUR-USERNAME/it-courses.git
cd it-courses

# Install dependencies (ci is faster + reproducible)
npm ci

# Build the production bundle (this is the longest step, ~2-4 min)
npm run build

# Quick sanity check — run it manually for 5 seconds
npm start &
sleep 5
curl -s http://localhost:3000 | head -c 200
# You should see the start of your homepage HTML.
# Stop the foreground process:
kill %1
i

If npm run build fails with "JavaScript heap out of memory," the instance ran out of RAM during the build. Fix on the next line:

10.3 — Add swap (only if build OOM'd)

# Skip if build worked. Otherwise, add 2 GB of swap.
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

Then re-run npm run build. The swap is "fake RAM" backed by disk — slower but enough to push past the spike during build.

11

Run the app under PM2 (auto-restart forever)

If you just npm start and close your SSH session, the app dies. PM2 keeps it running, restarts it on crash, and brings it back automatically when the server reboots.

# From the project folder
cd ~/it-courses

# Start Next.js under PM2, named "it-courses"
pm2 start npm --name "it-courses" -- start

# See it running
pm2 status

# Save the current process list so it auto-resurrects on reboot
pm2 save

# Generate the startup hook (PM2 prints a sudo command — copy and run it)
pm2 startup
# Example output: "sudo env PATH=$PATH:/usr/bin pm2 startup systemd -u ubuntu --hp /home/ubuntu"
# Run that exact command back. PM2 is now wired into systemd.

Verify everything's working:

curl -s http://localhost:3000 | head -c 100
# → first 100 chars of your HTML homepage. The app is live on port 3000 inside the box.

pm2 logs it-courses --lines 20
# → last 20 lines of stdout/stderr. Hit Ctrl+C to exit the log view.
★

PM2 also gives you pm2 monit — a live dashboard inside the terminal showing CPU/memory per process. Useful when troubleshooting.

12

Caddy as the public-facing reverse proxy

Right now Next.js listens on localhost:3000, which only the server itself can see. You need something on port 80 / 443 that takes web traffic and forwards it to Next.js. That's a "reverse proxy."

Use Caddy, not Nginx. Caddy automatically issues and renews a free Let's Encrypt SSL certificate for your domain. One config line, no Certbot, no cron jobs, no annual cert renewal anxiety.

12.1 — Install Caddy

# From the official Caddy install script (Ubuntu)
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' \
  | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' \
  | sudo tee /etc/apt/sources.list.d/caddy-stable.list

sudo apt update
sudo apt install -y caddy

# Verify
caddy version

12.2 — Write the Caddyfile

Replace yourname.com with your actual domain in the lines below.

sudo nano /etc/caddy/Caddyfile

Wipe the default content and replace with exactly this (use Ctrl+K repeatedly in nano to clear lines, then paste):

yourname.com, www.yourname.com {
  reverse_proxy localhost:3000

  encode gzip zstd

  header {
    Strict-Transport-Security "max-age=31536000; includeSubDomains"
    X-Frame-Options DENY
    X-Content-Type-Options nosniff
    Referrer-Policy strict-origin-when-cross-origin
  }
}

Save: Ctrl+O, Enter, then Ctrl+X to exit nano.

12.3 — Reload Caddy

# Validate the file first (catches typos before Caddy crashes)
sudo caddy validate --config /etc/caddy/Caddyfile

# Apply
sudo systemctl reload caddy

# Make sure it's running
sudo systemctl status caddy
# Look for: "active (running)" in green. Press q to exit.

Caddy is now waiting on ports 80 and 443. The moment your domain points here, it'll auto-fetch a Let's Encrypt cert. (We do that in the next step.)

i

Caddy stores its acquired certificates in /var/lib/caddy/.local/share/caddy/certificates/. They auto-renew 30 days before expiry, every 60 days. You'll never touch them.

13

Point your domain at the static IP

Open a new tab, log in to wherever you bought the domain (Namecheap, GoDaddy, Hostinger, etc.). Find the DNS or Advanced DNS section and add these two records:

TypeName / HostValueTTL
A@ (or empty / yourname.com)YOUR.STATIC.IP.HERE5 min (300s)
AwwwYOUR.STATIC.IP.HERE5 min (300s)

Two records, same IP, both type A. The @ covers yourname.com; the www covers www.yourname.com. Both will redirect-and-render through Caddy.

13.1 — Wait for propagation

Back in PowerShell on your Windows machine, check periodically:

nslookup yourname.com
nslookup www.yourname.com

You're waiting for both to resolve to your static IP. Usually under 10 minutes; sometimes up to 30 minutes; almost never the full 48 hours people warn about.

13.2 — Watch Caddy fetch the cert

The moment DNS resolves, Caddy notices and contacts Let's Encrypt. Watch it happen:

# On the server
sudo journalctl -u caddy -f
# Press Ctrl+C to exit when you see "certificate obtained successfully" lines.

Now visit https://yourname.com in your browser. Padlock in the address bar. Page loads. You shipped.

!

If you see "Caddy can't reach Let's Encrypt" in the logs, port 80 isn't open. Re-check step 7 — both 80 and 443 must be in the firewall.

Deploy updates — the daily flow

You make changes locally, push to GitHub, then SSH in and pull-build-reload. Here's the full update flow:

On your Windows machine

cd C:\Users\ADMIN\Desktop\claude-demo\IT-courses

# Make changes, test with: npm run dev

git add .
git commit -m "fix typo in lesson 11"
git push

On the server (SSH in, then)

cd ~/it-courses
git pull
npm ci
npm run build
pm2 reload it-courses

# 5-second verification
curl -s -o /dev/null -w "%{http_code}\n" https://yourname.com
# Should print 200

Make it one command — write a deploy script

Save yourself five steps. On the server:

nano ~/deploy.sh

Paste this:

#!/usr/bin/env bash
set -e
cd ~/it-courses
echo "Pulling..."     && git pull
echo "Installing..."  && npm ci
echo "Building..."    && npm run build
echo "Reloading..."   && pm2 reload it-courses
echo "Done."

Save (Ctrl+O, Enter, Ctrl+X). Make it executable:

chmod +x ~/deploy.sh

Now from any future SSH session, your full deploy is just:

~/deploy.sh
★

Want fully-auto deploys on every push? Set up a GitHub Actions workflow that SSHs into your Lightsail box and runs ~/deploy.sh. There's a popular action — appleboy/ssh-action — that does this in 15 lines of YAML. Worth doing once you're comfortable with the manual flow.

Snapshots & backups (recommended)

A snapshot is a full image of your instance — disk + config — saved into Lightsail's storage. If you ever break something badly (bad sudo, deleted the wrong folder, ransomware), you restore from the most recent snapshot in 5 minutes and your site is back.

Manual snapshot

Lightsail console → your instance → Snapshots tab → Create snapshot. Takes ~2 minutes. Storage is $0.05/GB/mo. A 60 GB Small instance using ~20 GB → ~$1/month per snapshot kept.

Automatic daily snapshots

Same tab → Enable automatic snapshots → pick a time (early morning when traffic is low). Lightsail keeps the last 7 daily snapshots automatically.

Cost calculation: 7 daily snapshots × ~$1/each = ~$7/month. Combined with your $10 instance, that's $17/month — over your $15 budget.

Three ways to fit under the cap:

  • Manual weekly snapshot only — keep one snapshot, delete the older one when you make a new one. ~$1/mo total. Good enough for a portfolio site.
  • Automatic snapshots, retention reduced to 3 days — ~$3/mo. Still under your budget at $13/mo total.
  • Skip snapshots entirely — your code is in GitHub, your config files are in this guide. You can rebuild from scratch in 90 minutes if disaster strikes. $0/mo extra.
i

Recommended starter posture: manual weekly snapshot before any risky change. Take one right now (after your first successful deploy) so you have a clean baseline to roll back to.

17

Troubleshooting — things that bite beginners

npm run build dies with "JavaScript heap out of memory"

The build briefly needs more than 2 GB of RAM. You have two fixes:

  1. Add 2 GB of swap (step 10.3 in this guide). Free; works fine for occasional builds.
  2. Bump NODE_OPTIONS for the build only:
NODE_OPTIONS="--max-old-space-size=4096" npm run build

Add this to ~/deploy.sh if it happens regularly.

I can't SSH — "Permission denied (publickey)"

Three things to verify:

  • The key file path in ssh -i is correct and the file actually exists.
  • The username is ubuntu (not root or ec2-user — those are other Linux distros).
  • The key file's permissions on Windows aren't too open. The icacls commands in step 8.1 fix this.

Still stuck? Use Lightsail console's browser SSH (orange button on the instance page) to get in, then debug from there.

Caddy logs say "no certificate available" or "tls: handshake failure"

Three checks, in order:

  1. nslookup yourname.com from your Windows machine — does it return your static IP? If not, DNS hasn't propagated. Wait.
  2. Lightsail firewall — both port 80 AND port 443 must be open. (Caddy needs port 80 to complete the Let's Encrypt challenge.)
  3. Caddyfile typo — run sudo caddy validate --config /etc/caddy/Caddyfile. Fix anything it complains about, then sudo systemctl reload caddy.
Site loads on the IP but not on the domain (or vice versa)

If http://YOUR.STATIC.IP.HERE works but https://yourname.com doesn't: DNS isn't pointing at the IP yet. Wait for propagation, then sudo systemctl reload caddy to retry the cert.

If the domain works but the IP doesn't: that's expected and correct. Caddy now redirects all non-domain traffic, which is good security hygiene — leave it.

PM2 says it's running but the site shows "Bad Gateway"

Caddy can't reach Next.js on localhost:3000. Check:

pm2 logs it-courses --lines 50
# If the app crashed at startup, the error is here.

curl -v http://localhost:3000
# Should print HTML. If "connection refused", Next.js isn't actually listening.

ss -tlnp | grep 3000
# Confirms a process owns port 3000.

Most common: npm run build wasn't run (or failed) before pm2 start. PM2 ran npm start, which crashed because there's no .next directory. Build first, then PM2.

The instance got slow and unresponsive

Almost always one of:

  • Out of disk — df -h shows / at 100%. Usually node_modules from old build attempts. npm cache clean --force and rm -rf ~/it-courses/.next then re-build.
  • Out of memory — free -h shows everything in swap. Stop PM2, build, restart PM2: pm2 stop all && npm run build && pm2 start all.
  • Runaway log — du -sh ~/.pm2/logs/*. If any log is over 100 MB, truncate: pm2 flush.
I want to roll back a bad deploy

Two paths, depending on what broke:

  1. Bad code commit — on the server: cd ~/it-courses && git log --oneline -10 shows recent commits. Revert with git reset --hard PREVIOUS_HASH then re-run ~/deploy.sh.
  2. Bad system change (broken Caddyfile, package install gone wrong) — Lightsail console → Snapshots tab → Restore the most recent good snapshot. Creates a new instance from that point. Detach the static IP from the broken one, attach to the new one, done.
I'm getting a higher Lightsail bill than expected

Lightsail bills are flat per plan. If yours is higher than $10/mo, the extras come from:

  • Snapshots — every kept snapshot is ~$0.05/GB/mo. Delete old ones from the Snapshots tab.
  • Detached static IPs — anything not attached to a running instance is ~$3.65/mo. Lightsail console → Networking → check for orphans, delete them.
  • Data transfer overage — only matters if you exceed 3 TB/month. A small site is nowhere near. Verify on the Metrics tab.
  • An accidentally created second instance — yes, this happens. Lightsail console homepage → check the Instances list.

What to learn next

You've done the hardest deployment a beginner does. From here, in order of "next worth learning":

  • GitHub Actions auto-deploy — push to main, GitHub SSHs into your box and runs ~/deploy.sh. 15 lines of YAML. Removes the manual SSH step.
  • fail2ban — auto-bans IPs that hammer your SSH port with bad credentials. sudo apt install fail2ban, default config is fine.
  • UFW firewall as a second layer — Lightsail's firewall blocks at the network edge; UFW blocks at the OS level. Belt-and-braces.
  • A real database — when your site needs persistence, Lightsail also has managed databases at flat rates ($15/mo for the smallest Postgres or MySQL). Or run SQLite right on the instance for free.
  • Lightsail Distribution (CDN) — $2.50/mo for 50 GB of cached delivery. Speeds up image-heavy sites globally. Add later if you have international traffic.
  • Multiple sites on one instance — your $10 box can host 3–5 small Next.js sites. Each gets its own folder, port, PM2 process, and Caddy block. Same hardware, no extra cost.

You shipped.

Next.js is running on a box you control, behind your own domain, with auto-renewing HTTPS. The bill is $10–11/month, every month, with no surprise spikes. That's the whole job — you can leave it alone for a year and it'll keep working.
