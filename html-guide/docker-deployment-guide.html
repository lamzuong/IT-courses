The big picture Bức tranh tổng thể

Before any commands, let's get the idea straight with a café analogy — it makes every later step click.

Imagine your website is a signature drink. To serve it to the whole world without surprises, you do three things — and that's exactly the three phases of this guide:

💻 Phase 1 — On your computer
You perfect the recipe and seal it into a meal-kit (a Docker image): the drink plus every ingredient, pre-measured. Then you put the recipe on a shared shelf (GitHub) so the server can grab it.
🖥️ Phase 2 — On the server
You travel to the actual shop (your server), grab the recipe off the shelf, open the kit, and start the kitchen (a running container) so it serves drinks 24/7.
☁️ Phase 3 — Connect your domain
You put a doorman (Cloudflare) on the street with your shop's name, who greets every customer, locks the door (HTTPS), and points them inside.

That's the whole story. Now let's walk it, slowly.

Trước khi gõ bất kỳ lệnh nào, mình nắm ý tưởng đã — dùng ví dụ quán cà phê cho dễ hình dung, rồi các bước sau sẽ "thông" hết.

Hãy tưởng tượng website của bạn là một ly nước signature. Để phục vụ cả thế giới mà không bị lỗi, bạn làm ba việc — và đó đúng là ba giai đoạn của hướng dẫn này:

💻 Giai đoạn 1 — Trên máy của bạn
Bạn hoàn thiện công thức rồi niêm phong thành một hộp nguyên liệu (Docker image): ly nước cộng mọi nguyên liệu đã cân đo. Sau đó đặt công thức lên kệ chung (GitHub) để server lấy về.
🖥️ Giai đoạn 2 — Trên server
Bạn tới tận quán (server của bạn), lấy công thức trên kệ xuống, mở hộp, và khởi động gian bếp (container đang chạy) để nó phục vụ 24/7.
☁️ Giai đoạn 3 — Nối tên miền
Bạn cho một anh lễ tân (Cloudflare) ra đứng ngoài đường với bảng tên quán, đón từng khách, khóa cửa (HTTPS), rồi chỉ khách vào trong.

Toàn bộ câu chuyện là vậy. Giờ mình đi từng bước thật chậm.

Test the build locally Chạy thử build trên máy

Why first? Never package something you haven't seen work. If the app builds and runs on your machine, Docker will almost certainly succeed too. If it fails here, you fix it now — comfortably, without a server in the mix.

In the project folder, run:

npm install
npm run build
npm start

Then open http://localhost:8888 in your browser (this project runs on port 8888). Click around — make sure pages load.

✓ Check it worked
npm run build ends with no red errors, and the site opens at localhost:8888. Press Ctrl + C in the terminal to stop it when you're happy.
⚠️ Fix build errors here, not later
A build error on your laptop is the same error Docker would hit — just easier to read. Solve it now before moving on.

Tại sao làm trước? Đừng bao giờ đóng gói thứ mà bạn chưa thấy nó chạy. Nếu app build và chạy được trên máy bạn, gần như chắc chắn Docker cũng sẽ thành công. Nếu hỏng ở đây, bạn sửa ngay — thoải mái, chưa dính tới server.

Trong thư mục dự án, chạy:

npm install
npm run build
npm start

Rồi mở http://localhost:8888 trên trình duyệt (dự án này chạy ở cổng 8888). Bấm dạo quanh — xem các trang có lên không.

✓ Kiểm tra đã chạy chưa
npm run build kết thúc không có lỗi đỏ, và web mở được ở localhost:8888. Khi ưng rồi, bấm Ctrl + C trong terminal để dừng.
⚠️ Sửa lỗi build ở đây, đừng để sau
Lỗi build trên laptop chính là lỗi Docker sẽ gặp — chỉ là dễ đọc hơn. Giải quyết ngay trước khi đi tiếp.
3

Turn on standalone output Bật chế độ standalone

Why? By default, a Next.js build needs the giant node_modules folder to run — like shipping your whole pantry with every meal-kit. The standalone option tells Next.js to trace exactly which files are needed and bundle a tiny self-contained server. That makes the Docker image dramatically smaller and faster.

Open next.config.mjs and make sure the output line is present inside the config object:

/** @type {import('next').NextConfig} */
const nextConfig = {
  output: "standalone",        // ← needed for Docker
  pageExtensions: ["ts", "tsx", "mdx"],
  turbopack: {
    root: __dirname,
  },
};
✓ What this gives you
After a build, Next.js creates .next/standalone/ containing a server.js you can run with plain Node — no node_modules install needed inside the final image.

Tại sao? Mặc định, bản build Next.js cần nguyên thư mục node_modules khổng lồ mới chạy — giống mỗi hộp nguyên liệu lại phải kèm cả cái kho. Tùy chọn standalone bảo Next.js dò chính xác file nào cần và gói thành một server tí hon tự chạy. Nhờ vậy image Docker nhỏ và nhanh hơn rất nhiều.

Mở next.config.mjs và đảm bảo dòng output có mặt bên trong object config:

/** @type {import('next').NextConfig} */
const nextConfig = {
  output: "standalone",        // ← cần cho Docker
  pageExtensions: ["ts", "tsx", "mdx"],
  turbopack: {
    root: __dirname,
  },
};
✓ Bạn được gì
Sau khi build, Next.js tạo thư mục .next/standalone/ chứa file server.js chạy được bằng Node thuần — không cần cài lại node_modules trong image cuối.
4

Create Dockerfile & .dockerignore Tạo Dockerfile & .dockerignore

The Dockerfile is the recipe card for your meal-kit — line by line, how to assemble the box. We use a two-stage build: one big kitchen to cook (install + build), then a tiny clean box that keeps only the finished result, so the final image stays small.

In the project root (next to package.json), create a file named exactly Dockerfile:

# ───── Stage 1: builder — install deps and build the app ─────
FROM node:22-alpine AS builder
WORKDIR /app

# Install dependencies first (better layer caching)
COPY package.json package-lock.json* ./
RUN npm ci

# Copy the rest of the source and build
COPY . .
RUN npm run build

# ───── Stage 2: runner — tiny image with only what runs ─────
FROM node:22-alpine AS runner
WORKDIR /app

ENV NODE_ENV=production
# The app listens on 8888 (see package.json)
ENV PORT=8888
ENV HOSTNAME=0.0.0.0

# Copy the self-contained server, static assets, and public files
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 8888
CMD ["node", "server.js"]

Next to it, create a .dockerignore so junk like your local node_modules isn't dragged into the build (slow, and can cause OS-mismatch bugs):

node_modules
.next
.git
.gitignore
npm-debug.log
Dockerfile
.dockerignore
README.md
.env*.local
html-guide
⚠️ The HOSTNAME line matters
HOSTNAME=0.0.0.0 tells the server to accept connections from outside the container. Without it, the site works inside the box but nothing can reach it.

File Dockerfile chính là tấm thẻ công thức cho hộp nguyên liệu — từng dòng ghi cách lắp ráp chiếc hộp. Mình dùng build hai tầng: một gian bếp lớn để nấu (cài + build), rồi một chiếc hộp nhỏ sạch sẽ chỉ giữ thành phẩm, nhờ vậy image cuối rất nhẹ.

Ở thư mục gốc dự án (ngay cạnh package.json), tạo file tên đúng là Dockerfile:

# ───── Tầng 1: builder — cài thư viện và build app ─────
FROM node:22-alpine AS builder
WORKDIR /app

# Cài thư viện trước (tận dụng cache tốt hơn)
COPY package.json package-lock.json* ./
RUN npm ci

# Copy phần mã nguồn còn lại rồi build
COPY . .
RUN npm run build

# ───── Tầng 2: runner — image tí hon chỉ chứa thứ để chạy ─────
FROM node:22-alpine AS runner
WORKDIR /app

ENV NODE_ENV=production
# App lắng nghe ở cổng 8888 (xem package.json)
ENV PORT=8888
ENV HOSTNAME=0.0.0.0

# Copy server tự chạy, file tĩnh và thư mục public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 8888
CMD ["node", "server.js"]

Bên cạnh đó, tạo file .dockerignore để rác như node_modules ở máy không bị lôi vào build (vừa chậm, vừa dễ gây lỗi lệch hệ điều hành):

node_modules
.next
.git
.gitignore
npm-debug.log
Dockerfile
.dockerignore
README.md
.env*.local
html-guide
⚠️ Dòng HOSTNAME rất quan trọng
HOSTNAME=0.0.0.0 bảo server chấp nhận kết nối từ bên ngoài container. Thiếu nó thì web vẫn chạy trong hộp nhưng không ai vào được.
5

Test the image locally Thử image ngay trên máy

Why bother before the server? Because catching a problem on your own laptop is ten times easier than debugging it over SSH. Let's seal the kit and open it right here first.

From the project root, build the image:

docker build -t it-courses .

Reading it: build = assemble the box, -t it-courses = name (tag) it, and the . = "use this folder as the recipe source." Now run it, mapping your local port 8888 to the container's 8888:

docker run -d --name it-courses -p 8888:8888 it-courses

Open http://localhost:8888 — you should see the exact same site, now running inside Docker. 🎉

✓ Check it worked
docker ps shows it-courses as "Up", and the site loads. When done testing, clean up: docker rm -f it-courses.
💡 If it works here, it works on the server
That's the whole point of Docker — the box behaves the same everywhere. Once this passes, you can ship with confidence.

Sao phải thử trước khi lên server? Vì bắt lỗi ngay trên laptop dễ hơn gấp mười lần ngồi gỡ qua SSH. Mình niêm phong hộp và mở nó ngay tại đây trước.

Ở thư mục gốc dự án, build image:

docker build -t it-courses .

Đọc cho hiểu: build = lắp ráp hộp, -t it-courses = đặt tên (tag), còn dấu . = "lấy thư mục này làm nguồn công thức". Giờ chạy nó, ánh xạ cổng 8888 của máy sang cổng 8888 của container:

docker run -d --name it-courses -p 8888:8888 it-courses

Mở http://localhost:8888 — bạn sẽ thấy đúng website đó, nhưng giờ đang chạy trong Docker. 🎉

✓ Kiểm tra đã chạy chưa
docker ps hiện it-courses trạng thái "Up", và web mở được. Thử xong thì dọn dẹp: docker rm -f it-courses.
💡 Chạy được ở đây thì chạy được trên server
Đó chính là điểm hay của Docker — chiếc hộp hành xử y hệt ở mọi nơi. Qua được bước này là bạn yên tâm mang đi.
6

Push your code to GitHub Đẩy code lên GitHub

Now we put the recipe on the shared shelf. The cleanest way to get code onto a server is Git: you push from your laptop, the server pulls it down. No messy file copying.

From the project root:

git add .
git commit -m "Add Docker deployment setup"
git push
💡 First time? Connect the repo once
If this project isn't linked to GitHub yet, create an empty repo there, then run: 
git remote add origin https://github.com/YOUR_NAME/it-courses.git
git branch -M main
git push -u origin main
✓ Check it worked
Refresh your repo page on GitHub — your latest files (including the Dockerfile) should be there.
⚠️ No secrets in Git
Never commit passwords or API keys. Keep them in .env*.local files (already git-ignored) and set them on the server separately.

Giờ mình đặt công thức lên kệ chung. Cách gọn nhất để đưa code lên server là dùng Git: bạn push từ laptop, server kéo về. Không phải copy file lằng nhằng.

Ở thư mục gốc dự án:

git add .
git commit -m "Add Docker deployment setup"
git push
💡 Lần đầu? Nối repo một lần thôi
Nếu dự án chưa nối với GitHub, tạo một repo trống trên đó, rồi chạy: 
git remote add origin https://github.com/TEN_CUA_BAN/it-courses.git
git branch -M main
git push -u origin main
✓ Kiểm tra đã chạy chưa
Tải lại trang repo trên GitHub — các file mới nhất (gồm cả Dockerfile) phải có ở đó.
⚠️ Không đưa thông tin bí mật lên Git
Đừng bao giờ commit mật khẩu hay API key. Để chúng trong file .env*.local (đã được git bỏ qua) và đặt riêng trên server.

Connect to your server (SSH) Kết nối vào server (SSH)

Everything from here happens on the server, not your laptop. SSH is how you open a remote terminal into it — like calling the shop and telling staff what to do. Your VPS provider gave you an IP address and a username (often root).

From your computer's terminal (PowerShell works fine on Windows):

ssh root@YOUR_SERVER_IP

The first time, it asks you to confirm the fingerprint — type yes. Then enter your password (or it logs in automatically if you set up an SSH key).

✓ Check it worked
Your prompt changes to something like root@server:~#. You're now typing commands on the server. Everything below runs here until Phase 3.
💡 Mental model
Two computers, two terminals. If a command isn't working, first ask: "am I on my laptop, or on the server right now?"

Từ đây mọi thứ diễn ra trên server, không phải laptop của bạn. SSH là cách mở một terminal từ xa vào server — như gọi điện tới quán và bảo nhân viên làm gì. Nhà cung cấp VPS đã cho bạn một địa chỉ IP và một tên đăng nhập (thường là root).

Từ terminal trên máy bạn (PowerShell trên Windows dùng tốt):

ssh root@IP_SERVER_CUA_BAN

Lần đầu, nó hỏi xác nhận fingerprint — gõ yes. Rồi nhập mật khẩu (hoặc tự đăng nhập nếu bạn đã cài SSH key).

✓ Kiểm tra đã chạy chưa
Dấu nhắc đổi thành kiểu root@server:~#. Giờ bạn đang gõ lệnh trên server. Mọi lệnh bên dưới chạy ở đây cho tới Giai đoạn 3.
💡 Cách hình dung
Hai máy tính, hai terminal. Nếu lệnh không chạy, hãy tự hỏi trước: "mình đang ở laptop, hay đang ở server?"
8

Install Docker on the server Cài Docker trên server

Your laptop has Docker; the server needs it too. The server is where the kitchen will actually run 24/7. On Ubuntu, the official install is one line:

curl -fsSL https://get.docker.com | sh

Then confirm it's alive:

docker --version
docker run hello-world
✓ Check it worked
docker --version prints a version number, and hello-world prints the friendly greeting. Docker is ready on the server.
💡 Not on Ubuntu?
For other Linux flavours, search "install Docker Engine on [your OS]" on the official Docker docs. The one-liner above covers Ubuntu/Debian, which most VPS providers use by default.

Laptop của bạn đã có Docker; server cũng cần. Server mới là nơi gian bếp thật sự chạy 24/7. Trên Ubuntu, lệnh cài chính thức chỉ một dòng:

curl -fsSL https://get.docker.com | sh

Rồi xác nhận nó sống:

docker --version
docker run hello-world
✓ Kiểm tra đã chạy chưa
docker --version in ra số phiên bản, và hello-world in ra lời chào thân thiện. Docker đã sẵn sàng trên server.
💡 Không dùng Ubuntu?
Với các bản Linux khác, tìm "install Docker Engine on [hệ điều hành của bạn]" trên tài liệu chính thức của Docker. Lệnh một dòng ở trên dùng cho Ubuntu/Debian, là loại đa số VPS dùng mặc định.
9

Get your code onto the server Đưa code lên server

Remember the recipe you put on the shelf (GitHub) in Step 6? Now the server grabs it. Still in your SSH session, clone the repo:

git clone https://github.com/YOUR_NAME/it-courses.git
cd it-courses

That downloads all your project files into a folder on the server and moves you inside it.

✓ Check it worked
Run ls — you should see your project files, including Dockerfile and package.json.
💡 Private repo?
If your GitHub repo is private, Git will ask for credentials. The easy modern way is a personal access token as the password, or set up an SSH deploy key. Public repos clone with no login.
💡 No GitHub? Two alternatives
You can skip Git and either (a) copy files straight over with scp -r ./it-courses root@YOUR_SERVER_IP:/root/ from your laptop, or (b) build the image locally and push it to Docker Hub, then docker pull on the server. Git is the simplest to keep updating, so we use it here.

Nhớ cái công thức bạn đặt lên kệ (GitHub) ở Bước 6 chứ? Giờ server lấy nó xuống. Vẫn trong phiên SSH, clone repo:

git clone https://github.com/TEN_CUA_BAN/it-courses.git
cd it-courses

Lệnh đó tải toàn bộ file dự án vào một thư mục trên server và đưa bạn vào trong đó.

✓ Kiểm tra đã chạy chưa
Chạy ls — bạn phải thấy các file dự án, gồm cả Dockerfile và package.json.
💡 Repo riêng tư?
Nếu repo GitHub của bạn ở chế độ riêng tư, Git sẽ hỏi thông tin đăng nhập. Cách hiện đại dễ nhất là dùng personal access token làm mật khẩu, hoặc cài SSH deploy key. Repo công khai thì clone không cần đăng nhập.
💡 Không dùng GitHub? Hai cách khác
Bạn có thể bỏ qua Git và hoặc (a) copy thẳng file qua bằng scp -r ./it-courses root@IP_SERVER:/root/ từ laptop, hoặc (b) build image trên máy rồi push lên Docker Hub, sau đó docker pull trên server. Git là cách dễ cập nhật nhất nên mình dùng ở đây.
10

Build & run the container Build & chạy container

This is the moment the kitchen opens. Inside the project folder on the server, build the image (same command as your local test):

docker build -t it-courses .

Then run it. Note the difference from your laptop test: here we map the server's port 80 (the normal web port Cloudflare talks to) to the app's port 8888:

docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 80:8888 \
  it-courses

Reading the flags:

  • -d — run in the background, so closing SSH doesn't stop it.
  • --restart unless-stopped — if the server reboots, Docker starts the site again automatically.
  • -p 80:8888 — connect the building's front door (80) to the kitchen window (8888). Format is host:container.
✓ Check it worked
Run docker ps (status "Up"), then visit http://YOUR_SERVER_IP in a browser. The site should load over the raw IP. 🎉
💡 Everyday commands
Logs: docker logs -f it-courses · Stop: docker stop it-courses · Start: docker start it-courses · Remove: docker rm -f it-courses

Đây là khoảnh khắc gian bếp mở cửa. Trong thư mục dự án trên server, build image (cùng lệnh với lúc thử ở máy):

docker build -t it-courses .

Rồi chạy nó. Lưu ý khác với lúc thử trên laptop: ở đây mình ánh xạ cổng 80 của server (cổng web tiêu chuẩn mà Cloudflare nói chuyện) sang cổng 8888 của app:

docker run -d \
  --name it-courses \
  --restart unless-stopped \
  -p 80:8888 \
  it-courses

Đọc từng tùy chọn:

  • -d — chạy nền, đóng SSH cũng không tắt web.
  • --restart unless-stopped — server khởi động lại thì Docker tự bật web lên lại.
  • -p 80:8888 — nối cửa chính tòa nhà (80) với cửa sổ gian bếp (8888). Định dạng host:container.
✓ Kiểm tra đã chạy chưa

Open the firewall Mở tường lửa

If the site loaded over the raw IP in the last step, your firewall is probably already open and you can skip ahead. If it didn't, the firewall is the usual culprit — it's the building's security guard, and by default it may turn web visitors away.

On Ubuntu's UFW, allow the web ports (and keep SSH open so you don't lock yourself out):

sudo ufw allow 22
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable
⚠️ Always allow port 22 first
Port 22 is SSH — your way in. If you enable the firewall without allowing it, you can lock yourself out of the server. Allow 22 before ufw enable.
💡 Cloud provider firewall too
On Cloudfly/AWS/etc. there's often a second firewall in the provider's web panel (a "security group"). Make sure ports 80 and 443 are allowed there as well.

Nếu ở bước trước web đã lên qua IP thô, nhiều khả năng tường lửa đã mở sẵn và bạn có thể bỏ qua. Nếu chưa lên, tường lửa thường là thủ phạm — nó là anh bảo vệ tòa nhà, và mặc định có thể chặn khách web.

Với UFW của Ubuntu, mở các cổng web (và giữ SSH mở để khỏi tự khóa mình ra ngoài):

sudo ufw allow 22
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable
⚠️ Luôn mở cổng 22 trước
Cổng 22 là SSH — đường vào của bạn. Nếu bật tường lửa mà chưa cho phép nó, bạn có thể tự khóa mình khỏi server. Hãy mở 22 trước khi ufw enable.
💡 Còn cả tường lửa của nhà cung cấp
Trên Cloudfly/AWS… thường có thêm một tường lửa thứ hai trong bảng điều khiển web của nhà cung cấp ("security group"). Nhớ mở cổng 80 và 443 ở đó nữa.

Point Cloudflare DNS Trỏ DNS Cloudflare

The site runs, but only by raw IP. Now we put the doorman out front so yourdomain.com leads people to it. This is a DNS record — a signpost mapping your domain to your server's IP.

In the Cloudflare dashboard, pick your domain, then DNS → Records → Add record:

TypeNameContentProxy
A@your server's public IPProxied (orange ☁️)
Awwwyour server's public IPProxied (orange ☁️)

@ means the bare domain (yourdomain.com); the www row covers www.yourdomain.com. Keep the cloud orange — that routes traffic through Cloudflare so you get HTTPS, caching, and your real server IP stays hidden.

💡 Orange vs grey cloud
Grey = Cloudflare only stores the signpost and steps aside; visitors hit your server directly (no HTTPS help, IP exposed). Orange = Cloudflare sits in the middle as the doorman. You want orange.
✓ Check it worked
DNS can take a few minutes. Then visit http://yourdomain.com — your site should appear. The doorman is on duty.

Web đã chạy, nhưng mới vào được bằng IP thô. Giờ mình cho anh lễ tân ra đứng trước cửa để tenmiencuaban.com dẫn khách tới đúng nơi. Đây là một bản ghi DNS — tấm biển chỉ đường nối tên miền với IP server.

Trong bảng điều khiển Cloudflare, chọn tên miền, rồi vào DNS → Records → Add record:

TypeNameContentProxy
A@IP công khai của serverProxied (mây cam ☁️)
AwwwIP công khai của serverProxied (mây cam ☁️)

@ nghĩa là tên miền trần (tenmiencuaban.com); dòng www lo cho www.tenmiencuaban.com. Giữ đám mây màu cam — đó là cách traffic đi qua Cloudflare để có HTTPS, có cache, và IP thật của server được giấu kín.

💡 Mây cam và mây xám
Xám = Cloudflare chỉ giữ tấm biển rồi né ra; khách vào thẳng server (không HTTPS, lộ IP). Cam = Cloudflare đứng giữa làm lễ tân. Bạn cần màu cam.
✓ Kiểm tra đã chạy chưa
DNS có thể mất vài phút. Sau đó vào http://tenmiencuaban.com — web phải hiện lên. Anh lễ tân đã vào ca.

Turn on HTTPS Bật HTTPS

The little padlock 🔒 is the lock on your shop's door — it tells visitors the connection is private. Cloudflare hands it out for free. The trip has two halves: visitor → Cloudflare and Cloudflare → your server.

Set the encryption mode

Go to SSL/TLS → Overview and choose:

Flexible — easiest start
Visitor → Cloudflare is encrypted; Cloudflare → server is plain HTTP on port 80. No certificate needed on your server. Works immediately with the -p 80:8888 setup from Step 10.
Full (strict) — most secure
Both halves encrypted. Needs a certificate on your server — a free Cloudflare Origin Certificate plus a reverse proxy. Aim for this once everything works.

Force HTTPS for everyone

Under SSL/TLS → Edge Certificates, turn on Always Use HTTPS. Now anyone typing http:// is gently redirected to the secure https:// version.

💡 Upgrading to Full (strict) later
Generate a free Origin Certificate in Cloudflare, install it in a reverse proxy like Caddy or Nginx in front of your container on port 443, then switch the mode. Optional — Flexible is fine to launch with.
⚠️ Redirect loop (ERR_TOO_MANY_REDIRECTS)?
Usually Flexible mode plus the app also forcing HTTPS, or leftover page rules. If it happens, switch the mode to Full and re-test.

Cái ổ khóa nhỏ 🔒 là ổ khóa cửa quán — báo cho khách biết kết nối là riêng tư. Cloudflare cấp miễn phí. Chuyến đi có hai chặng: khách → Cloudflare và Cloudflare → server của bạn.

Chọn chế độ mã hóa

Vào SSL/TLS → Overview và chọn:

Flexible — dễ bắt đầu nhất
Chặng khách → Cloudflare được mã hóa; chặng Cloudflare → server là HTTP thường ở cổng 80. Không cần chứng chỉ trên server. Chạy được ngay với cấu hình -p 80:8888 ở Bước 10.
Full (strict) — an toàn nhất
Cả hai chặng đều mã hóa. Cần chứng chỉ trên server — một Origin Certificate miễn phí của Cloudflare cộng một reverse proxy. Nhắm tới mức này khi mọi thứ đã chạy.

Bắt buộc HTTPS cho mọi người

Vào SSL/TLS → Edge Certificates, bật Always Use HTTPS. Từ giờ ai gõ http:// sẽ được tự chuyển nhẹ nhàng sang bản an toàn https://.

💡 Sau này nâng lên Full (strict)
Tạo một Origin Certificate miễn phí trong Cloudflare, cài vào reverse proxy như Caddy hay Nginx đứng trước container ở cổng 443, rồi đổi chế độ. Tùy chọn thôi — dùng Flexible để ra mắt là ổn.
⚠️ Bị lặp chuyển hướng (ERR_TOO_MANY_REDIRECTS)?
Thường do chế độ Flexible cộng app cũng ép HTTPS, hoặc page rule còn sót. Nếu gặp, đổi chế độ sang Full rồi thử lại.

Updating later Cập nhật về sau

When you change the site, the loop is always: edit on your laptop → push → pull on the server → rebuild → swap the container.

On your computer:

git add .
git commit -m "Update site"
git push

On the server (SSH in, then inside the project folder):

git pull
docker build -t it-courses .
docker rm -f it-courses
docker run -d --name it-courses --restart unless-stopped -p 80:8888 it-courses

Get the latest code, seal a new kit with the same name, throw away the old running kitchen, open a fresh one. Visitors see only a few seconds of downtime.

💡 Free up disk over time
Old images pile up. Every now and then run docker image prune -f to clean unused layers and reclaim space.

Khi sửa web, vòng lặp luôn là: sửa trên laptop → push → pull trên server → build lại → thay container.

Trên máy của bạn:

git add .

git push

Trên server (SSH vào, rồi vào trong thư mục dự án):

git pull
docker build -t it-courses .

docker run -d --name it-courses --restart unless-stopped -p 80:8888 it-courses

Lấy code mới nhất, niêm phong hộp mới cùng tên, vứt gian bếp cũ đang chạy, mở gian bếp mới. Khách chỉ thấy gián đoạn vài giây.

💡 Dọn ổ cứng theo thời gian
Image cũ sẽ dồn lại. Thỉnh thoảng chạy docker image prune -f để xóa layer không dùng và lấy lại dung lượng.

Many websites on one server? Nhiều website trên cùng server?

The steps above give the server's front door (port 80) to a single site with -p 80:8888. That's perfect when this is the only site. But if your server already hosts several websites told apart by subdomain, only one program can hold port 80/443 — so you can't give each container the front door.

The fix is a reverse proxy: one program that owns the front door, reads the domain on each request, and forwards it to the right container. Picture a building with one entrance and many shops inside — the proxy is the receptionist who hears which shop the guest wants and walks them there.

Visitor → Cloudflare → Server :443 → [Caddy reads domain] ─┬─ a.com  → localhost:8888
                                                          ├─ b.com  → localhost:8890
                                                          └─ c.com  → localhost:8892

In that setup each container binds to localhost only on its own port (e.g. -p 127.0.0.1:8888:8888) instead of -p 80:8888, and Caddy out front routes by domain.

📘 Full walkthrough in a separate guide
Setting this up — installing Caddy, the Caddyfile explained line by line, and adding new sites whether they run in Docker or not — is covered in its own page: caddy-reverse-proxy-guide.html (right next to this file).
💡 Just one site? You can skip this
If this server only ever runs the one site, the plain -p 80:8888 from Step 10 is all you need — no proxy required.

Các bước ở trên giao cửa chính của server (cổng 80) cho một site duy nhất bằng -p 80:8888. Cách đó tuyệt vời khi đây là site duy nhất. Nhưng nếu server của bạn đã chạy nhiều website phân biệt nhau bằng subdomain, chỉ một chương trình giữ được cổng 80/443 — nên bạn không thể giao cửa chính cho từng container.

Cách giải quyết là một reverse proxy: một chương trình giữ cửa chính, đọc tên miền trên mỗi request, rồi chuyển tới đúng container. Hãy hình dung một tòa nhà có một lối vào và nhiều quán bên trong — proxy chính là anh lễ tân nghe khách hỏi quán nào rồi dẫn tới đó.

Khách → Cloudflare → Server :443 → [Caddy đọc tên miền] ─┬─ a.com  → localhost:8888
                                                        ├─ b.com  → localhost:8890
                                                        └─ c.com  → localhost:8892

Trong cách này, mỗi container gắn vào chỉ localhost ở cổng riêng của nó (ví dụ -p 127.0.0.1:8888:8888) thay vì -p 80:8888, và Caddy đứng trước định tuyến theo tên miền.

📘 Hướng dẫn đầy đủ ở một file riêng
Cách cài đặt — cài Caddy, giải thích Caddyfile từng dòng, và thêm site mới dù chạy Docker hay không — nằm ở trang riêng: caddy-reverse-proxy-guide.html (ngay cạnh file này).
💡 Chỉ có một site? Bạn có thể bỏ qua
Nếu server này chỉ chạy duy nhất một site, dùng -p 80:8888 như ở Bước 10 là đủ — không cần proxy.

Build killed (out of memory) Build bị kill (hết RAM)

If docker build stops in the middle of Creating an optimized production build with an error like this:

npm error signal SIGKILL
npm error command sh -c next build
Error: building at STEP "RUN npm run build": ... exit status 1

...your code is fine. SIGKILL during next build almost always means the server ran out of RAM. The OS saw the build eating too much memory and killed it to save the machine.

☕ Café analogy
Building Next.js is like prepping a huge order in a cramped kitchen. When the counter (RAM) runs out of space, the manager (the OS) scraps the whole job to keep the café alive. Cheap VPSes often have just 1 GB of RAM — not enough to build Next.js.

First, confirm it's RAM

On the server, run:

free -h

Look at the Mem row. If total RAM is 1 GB (or less) and Swap is 0, that's the culprit.

Fix A (recommended) — add swap

Why: swap is a "spare storage room" — it borrows disk space to act as extra memory when RAM fills up. The build gets a little slower but stops getting killed. It's the cleanest fix and keeps your Git workflow. Run these one by one on the server:

# Create a 2GB swap file
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Make swap turn back on after every reboot
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Check: you should now see Swap = 2.0Gi
free -h

Then build again:

docker build -t it-courses .
💡 How much swap?
With 1 GB RAM, give 2 GB of swap (≈3 GB total is comfortable for a Next.js build). If it still gets killed, bump it to 4G.

Fix B (alternative) — build on your computer, ship the image

Why: your laptop has more RAM than the server. You build the sealed "meal-kit" at home and send the whole box up — the server just opens and runs it, no build needed, so it can't run out of RAM.

# On your computer (laptop)
docker build -t it-courses .
docker save it-courses | gzip > it-courses.tar.gz
scp it-courses.tar.gz root@YOUR_SERVER_IP:/root/

# On the server
docker load < it-courses.tar.gz
docker run -d --name it-courses --restart unless-stopped -p 80:8888 it-courses
💡 Which one to pick?
Go with Fix A (swap) — it's a one-time setup, after which git pull + rebuild on the server just works. Fix B avoids building on the server entirely, but you have to re-send the image file every time you update.

Nếu docker build dừng giữa chừng ở bước Creating an optimized production build với lỗi kiểu như:

npm error signal SIGKILL
npm error command sh -c next build
Error: building at STEP "RUN npm run build": ... exit status 1

...thì code của bạn không sao cả. SIGKILL khi đang next build gần như luôn nghĩa là server hết RAM. Hệ điều hành thấy tiến trình build ngốn quá nhiều bộ nhớ nên kill nó để cứu cả máy.

☕ Ví dụ quán cà phê
Build Next.js giống như chuẩn bị một đơn hàng cực lớn trong gian bếp chật. Khi mặt bàn (RAM) hết chỗ, anh quản lý (hệ điều hành) dẹp luôn việc đang làm để cứu cả quán. VPS giá rẻ thường chỉ có 1 GB RAM — không đủ để build Next.js.

Bước 1: xác nhận đúng là do RAM

Trên server, chạy:

free -h

Nhìn dòng Mem. Nếu tổng RAM là 1 GB (hoặc thấp hơn) và Swap là 0 — đúng là thủ phạm.

Cách A (khuyên dùng) — thêm swap

Tại sao: swap là "căn phòng kho tạm" — mượn ổ cứng làm bộ nhớ phụ khi RAM đầy. Build sẽ chậm hơn chút nhưng không bị kill nữa. Đây là cách gọn nhất, giữ nguyên quy trình Git của bạn. Chạy lần lượt trên server:

# Tạo file swap 2GB
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Cho swap tự bật lại mỗi khi server khởi động lại
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Kiểm tra: giờ phải thấy Swap = 2.0Gi
free -h

Rồi build lại:

docker build -t it-courses .
💡 Cấp bao nhiêu swap?
Nếu server có 1 GB RAM, hãy cấp 2 GB swap (tổng ~3 GB là build Next.js thoải mái). Nếu vẫn bị kill, tăng lên 4G.

Cách B (thay thế) — build trên máy, đẩy image lên server

Tại sao: laptop của bạn nhiều RAM hơn server. Bạn build sẵn "hộp nguyên liệu" ở nhà rồi gửi nguyên hộp lên — server chỉ việc mở ra chạy, không cần build nữa, nên không thể hết RAM.

# Trên máy của bạn (laptop)
docker build -t it-courses .
docker save it-courses | gzip > it-courses.tar.gz
scp it-courses.tar.gz root@IP_SERVER:/root/

# Trên server
docker load < it-courses.tar.gz
docker run -d --name it-courses --restart unless-stopped -p 80:8888 it-courses
💡 Nên chọn cách nào?
Hãy dùng Cách A (swap) — chỉ làm một lần, sau đó git pull + build lại trên server vẫn chạy bình thường. Cách B thì khỏi build trên server, nhưng mỗi lần cập nhật bạn phải gửi lại file image.

Port mapping (-p)Connecting a port on the host to a port inside the container. Format: host:container.
Firewall (UFW)The server's security guard, deciding which ports outsiders may reach.
DNS recordA signpost mapping a domain name to a server's IP. An "A record" points to an IPv4 address.
Proxied (orange cloud)Traffic flows through Cloudflare, which adds HTTPS, caching, and hides your real server IP.
SSL/TLSThe technology behind the padlock 🔒 — it encrypts the connection between visitor and site.
Reverse proxyA server (Caddy, Nginx) sitting in front of your app, often to handle HTTPS and routing.

ContainerMột bản đang chạy của image — "gian bếp" sống. Bật, tắt, xóa thoải mái.

Git / GitHubGit theo dõi thay đổi code; GitHub lưu chúng online — "cái kệ chung" mà server kéo về.
Ánh xạ cổng (-p)Nối một cổng trên host với một cổng trong container. Định dạng: host:container.
Tường lửa (UFW)Anh bảo vệ của server, quyết định cổng nào người ngoài được vào.

Proxied (mây cam)Traffic đi qua Cloudflare, được thêm HTTPS, cache, và giấu IP thật của server.
