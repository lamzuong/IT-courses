01

Why DigitalOcean (not AWS, not Vercel)

You're choosing a real VPS — a small Linux computer in a data centre that you rent by the month at a fixed price. Unlike Vercel (where overage is technically possible if you skip the spend cap) and unlike AWS Lightsail (where new accounts get blocked by vCPU quotas), DigitalOcean simply charges your card $12 on day one and gives you a server. There is no quota approval, no "set a cap or risk a surprise bill," no spend management to configure. The price is the price.

02

What you need before starting

  • Your IT-courses project building cleanly. Run npm run build in PowerShell first.
  • A GitHub account (the Droplet will pull from there).
  • Your domain, with access to its DNS settings page at the registrar.
  • A credit / debit card for DigitalOcean. Vietnamese-issued cards work; PayPal also accepted.
  • About 60 minutes of focused time. The Droplet creation takes 60 seconds; most of the time is in setup.
03

Sign up & add your card

Go to digitalocean.com and click Sign Up. Use email + password, or "Continue with Google".

3.1 — Verify your email

Check your inbox, click the verification link. Mandatory before you can create a Droplet.

3.2 — Add a payment method

You'll land on a "Payment Information" screen. Add your card. DigitalOcean places a temporary $1 authorization (instantly refunded) to verify the card works. Once that succeeds, you can create Droplets immediately — no waiting, no quota approval.

i

Look for new-customer credit. DigitalOcean periodically offers $100–$200 in free credit for first 60 days through their referral program. Search "DigitalOcean referral credit" before signing up — if you find a valid link, your first month or two are free.

04

Push your project to GitHub

Skip this step if your project is already on GitHub. Otherwise:

# in C:\Users\ADMIN\Desktop\claude-demo\IT-courses
git init
git add .
git commit -m "initial commit"
git branch -M main
git remote add origin https://github.com/YOUR-USERNAME/it-courses.git

Make the repo private. You'll give the Droplet a read-only deploy key in Step 10 so it can pull updates without storing your personal GitHub credentials.

5.1 — Generate an SSH key on Windows

Open PowerShell on your laptop:

# Generate a key pair if you don't have one yet
ssh-keygen -t ed25519 -C "vuonglam@itcourses"
# Press Enter for default path. Choose a passphrase or leave blank.

# Print the public key — copy this entire line
cat $env:USERPROFILE\.ssh\id_ed25519.pub

Paste the output (one line starting with ssh-ed25519 AAAA...) into DigitalOcean's SSH Key field. Name it "Windows laptop" or similar.

Click Create Droplet. A progress bar runs for ~60 seconds and then you see a green "Active" indicator. The Droplet is alive. Note its IP address — you'll use it in a moment.

06

Reserve a static IP (free, important)

The Droplet's IP can technically change if you ever destroy and recreate it. A Reserved IP (DigitalOcean's name for a static IP) pins an address to your account so future moves don't break your DNS. It's free while attached to a running Droplet.

  • Left sidebar → Networking
  • Tab: Reserved IPs
  • Click Assign Reserved IP
  • Pick your Droplet from the dropdown
  • Click Assign

The page now shows a Reserved IP like 165.22.xxx.xxx. This is the IP you point your domain at — not the original Droplet IP. Write it down.

07

Connect via SSH from PowerShell

Time to log into your server for the first time. Replace 165.22.xxx.xxx with your Reserved IP.

ssh root@165.22.xxx.xxx

First connection asks you to confirm the fingerprint — type yes. You're in. The prompt becomes something like root@it-courses-prod:~#. From now on, anything in a code block runs on the Droplet unless explicitly marked "PowerShell".

!

"Permissions are too open" error? Windows occasionally hands SSH a private key with overly permissive ACLs. Fix in PowerShell:

icacls $env:USERPROFILE\.ssh\id_ed25519 /inheritance:r
icacls $env:USERPROFILE\.ssh\id_ed25519 /grant:r "$env:USERNAME:F"

Then retry the ssh command.

08

Harden the server (10 minutes, do it once)

Don't run your site as root. Create a regular user with sudo rights, then disable root SSH.

# on the Droplet, as root
adduser deploy
# Set a strong password when prompted. Leave the other prompts blank.
usermod -aG sudo deploy

# Copy your SSH key to the new user so you can log in as them
rsync --archive --chown=deploy:deploy ~/.ssh /home/deploy

# Open a NEW PowerShell window and test:
#   ssh deploy@165.22.xxx.xxx
# It should work without a password. Keep the root session open until you confirm.

Once you can log in as deploy, disable root SSH and password auth:

# as deploy, sudo edit the SSH config
sudo nano /etc/ssh/sshd_config
# Find and set:
#   PermitRootLogin no
#   PasswordAuthentication no
# Save (Ctrl+O, Enter, Ctrl+X)

sudo systemctl restart ssh

Enable the firewall to allow only SSH, HTTP, and HTTPS:

sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
# type "y" to confirm
✕

Test before you trust. Open one more PowerShell window and try ssh deploy@165.22.xxx.xxx — confirm it still works. If not, you still have the original root session open as a fallback. Only close that one after the new deploy login is confirmed.

09

Install Node.js, Git, and PM2

All as the deploy user (use sudo where needed):

# Update packages
sudo apt update && sudo apt upgrade -y

# Install Node 22 LTS from NodeSource (Ubuntu's apt version is older)
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs

# Confirm version
node -v   # → v22.x.x
npm -v    # → 10.x

# Git, build tools (some npm deps need them), unzip, ufw helpers
sudo apt install -y git build-essential

# PM2 — keeps your Next.js app running and restarts on boot
sudo npm install -g pm2

If next build ever runs short on memory on a tiny instance, add a 2 GB swapfile:

sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

On the $12 (2 GB) tier you probably don't need this, but it's a cheap insurance policy.

10

Clone & build the project

Your private GitHub repo needs a way to authenticate from the Droplet. Use a Deploy Key — a read-only SSH key bound to just this one repo.

10.1 — Generate a deploy key on the Droplet

ssh-keygen -t ed25519 -C "droplet-it-courses" -f ~/.ssh/it_courses_deploy -N ""

# Print the public part
cat ~/.ssh/it_courses_deploy.pub

Copy the printed line.

10.2 — Add the deploy key on GitHub

Open your it-courses repo on GitHub → Settings → Deploy keys → Add deploy key. Title: "DO Droplet". Paste the key. Leave "Allow write access" off. Save.

10.3 — Tell SSH to use that key for github.com

cat >> ~/.ssh/config <<EOF
Host github.com
  IdentityFile ~/.ssh/it_courses_deploy
  IdentitiesOnly yes
EOF
chmod 600 ~/.ssh/config

# Confirm
ssh -T git@github.com   # → "Hi YOUR-USERNAME/it-courses! You've successfully authenticated..."

10.4 — Clone and build

cd ~
git clone git@github.com:YOUR-USERNAME/it-courses.git
cd it-courses

npm ci
npm run build
# Should complete in ~2-4 minutes on the $12 tier.
11

Run with PM2

PM2 starts your Next.js app and keeps it running across crashes, restarts, and reboots.

# in ~/it-courses
pm2 start npm --name it-courses -- start
pm2 save

# Make PM2 start on boot. Run the command it prints:
pm2 startup
# Copy-paste the "sudo env PATH=..." command it outputs. Run that, then:
pm2 save

Check it's alive:

pm2 status
# You should see "it-courses" in green, "online", port 3000

curl http://localhost:3000
# Should print the HTML of your homepage
12

Caddy: reverse proxy + free auto-HTTPS

Your Next.js app is on port 3000 inside the server. You want it reachable on https://yourname.com. Caddy is the simplest web server in the world and the only one that handles HTTPS certificates with zero config.

# Install Caddy from the official repo
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install -y caddy

Write the Caddyfile (replace yourname.com with your real domain):

sudo nano /etc/caddy/Caddyfile
# /etc/caddy/Caddyfile
yourname.com, www.yourname.com {
    reverse_proxy localhost:3000
    encode gzip zstd
}

Save (Ctrl+O, Enter, Ctrl+X), then reload Caddy:

sudo systemctl reload caddy

That's it. Once your DNS points at the Droplet (next step), Caddy automatically requests a Let's Encrypt SSL certificate, installs it, and renews it forever. You write zero lines of certificate config.

13

DNS records at your registrar

Log into wherever you bought the domain (Namecheap, GoDaddy, Tenten.vn, Mat Bao, P.A. Việt Nam, Cloudflare, etc.) and find the DNS management page.

13.1 — Delete any default parking record

Most registrars start with a placeholder A record. Delete it.

13.2 — Add these two records

TypeName / HostValueTTL
A@ (or blank)Your Reserved IP, e.g. 165.22.xxx.xxx3600
AwwwYour Reserved IP, e.g. 165.22.xxx.xxx3600

Save. DNS propagation is usually 5–30 minutes. Test from PowerShell:

nslookup yourname.com
# Should return your Reserved IP within an hour

Once it resolves, visit https://yourname.com — Caddy will silently fetch a cert from Let's Encrypt on first request (takes 5–10 seconds), then serve your site over HTTPS forever. Your site is live.

i

Cloudflare users: if your registrar is Cloudflare and the orange-cloud proxy is on, set it to DNS only (grey cloud) for both records. Caddy needs to talk to Let's Encrypt directly for certificate issuance.

14

One-command updates

Now that everything's wired, deploying a change is three commands. Write them once as a script.

nano ~/deploy.sh
#!/bin/bash
# ~/deploy.sh — pull, build, restart
set -e
cd ~/it-courses
git pull
npm ci
npm run build
pm2 restart it-courses
echo "✓ Deployed at $(date)"
chmod +x ~/deploy.sh

Update flow on your laptop:

# in PowerShell, after editing lessons locally
git add .
git commit -m "add new lesson"
git push

Then SSH in and run one command:

ssh deploy@165.22.xxx.xxx
./deploy.sh

Total update time: 2–4 minutes. The site stays online during build because PM2 only restarts the app at the very end.

★

Want auto-deploy on push? Add a GitHub Actions workflow that SSHes in and runs ./deploy.sh on every push to main. Out of scope for this guide, but search "GitHub Actions ssh deploy" once you're comfortable with the manual flow.

15

Backups (snapshots)

You enabled DigitalOcean's automated weekly backups in Step 05 (+$2.40/mo). These are full Droplet snapshots — restore-able in a single click if anything ever goes catastrophically wrong.

You can also take a manual snapshot before any risky change:

  • DigitalOcean dashboard → your Droplet → Snapshots tab
  • Click Take Snapshot (note: Droplet powers off briefly during snapshot — ~30s downtime)
  • Manual snapshots cost $0.06/GB/month — for a 50 GB Droplet that's ~$3 each per month, so delete old ones

17

Troubleshooting

Site shows "502 Bad Gateway" through Caddy

Caddy is up but Next.js on port 3000 isn't responding. Check PM2:

pm2 status
pm2 logs it-courses --lines 100

If the app is "errored" or "stopped", a runtime error crashed it. Read the logs, fix the code, push, run ./deploy.sh.

SSL certificate doesn't get issued

Caddy can only fetch a Let's Encrypt cert if (a) DNS already resolves to your IP, and (b) ports 80 and 443 are open in the firewall. Verify:

nslookup yourname.com       # should match your Reserved IP
sudo ufw status              # should list 80/tcp ALLOW, 443/tcp ALLOW
sudo journalctl -u caddy --since "10 minutes ago"

The journal tells you exactly why Let's Encrypt refused (most commonly: DNS hasn't fully propagated yet — wait 30 min and run sudo systemctl reload caddy again).

npm run build is killed mid-build

OOM killer. On the $12 (2 GB) tier this is rare but possible with large builds. Enable swap (Step 09 final code block) and retry. If it still fails, upgrade the Droplet to $18/mo from the dashboard — one click, ~60s downtime.

I can't SSH in anymore

Use DigitalOcean's Recovery Console: dashboard → your Droplet → Console. It logs you in as root through the browser, no SSH needed. From there you can fix whatever broke (usually a wrong line in /etc/ssh/sshd_config). Then restart SSH: sudo systemctl restart ssh.

The site is slow for users in Vietnam

You're already in Singapore, which is the closest data center to Vietnam — latency is ~30ms. If it still feels slow, the bottleneck is your code, not the network. Run Lighthouse on the live URL; common culprits are unoptimized images and unminified JS. Both fix automatically when you use Next.js's <Image> component and build in production mode (which you are).

Will Vietnamese cards work?

Yes. DigitalOcean accepts most Vietnamese-issued Visa/Mastercard for international online payments. PayPal is also available as a backup. The $1 verification charge runs immediately on signup; the first real charge is at the end of the first calendar month, prorated.

How do I cancel and stop being charged?

Dashboard → your Droplet → Destroy → Destroy this Droplet. Billing stops immediately — you only pay for hours used to date. The Reserved IP becomes free-while-unattached for a short period, then is auto-released; release it manually if you don't plan to redeploy.

Can I upgrade the Droplet later without losing data?

Yes. Dashboard → your Droplet → Resize. Pick the new tier. ~60 seconds of downtime, then you're on the bigger box with all data intact. You can resize both up and down.

18

What to learn next

Right away

  • GitHub Actions auto-deploy. Replace the manual ssh + ./deploy.sh step with a workflow that triggers on push to main.
  • Fail2ban. One sudo apt install fail2ban and SSH brute-force attempts get auto-banned for 10 minutes. Tiny config, huge security upgrade.
  • Caddy access logs. Add log { output file /var/log/caddy/access.log } to the Caddyfile for visitor logs.

Soon

  • UptimeRobot. Free 5-minute uptime monitoring with email/Telegram alerts when the site goes down.
  • Plausible or Umami analytics self-hosted on the same Droplet — privacy-friendly visitor stats, zero cookies, no per-event pricing.
  • Cloudflare in front of Caddy. Free CDN + DDoS protection. Set DNS proxy to "on" (orange cloud) and set Cloudflare's SSL mode to Full (strict). Adds caching and a layer of edge protection.

Bigger picture

  • A separate database Droplet (or managed Postgres on DigitalOcean for $15/mo) when you start storing user accounts or course progress.
  • Two Droplets + a load balancer when traffic grows past what one box can handle (you'll feel it long before $12 stops being enough).
  • Infrastructure as code. Once you've manually built one server, learn Terraform or Pulumi so the next one is a single command instead of an afternoon.
