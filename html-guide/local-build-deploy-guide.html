Why build locally instead of on the server Vì sao build trên máy bàn thay vì trên server

The original Cloudfly guide had you clone the repo on the VPS and run npm run build there. That works, but it has three pain points:

  • RAM. A Next.js 16 build with MDX can peak above 1.5 GB. On a 1 GB VPS it gets killed by the OOM killer mid-build — you end up needing a swapfile or a bigger plan.
  • Time. Cloudfly's vCPU is shared. A build that takes 40 seconds on your laptop can take 5–6 minutes on the VPS.
  • Source code on the server. If the VPS gets compromised, the attacker reads your whole codebase, env files, and Git history.

Building locally flips the trade-off. Your desktop has 16+ GB RAM and a fast CPU — the build finishes quickly. Then you upload only the compiled .next/ output. The server never sees source code, never runs npm install, and only needs enough RAM to run the app (~150 MB), not build it.

📝 When this is NOT the right approach
If multiple developers deploy from different machines (different Node versions, different OSes), local builds can drift apart and "works on my machine" bugs creep in. For a team, set up GitHub Actions instead. For a solo project, local-build is the simplest reliable workflow.

Hướng dẫn Cloudfly gốc bảo bạn clone repo lên VPS rồi chạy npm run build ngay trên đó. Cách này chạy được, nhưng có 3 điểm khó chịu:

  • RAM. Build Next.js 16 có MDX dễ ngốn trên 1.5 GB. Trên VPS 1 GB là bị OOM killer giết giữa chừng — cuối cùng phải tạo swapfile hoặc nâng gói.
  • Thời gian. vCPU của Cloudfly là loại chia sẻ. Một build chạy 40 giây trên laptop có thể mất 5–6 phút trên VPS.
  • Source code nằm trên server. Nếu VPS bị xâm nhập, kẻ tấn công đọc được toàn bộ code, file env, lịch sử git của bạn.

Build trên máy bàn lật ngược thế cờ. Máy bạn có 16+ GB RAM và CPU mạnh — build xong trong tích tắc. Sau đó bạn chỉ upload thư mục .next/ đã compile. Server không bao giờ thấy source code, không phải chạy npm install, và chỉ cần đủ RAM để chạy app (~150 MB) chứ không phải để build.

📝 Khi nào KHÔNG nên dùng cách này
Nếu nhiều dev deploy từ nhiều máy khác nhau (Node version khác, hệ điều hành khác), build trên máy mỗi người có thể lệch nhau và sinh ra bug kiểu "trên máy tao chạy được mà". Với team, hãy dùng GitHub Actions thay thế. Còn dự án solo thì build trên máy là cách đơn giản và đáng tin cậy nhất.

Turn on Next.js standalone output Bật chế độ standalone output của Next.js

By default, npm run build produces a .next/ folder that requires the full node_modules/ (often 400–800 MB) to run. The standalone output mode tells Next.js to trace exactly which dependencies your app uses at runtime, copy only those into .next/standalone/, and emit a tiny self-contained Node server.

Open next.config.mjs and add the output line:

/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'standalone',          // ← add this line
  pageExtensions: ['ts', 'tsx', 'mdx'],
  turbopack: {
    root: __dirname,
  },
};

export default withMDX(nextConfig);

Commit it now so the change is reproducible:

git add next.config.mjs
git commit -m "build: enable standalone output for local-build deploy"
💡 What "standalone" actually does
Next.js uses @vercel/nft to scan your code and figure out exactly which files inside node_modules/ are touched at runtime. Anything not traced (devDependencies, build-only tools, unused packages) is skipped. Result: .next/standalone/ is typically 30–80 MB instead of 500+ MB.

Mặc định, npm run build tạo ra thư mục .next/ cần CẢ node_modules/ (thường 400–800 MB) thì mới chạy được. Chế độ standalone bảo Next.js lần ra chính xác những dependency mà app cần lúc chạy, chỉ copy đúng những cái đó vào .next/standalone/, và xuất ra một Node server tí hon tự đứng được một mình.

Mở file next.config.mjs và thêm dòng output:

/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'standalone',          // ← thêm dòng này
  pageExtensions: ['ts', 'tsx', 'mdx'],
  turbopack: {
    root: __dirname,
  },
};

export default withMDX(nextConfig);

Commit ngay để thay đổi có thể tái lập:

git add next.config.mjs
git commit -m "build: enable standalone output for local-build deploy"
💡 "Standalone" làm gì cụ thể?
Next.js dùng @vercel/nft để quét code và tìm chính xác những file nào trong node_modules/ được dùng lúc chạy. Những thứ không bị dùng tới (devDependencies, công cụ chỉ phục vụ build, package thừa) sẽ bị bỏ qua. Kết quả: .next/standalone/ thường chỉ 30–80 MB thay vì 500+ MB.

Bundle the deploy folder (the gotcha step) Gói folder để deploy (bước cài bẫy)

Standalone output is almost self-contained — but Next.js deliberately does NOT copy two things into .next/standalone/:

  • .next/static/ — your hashed CSS, JS chunks, fonts
  • public/ — your images, favicon, robots.txt

The reason: in most production setups, you'd serve those from a CDN, not from Node. We're not using a CDN, so we have to copy them in by hand. Run this in PowerShell:

Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

After this, your .next/standalone/ looks like a complete, runnable Node app:

.next/standalone/ ├── server.js # the Node entry point ├── package.json ├── node_modules/ # only what's needed at runtime ├── .next/ │ ├── server/ # RSC, route handlers │ └── static/ # ← copied in this step └── public/ # ← copied in this step
💡 Why a "Copy-Item" instead of a build script?
Future you might forget. Add these two lines as a npm script so it's never skipped: 
// package.json
"scripts": {
  "build": "next build",
  "build:deploy": "next build && powershell -Command \"Copy-Item -Recurse -Force .next/static .next/standalone/.next/; Copy-Item -Recurse -Force public .next/standalone/\""
}
Then you just run npm run build:deploy instead of two separate commands.

Standalone output gần như tự đứng được — nhưng Next.js CỐ TÌNH không copy 2 thứ này vào .next/standalone/:

  • .next/static/ — file CSS, JS chunk có hash, font
  • public/ — ảnh, favicon, robots.txt

Lý do: ở môi trường production "chuẩn", người ta phục vụ mấy thứ này từ CDN chứ không từ Node. Bạn chưa dùng CDN, nên phải copy thủ công vào. Chạy trong PowerShell:

Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

Sau bước này, .next/standalone/ trông như một Node app hoàn chỉnh, chạy ngay được:

.next/standalone/ ├── server.js # điểm vào của Node ├── package.json ├── node_modules/ # chỉ chứa thứ cần lúc chạy ├── .next/ │ ├── server/ # RSC, route handler │ └── static/ # ← copy vào ở bước này └── public/ # ← copy vào ở bước này
💡 Sao không bỏ "Copy-Item" vào script luôn cho gọn?
Bạn sau này dễ quên lắm. Thêm 2 dòng này vào npm script để không bao giờ thiếu bước copy: 
// package.json
"scripts": {
  "build": "next build",
  "build:deploy": "next build && powershell -Command \"Copy-Item -Recurse -Force .next/static .next/standalone/.next/; Copy-Item -Recurse -Force public .next/standalone/\""
}
Sau đó chỉ cần chạy npm run build:deploy thay vì 2 lệnh riêng.

First-time server setup Chuẩn bị server lần đầu

You only do this once per server. SSH into the VPS:

ssh deploy@your-vps-ip

(Or whichever non-root user you created in step 8 of the Cloudfly guide. Below, deploy is a placeholder — substitute your real username.)

Create the deploy folder under /var/www/ and hand ownership to your SSH user so scp can write there without sudo:

sudo mkdir -p /var/www/it-courses
sudo chown -R $USER:$USER /var/www/it-courses
cd /var/www/it-courses
📝 Why the chown matters
/var/www/ is owned by root by default on Ubuntu. Without the chown, every scp upload would fail with Permission denied, or you'd have to prefix every operation with sudo. $USER is a shell variable that expands to your current Linux username — no need to type it.

While you're here, make sure PM2 is installed:

pm2 --version || sudo npm install -g pm2

Create a PM2 config file so future you doesn't have to remember the env vars and port number:

nano /var/www/it-courses/ecosystem.config.cjs

Paste this:

module.exports = {
  apps: [{
    name: 'it-courses',
    script: './server.js',          // standalone entry point
    cwd: '/var/www/it-courses',     // matches the upload destination
    instances: 1,
    autorestart: true,
    watch: false,
    max_memory_restart: '400M',
    env: {
      NODE_ENV: 'production',
      PORT: 8888,                    // matches package.json + nginx config
      HOSTNAME: '127.0.0.1'          // bind to loopback; nginx is the front door
    }
  }]
};

Save with Ctrl+O, Enter, Ctrl+X. Log out of the server — the rest happens from your laptop.

📝 Why HOSTNAME=127.0.0.1?
Binding Node to 127.0.0.1 (loopback only) means the world can't reach Node directly — only nginx running on the same machine can. This is exactly what you want: nginx terminates HTTPS, applies rate limits and gzip, then proxies cleartext to Node. If you bind to 0.0.0.0, anyone scanning port 8888 hits raw Node.

Phần này chỉ làm 1 lần cho mỗi server. SSH vào VPS:

ssh deploy@ip-cua-vps

(Hoặc tên user non-root mà bạn đã tạo ở bước 8 của hướng dẫn Cloudfly. Bên dưới, deploy chỉ là ví dụ — bạn thay bằng username thật của mình.)

Tạo folder deploy trong /var/www/ rồi giao quyền sở hữu cho user SSH của bạn để scp ghi được mà không cần sudo:

sudo mkdir -p /var/www/it-courses
sudo chown -R $USER:$USER /var/www/it-courses
cd /var/www/it-courses
📝 Vì sao phải chown?
Mặc định /var/www/ trên Ubuntu thuộc về root. Không chown thì mỗi lần scp sẽ báo Permission denied, hoặc bạn phải đặt sudo trước mọi thao tác. $USER là biến shell, tự bung ra thành username Linux hiện tại — không phải gõ tay.

Đang ở đây thì cài luôn PM2 cho chắc:

pm2 --version || sudo npm install -g pm2

Tạo file config PM2 để sau này bạn khỏi phải nhớ env var với port:

nano /var/www/it-courses/ecosystem.config.cjs

Dán nội dung này vào:

module.exports = {
  apps: [{
    name: 'it-courses',
    script: './server.js',          // điểm vào của standalone
    cwd: '/var/www/it-courses',     // trùng với nơi upload tới
    instances: 1,
    autorestart: true,
    watch: false,
    max_memory_restart: '400M',
    env: {
      NODE_ENV: 'production',
      PORT: 8888,                    // trùng với package.json + config nginx
      HOSTNAME: '127.0.0.1'          // chỉ lắng nghe trên loopback; nginx là cửa trước
    }
  }]
};

Lưu bằng Ctrl+O, Enter, Ctrl+X. Đăng xuất khỏi server — phần còn lại làm từ laptop.

📝 Vì sao HOSTNAME=127.0.0.1?
Cho Node bind vào 127.0.0.1 (chỉ loopback) nghĩa là thế giới bên ngoài không thể truy cập thẳng tới Node — chỉ có nginx chạy cùng máy mới gọi tới được. Đây chính xác là điều bạn muốn: nginx xử HTTPS, giới hạn rate, nén gzip, rồi mới chuyển plaintext sang Node. Nếu bind vào 0.0.0.0, ai quét được port 8888 là chạm thẳng vào Node luôn.
7

Upload the build with scp Upload bản build bằng scp

Back in PowerShell on your desktop, from the project root:

scp -r .next\standalone\* deploy@your-vps-ip:/var/www/it-courses/

Watch the file names scroll by. With the standalone output, the whole upload is usually 30–80 MB and takes 30 seconds to 3 minutes depending on your upload speed.

⚠ Two common gotchas with scp on Windows
  1. The trailing /* matters. scp -r .next\standalone\* copies the contents of standalone into it-courses/. Without the *, you'd get a nested it-courses/standalone/server.js — wrong path, app won't start.
  2. Backslash vs forward slash. On Windows the local path uses \, but the remote path after the colon always uses /. Don't mix them on the wrong side.

If scp asks for a password every time, set up an SSH key (see step 2). With keys, the upload is fully unattended.

Quay lại PowerShell trên máy bàn, từ thư mục gốc của project:

scp -r .next\standalone\* deploy@ip-cua-vps:/var/www/it-courses/

Bạn sẽ thấy tên file chạy qua. Với standalone output, tổng dung lượng upload thường 30–80 MB, mất 30 giây tới 3 phút tuỳ tốc độ upload nhà bạn.

⚠ Hai cái bẫy hay gặp khi dùng scp trên Windows
  1. Cái dấu /* cuối là quan trọng. scp -r .next\standalone\* copy nội dung bên trong standalone vào it-courses/. Nếu thiếu *, bạn sẽ có đường dẫn lồng it-courses/standalone/server.js — sai chỗ, app không khởi động được.
  2. Dấu sổ ngược vs sổ xuôi. Trên Windows đường dẫn cục bộ dùng \, nhưng đường dẫn remote (sau dấu hai chấm) luôn dùng /. Đừng nhầm lẫn giữa 2 bên.

Nếu scp hỏi mật khẩu mỗi lần, hãy thiết lập SSH key (xem bước 2). Có key rồi thì upload chạy luôn, không cần nhập gì.

Start (or reload) the app with PM2 Khởi động (hoặc reload) app bằng PM2

SSH back into the server one more time:

ssh deploy@your-vps-ip
cd /var/www/it-courses

First time, tell PM2 to start the app from your ecosystem file:

pm2 start ecosystem.config.cjs
pm2 save                       # remember this app across reboots
pm2 startup                    # only the FIRST time on a fresh VPS — copy and run the printed command

Check it's healthy:

pm2 status
pm2 logs it-courses --lines 30

You should see online in the status table, and recent log lines like ▲ Next.js 16.x - Local: http://127.0.0.1:8888.

Open http://your-domain.com (assuming nginx + DNS from the Cloudfly guide is already set up). Your site loads — running off a build your laptop made 2 minutes ago.

💡 Subsequent deploys use reload, not start
pm2 start would create a second instance every time. Use pm2 reload it-courses instead — it does a zero-downtime restart, swapping in the new code without dropping any in-flight requests.

SSH vào server lần nữa:

ssh deploy@ip-cua-vps
cd /var/www/it-courses

Lần đầu tiên, bảo PM2 khởi động app từ file ecosystem của bạn:

pm2 start ecosystem.config.cjs
pm2 save                       # ghi nhớ app này để khôi phục sau khi reboot
pm2 startup                    # chỉ làm LẦN ĐẦU trên VPS mới — copy và chạy lệnh nó in ra

Kiểm tra xem app có khoẻ không:

pm2 status
pm2 logs it-courses --lines 30

Bạn phải thấy online trong bảng status, và những dòng log gần đây kiểu ▲ Next.js 16.x - Local: http://127.0.0.1:8888.

Mở http://ten-mien-cua-ban.com (giả sử nginx + DNS từ hướng dẫn Cloudfly đã được setup). Website hiện lên — đang chạy từ bản build mà laptop bạn vừa tạo 2 phút trước.

💡 Những lần deploy sau dùng reload, không phải start
pm2 start sẽ tạo thêm 1 instance mới mỗi lần — sẽ có nhiều bản chạy song song. Dùng pm2 reload it-courses thay vào đó — nó restart không gián đoạn, swap code mới vào mà không làm rớt request nào đang xử lý.

The everyday deploy cycle Quy trình deploy hằng ngày

Once steps 6 and 8 are done, future deploys are just 4 commands. Run them from PowerShell in the project root every time you want to publish a change:

# 1. Build the project
npm run build

# 2. Bundle static + public into standalone
Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

# 3. Upload
scp -r .next\standalone\* deploy@your-vps-ip:/var/www/it-courses/

# 4. Reload PM2 over SSH (no need to log in)
ssh deploy@your-vps-ip "pm2 reload it-courses"

That's the whole loop. From save → live usually under 2 minutes once everything is set up.

📝 What's NOT in this loop
Notice there's no npm install on the server, no git pull, no migrations. If you added a new dependency in package.json, standalone traces it and copies the needed files automatically. If you added a database migration… that's a different conversation — see "Learn next" at the bottom.

Sau khi đã làm xong bước 6 và 8, mỗi lần deploy về sau chỉ còn 4 lệnh. Chạy từ PowerShell trong thư mục gốc của project mỗi khi muốn publish thay đổi:

# 1. Build project
npm run build

# 2. Gói static + public vào standalone
Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

# 3. Upload
scp -r .next\standalone\* deploy@ip-cua-vps:/var/www/it-courses/

# 4. Reload PM2 qua SSH (khỏi cần đăng nhập bằng tay)
ssh deploy@ip-cua-vps "pm2 reload it-courses"

Toàn bộ chu trình chỉ vậy. Từ lúc nhấn save đến lúc live thường dưới 2 phút khi mọi thứ đã setup xong.

📝 Cái KHÔNG có trong chu trình này
Để ý không có npm install trên server, không có git pull, không có chạy migration. Nếu bạn thêm dependency mới vào package.json, standalone tự lần ra và copy file cần thiết qua. Còn nếu bạn thêm migration database thì… là chuyện khác — xem mục "Học tiếp" ở cuối.
10

One-command deploy script (PowerShell) Script deploy một lệnh (PowerShell)

Typing 4 commands every time is fine — but it's a lot easier to mess up an scp path at midnight than at noon. Wrap it in a script.

Create deploy.ps1 in the project root:

# deploy.ps1 — build locally and ship to VPS
# Usage:  .\deploy.ps1

$ErrorActionPreference = 'Stop'
$VPS = 'deploy@your-vps-ip'
$REMOTE_DIR = '/var/www/it-courses/'

Write-Host "→ Building..." -ForegroundColor Cyan
npm run build

Write-Host "→ Bundling static + public..." -ForegroundColor Cyan
Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

Write-Host "→ Uploading to $VPS..." -ForegroundColor Cyan
scp -r .next\standalone\* "${VPS}:${REMOTE_DIR}"

Write-Host "→ Reloading PM2..." -ForegroundColor Cyan
ssh $VPS "pm2 reload it-courses"

Write-Host "✓ Deployed!" -ForegroundColor Green

The first time you run a PowerShell script, Windows may block it. Allow scripts in your user profile (one-time):

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

Now deploys are exactly one command:

.\deploy.ps1
💡 Add a guard so you don't deploy uncommitted code
At the top of the script, before the build, add: 
if ((git status --porcelain) -ne $null) {
  Write-Host "✗ Uncommitted changes. Commit or stash before deploying." -ForegroundColor Red
  exit 1
}
This catches the very common "I deployed but forgot to push to git" mistake — future-you sees outdated code on GitHub vs what's running live, and you can't figure out which version is which.

Gõ 4 lệnh mỗi lần cũng được — nhưng nửa đêm rất dễ gõ sai đường dẫn scp hơn là giữa trưa. Gói lại thành script cho gọn.

Tạo file deploy.ps1 ở thư mục gốc project:

# deploy.ps1 — build trên máy rồi đẩy lên VPS
# Cách dùng:  .\deploy.ps1

$ErrorActionPreference = 'Stop'
$VPS = 'deploy@ip-cua-vps'
$REMOTE_DIR = '/var/www/it-courses/'

Write-Host "→ Đang build..." -ForegroundColor Cyan
npm run build

Write-Host "→ Đang gói static + public..." -ForegroundColor Cyan
Copy-Item -Recurse -Force .next\static .next\standalone\.next\
Copy-Item -Recurse -Force public .next\standalone\

Write-Host "→ Đang upload lên $VPS..." -ForegroundColor Cyan
scp -r .next\standalone\* "${VPS}:${REMOTE_DIR}"

Write-Host "→ Đang reload PM2..." -ForegroundColor Cyan
ssh $VPS "pm2 reload it-courses"

Write-Host "✓ Deploy xong!" -ForegroundColor Green

Lần đầu chạy script PowerShell, Windows có thể chặn lại. Cho phép script trong profile user của bạn (chỉ làm 1 lần):

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

Bây giờ mỗi lần deploy chỉ còn đúng 1 lệnh:

.\deploy.ps1
💡 Thêm "chốt chặn" để không deploy code chưa commit
Thêm đoạn này vào đầu script, trước bước build: 
if ((git status --porcelain) -ne $null) {
  Write-Host "✗ Có file chưa commit. Commit hoặc stash trước khi deploy." -ForegroundColor Red
  exit 1
}
Cái này tránh lỗi cực phổ biến: "deploy xong rồi mà quên push lên git" — bạn sau này sẽ thấy code trên GitHub khác hẳn code đang chạy live, không biết bản nào là bản nào.

Troubleshooting Khắc phục sự cố

ProblemSolution
Build finishes but no .next/standalone/ appears You forgot to add output: 'standalone' in next.config.mjs (step 3). Add it, run npm run build again.
Site loads but all CSS/JS are 404 You forgot the bundle step (5) — .next/static/ was never copied into .next/standalone/.next/. Re-run the two Copy-Item commands and re-upload.
Images in public/ show as broken Same root cause as above — you forgot to copy public/ into .next/standalone/public/. Re-run the second Copy-Item.
PM2 status shows errored right after start pm2 logs it-courses --lines 50 shows why. Most common: missing server.js (wrong upload path — see scp gotchas in step 7), or port 8888 already taken by an old PM2 process (pm2 delete it-courses then start again).
scp: Permission denied The remote folder is still owned by root — you skipped or forgot the chown in step 6. Fix it: sudo chown -R $USER:$USER /var/www/it-courses on the server.
Different Node version on desktop vs VPS, app crashes Match the majors. Easiest: install nvm on the VPS and run nvm install 22 && nvm use 22, then re-create the PM2 app so it picks up the new Node.
Upload is painfully slow over scp scp opens one SSH channel per file. For faster transfer, tar+ssh in one stream: tar -cf - -C .next\standalone . | ssh deploy@ip "tar -xf - -C /var/www/it-courses". Requires tar on Windows (built into Windows 10+).
Site shows old version after deploy nginx or Cloudflare cached the old HTML. Hard-refresh in browser (Ctrl+Shift+R). If you use Cloudflare, "Purge Everything" from the Caching tab.
Vấn đềCách khắc phục
Build xong nhưng không có .next/standalone/ Bạn quên thêm output: 'standalone' trong next.config.mjs (bước 3). Thêm vào, chạy lại npm run build.
Website load được nhưng CSS/JS đều 404 Bạn quên bước gói (5) — .next/static/ chưa được copy vào .next/standalone/.next/. Chạy lại 2 lệnh Copy-Item và upload lại.
Ảnh trong public/ bị vỡ Cùng lý do như trên — bạn quên copy public/ sang .next/standalone/public/. Chạy lại lệnh Copy-Item thứ 2.
PM2 báo errored ngay sau khi start pm2 logs it-courses --lines 50 sẽ cho biết tại sao. Hay gặp nhất: thiếu file server.js (upload sai đường dẫn — xem mục bẫy scp ở bước 7), hoặc port 8888 đã bị process PM2 cũ chiếm (pm2 delete it-courses rồi start lại).
scp: Permission denied Folder trên server vẫn thuộc về root — bạn bỏ qua hoặc quên bước chown ở bước 6. Sửa nhanh: sudo chown -R $USER:$USER /var/www/it-courses trên server.
Node version khác nhau giữa máy bàn và VPS, app crash Cho khớp ít nhất ở major version. Dễ nhất: cài nvm trên VPS rồi chạy nvm install 22 && nvm use 22, sau đó tạo lại PM2 app để nó bắt Node mới.
Upload qua scp chậm khủng khiếp scp mở 1 channel SSH cho mỗi file. Để nhanh hơn, dùng tar+ssh thành 1 luồng: tar -cf - -C .next\standalone . | ssh deploy@ip "tar -xf - -C /var/www/it-courses". Cần lệnh tar trên Windows (có sẵn trong Windows 10+).
Site vẫn hiện bản cũ sau khi deploy nginx hoặc Cloudflare đang cache HTML cũ. Hard-refresh trên trình duyệt (Ctrl+Shift+R). Nếu dùng Cloudflare, vào tab Caching → "Purge Everything".

Glossary Bảng thuật ngữ

TermMeaning
Build artifactThe compiled output of npm run build — the files you actually deploy. For Next.js, that's .next/ (or with standalone, .next/standalone/).
Standalone outputA Next.js build mode that produces a self-contained, minimal Node app in .next/standalone/ — server + only the needed dependencies, around 30–80 MB instead of 500+ MB.
scpSecure Copy. Built-in SSH-based file transfer — scp -r local/folder user@host:/remote/path. Ships with Windows 10/11.
SSH keyA pair of cryptographic files (public + private) that prove your identity to a server without typing a password.
PM2A process manager for Node.js. Keeps the app running, restarts it on crash, brings it back after server reboot, and provides logs.
pm2 reloadZero-downtime restart of a PM2 app. The new code replaces the old in-place without dropping in-flight requests. Always use this after a deploy, not start.
ecosystem filePM2's config file (ecosystem.config.cjs). Defines app name, entry script, env vars, memory limits — so you don't have to remember them every time you run pm2 start.
loopback (127.0.0.1)Special IP address that only the local machine can reach. Binding Node to loopback hides it from the public internet — only nginx on the same VPS can reach it.
OOM killerOut-Of-Memory killer. The Linux kernel's last-resort defense — when RAM is exhausted, it kills the biggest hungry process to keep the system alive. Frequently kills npm run build on small VPS instances.
nginxThe reverse proxy on the VPS that accepts public HTTPS requests and forwards them to Node on 127.0.0.1:8888. See the Cloudfly guide step 12B for setup.
PowerShellThe default shell on modern Windows. Different syntax from bash — Copy-Item instead of cp, $env:NAME instead of $NAME.

SSH keyCặp file mã hoá (public + private) chứng minh danh tính của bạn với server mà không cần gõ mật khẩu.
PM2Process manager cho Node.js. Giữ app chạy 24/7, restart khi crash, tự bật lại sau khi server reboot, và cung cấp log.
