Why Cloudfly (and not the international providers) Tại sao chọn Cloudfly (thay vì nhà cung cấp nước ngoài)

Cloudfly is a Vietnamese cloud provider with data centres in Ho Chi Minh City and Hanoi. For a course site whose readers are mostly in Vietnam, that proximity gives noticeably faster first-byte time — usually under 30 ms instead of 80–200 ms from US or Singapore. You also get three practical advantages over DigitalOcean, AWS, or Vercel:

  • Pay in VND — Momo, ZaloPay, VNPay, or domestic bank transfer. No international card needed.
  • Vietnamese support — chat and tickets in Vietnamese, replies in minutes during business hours.
  • Prepaid balance — load money once, the VPS deducts the monthly fee. No surprise USD bills.

Prep checklist (15 minutes) Chuẩn bị trước (15 phút)

Before you spend a single đồng, line these up. Each is small individually but each is a blocker if you skip it.

  • An email and a Vietnamese phone number for the Cloudfly account.
  • A way to pay in VND: Momo, VNPay, ZaloPay, or domestic bank transfer.
  • A domain name you already own — .com, .vn, .id.vn, anything where you control DNS.
  • A GitHub account with your IT-courses repository pushed (or we'll push it in step 4).

Register at Cloudfly and top up your balance Đăng ký Cloudfly và nạp tiền vào tài khoản

Cloudfly works on a prepaid wallet — you load money in once, and resources deduct from the balance.

Create the account

Open cloudfly.vn and click Đăng ký. Use your real name and Vietnamese phone number — they may send an OTP to verify.

Verify your identity (KYC)

For higher limits and to unlock VAT-invoice (hóa đơn đỏ) requests, upload a photo of your CCCD/CMND in Tài khoản → Xác minh danh tính. Optional for a single VPS, but recommended.

Top up

Go to Số dư → Nạp tiền and load at least 300,000 VND. That covers one month of the suggested VPS plus a buffer. Methods: Momo, ZaloPay, VNPay, ATM nội địa, or bank transfer (slowest — up to 10 minutes to credit).

★ Pro tip
If you plan to keep the VPS long-term, top up 6 months at once. Cloudfly sometimes runs khuyến mãi that gives bonus credit when you load 1,000,000 VND or more in one transaction.

Cloudfly hoạt động kiểu ví trả trước — bạn nạp tiền vào một lần, các dịch vụ tự trừ dần.

Note your VPS's static public IP Ghi lại địa chỉ IP tĩnh của VPS

Unlike DigitalOcean, every Cloudfly Cloud Server already comes with a static IPv4 attached — no separate Reserved IP step. Click into the server detail page and copy the value under IP Public. It looks like 103.124.95.42.

Write it down — you'll paste it into SSH commands, your Caddyfile, and your DNS provider over the next ten minutes.

★ Good news
This IP stays with the server even if you reboot, rebuild, or restore from a snapshot. Cloudfly only changes it if you explicitly destroy the server or migrate to a different region.

Khác với DigitalOcean, mỗi Cloud Server của Cloudfly đã đi kèm sẵn một IPv4 tĩnh — không cần bước cấp Reserved IP riêng. Bấm vào trang chi tiết server và copy giá trị ở mục IP Public. Trông như 103.124.95.42.

Hãy ghi nó lại — bạn sẽ dán vào lệnh SSH, vào Caddyfile, và vào nhà cung cấp DNS trong 10 phút tới.

★ Tin vui
IP này gắn cố định với server kể cả khi bạn khởi động lại, cài lại, hay khôi phục từ snapshot. Cloudfly chỉ đổi IP khi bạn xoá hẳn server hoặc chuyển sang region khác.

Open PowerShell on your Windows machine and run:

The first time, SSH asks "The authenticity of host ... can't be established. Are you sure?" — type yes and press Enter. Then paste the root password from step 5.

You should see a green prompt:

You're inside the server. Everything from now until step 12 happens here.

Wait another minute — the SSH daemon sometimes needs a moment after first boot. If it persists past 5 minutes, double-check the IP and that your home network isn't blocking outbound port 22.

Mở PowerShell trên máy Windows và chạy:

Lần đầu kết nối, SSH sẽ hỏi "The authenticity of host ... can't be established. Are you sure?" — gõ yes rồi Enter. Sau đó dán mật khẩu root từ bước 5.

Bạn sẽ thấy dấu nhắc lệnh màu xanh:

root@it-courses-prod:~#

Bạn đã vào trong server. Mọi thứ từ giờ đến bước 12 đều thực hiện ở đây.

Đợi thêm một phút — đôi khi SSH daemon cần thời gian sau khi server khởi động lần đầu. Nếu sau 5 phút vẫn lỗi, kiểm tra lại IP và xem mạng nhà bạn có chặn cổng 22 đi ra không.
8

Harden the server (don't skip this) Bảo mật máy chủ (đừng bỏ qua bước này)

A root login over password is the single biggest attack surface a fresh VPS has. Bots find it within minutes of the IP becoming reachable. Replace it with a non-root user that uses an SSH key.

8a. Update everything

apt update && apt upgrade -y

8b. Create a non-root user with sudo

adduser deploy
usermod -aG sudo deploy

Pick a strong password when prompted; you'll rarely type it.

8c. Generate an SSH key on Windows (second PowerShell window)

ssh-keygen -t ed25519 -C "deploy@cloudfly"

Accept the default location. Set a passphrase or leave empty. The public key lands at C:\Users\YOU\.ssh\id_ed25519.pub.

8d. Copy that public key onto the VPS

su - deploy
mkdir -p ~/.ssh && chmod 700 ~/.ssh
nano ~/.ssh/authorized_keys

Paste the contents of id_ed25519.pub, save with Ctrl+O Enter Ctrl+X, then:

chmod 600 ~/.ssh/authorized_keys
exit

8e. Test the key-based login from a fresh PowerShell window

ssh deploy@YOUR.SERVER.IP

If you land in a deploy@it-courses-prod:~$ prompt without typing a password — you're set. Do not skip this test.

8f. Disable root login & password auth

Back in the root SSH session:

nano /etc/ssh/sshd_config

Find and set:

PermitRootLogin no
PasswordAuthentication no

Save, then reload SSH:

systemctl reload ssh

8g. Turn on the firewall

ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable

Type y to confirm.

! Critical
Only do step 8f after step 8e succeeds. If you disable password login before key login works, you'll lock yourself out and have to use Cloudfly's web console (VNC tab) to fix it.

Đăng nhập root bằng mật khẩu là điểm yếu lớn nhất của một VPS mới. Bot sẽ tìm ra trong vài phút sau khi IP có thể truy cập. Thay nó bằng một user không phải root, đăng nhập bằng SSH key.

8a. Cập nhật toàn bộ hệ thống

apt update && apt upgrade -y

8b. Tạo user mới có quyền sudo

usermod -aG sudo deploy

Chọn một mật khẩu mạnh; bạn sẽ hiếm khi phải gõ nó.

8c. Tạo SSH key trên Windows (cửa sổ PowerShell mới)

ssh-keygen -t ed25519 -C "deploy@cloudfly"

Bấm Enter để chấp nhận đường dẫn mặc định. Đặt passphrase hoặc bỏ trống. File khoá công khai nằm ở C:\Users\TEN\.ssh\id_ed25519.pub.

8d. Đưa khoá công khai lên VPS

mkdir -p ~/.ssh && chmod 700 ~/.ssh
nano ~/.ssh/authorized_keys

Dán nội dung file id_ed25519.pub, lưu lại bằng Ctrl+O Enter Ctrl+X, rồi:

chmod 600 ~/.ssh/authorized_keys
exit

8e. Thử đăng nhập bằng key từ một cửa sổ PowerShell mới

Nếu vào được dấu nhắc deploy@it-courses-prod:~$ mà không cần gõ mật khẩu — bạn đã làm đúng. Đừng bỏ qua bước thử này.

8f. Tắt đăng nhập root & đăng nhập bằng mật khẩu

Quay lại phiên SSH root:

Tìm và sửa:

Lưu, sau đó nạp lại SSH:

Gõ y để xác nhận.

Chỉ làm bước 8f sau khi bước 8e thành công. Nếu tắt mật khẩu trước khi key hoạt động, bạn sẽ tự khoá mình ra ngoài và phải dùng console web Cloudfly (tab VNC) để sửa.

10

Clone the project and build it Tải code về và build dự án

Your repo on GitHub is private, so the VPS needs read access. The cleanest way is a deploy key — a read-only SSH key scoped to one repository.

10a. Generate a deploy key on the VPS

ssh-keygen -t ed25519 -f ~/.ssh/it_courses_deploy -N "" -C "vps-deploy"
cat ~/.ssh/it_courses_deploy.pub

Copy the output line.

10b. Add it to GitHub as a deploy key

Open your repo on GitHub → Settings → Deploy keys → Add deploy key. Title: cloudfly-prod. Paste the key. Do not tick "Allow write access". Save.

10c. Tell SSH which key to use for github.com

nano ~/.ssh/config

Paste:

Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/it_courses_deploy
  IdentitiesOnly yes
chmod 600 ~/.ssh/config

10d. Clone and build

cd ~
git clone git@github.com:YOUR-USERNAME/IT-courses.git it-courses
cd it-courses
npm ci
npm run build

The build prints Next.js compilation progress and ends with a route summary. If you see it, you're past the hardest part.

10e. Build failed on the VPS? Read this

If you see one of these — your VPS doesn't have enough RAM to build:

  • The terminal prints Killed and drops back to the prompt with no other message.
  • SSH disconnects in the middle of the build, then reconnects fine afterwards.
  • Next.js prints FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory.
  • dmesg | tail shows Out of memory: Killed process … (node).

This is the OOM killer doing its job. next build on this project peaks around 1.2–1.5 GB of RAM because of MDX compilation + Shiki syntax highlighting. On a 1 GB VPS that's a guaranteed kill. Pick one of the three fixes below, ordered by effort:

💡 Fix 1 (quickest) — add 2 GB of swap, build through it

Swap is disk space that Linux uses as overflow RAM. The build will be slower (disk is ~50× slower than RAM), but it will finish. Already covered in step 9d — if you skipped it, jump back and run those commands now, then return here and retry npm run build.

Trade-off: the build can take 4–8 minutes instead of 40 seconds, and writing/reading swap shortens SSD life slightly. Fine for a hobby site you redeploy a few times a month.

💡 Fix 2 — cap Node's heap so it gives up earlier than the OOM killer

Sometimes Node grabs more memory than it actually needs and triggers the OOM killer before the swap kicks in. Telling Node a hard ceiling makes it garbage-collect more aggressively and stay under the line:

NODE_OPTIONS="--max-old-space-size=1536" npm run build

Use 1536 on a 2 GB VPS (leaves ~500 MB for Linux + ssh + nginx). On 1 GB + 2 GB swap, try 1024 first. This is a one-line fix, no system changes — try it before reaching for the bigger hammers.

📝 Fix 3 (recommended for a 1 GB VPS) — don't build on the server at all

Why: your laptop has 16+ GB of RAM and a real CPU. The build that takes 5 minutes on the VPS finishes in 30 seconds on your machine. Then you upload only the compiled output — 30–80 MB — and the server just runs node server.js. No more OOM, no more swap thrashing, and the server never sees your source code.

When to pick this: solo developer, 1 GB or smaller VPS, or you just want deploys to be fast. When NOT to: if multiple teammates deploy from different OSes — drift between machines causes "works on my laptop" bugs; use GitHub Actions instead.

💡 Cách 1 (nhanh nhất) — tạo 2 GB swap rồi build xuyên qua nó

Đánh đổi: build có thể mất 4–8 phút thay vì 40 giây, và việc đọc/ghi swap làm giảm tuổi thọ SSD một chút. Ổn cho site cá nhân, mỗi tháng deploy vài lần.

Đôi khi Node tham lam, xin nhiều bộ nhớ hơn nó thực sự cần và bị OOM killer tóm trước khi swap kịp đỡ. Bảo Node một giới hạn cứng sẽ buộc nó dọn rác (garbage collect) mạnh tay hơn và nằm dưới ngưỡng:

📝 Cách 3 (khuyên dùng cho VPS 1 GB) — không build trên server nữa

Vì sao: laptop của bạn có 16+ GB RAM và CPU thật. Cùng một bản build mất 5 phút trên VPS thì máy bạn chỉ cần 30 giây. Khi nào nên dùng: dev một mình, VPS 1 GB hoặc nhỏ hơn, hoặc đơn giản là muốn deploy nhanh gọn. Khi nào KHÔNG nên: nếu nhiều người trong team deploy từ máy khác nhau (OS khác, Node version khác) — máy ai nấy build dễ sinh bug kiểu "máy tao chạy được"; lúc đó nên chuyển sang GitHub Actions.

Alternative — nginx + Certbot (instead of Caddy) Tuỳ chọn thay thế — nginx + Certbot (thay cho Caddy)

⚠ Choose one: Caddy OR nginx, never both ⚠ Chỉ chọn một: Caddy HOẶC nginx, không bao giờ cả hai
Both want to own ports 80 and 443. If you already installed Caddy in step 12 and want to switch, stop and remove it first: sudo systemctl stop caddy && sudo systemctl disable caddy && sudo apt remove -y caddy. Then come back here. Cả hai đều muốn chiếm port 80 và 443. Nếu bạn đã cài Caddy ở bước 12 và muốn đổi sang nginx, dừng và gỡ Caddy trước: sudo systemctl stop caddy && sudo systemctl disable caddy && sudo apt remove -y caddy. Sau đó quay lại bước này.

nginx is the classic reverse proxy — older, more configurable, more widely documented. The trade-off vs Caddy: you write a bit more config, and HTTPS is a separate tool (Certbot) instead of automatic. For most beginners Caddy is easier; pick nginx if your team already knows it, or if you want fine-grained control over headers, caching, and rate limits later.

1. Install nginx

sudo apt update
sudo apt install -y nginx
sudo systemctl enable --now nginx

Open your VPS IP in a browser — you should see the "Welcome to nginx!" page. That confirms nginx is running and port 80 is open.

2. Write the server block

Create a config file for your site. Replace yourname.com with your real domain everywhere:

sudo nano /etc/nginx/sites-available/itcourses

Paste this in — note the proxy points to localhost:8888 because that's the port PM2 runs Next.js on (see package.json):

server {
    listen 80;
    listen [::]:80;
    server_name yourname.com www.yourname.com;

    # Max upload size for MDX / images
    client_max_body_size 20M;

    # gzip for HTML, CSS, JS, JSON
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css application/json application/javascript
               text/xml application/xml application/xml+rss text/javascript;

    location / {
        proxy_pass http://localhost:8888;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 60s;
    }

    # Cache Next.js static assets aggressively (filenames are hashed)
    location /_next/static/ {
        proxy_pass http://localhost:8888;
        proxy_cache_valid 200 365d;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }
}

3. Enable it and test

# Enable the site
sudo ln -s /etc/nginx/sites-available/itcourses /etc/nginx/sites-enabled/

# Remove the default welcome page
sudo rm /etc/nginx/sites-enabled/default

# Test for syntax errors — ALWAYS do this before reload
sudo nginx -t

# Apply the new config
sudo systemctl reload nginx
💡 Why nginx -t first?
A bad config on reload will not kill nginx (it keeps the old config), but it's still good hygiene. nginx -t tells you the exact line number of any typo before you touch the live server.

If DNS in step 13 is already pointed at the VPS, visit http://yourname.com — you should see your Next.js site (over plain HTTP for now).

4. Add free HTTPS with Certbot

Certbot is the official Let's Encrypt client. It will edit your nginx config to add a listen 443 ssl block, request a real certificate, and set up auto-renewal — all in one command.

sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d yourname.com -d www.yourname.com

Certbot will ask:

  • Your email (used only for expiry warnings)
  • Agree to ToS — type Y
  • Whether to redirect HTTP → HTTPS — choose 2 (Redirect)

That's it. Reload your browser at https://yourname.com and you'll see the padlock.

5. Verify auto-renewal works

Certbot installs a systemd timer that renews any certificate < 30 days from expiry. Confirm it's armed:

sudo systemctl list-timers | grep certbot
sudo certbot renew --dry-run

If the dry-run says "Congratulations, all simulated renewals succeeded", you're done forever — you'll never have to think about HTTPS again.

Snapshots: your "undo button" Snapshot: nút "Undo" của bạn

A snapshot is a frozen copy of your entire VPS disk at one point in time. If a bad deploy or a bad rm wrecks the server, you restore the snapshot and you're back within minutes.

Manual snapshot before risky changes

In the Cloudfly dashboard, open your server's detail page → Snapshot tab → Tạo snapshot mới. Give it a name like before-node-upgrade and click create. Costs a few thousand đồng per GB.

Scheduled auto-snapshot

Same page → Backup tự động → enable. Pick weekly. Cloudfly keeps the last few rotations and bills by storage.

ℹ Limits of snapshots
Snapshots live inside Cloudfly's infrastructure. They protect against software accidents — they don't protect against account suspension or a regional outage. Your code is already on GitHub, and you can periodically scp any uploaded content to your laptop.

Snapshot là bản sao đông cứng toàn bộ ổ đĩa VPS tại một thời điểm. Nếu một lần deploy lỗi hay một lệnh rm sai làm hỏng server, bạn khôi phục snapshot và mọi thứ quay lại như cũ trong vài phút.

Snapshot thủ công trước thay đổi rủi ro

Trong dashboard Cloudfly, vào trang chi tiết server → tab Snapshot → Tạo snapshot mới. Đặt tên như before-node-upgrade rồi tạo. Tốn vài nghìn đồng/GB.

Snapshot tự động theo lịch

Cùng trang → Backup tự động → bật. Chọn hằng tuần. Cloudfly giữ vài bản gần nhất và tính tiền theo dung lượng lưu trữ.

ℹ Giới hạn của snapshot
Snapshot lưu bên trong hạ tầng Cloudfly. Chúng giúp bạn khỏi sai sót phần mềm — nhưng không bảo vệ nếu tài khoản bị khoá hay cả region gặp sự cố. Code của bạn vẫn nằm trên GitHub, và bạn có thể scp nội dung upload về laptop định kỳ.

Renewal & cost tracking Gia hạn & theo dõi chi phí

Cloudfly bills monthly from the date you created the server. Two habits keep you from waking up to a suspended VPS:

  • Turn on email reminders for low balance in Tài khoản → Cài đặt thông báo, threshold 100,000 VND.
  • Every month, check Hóa đơn / Lịch sử giao dịch for the past 30 days. Line items should match your VPS plan plus snapshot storage; anything else is worth investigating.

If the balance hits zero, the VPS isn't destroyed immediately — Cloudfly suspends it for a grace period (usually a few days) before reclaiming the disk. Top up before the grace window closes and the server resumes exactly as it was.

Cloudfly tính tiền hằng tháng kể từ ngày bạn tạo server. Hai thói quen giúp bạn khỏi giật mình thấy VPS bị tạm dừng:

  • Bật thông báo email khi số dư thấp ở Tài khoản → Cài đặt thông báo, ngưỡng 100.000đ.
  • Mỗi tháng, mở Hóa đơn / Lịch sử giao dịch xem 30 ngày qua. Các khoản trừ phải khớp với gói VPS + snapshot; bất kỳ khoản lạ nào cũng nên kiểm tra.

Khi số dư về 0, VPS không bị huỷ ngay — Cloudfly tạm dừng trong vài ngày (grace period) trước khi thu hồi ổ đĩa. Nạp tiền trước khi hết hạn grace là server tự bật lại nguyên trạng.

Browser shows 502 Bad Gateway Reverse proxy (Caddy or nginx) is up but can't reach Node on port 8888. SSH in, run pm2 status. If it-courses is errored or stopped, check pm2 logs it-courses --lines 100 — usually a missing env var or stale build. Run ~/deploy.sh again.
Caddy can't get an HTTPS certificate sudo journalctl -u caddy -n 50 shows why. Most common: DNS not yet pointing at the VPS — wait, then sudo systemctl reload caddy. Or Cloudflare's orange-cloud proxy blocked the challenge — switch to grey "DNS only".
nginx: bind() to 0.0.0.0:80 failed (98: Address already in use) Something else owns port 80 — usually Caddy or Apache. Find the culprit: sudo ss -tlnp | grep :80. Stop and disable it (e.g. sudo systemctl disable --now caddy), then sudo systemctl restart nginx.
Certbot fails with "challenge failed" / timeout Let's Encrypt can't reach your VPS on port 80. Check: (1) DNS in step 13 actually points to this VPS — nslookup yourname.com from your laptop. (2) UFW allows HTTP — sudo ufw status should show 80/tcp ALLOW. (3) Cloudflare proxy is OFF (grey cloud) during certificate issuance.
sudo nginx -t reports a syntax error The message tells you the exact file and line — open it with sudo nano /etc/nginx/sites-available/itcourses, fix the typo (usually a missing semicolon ; or unmatched brace }), and re-run sudo nginx -t until it says "syntax is ok, test is successful".
