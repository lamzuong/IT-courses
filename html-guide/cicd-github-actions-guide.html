What CI/CD actually is CI/CD thực ra là gì

Imagine a café kitchen. Every dish that leaves the kitchen passes a checkpoint: a chef tastes it, checks it's not burnt, then a runner carries it to the right table. Nobody serves a plate straight from the pan without that check.

CI/CD is that checkpoint and that runner, but for your code.

  • CI (Continuous Integration) = the taste-test. Every time you push code, it gets built and tested automatically. If something is broken, you find out in minutes — not from an angry user.
  • CD (Continuous Deployment) = the runner. If the taste-test passes, the new version is carried to your server and goes live, all by itself.

The tool that does this for us is GitHub Actions — it lives inside your GitHub repo for free and reacts whenever you push.

💡 Why bother?
Right now you deploy by hand: build, copy files, restart. That's fine until the day you forget a step or push a typo at 11pm. CI/CD removes the human-error part and makes every release boringly identical.

Hãy tưởng tượng bếp của một quán café. Mỗi món trước khi rời bếp đều qua một trạm kiểm tra: đầu bếp nếm thử, xem có bị cháy không, rồi nhân viên mới bưng ra đúng bàn. Không ai bưng thẳng món từ chảo ra mà chưa kiểm tra cả.

CI/CD chính là cái trạm kiểm tra và người bưng món đó, nhưng dành cho code của bạn.

  • CI (Continuous Integration) = công đoạn nếm thử. Mỗi lần bạn push code, nó được build và kiểm thử tự động. Có gì hỏng, bạn biết trong vài phút — chứ không phải nghe người dùng la.
  • CD (Continuous Deployment) = người bưng món. Nếu phần nếm thử đạt, phiên bản mới được đưa lên server và chạy thật, hoàn toàn tự động.

Công cụ làm việc này cho mình là GitHub Actions — nó nằm sẵn trong repo GitHub của bạn, miễn phí, và tự phản ứng mỗi khi bạn push.

💡 Tại sao phải làm?
Hiện tại bạn deploy thủ công: build, copy file, restart. Vẫn ổn — cho tới ngày bạn quên một bước hoặc push nhầm lúc 11 giờ đêm. CI/CD bỏ đi phần "lỗi do con người" và khiến mỗi lần release đều giống hệt nhau, nhàm chán một cách an toàn.

What you need first Cần chuẩn bị gì trước

Before we write a single line, let's make sure the table is set. You need these five things — most you already have from earlier deploy work.

  • Your IT-courses code is on GitHub (a repo you can push to).
  • A server (VPS) you can reach over SSH — the same one from your DigitalOcean / CloudFly guide.
  • Docker is installed on that server (Step 3 shows how if it isn't yet), and your Dockerfile is in the repo (it already is ✅).
  • The project is already cloned on the server, e.g. at ~/IT-courses.
  • You can log in to your server as a user, e.g. git pull works there.
📦 A quick word on the plan
We'll teach the most common setup: GitHub Actions runs your tests, then SSHes into your own server and rebuilds the Docker container. This reuses everything you already built — no new hosting bill.
⚠️ Don't have a server yet?
Then the simplest CI/CD is Vercel — connect the repo once and it auto-deploys on every push, no workflow file needed. See your vercel-deployment-guide.html. This guide is for the self-hosted-on-your-own-VPS path.

Trước khi viết dòng nào, mình dọn bàn cho gọn đã. Bạn cần năm thứ sau — phần lớn bạn đã có từ lần deploy trước rồi.

  • Code IT-courses đã nằm trên GitHub (một repo bạn push được).
  • Một server (VPS) bạn vào được bằng SSH — chính cái server trong hướng dẫn DigitalOcean / CloudFly của bạn.
  • Docker đã cài trên server đó (Bước 3 sẽ chỉ cách nếu chưa có), và file Dockerfile có sẵn trong repo (đã có rồi ✅).
  • Project đã được clone sẵn trên server, ví dụ tại ~/IT-courses.
  • Bạn đăng nhập được vào server, ví dụ git pull chạy được ở đó.
📦 Nói nhanh về kế hoạch
Mình sẽ dạy kiểu phổ biến nhất: GitHub Actions chạy test, rồi SSH vào chính server của bạn và build lại container Docker. Cách này tận dụng mọi thứ bạn đã làm — không phát sinh tiền hosting mới.
⚠️ Chưa có server?
Vậy CI/CD đơn giản nhất là Vercel — kết nối repo một lần, nó tự deploy mỗi lần push, khỏi cần file workflow. Xem vercel-deployment-guide.html. Hướng dẫn này dành cho hướng tự host trên VPS của riêng bạn.

Install Docker on your server Cài Docker lên server

Think of Docker as a sealed lunchbox. Instead of cooking on the customer's stove and hoping their kitchen has the right pans, you pack the whole meal — app, Node.js, every setting — into one box that runs the same everywhere. The robot in Step 8 will say docker build on your server, so Docker has to be living there first. Let's install it.

Step in the door first

From your own computer, SSH into the server (same login you use for git pull there):

ssh your-user@your-server-ip

Everything below runs on the server, in that SSH session.

Install Docker the official, no-guessing way

Docker publishes a little script that looks at your Linux and installs the correct version for you — no hunting for the right package name. Download it, then run it:

curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

It chats for a minute, then you're done. sudo just means "do this as the admin" — installing system software needs that.

Let yourself (and the robot) skip sudo

Out of the box, only the admin can talk to Docker, so you'd type sudo before every command. Add your user to the docker group once and that annoyance disappears — important, because the deploy robot logs in as you and won't be typing any passwords.

sudo usermod -aG docker $USER

This only takes effect on a fresh login. The simplest fix: log out and SSH back in. (Or run newgrp docker to refresh the current session.)

Make Docker wake up on its own

If the server ever reboots, you want Docker (and your site) to come back without you. This one line turns it on now and sets it to auto-start on boot:

sudo systemctl enable --now docker

Check it actually works

Two quick tests. The first prints the version; the second downloads a tiny "hello" image and runs it:

docker --version
docker run hello-world

If you see a friendly "Hello from Docker!" message, the lunchbox factory is open for business. 🎉

📦 This assumes Ubuntu / Debian
The script above also handles most common Linux servers (Ubuntu, Debian, CentOS, Fedora). The DigitalOcean and CloudFly servers from your earlier guides are Ubuntu, so you're covered.
⚠️ Still asked for sudo after adding the group?
That's normal until you log out and back in — the group membership only loads on a new session. Re-SSH and try docker run hello-world again without sudo.
✓ One prerequisite ticked
That's the "Docker is installed on the server" item from Step 2 done. The rest of this guide can now safely call docker build and docker run over SSH.

Hãy hình dung Docker như một hộp cơm niêm phong. Thay vì nấu nhờ trên bếp của khách rồi cầu mong bếp họ có đủ xoong chảo, bạn gói cả bữa ăn — app, Node.js, mọi cấu hình — vào một cái hộp chạy giống hệt nhau ở mọi nơi. Robot ở Bước 8 sẽ gõ docker build ngay trên server của bạn, nên Docker phải có sẵn ở đó trước đã. Mình cài nó nhé.

Bước vào cửa trước đã

Từ máy của bạn, SSH vào server (dùng đúng tài khoản bạn vẫn git pull ở đó):

ssh your-user@your-server-ip

Mọi lệnh bên dưới đều chạy trên server, trong phiên SSH đó.

Cài Docker theo cách chính chủ, khỏi đoán

Docker có sẵn một đoạn script tự nhìn xem Linux của bạn là gì rồi cài đúng phiên bản — bạn khỏi phải mò tên gói. Tải về, rồi chạy:

curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

Nó chạy lách tách một lúc là xong. sudo chỉ có nghĩa là "làm việc này với quyền quản trị" — cài phần mềm hệ thống thì cần quyền đó.

Cho bạn (và robot) khỏi phải gõ sudo

Mặc định chỉ quản trị viên mới nói chuyện được với Docker, nên bạn sẽ phải gõ sudo trước mỗi lệnh. Thêm user của bạn vào nhóm docker một lần là hết phiền — điều này quan trọng, vì robot deploy đăng nhập với tư cách là bạn và sẽ không gõ mật khẩu nào cả.

sudo usermod -aG docker $USER

Việc này chỉ có hiệu lực khi bạn đăng nhập lại. Cách đơn giản nhất: đăng xuất rồi SSH vào lại. (Hoặc chạy newgrp docker để làm mới phiên hiện tại.)

Để Docker tự thức dậy

Lỡ server có khởi động lại, bạn muốn Docker (và website) tự bật lại mà không cần bạn. Một dòng này vừa bật ngay bây giờ, vừa đặt cho nó tự chạy mỗi lần khởi động:

sudo systemctl enable --now docker

Kiểm tra xem chạy thật chưa

Hai phép thử nhanh. Cái đầu in ra phiên bản; cái sau tải một image "hello" tí hon về rồi chạy:

docker --version
docker run hello-world

Nếu thấy dòng chữ thân thiện "Hello from Docker!" thì xưởng đóng hộp cơm đã mở cửa hoạt động. 🎉

📦 Phần này giả định Ubuntu / Debian
Đoạn script ở trên cũng xử lý được hầu hết server Linux phổ biến (Ubuntu, Debian, CentOS, Fedora). Server DigitalOcean và CloudFly trong các hướng dẫn trước của bạn đều là Ubuntu, nên yên tâm nhé.
⚠️ Thêm nhóm rồi mà vẫn bị đòi sudo?
Bình thường thôi, cho tới khi bạn đăng xuất rồi đăng nhập lại — tư cách nhóm chỉ nạp khi có phiên mới. SSH vào lại và thử docker run hello-world lần nữa, lần này khỏi sudo.
✓ Đã tick xong một thứ cần chuẩn bị
Vậy là mục "Docker đã cài trên server" ở Bước 2 coi như xong. Phần còn lại của hướng dẫn giờ có thể yên tâm gọi docker build và docker run qua SSH.

Create the workflow file Tạo file workflow

GitHub Actions looks for instruction files in one exact folder: .github/workflows/. Anything ending in .yml in there becomes a "workflow" it can run. Let's make that folder and an empty file.

In your project folder, run:

mkdir -p .github/workflows
ni .github/workflows/deploy.yml
🪟 On Windows PowerShell
ni is short for New-Item — it creates the empty file. If you're on Git Bash or macOS, use touch .github/workflows/deploy.yml instead. Either way you now have an empty file to fill in.

The file is written in YAML — a format where indentation matters. Think of it like a bullet-point outline: things lined up under each other belong together. We'll fill this file across the next steps.

⚠️ Spaces, not tabs
YAML hates the Tab key. Always indent with 2 spaces. If a workflow mysteriously refuses to run, a stray tab is the usual culprit.

GitHub Actions chỉ tìm file hướng dẫn ở đúng một thư mục: .github/workflows/. Mọi file kết thúc bằng .yml trong đó sẽ thành một "workflow" mà nó chạy được. Mình tạo thư mục đó và một file rỗng nhé.

Trong thư mục project, chạy:

mkdir -p .github/workflows
ni .github/workflows/deploy.yml
🪟 Trên Windows PowerShell
ni là viết tắt của New-Item — nó tạo ra file rỗng. Nếu bạn dùng Git Bash hoặc macOS thì thay bằng touch .github/workflows/deploy.yml. Kiểu nào thì giờ bạn cũng có một file rỗng để điền vào.

File này viết bằng YAML — một định dạng mà thụt lề rất quan trọng. Hãy hình dung như một dàn ý gạch đầu dòng: những thứ thẳng hàng dưới nhau là cùng một nhóm. Mình sẽ điền file này dần qua các bước sau.

⚠️ Dùng dấu cách, đừng dùng Tab
YAML rất ghét phím Tab. Luôn thụt lề bằng 2 dấu cách. Nếu workflow tự nhiên không chịu chạy, thủ phạm thường là một dấu Tab lạc vào.
5

Write the CI part (tests) Viết phần CI (kiểm thử)

Let's start with the taste-test. Open .github/workflows/deploy.yml and paste this. It's the whole "CI" half — we'll add deploy in Step 8.

name: CI/CD

# When should the robot wake up?
on:
  push:
    branches: [main]   # every push to main
  pull_request:        # and every PR, to catch bugs early

jobs:
  test:
    name: Build & test
    runs-on: ubuntu-latest   # a fresh rented computer
    steps:
      - uses: actions/checkout@v4        # 1. grab your code
      - uses: actions/setup-node@v4      # 2. install Node.js
        with:
          node-version: 22
          cache: npm
      - run: npm ci                      # 3. install deps (clean)
      - run: npm run typecheck           # 4. TypeScript sanity check
      - run: npm run lint                # 5. style / mistakes check
      - run: npm run test                # 6. run the unit tests

Reading it top to bottom, like a recipe:

  • on: — the trigger. "Run whenever someone pushes to main, or opens a pull request."
  • runs-on: — GitHub rents you a brand-new, clean Ubuntu computer for each run. Nothing from your laptop leaks in, so it catches the "works on my machine" bugs.
  • steps: — run in order. Check out code → install Node → install packages → typecheck → lint → test. Each uses the exact scripts from your package.json, so it matches what you run locally.
✓ Why npm ci not npm install?
npm ci installs the exact versions from package-lock.json and never quietly changes them. On a robot you always want the predictable, locked version.
📝 Note on E2E tests
We left out npm run test:e2e (Playwright) on purpose — it needs a running browser and is slower. You can add it later once the basics are green.

Bắt đầu với phần nếm thử. Mở .github/workflows/deploy.yml và dán đoạn này vào. Đây là toàn bộ nửa "CI" — phần deploy mình thêm ở Bước 8.

name: CI/CD

# Khi nào robot thức dậy?
on:
  push:
    branches: [main]   # mỗi lần push lên main
  pull_request:        # và mỗi pull request, để bắt lỗi sớm

jobs:
  test:
    name: Build & test
    runs-on: ubuntu-latest   # một máy tính thuê mới tinh
    steps:
      - uses: actions/checkout@v4        # 1. lấy code của bạn về
      - uses: actions/setup-node@v4      # 2. cài Node.js
        with:
          node-version: 22
          cache: npm
      - run: npm ci                      # 3. cài thư viện (sạch)
      - run: npm run typecheck           # 4. kiểm tra TypeScript
      - run: npm run lint                # 5. kiểm tra lỗi / phong cách
      - run: npm run test                # 6. chạy unit test

Đọc từ trên xuống, như một công thức nấu ăn:

  • on: — điều kiện kích hoạt. "Chạy mỗi khi có người push lên main, hoặc mở pull request."
  • runs-on: — GitHub cho bạn thuê một máy Ubuntu mới tinh, sạch sẽ cho mỗi lần chạy. Không có gì từ máy bạn lọt vào, nên nó bắt được mấy lỗi kiểu "máy mình chạy tốt mà".
  • steps: — chạy lần lượt. Lấy code → cài Node → cài package → typecheck → lint → test. Mỗi bước dùng đúng script trong package.json của bạn, nên giống hệt lúc bạn chạy ở máy.
✓ Sao lại npm ci chứ không phải npm install?
npm ci cài đúng phiên bản ghi trong package-lock.json và không tự ý đổi. Trên robot, bạn luôn muốn phiên bản khóa cứng, đoán trước được.
📝 Về test E2E
Mình cố tình bỏ npm run test:e2e (Playwright) — nó cần mở trình duyệt và chạy chậm hơn. Khi mọi thứ cơ bản đã xanh, bạn có thể thêm sau.
6

Make an SSH key for the robot Tạo khóa SSH cho robot

For CD, the robot needs to log into your server — but a robot can't type a password. Instead we give it a key. Think of an SSH key as a pair: a padlock (public key) you bolt onto your server's door, and a matching key (private key) the robot keeps secret. Only that key opens that padlock.

On your own computer, make a fresh key pair just for deploys:

ssh-keygen -t ed25519 -C "github-actions" -f deploy_key

Press Enter twice when it asks for a passphrase (leave it empty — the robot can't type one). You now have two files:

  • deploy_key — the private key (the secret half — goes to GitHub).
  • deploy_key.pub — the public key (the padlock — goes to the server).

Bolt the padlock onto your server

Copy the public key onto the server so it trusts this key. Open deploy_key.pub, copy its one line, then on the server run:

echo "PASTE_THE_PUBLIC_KEY_LINE_HERE" >> ~/.ssh/authorized_keys
🔒 Never share the private key
deploy_key (no .pub) is like your house key. Never commit it to git, never paste it in chat. It only ever goes into GitHub's encrypted Secrets box in the next step.

Để làm CD, robot cần đăng nhập vào server của bạn — nhưng robot không gõ mật khẩu được. Thay vào đó mình đưa cho nó một cái khóa. Hãy hình dung khóa SSH như một cặp: một ổ khóa (public key) bạn gắn lên cửa server, và một chìa khớp với ổ đó (private key) robot giữ bí mật. Chỉ chìa đó mới mở được ổ đó.

Trên máy tính của bạn, tạo một cặp khóa mới chỉ dùng để deploy:

ssh-keygen -t ed25519 -C "github-actions" -f deploy_key

Khi nó hỏi passphrase, bấm Enter hai lần (để trống — robot không gõ được). Giờ bạn có hai file:

  • deploy_key — khóa private (nửa bí mật — đưa cho GitHub).
  • deploy_key.pub — khóa public (cái ổ khóa — gắn lên server).

Gắn ổ khóa lên server

Chép khóa public lên server để nó tin tưởng khóa này. Mở deploy_key.pub, copy dòng duy nhất bên trong, rồi trên server chạy:

echo "DÁN_DÒNG_PUBLIC_KEY_VÀO_ĐÂY" >> ~/.ssh/authorized_keys
🔒 Đừng bao giờ chia sẻ khóa private
deploy_key (không có .pub) giống như chìa khóa nhà bạn. Đừng commit vào git, đừng dán vào chat. Nó chỉ được đưa vào hộp Secrets mã hóa của GitHub ở bước tiếp theo.
7

Add secrets to GitHub Thêm secret vào GitHub

The robot needs three private facts to reach your server: where it is, who to log in as, and the key. We store these in GitHub's Secrets — a locked drawer that never shows up in logs and never leaks into pull requests.

In your repo on GitHub, go to:

Settings → Secrets and variables → Actions → New repository secret

Add these three, one at a time:

Secret nameValue
SSH_HOSTYour server's IP, e.g. 203.0.113.10
SSH_USERThe login user, e.g. root or deploy
SSH_KEYThe full contents of the private deploy_key file
⚠️ Paste the whole key
For SSH_KEY, copy everything in the deploy_key file — including the -----BEGIN... and -----END... lines. Miss a line and the login fails.
✓ Why secrets and not just put it in the file?
Anything in your code is visible to anyone who can see the repo. Secrets are encrypted and only revealed to the running robot, masked even in the logs. Passwords and keys always go here.

Robot cần ba thông tin riêng tư để vào được server: server ở đâu, đăng nhập bằng ai, và khóa nào. Mình cất chúng trong Secrets của GitHub — một ngăn kéo khóa kín, không bao giờ hiện trong log và không lọt vào pull request.

Trong repo trên GitHub, vào:

Settings → Secrets and variables → Actions → New repository secret

Thêm ba cái sau, lần lượt từng cái:

Tên secretGiá trị
SSH_HOSTIP server của bạn, ví dụ 203.0.113.10
SSH_USERUser đăng nhập, ví dụ root hoặc deploy
SSH_KEYToàn bộ nội dung file khóa private deploy_key
⚠️ Dán nguyên cả khóa
Với SSH_KEY, copy toàn bộ nội dung file deploy_key — kể cả dòng -----BEGIN... và -----END.... Thiếu một dòng là đăng nhập thất bại.
✓ Sao phải dùng secret mà không ghi thẳng vào file?
Bất cứ thứ gì nằm trong code đều bị nhìn thấy bởi ai xem được repo. Secrets thì được mã hóa, chỉ hé lộ cho robot lúc chạy, và che cả trong log. Mật khẩu với khóa luôn để ở đây.
8

Write the CD part (deploy) Viết phần CD (deploy)

Now the runner. Add a second job to the same deploy.yml file, under jobs:, lined up with test:. This job logs into your server and rebuilds the Docker container.

  deploy:
    name: Deploy to server
    needs: test                              # only if tests passed
    if: github.ref == 'refs/heads/main'      # only on main, not PRs
    runs-on: ubuntu-latest
    steps:
      - name: Deploy over SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SSH_HOST }}
          username: ${{ secrets.SSH_USER }}
          key: ${{ secrets.SSH_KEY }}
          script: |
            cd ~/IT-courses
            git pull origin main
            docker build -t it-courses .
            docker stop it-courses || true
            docker rm it-courses || true
            docker run -d --name it-courses \
              -p 8888:8888 --restart unless-stopped it-courses

The two small lines at the top are the whole magic:

  • needs: test — deploy waits for the test job and only runs if it was green. A broken build never reaches your users. This is the "gate."
  • if: ... main — deploy runs for pushes to main, but skips pull requests. You don't want every draft PR going live.

The script: block is exactly what you'd type on the server by hand, now done for you: pull the new code, build a fresh image, stop and remove the old container, start the new one on port 8888. The || true means "don't panic if there's no old container to stop" (like the very first deploy).

📂 Match your real path
Change ~/IT-courses if your project lives somewhere else on the server, and the container/image name it-courses to whatever you used in your Docker guide.
✓ Why rebuild on the server?
Your Dockerfile already does a clean multi-stage build with output: "standalone". Rebuilding there keeps the image small and avoids shipping big files across the internet. Simple and reliable for a single server.

Giờ tới người bưng món. Thêm một job thứ hai vào cùng file deploy.yml, nằm dưới jobs:, thẳng hàng với test:. Job này đăng nhập vào server và build lại container Docker.

  deploy:
    name: Deploy to server
    needs: test                              # chỉ chạy nếu test đã đạt
    if: github.ref == 'refs/heads/main'      # chỉ trên main, không phải PR
    runs-on: ubuntu-latest
    steps:
      - name: Deploy over SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SSH_HOST }}
          username: ${{ secrets.SSH_USER }}
          key: ${{ secrets.SSH_KEY }}
          script: |
            cd ~/IT-courses
            git pull origin main
            docker build -t it-courses .
            docker stop it-courses || true
            docker rm it-courses || true
            docker run -d --name it-courses \
              -p 8888:8888 --restart unless-stopped it-courses

Hai dòng nhỏ ở đầu chính là toàn bộ phép màu:

  • needs: test — deploy chờ job test và chỉ chạy nếu nó đã xanh. Một bản build hỏng không bao giờ tới tay người dùng. Đây chính là "cái cổng".
  • if: ... main — deploy chạy khi push lên main, nhưng bỏ qua pull request. Bạn đâu muốn mỗi PR nháp đều lên thật.

Khối script: chính là những gì bạn tự gõ trên server, giờ được làm hộ: kéo code mới về, build image mới, dừng và xóa container cũ, chạy container mới ở cổng 8888. Dòng || true nghĩa là "đừng hoảng nếu chưa có container cũ để dừng" (như lần deploy đầu tiên).

📂 Chỉnh cho khớp đường dẫn thật
Đổi ~/IT-courses nếu project của bạn nằm chỗ khác trên server, và đổi tên container/image it-courses thành tên bạn đã dùng trong hướng dẫn Docker.
✓ Sao lại build ngay trên server?
Dockerfile của bạn đã build sạch theo nhiều tầng với output: "standalone". Build ngay tại đó giúp image nhỏ gọn và khỏi phải gửi file lớn qua mạng. Đơn giản và đáng tin cho một server đơn lẻ.
9

Push & watch it run Push & xem nó chạy

Everything is in one file now. Commit it and push — and this push itself becomes the first thing the robot reacts to.

git add .github/workflows/deploy.yml
git commit -m "Add CI/CD with GitHub Actions"
git push origin main

Now open your repo on GitHub and click the Actions tab. You'll see your run appear with a spinning yellow dot. Click into it to watch each step live, like a progress bar:

  • 🟡 Yellow = running right now.
  • ✅ Green check = that step passed.
  • ❌ Red X = something failed — click it to read the exact error line.

When both test and deploy turn green, open your site in the browser. The new version is live — and you never touched the server. 🎉

✓ Your new everyday routine
From now on: edit code → git push → done. The robot tests and deploys for you. If tests fail, nothing deploys and you get an email. That's the whole payoff.
💡 Nice next step: protect main
In Settings → Branches you can add a rule so code can't merge into main unless the test job is green. Now broken code literally can't get in.

Mọi thứ giờ nằm gọn trong một file. Commit rồi push — và chính cú push này trở thành điều đầu tiên robot phản ứng.

git add .github/workflows/deploy.yml
git commit -m "Add CI/CD with GitHub Actions"
git push origin main

Giờ mở repo trên GitHub và bấm tab Actions. Bạn sẽ thấy lần chạy của mình hiện ra với một chấm vàng đang quay. Bấm vào để xem từng bước chạy trực tiếp, như một thanh tiến trình:

  • 🟡 Vàng = đang chạy ngay lúc này.
  • ✅ Dấu xanh = bước đó đã đạt.
  • ❌ Dấu X đỏ = có lỗi — bấm vào để đọc đúng dòng báo lỗi.

Khi cả test và deploy đều xanh, mở trang web trên trình duyệt. Phiên bản mới đã chạy thật — mà bạn không hề đụng tới server. 🎉

✓ Thói quen mới mỗi ngày của bạn
Từ giờ: sửa code → git push → xong. Robot tự test và deploy. Nếu test fail thì không có gì được deploy và bạn nhận email báo. Đó chính là toàn bộ phần thưởng.
💡 Bước tiếp theo hay ho: bảo vệ main
Trong Settings → Branches, bạn có thể thêm luật để code không merge được vào main trừ khi job test đã xanh. Lúc đó code hỏng không cách nào lọt vào được.

Troubleshooting Khắc phục sự cố

ProblemSolution
Workflow doesn't show up in Actions tabThe file must be at .github/workflows/deploy.yml exactly, and pushed to GitHub. Check the folder name has a leading dot.
Invalid workflow file errorAlmost always indentation. Use 2 spaces, never Tab. Paste the YAML into a validator if unsure.
Deploy step: Permission denied (publickey)The SSH_KEY secret is wrong or incomplete, or the public key wasn't added to the server's ~/.ssh/authorized_keys. Re-do Step 6 & 7.
Host key verification failedSome setups need the server's fingerprint. The appleboy/ssh-action handles this by default; make sure SSH_HOST is the IP, not a URL with https://.
Tests pass locally but fail on the robotThe robot is clean — a missing dependency or a file you forgot to commit. Read the red step's log; it names the exact file.
docker: command not found on serverDocker isn't installed (or your user can't run it). Follow Step 3 to install Docker and add your user to the docker group.
Site didn't update after green deployThe old container may still hold the port. SSH in and run docker ps to confirm it-courses is the new one; check the build actually rebuilt.

Lỗi Invalid workflow fileGần như luôn là do thụt lề. Dùng 2 dấu cách, đừng dùng Tab. Không chắc thì dán YAML vào một trình kiểm tra.

Glossary Bảng thuật ngữ

TermMeaning
CIContinuous Integration — automatically build & test your code on every change.
CDContinuous Deployment — automatically ship that code to your server once tests pass.
GitHub ActionsGitHub's free robot that runs your workflows when events (like a push) happen.
WorkflowA .yml recipe in .github/workflows/ telling Actions what to do.
JobA group of steps that run together on one rented computer (e.g. test, deploy).
StepA single command or action inside a job, run in order.
RunnerThe fresh virtual computer GitHub gives you for each run (ubuntu-latest).
YAMLThe text format workflows are written in — indentation matters, use spaces.
SecretAn encrypted value (password, key) stored in GitHub, hidden from logs.
SSH keyA padlock (public) + key (private) pair that lets the robot log in without a password.
DockerA tool that packs your app + everything it needs into one portable "box" that runs the same on any server.
docker groupA Linux group whose members can run Docker without typing sudo every time.
Docker imageA sealed box holding your app + everything it needs to run, built from the Dockerfile.
ContainerA running copy of an image — the live process serving your site on port 8888.
