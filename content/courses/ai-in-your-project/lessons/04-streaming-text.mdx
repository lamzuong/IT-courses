Streaming is the difference between "the model is thinking" and "the model is talking." Same wait time, very different feel. The user who watches words appear is patient; the user who stares at a spinner is not. Nothing about the model changes — only what the browser does with the bytes as they arrive.

This lesson takes the SSE handler from lesson 3 and connects it to a real React UI. By the end, your app will drip tokens in as the model produces them, and you'll have a small `useStream` hook you can drop into every page that needs it.

## 01 — Why streaming is worth the small extra work

A model that emits 200 tokens at 30 tokens per second takes about seven seconds. If you wait for the whole response, the user stares at a spinner for seven seconds and then sees a wall of text. If you stream, the first word appears in ~50ms and the user reads while the rest is generating. The total wall-clock time is identical, but the *perceived* latency is the time-to-first-token.

The reading speed of a typical adult is roughly 250 words per minute, or about four words per second. Most modern open-weight models emit faster than that. Once the user starts reading, the model has effectively *caught up to them* — they finish the sentence at the same moment the model finishes generating it. The wait disappears.

Streaming also gives you a free progress signal. If tokens stop arriving you know something is wrong; if they're trickling in, the system is alive. You don't need a separate "is it still working?" indicator.

A concrete side-by-side, asking the same question — *"Summarise this customer's last three orders"* — that produces ~200 tokens:

| | Non-streaming | Streaming |
|---|---|---|
| t = 0.0s | spinner | spinner |
| t = 0.05s | spinner | *"Phuoc has placed…"* |
| t = 1.5s | spinner | *"Phuoc has placed three orders in the last 6 weeks. The most recent…"* |
| t = 7.0s | full reply appears | *"…and totalled $312."* (reply ends) |
| Felt as | seven seconds of nothing | half a second of nothing, then reading |

Same model, same answer, same wall-clock cost. Only the user's experience changed.

## 02 — `model.stream()` on the server

You wrote this handler in lesson 3. Read it again with streaming in mind:

```ts
const stream = await model.stream(message);     // → AsyncIterable<AIMessageChunk>

const body = new ReadableStream({                // → Web ReadableStream we'll attach to Response
  async start(controller) {
    for await (const chunk of stream) {          // → each chunk: AIMessageChunk { content: 'Hi' }
      controller.enqueue(encoder.encode(`data: ${JSON.stringify(chunk.content)}\n\n`)); // → wire frame: 'data: "Hi"\n\n'
    }
    controller.enqueue(encoder.encode('data: [DONE]\n\n')); // → terminator the client looks for
    controller.close();                          // → closes the stream cleanly
  },
});
```

`model.stream()` returns an async iterator of `AIMessageChunk` objects. Each iteration is one chunk — usually a single token or a small group of them. You wrap them in the SSE format (`data: ...\n\n`) and push them through a `ReadableStream`. The `[DONE]` sentinel is a convention, not a spec — but every browser SSE consumer expects something like it, so we follow the crowd.

Three things to notice:

- The handler is *push-shaped*, not pull-shaped. The model decides when chunks arrive; you forward them as they show up.
- `JSON.stringify` is mandatory — newlines in raw text would corrupt the SSE framing.
- The stream closes itself when the model finishes. No timeout, no `setTimeout`, no manual cleanup.

## 03 — The React side: a tiny `useStream` hook

The browser side is shorter than people expect. `fetch` returns a `Response` whose `body` is a `ReadableStream`. You read it chunk by chunk, parse the `data: ` lines, and append to state.

```tsx
'use client';
import { useState } from 'react';

export function useStream() {
  const [text, setText] = useState('');          // → '' initially; grows as chunks arrive
  const [running, setRunning] = useState(false); // → false; flips true while a request is in flight

  async function send(message: string) {
    setText('');                                 // → clears any prior reply
    setRunning(true);                            // → UI shows "running" state
    const res = await fetch('/api/chat', {       // → Response with streaming body
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ message }),         // → '{"message":"..."}'
    });
    const reader = res.body!.getReader();        // → ReadableStreamDefaultReader<Uint8Array>
    const decoder = new TextDecoder();           // → utf-8 decoder
    let buf = '';                                // → accumulates leftover bytes between reads
    while (true) {
      const { value, done } = await reader.read(); // → { value: Uint8Array, done: boolean }
      if (done) break;                           // → stream finished — exit loop
      buf += decoder.decode(value, { stream: true }); // → buf now contains 'data: "Hi"\n\ndata: "!"\n\n...'
      for (const line of buf.split('\n\n')) {    // → split into individual SSE frames
        if (!line.startsWith('data: ')) continue;// → skip blank/keepalive lines
        const payload = line.slice(6);           // → '"Hi"' (the JSON-encoded chunk)
        if (payload === '[DONE]') break;         // → stop on terminator
        setText((t) => t + JSON.parse(payload)); // → append decoded string ('Hi') to UI text
      }
      buf = buf.split('\n\n').slice(-1)[0];      // → keep the unfinished tail for the next read
    }
    setRunning(false);                           // → UI exits "running" state
  }

  return { text, running, send };                // → consumer reads `text`, calls `send(msg)`
}
```

Twenty-five lines, no dependency. The only subtle bit is the trailing-buffer trick: SSE frames are separated by `\n\n`, but a network read can split a frame in the middle. We keep the unfinished tail in `buf` and prepend it next time around. Without that, you'll randomly drop tokens whenever the network gives you a small chunk.

Once you have this hook, every chat-shaped page is roughly the same: a textarea, a button, and `<div>{text}</div>`. The streaming machinery becomes invisible.

Người dùng nhìn chữ hiện ra dần thì kiên nhẫn; người dùng nhìn spinner xoay thì không. Bản thân model không thay đổi gì — chỉ là trình duyệt làm gì với những byte khi chúng tới.

## 01 — Vì sao streaming xứng đáng với chút công sức thêm

Một model phát 200 token với tốc độ 30 token/giây sẽ mất khoảng bảy giây. Nếu bạn chờ cho xong cả response, người dùng nhìn spinner trong bảy giây rồi đột ngột thấy một bức tường chữ. Nếu bạn streaming, từ đầu tiên xuất hiện sau ~50ms và người dùng đọc trong khi phần còn lại đang được sinh ra. Wall-clock time tổng cộng *y hệt*, nhưng độ trễ *cảm nhận được* chính là time-to-first-token.

Tốc độ đọc của một người lớn trung bình là khoảng 250 từ/phút, tức ~4 từ/giây. Hầu hết model mã nguồn mở hiện đại phát chữ ra nhanh hơn thế. Khi người dùng đã bắt đầu đọc, model coi như *bắt kịp họ* — họ đọc xong câu cuối ngay lúc model sinh xong câu cuối. Sự chờ đợi biến mất.

Streaming còn cho bạn một tín hiệu tiến độ miễn phí. Token ngừng tới thì biết có vấn đề; token đang trickle vào thì biết hệ thống vẫn sống. Bạn không cần thêm chỉ báo "có còn đang chạy không?" tách rời.

So sánh cụ thể, cùng một câu hỏi — *"Tóm tắt ba đơn hàng gần nhất của khách hàng này"* — sản sinh ~200 token:

| | Không streaming | Có streaming |
|---|---|---|
| t = 0.0s | spinner | spinner |
| t = 0.05s | spinner | *"Phước đã đặt…"* |
| t = 1.5s | spinner | *"Phước đã đặt ba đơn trong 6 tuần qua. Gần nhất là…"* |
| t = 7.0s | toàn bộ reply hiện ra | *"…và tổng là $312."* (reply kết thúc) |
| Cảm giác | bảy giây trống rỗng | nửa giây trống rỗng, rồi vừa đọc vừa chờ |

Cùng model, cùng câu trả lời, cùng chi phí wall-clock. Chỉ trải nghiệm người dùng thay đổi.

## 02 — `model.stream()` ở phía server

Bạn đã viết handler này ở bài 3. Đọc lại nhưng để ý mặt streaming:

```ts
const stream = await model.stream(message);     // → AsyncIterable<AIMessageChunk>

const body = new ReadableStream({                // → Web ReadableStream we'll attach to Response
  async start(controller) {
    for await (const chunk of stream) {          // → each chunk: AIMessageChunk { content: 'Hi' }
      controller.enqueue(encoder.encode(`data: ${JSON.stringify(chunk.content)}\n\n`)); // → wire frame: 'data: "Hi"\n\n'
    }
    controller.enqueue(encoder.encode('data: [DONE]\n\n')); // → terminator the client looks for
    controller.close();                          // → closes the stream cleanly
  },
});
```

`model.stream()` trả về một async iterator của các object `AIMessageChunk`. Mỗi vòng lặp là một chunk — thường là một token hoặc vài token. Bạn bọc chúng theo định dạng SSE (`data: ...\n\n`) và đẩy qua một `ReadableStream`. Sentinel `[DONE]` là quy ước, không phải spec — nhưng mọi SSE consumer trên trình duyệt đều mong đợi một dấu kết thúc kiểu vậy, nên ta đi theo số đông.

Ba điểm cần để ý:

- Handler có hình dạng *push*, không phải *pull*. Model quyết định khi nào chunk tới; bạn forward khi chúng xuất hiện.
- `JSON.stringify` là bắt buộc — ký tự xuống dòng trong text gốc sẽ phá khung SSE.
- Stream tự đóng khi model kết thúc. Không timeout, không `setTimeout`, không cleanup thủ công.

## 03 — Phía React: một hook `useStream` nho nhỏ

Phía trình duyệt ngắn hơn người ta tưởng. `fetch` trả về `Response` mà `body` của nó là một `ReadableStream`. Bạn đọc từng chunk, parse các dòng `data: `, và append vào state.

```tsx
'use client';
import { useState } from 'react';

export function useStream() {
  const [text, setText] = useState('');          // → '' initially; grows as chunks arrive
  const [running, setRunning] = useState(false); // → false; flips true while a request is in flight

  async function send(message: string) {
    setText('');                                 // → clears any prior reply
    setRunning(true);                            // → UI shows "running" state
    const res = await fetch('/api/chat', {       // → Response with streaming body
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ message }),         // → '{"message":"..."}'
    });
    const reader = res.body!.getReader();        // → ReadableStreamDefaultReader<Uint8Array>
    const decoder = new TextDecoder();           // → utf-8 decoder
    let buf = '';                                // → accumulates leftover bytes between reads
    while (true) {
      const { value, done } = await reader.read(); // → { value: Uint8Array, done: boolean }
      if (done) break;                           // → stream finished — exit loop
      buf += decoder.decode(value, { stream: true }); // → buf now contains 'data: "Hi"\n\ndata: "!"\n\n...'
      for (const line of buf.split('\n\n')) {    // → split into individual SSE frames
        if (!line.startsWith('data: ')) continue;// → skip blank/keepalive lines
        const payload = line.slice(6);           // → '"Hi"' (the JSON-encoded chunk)
        if (payload === '[DONE]') break;         // → stop on terminator
        setText((t) => t + JSON.parse(payload)); // → append decoded string ('Hi') to UI text
      }
      buf = buf.split('\n\n').slice(-1)[0];      // → keep the unfinished tail for the next read
    }
    setRunning(false);                           // → UI exits "running" state
  }

  return { text, running, send };                // → consumer reads `text`, calls `send(msg)`
}
```

Hai mươi lăm dòng, không cần dependency. Phần tinh tế duy nhất là mẹo "buffer đuôi": các SSE frame ngăn cách bằng `\n\n`, nhưng một lần đọc network có thể cắt một frame ở giữa. Ta giữ phần đuôi chưa xong trong `buf` và ghép lại ở vòng đọc tiếp theo. Không có nó, bạn sẽ ngẫu nhiên bị mất token mỗi khi mạng đẩy về một chunk nhỏ.

Khi đã có hook này, mọi chat-page về cơ bản đều giống nhau: một textarea, một button, và `<div>{text}</div>`. Bộ máy streaming trở nên vô hình.
