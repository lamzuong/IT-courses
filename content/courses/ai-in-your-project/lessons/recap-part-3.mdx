## How the pieces fit

**Your first tool (L09)** introduces the trio that defines a tool: `name`, `description`, and Zod `schema`. The model emits a `tool_call` in JSON, your runtime executes the function, the result goes back as a `ToolMessage`. Two model calls + one tool execution = one turn.

**Reads at scale (L10)** is the playbook for the moment your dev seed graduates to a production DB. Three patterns — pagination (cursor, not offset), filtering (typed object, never raw SQL), aggregation (pre-compute summaries). Each one shrinks the data flowing into context by an order of magnitude.

**Design rules (L11)** is the principles file. Narrow tools beat broad. Idempotent by default. The description IS the prompt. Errors are recovery paths. And the `5–15 sweet spot` for how many tools an agent should hold at once.

**Write danger (L12)** is the moment the safety stance from L01 gets teeth. Every mutating tool — *create*, *update*, *delete*, *charge*, *send* — needs a confirmation gate before it commits. Reads run automatically; writes wait for approval. The gate machinery is built in Part IV with `interrupt()`.

**Idempotency (L13)** is what keeps retries from billing twice. A request key per logical operation, a check-and-insert pattern in the DB. The first call lands; the second call returns the first call's result. The model never has to know it retried.

**Errors & retries (L14)** is the lesson that elevates error messages to first-class prompt material. *What failed. Why. What to do instead.* Three sentences, and most of the time the model fixes its call on the next turn — a self-healing recovery path with no scaffolding.

<Recap>
- **A tool is `name + description + Zod schema + function`.** The Zod schema is the contract that tells the model what to emit and lets your function trust what it receives.
- **Reads at scale: paginate, filter, aggregate.** Cursor-based pagination, typed filter objects translated to parameterised queries, and pre-computed summaries instead of raw rows.
- **Never interpolate model output into SQL.** Parse with Zod, validate, translate. Your function writes the SQL; the model only describes intent.
- **Narrow tools beat broad ones.** `promotions.create_for_lapsed_customers` is a worse tool from a code-aesthetic standpoint and a much better one for an LLM. The name *is* the use case.
- **Writes need a confirmation gate.** No mutating tool runs unattended in this course. Even your first write goes through `interrupt()` before commit.
- **Idempotent by default.** Every write tool takes a `requestKey`; the DB check-and-insert pattern keeps retries safe.
- **Typed errors are recovery paths.** *What failed. Why. What to do.* Often the model reads your error message and fixes its own call.
- **5–15 tools is the sweet spot.** Past 30 the model picks near-siblings; past 50 you need scoping, retrieval, or sub-agents.
</Recap>

**Up next — Part IV: LangGraph & confirmation.** The pieces stop being separate. State, nodes, edges, conditional routing, and `interrupt()` come together into a graph that can pause for a human and resume from where it stopped.

## Các mảnh ghép vào nhau ra sao

**Tool đầu tiên (L09)** giới thiệu bộ ba định nghĩa một tool: `name`, `description`, và Zod `schema`. Model phát ra một `tool_call` dạng JSON, runtime của bạn chạy hàm, kết quả quay về dưới dạng `ToolMessage`. Hai lần gọi model + một lần thực thi tool = một lượt.

**Đọc ở quy mô (L10)** là playbook cho khoảnh khắc DB seed của bạn "tốt nghiệp" lên DB production. Ba pattern — phân trang (cursor, không offset), filter (object có kiểu, không bao giờ raw SQL), aggregate (gộp sẵn summary). Mỗi pattern thu nhỏ lượng dữ liệu chảy vào context theo cấp số mười.

**Quy tắc thiết kế (L11)** là file nguyên tắc. Tool hẹp thắng tool rộng. Mặc định idempotent. Description CHÍNH LÀ prompt. Lỗi là lối thoát. Và `vùng vàng 5–15` cho số tool tối ưu mà một agent nên nắm cùng lúc.

**Vùng nguy hiểm khi ghi (L12)** là khoảnh khắc quan điểm an toàn từ L01 có "răng". Mọi tool có khả năng thay đổi data — *create*, *update*, *delete*, *charge*, *send* — đều cần một cổng xác nhận trước khi commit. Đọc thì chạy tự động; ghi thì chờ duyệt. Phần "máy móc" của cổng này được dựng ở Phần IV với `interrupt()`.

**Idempotency (L13)** là thứ giữ cho retry không bị tính tiền hai lần. Một request key cho mỗi thao tác logic, một pattern check-and-insert trong DB. Lần gọi đầu tiên hạ cánh; lần gọi thứ hai trả lại kết quả của lần đầu. Model không cần biết là nó đã retry.

**Lỗi & retry (L14)** là bài nâng "thông điệp lỗi" lên thành "vật liệu prompt hạng nhất". *Cái gì hỏng. Tại sao. Nên làm gì thay vào đó.* Ba câu, và phần lớn thời gian model tự sửa lệnh gọi của nó ở lượt sau — một lối tự hồi phục, không cần dựng giàn giáo gì cả.

<Recap>
- **Một tool = `name + description + Zod schema + function`.** Zod schema là hợp đồng nói cho model biết phải phát ra gì, và cho hàm của bạn quyền tin tưởng những gì nhận về.
- **Đọc ở quy mô: phân trang, filter, gộp.** Phân trang cursor-based, object filter có kiểu được dịch sang query có tham số, và summary tính sẵn thay vì kéo từng row.
- **Không bao giờ nhét output của model vào SQL.** Parse bằng Zod, validate, dịch. Hàm của bạn viết SQL; model chỉ mô tả ý định.
- **Tool hẹp thắng tool rộng.** `promotions.create_for_lapsed_customers` xét về thẩm mỹ code thì kém, nhưng đối với LLM thì tốt hơn rất nhiều. Cái tên *chính là* use case.
- **Mọi lệnh ghi cần cổng xác nhận.** Trong khoá này, không tool nào có khả năng thay đổi data được chạy không người giám sát. Ngay cả lệnh ghi đầu tiên cũng đi qua `interrupt()` trước khi commit.
- **Mặc định idempotent.** Mỗi tool ghi nhận một `requestKey`; pattern check-and-insert trong DB giữ cho retry an toàn.
- **Lỗi có kiểu là lối thoát.** *Cái gì hỏng. Tại sao. Nên làm gì.* Thường thì model đọc thông điệp lỗi của bạn rồi tự sửa lệnh gọi của nó.
- **5–15 tool là vùng vàng.** Quá 30 thì model bắt đầu chọn nhầm tool gần giống nhau; quá 50 thì bạn cần phân vùng, retrieval, hoặc sub-agent.
</Recap>
