Testing AI integrations is testing a stochastic system. The patterns are different from unit-testing a sort function. A sort function has one right answer for every input; a model has a distribution. You can lower the temperature, you can pin the seed, you can swear at the API — and on a flaky day a model will still emit a slightly different sentence than yesterday. The trick is to stop testing the parts that vary and start testing the parts that don't. This lesson is about which parts are which.

The promise: by the end you'll have golden tests for tool-call routing, snapshot tests for prose shape, fake LLMs in your unit tests, and an eval dataset on every PR. None of those four things is "test that the AI is smart." All four are testing software, with the model as a deterministic-enough collaborator.

## 01 — Golden tool-call tests

Given a fixed prompt, fixed system prompt, and fixed tool definitions, the model should call the *right tool* with the *right args*. That is the part of your AI integration that is closest to deterministic, and it is the part most worth pinning. Set `temperature: 0`, run a small suite of inputs through the agent, and assert on the tool calls that come out:

```ts
test('lapsed-customer prompt produces a four-step plan', async () => {
  const llm = new ChatAnthropic({ model: 'claude-sonnet-4', temperature: 0 }); // → deterministic ChatAnthropic
  const agent = createAgent({ llm, tools: allTools, systemPrompt: SYSTEM });   // → CompiledStateGraph

  const result = await agent.invoke({
    messages: [new HumanMessage('Create a 15% promo for customers who haven\'t ordered in 60 days.')],
  });
  // → result: GraphState with full transcript

  const calls = result.messages.flatMap((m) =>
    isAIMessage(m) ? m.tool_calls ?? [] : [],     // → flatten tool_calls across every AIMessage
  );
  // → calls: ToolCall[] in the order the agent emitted them

  expect(calls.map((c) => c.name)).toEqual([      // → assertion: exact ordered tool names
    'promotions.create',
    'customers.lapsed',
    'email_drafts.create',
  ]);
  expect(calls[0].args).toMatchObject({ discountValue: 15 }); // → first tool args contain { discountValue: 15 }
});
```

`temperature: 0` plus `isAIMessage` from `@langchain/core/messages` is the workhorse pattern. Run this in CI on every PR; when a prompt change breaks the routing, the assertion fires and you find out before a customer does.

## 02 — Snapshot regression for prose

Sometimes the model has to generate prose: an email body, an explanation, a summary. You cannot assert on the exact string — punctuation will drift between runs. You *can* assert on the structure:

```ts
test('email body has the expected shape', async () => {
  const out = await agent.invoke({ messages: [new HumanMessage(PROMPT)] }); // → GraphState
  const body = out.draft.body;                    // → string — the AI-drafted email body

  expect(body.split(/[.!?]/).filter(Boolean).length).toBeGreaterThanOrEqual(3); // → at least 3 sentences
  expect(body).toMatch(/15%/);                    // → mentions the discount
  expect(body).toMatch(/code\s+LAPSED15/i);       // → mentions the promo code
  expect(body).not.toMatch(/\{\{|<placeholder>/i); // → no leftover template placeholders
});
```

Three sentences or more. Mentions the discount. Mentions the code. Has no leftover template placeholders. Those four shape assertions catch most regressions without chaining your test suite to one specific phrasing.

## 03 — Fake LLMs in tests

Your graph logic — routing, interrupts, retry loops — does not need a real model to be tested. Real models are slow and expensive and flaky in CI. Use a fake:

```ts
import { FakeListChatModel } from '@langchain/core/utils/testing'; // → scriptable LLM stub

test('graph routes write tool calls to await_confirmation', async () => {
  const fake = new FakeListChatModel({            // → returns its `responses` in order
    responses: [
      // The model "decides" to call promotions.create. Encoded as a
      // tool-call message your graph parses normally.
      JSON.stringify({
        tool_calls: [{ name: 'promotions.create', args: { code: 'TEST' }, id: '1' }],
      }),                                         // → scripted "first turn" — proposes one write
    ],
  });

  const graph = buildGraph({ llm: fake });        // → CompiledStateGraph wired to the fake LLM
  const result = await graph.invoke({ messages: [new HumanMessage('go')] }); // → GraphState (paused on interrupt)

  expect(result.__interrupt__).toBeDefined();     // → graph hit the confirmation interrupt
  expect(result.__interrupt__[0].value.kind).toBe('confirm_writes'); // → payload kind matches our shape
});
```

`FakeListChatModel` returns its `responses` in order. You can script a five-turn conversation, including parallel tool calls, simulated errors, anything. None of it costs tokens and none of it varies between runs. Your graph code thinks it's talking to a real LLM; it never has to know.

## 04 — Eval datasets

Beyond unit tests, you want a corpus of representative prompts replayed on every PR. A few dozen `(prompt, expected_tool_calls)` pairs in a JSON file:

```json
[
  {
    "id": "lapsed-customer-basic",
    "prompt": "Create a 15% promo for customers who haven't ordered in 60 days.",
    "expected": {
      "tools": ["promotions.create", "customers.lapsed", "email_drafts.create"],
      "first_args": { "discountValue": 15, "audience": "lapsed-60d" }
    }
  },
  {
    "id": "vip-thank-you",
    "prompt": "Send a thank-you email to all VIP customers.",
    "expected": {
      "tools": ["customers.byTag", "email_drafts.create"]
    }
  }
]
```

A small script loads the file, runs each prompt through the live agent, and reports a pass/fail per row plus a summary. When you tune a prompt or swap a model, you watch the score on the dataset go up or down. This is the evaluation layer; it is what lets you make a prompt change with confidence instead of crossed fingers. Several frameworks (LangSmith, Promptfoo) automate this loop, but a fifty-line script in your own repo is enough to start.

<Callout type="tip">
  Don't try to test "is the AI nice." Test "does the graph reach the END node when the user rejects." One is fuzzy; the other is software. Fuzzy properties are evaluated by humans skimming sample outputs; deterministic properties are tested by your CI. Don't mix them up.
</Callout>

## 05 — What's worth testing, what isn't

A short triage list:

- **Worth testing** — Graph routing logic. Tool args validation. Error recovery paths. Interrupt and resume. The deterministic surface is large; treat it like any other state machine.
- **Worth evaluating, not unit-testing** — Prose quality, tone, helpfulness. These belong in your eval dataset with sample-and-review, not in a CI assertion that fails on a comma change.
- **Not worth testing** — The model's specific word choice. Whether it uses "however" or "but". Whether it puts the discount before or after the deadline in the email. You will lose your mind chasing these flakes.

The mental model: your graph is software, your tools are software, your validators are software — test those rigorously. The model itself is a probabilistic component you wire between them; you bound its inputs and outputs, then you trust it within those bounds. You don't *test* trust; you *measure* it on the eval dataset.

<Recap>
- Test the deterministic surface: routing, args, recovery, interrupt and resume. Skip the model's prose creativity.
- Golden tool-call tests: `temperature: 0`, fixed prompt, assert on `tool_calls` from `isAIMessage` results.
- Snapshot prose by shape (sentence count, key phrases, JSON structure) — not exact text.
- Use `FakeListChatModel` from `@langchain/core/utils/testing` to test graph logic without real tokens.
- Build a small eval dataset of `(prompt, expected_tool_calls)` and replay it on every PR; tune prompts against the score.
- *"Does the graph reach END when the user rejects?"* is testable. *"Is the AI nice?"* is evaluable. Don't confuse them.
</Recap>

Bạn có thể hạ temperature, ghim seed, có chửi cả cái API — đến một ngày đỏng đảnh mô hình vẫn sẽ phát ra một câu hơi khác hôm qua. Mánh là dừng test mấy phần "thay đổi" và bắt đầu test mấy phần "không đổi". Bài này nói về phần nào là phần nào.

Lời hứa: cuối bài, bạn sẽ có golden test cho routing tool-call, snapshot test cho hình dạng văn xuôi, fake LLM trong unit test, và một eval dataset chạy trên mỗi PR. Không cái nào trong bốn cái đó là "test xem AI có thông minh không". Cả bốn đều là test phần mềm, với mô hình đóng vai một cộng tác viên đủ tất định.

## 01 — Golden test cho tool-call

Cho một prompt cố định, system prompt cố định, và định nghĩa tool cố định, mô hình *phải* gọi đúng tool với đúng args. Đó là phần gần với tất định nhất trong tích hợp AI của bạn, và là phần đáng ghim nhất. Đặt `temperature: 0`, chạy một bộ input nhỏ qua agent, và assert vào các tool call ra:

## 02 — Snapshot regression cho văn xuôi

Đôi khi mô hình phải sinh văn xuôi: body email, một đoạn giải thích, một bản tóm tắt. Bạn không thể assert nguyên văn — dấu câu sẽ trôi giữa các lần chạy. Bạn *có thể* assert vào cấu trúc:

```ts
test('email body has the expected shape', async () => {
  const out = await agent.invoke({ messages: [new HumanMessage(PROMPT)] }); // → GraphState

  expect(body.split(/[.!?]/).filter(Boolean).length).toBeGreaterThanOrEqual(3); // → at least 3 sentences

```

Tối thiểu ba câu. Nhắc tới mức giảm giá. Nhắc tới mã. Không còn placeholder template sót lại. Bốn assertion về hình dạng đó bắt được hầu hết regression mà không khoá test suite của bạn vào một cách diễn đạt cụ thể.

## 03 — Fake LLM trong test

Logic graph của bạn — routing, interrupt, vòng lặp retry — không cần mô hình thật để test. Mô hình thật thì chậm, đắt, và đỏng đảnh trong CI. Dùng đồ giả:

## 04 — Eval dataset

Ngoài unit test, bạn còn cần một corpus các prompt đại diện được chạy lại trên mỗi PR. Vài chục cặp `(prompt, expected_tool_calls)` trong một file JSON:

```json
[
  {
    "id": "lapsed-customer-basic",
    "prompt": "Create a 15% promo for customers who haven't ordered in 60 days.",
    "expected": {
      "tools": ["promotions.create", "customers.lapsed", "email_drafts.create"],
      "first_args": { "discountValue": 15, "audience": "lapsed-60d" }
    }
  },
  {
    "id": "vip-thank-you",
    "prompt": "Send a thank-you email to all VIP customers.",
    "expected": {
      "tools": ["customers.byTag", "email_drafts.create"]
    }
  }
]
```

Một script nhỏ load file, chạy mỗi prompt qua agent thật, và báo cáo pass/fail theo từng dòng cộng với tóm tắt. Khi bạn tinh chỉnh prompt hay đổi mô hình, bạn nhìn score trên dataset đi lên hay đi xuống. Đây là tầng evaluation; chính nó cho phép bạn đổi prompt với sự tự tin thay vì khấn vái. Một số framework (LangSmith, Promptfoo) tự động hoá vòng lặp này, nhưng một script năm chục dòng trong repo của bạn là đủ để bắt đầu.

<Callout type="tip">
  Đừng cố test "AI có dễ thương không". Hãy test "graph có tới node END khi user reject không". Một bên mơ hồ; một bên là phần mềm. Các thuộc tính mơ hồ thì để con người lướt xem mẫu đầu ra; các thuộc tính tất định thì để CI assert. Đừng trộn lẫn.
</Callout>

## 05 — Cái gì đáng test, cái gì không

Một bảng phân loại ngắn:

- **Đáng test** — Logic routing của graph. Validate args tool. Đường hồi phục lỗi. Interrupt và resume. Bề mặt tất định lớn lắm; cứ coi như một state machine bình thường.
- **Đáng evaluate, không phải unit-test** — Chất lượng văn xuôi, giọng điệu, mức hữu ích. Mấy thứ này thuộc về eval dataset với cơ chế "lấy mẫu rồi review", không phải assertion CI fail vì một cái dấu phẩy.
- **Không đáng test** — Lựa chọn từ ngữ cụ thể của mô hình. Nó dùng "however" hay "but". Nó đặt mức giảm giá trước hay sau deadline trong email. Bạn sẽ điên cuồng đuổi mấy flake này.

Mô hình tư duy: graph của bạn là phần mềm, tool của bạn là phần mềm, validator của bạn là phần mềm — test chúng nghiêm túc. Bản thân mô hình là một thành phần xác suất bạn nối giữa chúng; bạn ràng buộc input và output của nó, rồi tin tưởng nó trong các ràng buộc đó. Bạn không *test* sự tin tưởng; bạn *đo* sự tin tưởng trên eval dataset.
