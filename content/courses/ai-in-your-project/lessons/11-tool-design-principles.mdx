Tool design is API design with one extra constraint — your caller is a probabilistic interpreter, not a deterministic one. Every other API you've ever shipped has been called by code that does exactly what its author told it to. This one is called by a model that read your description, formed a guess about what the tool does, and is now trying. If your description was vague, the guess will be vague. If your name was clever, the guess will be wrong. Tool design rewards the dull, specific, and obvious in a way most software design does not.

This lesson is the principles file. Four rules to follow and five mistakes to recognize. None of them are revolutionary — they're the same things experienced API designers have always preached — but the cost of getting them wrong is higher when the caller can't read your source code and won't ask clarifying questions.

## 01 — Narrow beats broad

The instinct from human library design is to build one general tool and let the caller compose. *"`promotions.create` with sixteen optional fields covers everything."* For an LLM this is the wrong move. The model has to discover, from your description and schema, that a particular combination of fields means *"a 15% off code targeted at customers who haven't ordered in 60 days."* It will sometimes get it. It will sometimes not.

`promotions.create_for_lapsed_customers` is a worse tool from a code-aesthetic standpoint. Six tools instead of one, each a thin wrapper over the same primitive. But it's a much better tool for an LLM, because *the name itself tells the model what the tool does and when to use it*. The model picks the right tool by reading the directory; you don't have to teach it your composition rules.

A practical rule: if you find yourself writing *"to do X, set field A to Y and field B to Z"* in a tool description, that's two tools, not one. Split it.

**The broad version (the temptation)**

```ts
export const createPromotion = tool(
  async (args) => { /* ... */ },
  {
    name: 'promotions.create',
    description: 'Create a promotion. See schema for all options.',
    schema: z.object({
      type: z.enum(['percent', 'fixed', 'bogo', 'free_shipping']),
      value: z.number().optional(),
      audience: z.enum(['all', 'lapsed', 'vip', 'new', 'tag']).optional(),
      audienceTag: z.string().optional(),
      lapsedDays: z.number().optional(),
      minOrderCents: z.number().optional(),
      validFrom: z.string().datetime().optional(),
      validUntil: z.string().datetime().optional(),
      code: z.string().optional(),
      maxUses: z.number().optional(),
      stackable: z.boolean().optional(),
      // ...four more fields
    }),
  },
);
```

When the user asks *"15% off for customers who haven't ordered in 60 days"*, the model has to derive a sixteen-field shape: `type: 'percent'`, `value: 15`, `audience: 'lapsed'`, `audienceTag: undefined`, `lapsedDays: 60`, plus a default code. It will get this right most of the time and wrong some of the time, and the failures look like *"audience is 'tag' but audienceTag is missing"* — silent semantic errors, not validation errors.

**The narrow version (the right move)**

```ts
export const createLapsedCustomerPromo = tool(
  async ({ discountPercent, lapsedDays, code }) => { /* ... */ },
  {
    name: 'promotions.create_for_lapsed_customers',
    description: 'Create a percentage-off promo code targeted at customers who have not ordered in N days.',
    schema: z.object({
      discountPercent: z.number().int().min(1).max(75),
      lapsedDays: z.number().int().min(7).max(365),
      code: z.string().regex(/^[A-Z0-9]{4,16}$/),
    }),
  },
);

export const createVipThankYouPromo = tool(/* ... */);
export const createFlashSale = tool(/* ... */);
export const createFreeShippingPromo = tool(/* ... */);
```

Three fields, all required, all bounded. The name *is* the use case — the model picks `promotions.create_for_lapsed_customers` from the directory the same way you'd pick a function from autocomplete. Six tools instead of one, but each is twenty lines, each is independently testable, and the model never has to compose a sixteen-field call from scratch.

## 02 — Idempotent by default

The model will retry. Sometimes because the first call timed out, sometimes because it parsed your error response and decided the call didn't take, sometimes for no reason a logfile can explain. If your tool is `orders.charge` and a retry double-charges, you have a billing incident waiting for you.

The fix is the same fix you'd apply to a webhook handler: an idempotency key supplied (or generated) per logical operation, and a `check-and-insert` pattern in the database. The first call goes through. The second call sees the existing row and returns the cached result. The model gets the same answer either way and doesn't know — or need to know — that it was a retry.

For read tools this is almost free, because reads are naturally idempotent. The discipline matters most for the write tools we get to in lesson 12, but instilling it now means you don't have to retrofit it later.

A concrete sketch of the difference (full treatment in lesson 13):

```ts
// ❌ Not idempotent — every retry charges again
async function chargeOrder({ orderId, amountCents }) {
  return await stripe.charges.create({ amount: amountCents, ... });
}

// ✅ Idempotent — second call returns the first call's result
async function chargeOrder({ orderId, amountCents, requestKey }) {
  const existing = await db.charges.findUnique({ where: { requestKey } });
  if (existing) return existing;
  const charge = await stripe.charges.create({
    amount: amountCents,
    idempotency_key: requestKey,
  });
  return await db.charges.create({ data: { ...charge, requestKey } });
}
```

Same signature plus one `requestKey` argument; the customer is no longer at risk of being double-charged when the model retries.

## 03 — The description IS the prompt

The single highest-leverage thing you'll write in your AI codebase is the description string on each tool. It is sent to the model on every turn. It is what the model uses to decide *whether* to call the tool, *when* to call it, and *what arguments* to send. A vague description guarantees a confused model.

The bad version: `"Search customers."` Five characters of signal. The model has to guess what fields are searchable, what's returned, what the limits are.

The good version: `"Search customers by lapsed-days threshold, tag membership, or email domain. Returns up to 25 rows ordered by last_order_at descending. Use customers.list for unfiltered listing."` Three sentences, three things the model now knows it didn't have to guess. Notice the cross-reference at the end — pointing at a sibling tool helps the model pick the right one.

**The same tool, three versions of the description**

```ts
// ❌ Vague — the model guesses what's searchable and what comes back
description: 'Search customers.';

// ⚠ Better, but missing the boundary
description: 'Search customers by tag, email, or lapsed days.';

// ✅ Tells the model what it filters on, what it returns,
//    how many, and which sibling to use when this isn't right
description: [
  'Search customers by lapsed-days threshold, tag membership, or email domain.',
  'Returns up to 25 rows ordered by last_order_at descending.',
  'Use customers.list for unfiltered listing or customers.byId for a single lookup.',
].join(' ');
```

A second example for a write tool, where stakes are higher and the description does double duty as a guard rail:

```ts
// ❌ Tells the model nothing about side effects
description: 'Send an email.';

// ✅ States the side effect, the safety net, and the boundary
description: [
  'Draft an email to a single customer and save it as a DRAFT.',
  'Does NOT send — a human reviews drafts before they go out.',
  'For bulk drafts use email_drafts.create_for_segment.',
].join(' ');
```

The pattern that holds across all good descriptions: *what it does · what it returns / what its side effect is · which sibling to pick if this isn't the right one*. Three sentences, big leverage.

<Callout type="tip" title="Read your descriptions out loud">
The fastest test for tool descriptions: read them aloud as if you were briefing a coworker. If a sentence makes you stumble, the model will stumble. If you used jargon you'd normally explain, the model didn't get the explanation. If you said "etc." or "and so on", the model has no idea what was elided. Tool descriptions are technical writing, and the same instincts that make a good README make a good description.
</Callout>

## 04 — Typed errors

When something goes wrong inside a tool, the model receives the thrown error message verbatim as a `ToolMessage` and decides what to do next. That decision is going to be informed by what your error says.

`Error: ENOENT` tells the model nothing useful. It will retry, or give up, or hallucinate that the customer doesn't exist.

`Error: Customer with id 7041 not found. Use customers.search to look up customers by name or email.` tells the model exactly what happened and exactly what to do instead. Often the model will *follow the suggestion in your error message* and fix the call on the next turn — which means a well-written error becomes a self-healing recovery path with zero scaffolding code.

The shape that works: `<what failed>. <why>. <suggested next step>.` Treat error messages as part of the prompt — because that's what they are.

## 05 — How many tools is enough?

There is no magic number, but there are clear failure modes at both ends. Too few and the model has to compose primitives it doesn't understand; too many and it confuses sibling tools that sound alike. Most production agents live in what we'll call the <Term def="A metaphor borrowed from sports — the comfortable range where the model picks tools confidently and all descriptions fit in context without competing.">sweet spot</Term> — the bands below are what shows up in practice across small and mid-sized agents:

| Tool count | What you'll see |
| --- | --- |
| **5–15** | The sweet spot. The model picks confidently. Descriptions fit in context comfortably. |
| **15–30** | Still fine if names are tight and tools cluster by domain (`customers.*`, `promotions.*`, `orders.*`). |
| **30–50** | Starts to hurt. The model picks a near-sibling instead of the right tool, especially when two descriptions overlap. |
| **50+** | Stop adding. Switch strategy. |

**The unit is *intent*, not *endpoint*.** If your backend has 200 REST routes, you do not expose 200 tools. You expose the 10–20 things a user will actually ask the agent to do. A tool no real prompt ever triggers is just tokens you pay for on every turn — every turn — for the life of the agent.

When the system is genuinely big, three escape hatches in order of complexity:

- **Scope per session.** Bind only the tools relevant to the current task. A *support* agent gets 8 tools; a *billing* agent gets a different 8. Same codebase, different `.bindTools()` call at construction time.
- **Tool retrieval.** Index tool descriptions in a vector store, retrieve the top 10 per user message, bind those for that turn. Useful past ~50 tools where scoping isn't enough.
- **Sub-agents.** A router agent delegates to a `BillingAgent` whose internal tools are hidden from the parent. This is the LangGraph pattern in Part IV — each agent stays inside its own 5–15 sweet spot.

The deciding question for any tool you're tempted to add: *"would a fresh user prompt ever trigger this, and would the model pick it over its five nearest neighbours from the name and description alone?"* If no, rename, merge, or delete it.

## 06 — Common mistakes

A short, brutal checklist. If your tool matches any of these, refactor before you ship:

- **Tools that take 12 arguments.** The model will fill three correctly, two incorrectly, and forget seven. Split into smaller tools.
- **Tools that mutate without acknowledgment.** Every write tool should return what it wrote — the row, the ID, the timestamp. Returning `void` leaves the model with no way to confirm the call landed and no way to reference what it just did.
- **Tools that return free-form text instead of structured data.** *"I sent the email to ana@example.com at 14:32"* is a paragraph; the model has to re-parse it. Return `{ sentTo, sentAt }` and the model can quote the values directly.
- **Tools that call other tools recursively.** The model orchestrates tool sequences. Tools should not. A tool that internally calls three other tools is hiding work from the model and making the round-trip un-debuggable.
- **Tools that depend on global state.** *"This tool uses the customer set in the previous call."* No. Pass the customer ID explicitly every time. Implicit state across calls is how you build features that work in development and silently fail in production.

**Refactor pairs — what each anti-pattern looks like, and the fix**

*Silent writes — return what you wrote*

```ts
// ❌ Returns void; the model can't reference the row it just created
export const createDraft = tool(
  async ({ customerId, body }) => {
    await db.emailDrafts.create({ data: { customerId, body } });
    return 'OK';
  },
  { name: 'email_drafts.create', /* ... */ },
);

// ✅ Returns the row; the next turn can quote draftId, sentAt, etc.
export const createDraft = tool(
  async ({ customerId, body }) => {
    const draft = await db.emailDrafts.create({ data: { customerId, body } });
    return JSON.stringify({
      draftId: draft.id,
      customerId: draft.customerId,
      createdAt: draft.createdAt.toISOString(),
      status: 'draft',
    });
  },
  { name: 'email_drafts.create', /* ... */ },
);
```

*Free-form returns — give the model JSON, not prose*

```ts
// ❌ The model has to parse English to find the values
return `Sent the email to ${email} at ${new Date().toISOString()}.`;

// ✅ The model quotes structured fields directly
return JSON.stringify({ sentTo: email, sentAt: new Date().toISOString() });
```

*Recursive tools — surface the steps to the model instead*

```ts
// ❌ One tool hides three operations; the model can't reason about partial failure
export const createPromoAndDraftEmail = tool(
  async ({ discount, lapsedDays }) => {
    const promo = await createPromotion({ discount, lapsedDays });
    const customers = await findLapsedCustomers({ lapsedDays });
    const drafts = await Promise.all(customers.map((c) => createDraft({ customerId: c.id, /* ... */ })));
    return JSON.stringify({ promo, draftCount: drafts.length });
  },
  { /* ... */ },
);

// ✅ Three narrow tools the model orchestrates itself; each step is observable
export const createPromotion = tool(/* ... */);
export const customersLapsed = tool(/* ... */);
export const createDraft = tool(/* ... */);
```

*Implicit global state — pass the ID every time*

```ts
// ❌ Relies on whatever was set on the last call
let CURRENT_CUSTOMER: Customer | null = null;

export const setCustomer = tool(/* sets CURRENT_CUSTOMER */);
export const draftEmail = tool(
  async ({ body }) => {
    if (!CURRENT_CUSTOMER) throw new Error('No customer selected');
    return await createDraft({ customerId: CURRENT_CUSTOMER.id, body });
  },
  /* ... */
);

// ✅ Pure: every call carries its own context
export const draftEmail = tool(
  async ({ customerId, body }) => createDraft({ customerId, body }),
  {
    schema: z.object({
      customerId: z.string().uuid(),
      body: z.string().min(20),
    }),
    /* ... */
  },
);
```

A useful exercise after writing a tool: read it back and ask *"could a model that has never seen my code call this correctly using only the name, description, and schema?"* If the answer is no, you have more work to do.

<Recap>
- Narrow tools beat broad tools. The name should describe the situation, not just the verb.
- Idempotent by default. The model will retry, often invisibly. Build for that.
- Tool descriptions are part of the prompt. Spend real time on them; cross-reference siblings.
- Typed errors are recovery paths. *What failed. Why. What to do instead.* Three sentences, big leverage.
- How many tools? 5–15 is the sweet spot, 30+ starts to hurt, 50+ needs scoping, retrieval, or sub-agents. Count intents, not endpoints.
- Five anti-patterns to avoid: 12-arg tools, silent writes, free-form returns, recursive tools, implicit global state.
</Recap>
