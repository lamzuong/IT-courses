Tool design is API design with one extra constraint — your caller is a probabilistic interpreter, not a deterministic one. Every other API you've ever shipped has been called by code that does exactly what its author told it to. This one is called by a model that read your description, formed a guess about what the tool does, and is now trying. If your description was vague, the guess will be vague. If your name was clever, the guess will be wrong. Tool design rewards the dull, specific, and obvious in a way most software design does not.

This lesson is the principles file. Four rules to follow and five mistakes to recognize. None of them are revolutionary — they're the same things experienced API designers have always preached — but the cost of getting them wrong is higher when the caller can't read your source code and won't ask clarifying questions.

## 01 — Narrow beats broad

The instinct from human library design is to build one general tool and let the caller compose. *"`promotions.create` with sixteen optional fields covers everything."* For an LLM this is the wrong move. The model has to discover, from your description and schema, that a particular combination of fields means *"a 15% off code targeted at customers who haven't ordered in 60 days."* It will sometimes get it. It will sometimes not.

`promotions.create_for_lapsed_customers` is a worse tool from a code-aesthetic standpoint. Six tools instead of one, each a thin wrapper over the same primitive. But it's a much better tool for an LLM, because *the name itself tells the model what the tool does and when to use it*. The model picks the right tool by reading the directory; you don't have to teach it your composition rules.

A practical rule: if you find yourself writing *"to do X, set field A to Y and field B to Z"* in a tool description, that's two tools, not one. Split it.

**The broad version (the temptation)**

```ts
export const createPromotion = tool(             // → StructuredTool { name: 'promotions.create' }
  async (args) => { /* ... */ },                 // → handler receives 16-field object, must branch on `type`+`audience`
  {
    name: 'promotions.create',                   // → one identifier covering 4 promo types × 5 audiences
    description: 'Create a promotion. See schema for all options.', // → vague — model must guess valid combos
    schema: z.object({
      type: z.enum(['percent', 'fixed', 'bogo', 'free_shipping']), // → 4 disjoint promo kinds
      value: z.number().optional(),              // → meaning depends on `type` — model has to know that
      audience: z.enum(['all', 'lapsed', 'vip', 'new', 'tag']).optional(), // → 5 segments
      audienceTag: z.string().optional(),        // → required when audience='tag', else ignored
      lapsedDays: z.number().optional(),         // → required when audience='lapsed'
      minOrderCents: z.number().optional(),      // → optional gate
      validFrom: z.string().datetime().optional(), // → ISO 8601
      validUntil: z.string().datetime().optional(),// → ISO 8601
      code: z.string().optional(),               // → the promo code string
      maxUses: z.number().optional(),            // → optional cap
      stackable: z.boolean().optional(),         // → can it combine with other promos
      // ...four more fields
    }),
  },
);
// → one tool the model picks for any promotion — high failure rate on 60s-lapsed-15%-off prompts
```

When the user asks *"15% off for customers who haven't ordered in 60 days"*, the model has to derive a sixteen-field shape: `type: 'percent'`, `value: 15`, `audience: 'lapsed'`, `audienceTag: undefined`, `lapsedDays: 60`, plus a default code. It will get this right most of the time and wrong some of the time, and the failures look like *"audience is 'tag' but audienceTag is missing"* — silent semantic errors, not validation errors.

**The narrow version (the right move)**

```ts
export const createLapsedCustomerPromo = tool(   // → StructuredTool — name encodes the use case
  async ({ discountPercent, lapsedDays, code }) => { /* ... */ }, // → 3 required fields, all bounded
  {
    name: 'promotions.create_for_lapsed_customers', // → model picks this by reading the directory
    description: 'Create a percentage-off promo code targeted at customers who have not ordered in N days.', // → one job, one sentence
    schema: z.object({
      discountPercent: z.number().int().min(1).max(75), // → integer 1..75 (no 0%, no 90%)
      lapsedDays: z.number().int().min(7).max(365),     // → integer 7..365
      code: z.string().regex(/^[A-Z0-9]{4,16}$/),       // → uppercase alnum, 4-16 chars
    }),
  },
);

export const createVipThankYouPromo = tool(/* ... */);   // → StructuredTool 'promotions.create_vip_thank_you'
export const createFlashSale = tool(/* ... */);          // → StructuredTool 'promotions.create_flash_sale'
export const createFreeShippingPromo = tool(/* ... */);  // → StructuredTool 'promotions.create_free_shipping'
// → 4 narrow tools the model picks by name; failure mode is "no tool matched" (catchable), not "wrong combo" (silent)
```

Three fields, all required, all bounded. The name *is* the use case — the model picks `promotions.create_for_lapsed_customers` from the directory the same way you'd pick a function from autocomplete. Six tools instead of one, but each is twenty lines, each is independently testable, and the model never has to compose a sixteen-field call from scratch.

## 02 — Idempotent by default

The model will retry. Sometimes because the first call timed out, sometimes because it parsed your error response and decided the call didn't take, sometimes for no reason a logfile can explain. If your tool is `orders.charge` and a retry double-charges, you have a billing incident waiting for you.

The fix is the same fix you'd apply to a webhook handler: an idempotency key supplied (or generated) per logical operation, and a `check-and-insert` pattern in the database. The first call goes through. The second call sees the existing row and returns the cached result. The model gets the same answer either way and doesn't know — or need to know — that it was a retry.

For read tools this is almost free, because reads are naturally idempotent. The discipline matters most for the write tools we get to in lesson 12, but instilling it now means you don't have to retrofit it later.

A concrete sketch of the difference (full treatment in lesson 13):

```ts
// ❌ Not idempotent — every retry charges again
async function chargeOrder({ orderId, amountCents }) {
  return await stripe.charges.create({ amount: amountCents, ... }); // → new Charge each call (retry = double-charge)
}

// ✅ Idempotent — second call returns the first call's result
async function chargeOrder({ orderId, amountCents, requestKey }) {
  const existing = await db.charges.findUnique({ where: { requestKey } }); // → Charge | null
  if (existing) return existing;                 // → fast-path: retry hits here, returns first result
  const charge = await stripe.charges.create({   // → Stripe Charge (only on first call)
    amount: amountCents,
    idempotency_key: requestKey,                 // → Stripe also dedups server-side via this header
  });
  return await db.charges.create({ data: { ...charge, requestKey } }); // → persisted Charge row
}
// → second call with same requestKey returns the first Charge; customer charged exactly once
```

Same signature plus one `requestKey` argument; the customer is no longer at risk of being double-charged when the model retries.

**Where does `requestKey` come from?**

It is *not* a fresh UUID the model picks each time — that would defeat the whole pattern, since a retry would generate a different key and miss the dedup. Three places it can come from, in rough order of how often you'll use them:

- **The orchestrator (runtime layer above the tool).** A small wrapper sees `(toolName, args, turnId)` and produces a stable key — typically a UUID minted once per logical operation and reused across retries. This is the default; the tool body just receives `requestKey` as an argument.
- **The tool itself, derived from inputs.** When the args naturally identify the operation (`charge:${orderId}:${amountCents}`), the tool can hash them into a key. Cheap and good enough for most write tools.
- **The model, via the schema.** The schema declares `requestKey: z.string().uuid()` so the orchestrator can fill it before invocation. The model *sees* the field but doesn't pick its value — the wrapper writes the key into args before the tool runs.

The full pattern, including the `INSERT … ON CONFLICT` race-safe variant, is the subject of lesson 13. For now, every write tool you design should have a `requestKey` slot; treat it as the safety belt that makes retries cost nothing.

## 03 — The description IS the prompt

The single highest-leverage thing you'll write in your AI codebase is the description string on each tool. It is sent to the model on every turn. It is what the model uses to decide *whether* to call the tool, *when* to call it, and *what arguments* to send. A vague description guarantees a confused model.

The bad version: `"Search customers."` Five characters of signal. The model has to guess what fields are searchable, what's returned, what the limits are.

The good version: `"Search customers by lapsed-days threshold, tag membership, or email domain. Returns up to 25 rows ordered by last_order_at descending. Use customers.list for unfiltered listing."` Three sentences, three things the model now knows it didn't have to guess. Notice the cross-reference at the end — pointing at a sibling tool helps the model pick the right one.

**The same tool, three versions of the description**

```ts
// ❌ Vague — the model guesses what's searchable and what comes back
description: 'Search customers.';                // → 17 chars; model invents fields and shape

// ⚠ Better, but missing the boundary
description: 'Search customers by tag, email, or lapsed days.'; // → fields named, but no return shape, no sibling hint

// ✅ Tells the model what it filters on, what it returns,
//    how many, and which sibling to use when this isn't right
description: [
  'Search customers by lapsed-days threshold, tag membership, or email domain.', // → what it filters on
  'Returns up to 25 rows ordered by last_order_at descending.',                  // → what it returns + cap
  'Use customers.list for unfiltered listing or customers.byId for a single lookup.', // → sibling pointers
].join(' ');                                     // → "Search customers by ... 25 rows ... Use customers.list ..."
```

A second example for a write tool, where stakes are higher and the description does double duty as a guard rail:

```ts
// ❌ Tells the model nothing about side effects
description: 'Send an email.';                   // → 14 chars; model assumes it actually sends — dangerous

// ✅ States the side effect, the safety net, and the boundary
description: [
  'Draft an email to a single customer and save it as a DRAFT.',  // → what it does (saves a draft)
  'Does NOT send — a human reviews drafts before they go out.',   // → safety net (no surprise sends)
  'For bulk drafts use email_drafts.create_for_segment.',         // → sibling pointer for the bulk case
].join(' ');                                     // → "Draft an email ... Does NOT send ... For bulk ..."
```

The pattern that holds across all good descriptions: *what it does · what it returns / what its side effect is · which sibling to pick if this isn't the right one*. Three sentences, big leverage.

<Callout type="tip" title="Read your descriptions out loud">
The fastest test for tool descriptions: read them aloud as if you were briefing a coworker. If a sentence makes you stumble, the model will stumble. If you used jargon you'd normally explain, the model didn't get the explanation. If you said "etc." or "and so on", the model has no idea what was elided. Tool descriptions are technical writing, and the same instincts that make a good README make a good description.
</Callout>

## 04 — Typed errors

When something goes wrong inside a tool, the model receives the thrown error message verbatim as a `ToolMessage` and decides what to do next. That decision is going to be informed by what your error says.

`Error: ENOENT` tells the model nothing useful. It will retry, or give up, or hallucinate that the customer doesn't exist.

`Error: Customer with id 7041 not found. Use customers.search to look up customers by name or email.` tells the model exactly what happened and exactly what to do instead. Often the model will *follow the suggestion in your error message* and fix the call on the next turn — which means a well-written error becomes a self-healing recovery path with zero scaffolding code.

The shape that works: `<what failed>. <why>. <suggested next step>.` Treat error messages as part of the prompt — because that's what they are.

## 05 — How many tools is enough?

There is no magic number, but there are clear failure modes at both ends. Too few and the model has to compose primitives it doesn't understand; too many and it confuses sibling tools that sound alike. Most production agents live in what we'll call the <Term def="A metaphor borrowed from sports — the comfortable range where the model picks tools confidently and all descriptions fit in context without competing.">sweet spot</Term> — the bands below are what shows up in practice across small and mid-sized agents:

| Tool count | What you'll see |
| --- | --- |
| **5–15** | The sweet spot. The model picks confidently. Descriptions fit in context comfortably. |
| **15–30** | Still fine if names are tight and tools cluster by domain (`customers.*`, `promotions.*`, `orders.*`). |
| **30–50** | Starts to hurt. The model picks a near-sibling instead of the right tool, especially when two descriptions overlap. |
| **50+** | Stop adding. Switch strategy. |

**The unit is *intent*, not *endpoint*.** If your backend has 200 REST routes, you do not expose 200 tools. You expose the 10–20 things a user will actually ask the agent to do. A tool no real prompt ever triggers is just tokens you pay for on every turn — every turn — for the life of the agent.

When the system is genuinely big, three escape hatches in order of complexity:

- **Scope per session.** Bind only the tools relevant to the current task. A *support* agent gets 8 tools; a *billing* agent gets a different 8. Same codebase, different `.bindTools()` call at construction time.
- **Tool retrieval.** Index tool descriptions in a vector store, retrieve the top 10 per user message, bind those for that turn. Useful past ~50 tools where scoping isn't enough.
- **Sub-agents.** A router agent delegates to a `BillingAgent` whose internal tools are hidden from the parent. This is the LangGraph pattern in Part IV — each agent stays inside its own 5–15 sweet spot.

A few things worth noticing as you flip between strategies:

- **Bind all 50** sends every tool definition on every turn. The model has to discriminate between near-siblings and the prompt is heavy.
- **Scope per session** is a one-line change at construction (`.bindTools(billingTools)`). The model sees ten clean choices instead of fifty noisy ones. No new infrastructure.
- **Tool retrieval** scores every tool against the user prompt and binds the top ten. Note the *adjacent* domain (e.g. support tools when the task is billing) gets moderate scores too — that's the realistic case where retrieval recovers from a slightly off classifier.
- **Sub-agents** pays a small fixed cost (the router's five delegate tools) but the sub-agent then runs in its own 5–15 sweet spot. Use it when one task can span domains and you still want each domain to be tight.

**How to implement each one**

Working LangChain sketches for every strategy in the demo. The shared starting point is a tool catalog organised by domain.

```ts
// Shared setup — same for every strategy
import { ChatAnthropic } from '@langchain/anthropic'; // → ChatAnthropic class imported
import { tool } from '@langchain/core/tools';         // → tool() factory imported

const billingTools  = [refundsCreate, chargesCreate, /* ... 10 total */];      // → StructuredTool[10]
const supportTools  = [ticketsCreate, ticketsAssign,  /* ... 10 total */];     // → StructuredTool[10]
const scheduleTools = [calendarCreate, availabilityCheck, /* ... 10 total */]; // → StructuredTool[10]
const invTools      = [productsList, stockAdjust,      /* ... 10 total */];    // → StructuredTool[10]
const reportTools   = [reportDaily, exportsCsv,        /* ... 10 total */];    // → StructuredTool[10]
const ALL_TOOLS = [...billingTools, ...supportTools, ...scheduleTools, ...invTools, ...reportTools]; // → StructuredTool[50]

const baseModel = new ChatAnthropic({ model: 'claude-sonnet-4', temperature: 0 }); // → ChatAnthropic instance, no tools yet
```

*Bind all 50 — the baseline (don't do this past ~15 tools)*

```ts
const agent = baseModel.bindTools(ALL_TOOLS);    // → Runnable bound with all 50 tools

// Every user message now sends ~3,000 tokens of tool definitions before the
// prompt even starts. The model has to discriminate between near-siblings
// (e.g. support.tickets.create vs support.tickets.assign) on every turn.
await agent.invoke([new HumanMessage(userMessage)]); // → AIMessage; ~3,000 tool-def tokens billed every turn
```

*Scope per session — one extra line at construction*

```ts
const TOOLS_BY_DOMAIN = {                        // → Record<Domain, StructuredTool[]>
  billing:  billingTools,                        // → 10 tools
  support:  supportTools,                        // → 10 tools
  schedule: scheduleTools,                       // → 10 tools
  inv:      invTools,                            // → 10 tools
  report:   reportTools,                         // → 10 tools
} as const;
type Domain = keyof typeof TOOLS_BY_DOMAIN;      // → 'billing' | 'support' | 'schedule' | 'inv' | 'report'

function agentForSession(session: { domain: Domain }) {
  return baseModel.bindTools(TOOLS_BY_DOMAIN[session.domain]); // → Runnable bound with that domain's 10 tools
}
// → call agentForSession({ domain: 'billing' }) → agent that sees 10 tools, not 50
```

The crucial detail: **`domain` is *not* derived from the user's prompt.** Scope-per-session works precisely because something else — the session itself — already tells you which domain you're in. Four signals, in priority order:

```ts
// 1. Route / URL — the request hit /admin/billing/...
function pathToDomain(pathname: string): Domain {
  if (pathname.startsWith('/admin/billing'))  return 'billing';   // → 'billing'  for /admin/billing/...
  if (pathname.startsWith('/admin/support'))  return 'support';   // → 'support'  for /admin/support/...
  if (pathname.startsWith('/admin/schedule')) return 'schedule';  // → 'schedule' for /admin/schedule/...
  if (pathname.startsWith('/admin/inventory')) return 'inv';      // → 'inv'      for /admin/inventory/...
  if (pathname.startsWith('/admin/reports'))  return 'report';    // → 'report'   for /admin/reports/...
  throw new Error(`No domain mapping for ${pathname}`);           // → throws on an unknown route
}

// 2. User role — a "Billing Specialist" never sees support tools, even if they ask
const ROLE_TO_DOMAIN: Record<UserRole, Domain> = {  // → Record<UserRole, Domain> lookup table
  billing_specialist: 'billing',                    // → role → 'billing'
  support_agent:      'support',                    // → role → 'support'
  ops_manager:        'schedule',                   // → role → 'schedule'
  inventory_clerk:    'inv',                        // → role → 'inv'
  analyst:            'report',                     // → role → 'report'
};

// 3. Workspace / tenant config — a finance tenant binds finance tools
const domainFromTenant = (tenant: Tenant): Domain => tenant.assistantDomain; // → reads pre-configured Domain off the tenant row

// 4. Explicit UI affordance — user picks "What can I help with? [Billing | Support | …]"
const domainFromPicker = (selected: Domain): Domain => selected; // → identity — UI already gave us a Domain
```

Putting it together at the request boundary:

```ts
// app/api/chat/route.ts
export async function POST(req: NextRequest) {
  const { messages, tenantId } = await req.json();    // → { messages: ChatMessage[], tenantId: string }
  const user = await getCurrentUser(req);             // → User { role, ... }

  // Pick the domain from session signals. Try them in priority order.
  const domain: Domain =
    pathToDomain(req.nextUrl.pathname) ??             // → 'billing' if URL says so, else fall through
    ROLE_TO_DOMAIN[user.role]          ??             // → role-based fallback
    domainFromTenant(await getTenant(tenantId));      // → tenant-config fallback

  const agent = agentForSession({ domain });          // → Runnable with this domain's 10 tools
  return agent.stream(messages);  // → ReadableStream of AIMessage chunks; model sees 10 tools, not 50
}
```

No classifier call, no embedding lookup, no extra round-trip. The route handler already knows the domain by the time it constructs the agent.

**When scope-per-session is *not* the right strategy.** If the same chat box has to handle anything the user types — billing one minute, scheduling the next, no UI cue between them — then session signals can't tell you the domain. You've left the *scope* zone and entered the *retrieval* (read the prompt) or *sub-agents* (let a router LLM read the prompt) zone, both of which appear below.

**If you really do have to classify the prompt**

Sometimes there is no session signal — a single chat box, anonymous-ish user, no panel context. You can still scope-per-session, but you need a `prompt → domain` classifier in front of the agent factory. Three ways, picked by latency, accuracy, and cost.

*Approach A — keyword heuristic (fastest, brittle)*

Zero round-trips. A regex per domain, vote-counts decide. Good enough as a prototype or fallback.

```ts
const DOMAIN_KEYWORDS: Record<Domain, RegExp> = {  // → Record<Domain, RegExp> table
  billing:  /\b(refund|charge|invoice|subscription|payment|dispute)\b/i, // → matches billing nouns/verbs
  support:  /\b(ticket|bug|complaint|escalat|csat|customer|issue)\b/i,   // → matches support vocab
  schedule: /\b(meeting|calendar|availability|book|invite|time off)\b/i, // → matches calendar vocab
  inv:      /\b(product|stock|inventory|supplier|sku|warehouse)\b/i,     // → matches inventory vocab
  report:   /\b(report|chart|kpi|export|dashboard|trend)\b/i,            // → matches reporting vocab
};

function domainFromPrompt(message: string): Domain | null {
  const matches = (Object.entries(DOMAIN_KEYWORDS) as [Domain, RegExp][]) // → [['billing', /.../], ...]
    .map(([d, re]) => [d, (message.match(re) || []).length] as const)    // → [['billing', 2], ['support', 0], ...]
    .filter(([, n]) => n > 0)                                            // → keep domains with ≥1 hit
    .sort(([, a], [, b]) => b - a);                                      // → highest-count first
  return matches[0]?.[0] ?? null;                                        // → top domain or null
}

domainFromPrompt('Refund customer 7041\'s last charge of $48');
// → 'billing'  (matched: refund, charge)
```

*Approach B — cheap LLM classifier (one extra round-trip)*

A small fast model classifies the message before the main agent runs. ~50–150 ms, very accurate, easy to test.

```ts
import { ChatAnthropic } from '@langchain/anthropic'; // → ChatAnthropic class imported
import { z } from 'zod';                              // → Zod schema builder imported

// Use a cheap fast model — NOT the same model running the agent.
const classifier = new ChatAnthropic({ model: 'claude-haiku-4-5', temperature: 0 }) // → Haiku instance, deterministic
  .withStructuredOutput(                              // → Runnable that returns the typed object directly
    z.object({
      domain: z.enum(['billing', 'support', 'schedule', 'inv', 'report']), // → 5-way enum
      confidence: z.number().min(0).max(1),           // → 0..1 score
    }),
  );

async function domainFromPromptLLM(message: string): Promise<Domain> {
  const { domain } = await classifier.invoke([        // → { domain: Domain, confidence: number }
    {
      role: 'system',
      content: [
        'Classify the user task into one domain.',
        'billing  — refunds, charges, invoices, subscriptions, payments, disputes',
        'support  — tickets, customer issues, escalations, CSAT',
        'schedule — meetings, calendar, availability, time off',
        'inv      — products, stock, suppliers, purchase orders',
        'report   — analytics, dashboards, exports, KPIs',
      ].join('\n'),                                   // → one system string with the domain rubric
    },
    { role: 'user', content: message },               // → the actual user prompt
  ]);
  return domain;                                      // → 'billing' | 'support' | 'schedule' | 'inv' | 'report'
}

// Wired into the route handler:
export async function POST(req: NextRequest) {
  const { messages } = await req.json();              // → ChatMessage[] from the client
  const lastUser = messages.findLast((m) => m.role === 'user').content; // → newest user message text

  const domain = await domainFromPromptLLM(lastUser); // → 'billing' (≈80ms classifier round-trip)
  const agent = agentForSession({ domain });          // → Runnable with that domain's 10 tools
  return agent.stream(messages);                      // → streamed AIMessage chunks
}
```

Two round-trips per turn (classifier + main agent), but the main agent now sees only its 10 tools — usually a net token saving.

*Approach C — embedding similarity (no LLM call)*

Pre-compute one embedding per domain (a description of what it handles). Embed the user message at request time and pick the closest. ~30–80 ms, no per-turn LLM charge.

```ts
import { OpenAIEmbeddings } from '@langchain/openai'; // → OpenAIEmbeddings class imported

const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-3-small' }); // → embedder instance, 1536 dims

const DOMAIN_DESCRIPTIONS: Record<Domain, string> = {
  billing:  'Refunds, charges, invoices, subscriptions, payment methods, disputes.', // → corpus for billing centroid
  support:  'Support tickets, complaints, bug reports, escalations, CSAT.',          // → corpus for support centroid
  schedule: 'Meetings, calendar events, availability, time-off requests.',           // → corpus for schedule centroid
  inv:      'Products, stock, suppliers, purchase orders, SKU lookups.',             // → corpus for inv centroid
  report:   'Analytics reports, dashboards, KPI charts, CSV exports.',               // → corpus for report centroid
};

// Pre-computed once at boot.
const DOMAIN_VECTORS: Promise<Record<Domain, number[]>> = (async () => {
  const entries = Object.entries(DOMAIN_DESCRIPTIONS) as [Domain, string][]; // → [['billing', '...'], ...]
  const vectors = await embeddings.embedDocuments(entries.map(([, d]) => d)); // → number[][] (5 × 1536)
  return Object.fromEntries(entries.map(([d], i) => [d, vectors[i]])) as Record<Domain, number[]>; // → { billing: number[1536], ... }
})();

function cosine(a: number[], b: number[]): number {
  let dot = 0, na = 0, nb = 0;                       // → running sums
  for (let i = 0; i < a.length; i++) { dot += a[i] * b[i]; na += a[i] * a[i]; nb += b[i] * b[i]; } // → dot, ||a||², ||b||²
  return dot / (Math.sqrt(na) * Math.sqrt(nb));      // → cosine similarity in [-1, 1]
}

async function domainFromPromptEmbedding(message: string): Promise<Domain> {
  const [vec, vectors] = await Promise.all([embeddings.embedQuery(message), DOMAIN_VECTORS]); // → [number[1536], Record<Domain,number[]>]
  const ranked = (Object.entries(vectors) as [Domain, number[]][])
    .map(([d, v]) => [d, cosine(vec, v)] as const)   // → [['billing', 0.83], ['support', 0.41], ...]
    .sort(([, a], [, b]) => b - a);                  // → highest similarity first
  return ranked[0][0];                               // → top Domain (e.g. 'billing')
}
```

**Pick a classifier by latency, accuracy, cost**

| Approach | Latency | Accuracy | Cost | Use when |
| --- | --- | --- | --- | --- |
| Keyword heuristic | ~0 ms | Brittle (misses paraphrases) | Free | Prototype, or fallback when other paths fail |
| Cheap LLM classifier | ~50–150 ms | Very high | $/turn | Production default; you already have an LLM provider |
| Embedding similarity | ~30–80 ms | High | ≈1/30 of an LLM call | When you also need similarity for tool retrieval — reuse the store |

**The deeper point**

Once you're classifying the prompt to pick a toolset, you've effectively built a tiny *router*. At that point the **sub-agents** strategy below is the same idea written more cleanly: instead of a separate classifier call, the router *is* an LLM agent whose tools are `delegate.<domain>(task)`. One round-trip, one model, the routing logic lives in the system prompt and the delegate descriptions instead of in your TS code.

The migration path most teams walk:

```text
session signal       →   pure scope-per-session (no prompt classification)
↓ no session signal
keyword fallback     →   scope-per-session + regex classifier   (cheap to start)
↓ keywords miss too often
cheap LLM classifier →   scope-per-session + LLM classifier     (still classical)
↓ prompts span domains in one turn
sub-agents (router)  →   no separate classifier; one LLM picks the delegate
```

*Tool retrieval — vector index of tool descriptions, top-K per turn*

```ts
import { MemoryVectorStore } from 'langchain/vectorstores/memory'; // → in-process vector store class
import { OpenAIEmbeddings } from '@langchain/openai';              // → OpenAIEmbeddings class
import { Document } from '@langchain/core/documents';              // → Document wrapper

// Indexed once at boot. Re-index when tool descriptions change.
const vectorStore = await MemoryVectorStore.fromDocuments(  // → MemoryVectorStore with 50 indexed tool docs
  ALL_TOOLS.map((t) =>
    new Document({
      pageContent: `${t.name}: ${t.description}`,           // → "billing.refunds.create: Issue a refund ..."
      metadata: { name: t.name },                           // → { name: 'billing.refunds.create' }
    }),
  ),
  new OpenAIEmbeddings({ model: 'text-embedding-3-small' }), // → embedder used for indexing + queries
);

async function agentForMessage(userMessage: string) {
  const hits = await vectorStore.similaritySearch(userMessage, 10); // → Document[10] sorted by similarity
  const names = new Set(hits.map((h) => h.metadata.name as string)); // → Set of 10 tool names
  const subset = ALL_TOOLS.filter((t) => names.has(t.name));         // → StructuredTool[10]
  return baseModel.bindTools(subset);                                // → Runnable bound with the top-10 tools
}

// Per turn:
const agent = await agentForMessage(latestUserMessage); // → Runnable customised to this message
await agent.invoke(messages);                           // → AIMessage; only ~600 tokens of tool defs sent
```

The vector store is the new piece. `MemoryVectorStore` is fine for a few hundred tools; for thousands use Chroma, pgvector, or any production store — the call shape is identical.

*Sub-agents — a router with delegate tools*

```ts
import { createReactAgent } from '@langchain/langgraph/prebuilt'; // → prebuilt ReAct agent factory
import { z } from 'zod';                                          // → Zod schema builder

// Each sub-agent stays in its own 5–15 sweet spot.
const billingAgent  = createReactAgent({ llm: baseModel, tools: billingTools,  systemPrompt: 'You handle billing only.' });   // → CompiledStateGraph (10 tools)
const supportAgent  = createReactAgent({ llm: baseModel, tools: supportTools,  systemPrompt: 'You handle support only.' });   // → CompiledStateGraph (10 tools)
const scheduleAgent = createReactAgent({ llm: baseModel, tools: scheduleTools, systemPrompt: 'You handle scheduling only.' }); // → CompiledStateGraph (10 tools)
// ... one per domain

// One delegate tool per sub-agent. The router's whole catalog is these five.
const delegateBilling = tool(                          // → StructuredTool { name: 'router.delegate.billing' }
  async ({ task }) => {
    const result = await billingAgent.invoke({ messages: [new HumanMessage(task)] }); // → { messages: BaseMessage[] }
    return String(result.messages.at(-1)?.content ?? ''); // → final assistant message text
  },
  {
    name: 'router.delegate.billing',                   // → identifier the router uses
    description:
      'Delegate billing tasks — refunds, charges, invoices, subscriptions, payment methods, disputes — to the billing specialist.', // → router's only signal
    schema: z.object({ task: z.string() }),            // → router passes a single freeform task string
  },
);
// ... delegateSupport, delegateSchedule, delegateInv, delegateReport

const router = createReactAgent({                      // → CompiledStateGraph — the router itself
  llm: baseModel,
  tools: [delegateBilling, delegateSupport, delegateSchedule, delegateInv, delegateReport], // → 5 delegate tools, that's it
  systemPrompt: 'You are a router. Pick the right specialist and forward the task.',
});

// Top-level call:
await router.invoke({ messages: [new HumanMessage(userMessage)] }); // → final assistant message; sub-agent did the work
```

The delegate's `description` is what the router reads to decide. Spend real time on those five strings — they're the entire UX of the routing layer.

**Pick by total tool count, not by aesthetics**

| Tools | Use |
| --- | --- |
| 5–15 | Bind all. Simplest is best. |
| 15–30 | Scope per session if sessions naturally split by domain. |
| 30–80 | Tool retrieval. The vector index is one new dep and a small per-turn latency hit. |
| 80+ or genuine cross-domain tasks | Sub-agents. The router's delegate descriptions become the new UX. |

The deciding question for any tool you're tempted to add: *"would a fresh user prompt ever trigger this, and would the model pick it over its five nearest neighbours from the name and description alone?"* If no, rename, merge, or delete it.

## 06 — Common mistakes

A short, brutal checklist. If your tool matches any of these, refactor before you ship:

- **Tools that take 12 arguments.** The model will fill three correctly, two incorrectly, and forget seven. Split into smaller tools.
- **Tools that mutate without acknowledgment.** Every write tool should return what it wrote — the row, the ID, the timestamp. Returning `void` leaves the model with no way to confirm the call landed and no way to reference what it just did.
- **Tools that return free-form text instead of structured data.** *"I sent the email to ana@example.com at 14:32"* is a paragraph; the model has to re-parse it. Return `{ sentTo, sentAt }` and the model can quote the values directly.
- **Tools that call other tools recursively.** The model orchestrates tool sequences. Tools should not. A tool that internally calls three other tools is hiding work from the model and making the round-trip un-debuggable.
- **Tools that depend on global state.** *"This tool uses the customer set in the previous call."* No. Pass the customer ID explicitly every time. Implicit state across calls is how you build features that work in development and silently fail in production.

**Refactor pairs — what each anti-pattern looks like, and the fix**

*Silent writes — return what you wrote*

```ts
// ❌ Returns void; the model can't reference the row it just created
export const createDraft = tool(
  async ({ customerId, body }) => {
    await db.emailDrafts.create({ data: { customerId, body } }); // → row created (id discarded)
    return 'OK';                                                 // → 'OK' — model has no draftId to reference
  },
  { name: 'email_drafts.create', /* ... */ },
);

// ✅ Returns the row; the next turn can quote draftId, sentAt, etc.
export const createDraft = tool(
  async ({ customerId, body }) => {
    const draft = await db.emailDrafts.create({ data: { customerId, body } }); // → EmailDraft row
    return JSON.stringify({
      draftId: draft.id,                                         // → 'draft_01HX...'
      customerId: draft.customerId,                              // → 'cus_7041'
      createdAt: draft.createdAt.toISOString(),                  // → '2026-05-09T13:42:11.000Z'
      status: 'draft',                                           // → literal 'draft' so the model knows it didn't send
    });
    // → '{"draftId":"draft_01HX...","customerId":"cus_7041","createdAt":"2026-05-09T...","status":"draft"}'
  },
  { name: 'email_drafts.create', /* ... */ },
);
```

*Free-form returns — give the model JSON, not prose*

```ts
// ❌ The model has to parse English to find the values
return `Sent the email to ${email} at ${new Date().toISOString()}.`;
// → 'Sent the email to ana@example.com at 2026-05-09T13:42:11.000Z.'

// ✅ The model quotes structured fields directly
return JSON.stringify({ sentTo: email, sentAt: new Date().toISOString() });
// → '{"sentTo":"ana@example.com","sentAt":"2026-05-09T13:42:11.000Z"}'
```

*Recursive tools — surface the steps to the model instead*

```ts
// ❌ One tool hides three operations; the model can't reason about partial failure
export const createPromoAndDraftEmail = tool(
  async ({ discount, lapsedDays }) => {
    const promo = await createPromotion({ discount, lapsedDays });   // → Promotion
    const customers = await findLapsedCustomers({ lapsedDays });     // → Customer[] (could be 0 or 5,000)
    const drafts = await Promise.all(customers.map((c) => createDraft({ customerId: c.id, /* ... */ }))); // → Draft[] (any one can fail mid-flight)
    return JSON.stringify({ promo, draftCount: drafts.length });     // → '{"promo":{...},"draftCount":423}' — model can't see which customer failed
  },
  { /* ... */ },
);

// ✅ Three narrow tools the model orchestrates itself; each step is observable
export const createPromotion = tool(/* ... */);    // → StructuredTool 'promotions.create_for_lapsed_customers'
export const customersLapsed = tool(/* ... */);    // → StructuredTool 'customers.lapsed'
export const createDraft = tool(/* ... */);        // → StructuredTool 'email_drafts.create'
// → model picks them in sequence; each step's result lands in the transcript and is debuggable
```

*Implicit global state — pass the ID every time*

```ts
// ❌ Relies on whatever was set on the last call
let CURRENT_CUSTOMER: Customer | null = null;        // → mutable module-level state, shared across requests in serverless

export const setCustomer = tool(/* sets CURRENT_CUSTOMER */); // → StructuredTool — side-effects the global
export const draftEmail = tool(
  async ({ body }) => {
    if (!CURRENT_CUSTOMER) throw new Error('No customer selected'); // → throws when caller forgot setCustomer first
    return await createDraft({ customerId: CURRENT_CUSTOMER.id, body }); // → Draft for whichever customer happened to be set
  },
  /* ... */
);

// ✅ Pure: every call carries its own context
export const draftEmail = tool(
  async ({ customerId, body }) => createDraft({ customerId, body }), // → Draft, no globals read
  {
    schema: z.object({
      customerId: z.string().uuid(),                 // → UUID enforced; can't reach this with a bad id
      body: z.string().min(20),                      // → at least 20 chars of body
    }),
    /* ... */
  },
);
// → call is reproducible from args alone; works the same in dev, prod, and serverless cold starts
```

A useful exercise after writing a tool: read it back and ask *"could a model that has never seen my code call this correctly using only the name, description, and schema?"* If the answer is no, you have more work to do.

<Recap>
- Narrow tools beat broad tools. The name should describe the situation, not just the verb.
- Idempotent by default. The model will retry, often invisibly. Build for that.
- Tool descriptions are part of the prompt. Spend real time on them; cross-reference siblings.
- Typed errors are recovery paths. *What failed. Why. What to do instead.* Three sentences, big leverage.
- How many tools? 5–15 is the sweet spot, 30+ starts to hurt, 50+ needs scoping, retrieval, or sub-agents. Count intents, not endpoints.
- Five anti-patterns to avoid: 12-arg tools, silent writes, free-form returns, recursive tools, implicit global state.
</Recap>

Thiết kế tool là thiết kế API có thêm một ràng buộc — caller của bạn là một interpreter mang tính xác suất, không phải tất định. Mọi API khác bạn từng ship đều được gọi bởi code làm đúng những gì tác giả của nó bảo nó làm. Cái này được gọi bởi một model đã đọc description của bạn, hình thành một phỏng đoán về việc tool đó làm gì, và bây giờ đang thử. Description mơ hồ thì phỏng đoán mơ hồ. Đặt tên "khôn quá" thì phỏng đoán sai. Thiết kế tool tưởng thưởng cho cái nhạt, cụ thể, và hiển nhiên — theo một cách mà hầu hết các thiết kế phần mềm khác không có.

Bài này là file nguyên tắc. Bốn quy tắc cần theo và năm sai lầm cần nhận diện. Không có quy tắc nào mang tính cách mạng — đó là những điều các API designer kỳ cựu vẫn rao giảng từ xưa — nhưng cái giá khi làm sai cao hơn nhiều khi caller không đọc được source code của bạn và cũng không hỏi lại để làm rõ.

## 01 — Hẹp thắng rộng

Bản năng từ thiết kế thư viện cho người là xây một tool tổng quát và để caller tự ráp. *"`promotions.create` với mười sáu field tuỳ chọn là bao trùm hết."* Với một LLM thì đây là nước cờ sai. Model phải tự khám phá, từ description và schema của bạn, rằng một tổ hợp cụ thể của các field có nghĩa là *"mã giảm 15% nhắm vào khách hàng không đặt hàng trong 60 ngày"*. Đôi khi nó đoán đúng. Đôi khi không.

`promotions.create_for_lapsed_customers` là một tool kém đẹp hơn theo tiêu chí thẩm mỹ code. Sáu tool thay vì một, mỗi cái là một lớp wrapper mỏng trên cùng một primitive. Nhưng nó là tool *tốt hơn nhiều* cho LLM, vì *bản thân cái tên đã nói cho model biết tool đó làm gì và khi nào dùng nó*. Model chọn đúng tool bằng cách đọc danh bạ; bạn không cần dạy nó các luật ráp nối.

Một quy tắc thực dụng: nếu bạn thấy mình đang viết *"để làm X, set field A thành Y và field B thành Z"* trong description của một tool, đó là hai tool, không phải một. Tách ra.

**Phiên bản rộng (cám dỗ)**

```ts
export const createPromotion = tool(             // → StructuredTool { name: 'promotions.create' }
  async (args) => { /* ... */ },                 // → handler receives 16-field object, must branch on `type`+`audience`
  {
    name: 'promotions.create',                   // → one identifier covering 4 promo types × 5 audiences
    description: 'Create a promotion. See schema for all options.', // → vague — model must guess valid combos
    schema: z.object({
      type: z.enum(['percent', 'fixed', 'bogo', 'free_shipping']), // → 4 disjoint promo kinds
      value: z.number().optional(),              // → meaning depends on `type` — model has to know that
      audience: z.enum(['all', 'lapsed', 'vip', 'new', 'tag']).optional(), // → 5 segments
      audienceTag: z.string().optional(),        // → required when audience='tag', else ignored
      lapsedDays: z.number().optional(),         // → required when audience='lapsed'
      minOrderCents: z.number().optional(),      // → optional gate
      validFrom: z.string().datetime().optional(), // → ISO 8601
      validUntil: z.string().datetime().optional(),// → ISO 8601
      code: z.string().optional(),               // → the promo code string
      maxUses: z.number().optional(),            // → optional cap
      stackable: z.boolean().optional(),         // → can it combine with other promos
      // ...four more fields
    }),
  },
);
// → one tool the model picks for any promotion — high failure rate on 60s-lapsed-15%-off prompts
```

Khi người dùng yêu cầu *"giảm 15% cho khách hàng không đặt hàng trong 60 ngày"*, model phải suy ra một shape mười sáu field: `type: 'percent'`, `value: 15`, `audience: 'lapsed'`, `audienceTag: undefined`, `lapsedDays: 60`, cộng thêm một code mặc định. Nó sẽ đúng phần lớn lúc và sai một số lúc, và các fail trông như *"audience là 'tag' nhưng audienceTag bị thiếu"* — lỗi ngữ nghĩa âm thầm, không phải lỗi validation.

**Phiên bản hẹp (nước cờ đúng)**

```ts
export const createLapsedCustomerPromo = tool(   // → StructuredTool — name encodes the use case
  async ({ discountPercent, lapsedDays, code }) => { /* ... */ }, // → 3 required fields, all bounded
  {
    name: 'promotions.create_for_lapsed_customers', // → model picks this by reading the directory
    description: 'Create a percentage-off promo code targeted at customers who have not ordered in N days.', // → one job, one sentence
    schema: z.object({
      discountPercent: z.number().int().min(1).max(75), // → integer 1..75 (no 0%, no 90%)
      lapsedDays: z.number().int().min(7).max(365),     // → integer 7..365
      code: z.string().regex(/^[A-Z0-9]{4,16}$/),       // → uppercase alnum, 4-16 chars
    }),
  },
);

export const createVipThankYouPromo = tool(/* ... */);   // → StructuredTool 'promotions.create_vip_thank_you'
export const createFlashSale = tool(/* ... */);          // → StructuredTool 'promotions.create_flash_sale'
export const createFreeShippingPromo = tool(/* ... */);  // → StructuredTool 'promotions.create_free_shipping'
// → 4 narrow tools the model picks by name; failure mode is "no tool matched" (catchable), not "wrong combo" (silent)
```

Ba field, đều bắt buộc, đều có chặn. Cái tên *chính là* use case — model chọn `promotions.create_for_lapsed_customers` từ danh bạ y hệt cách bạn chọn một hàm từ autocomplete. Sáu tool thay vì một, nhưng mỗi cái hai mươi dòng, mỗi cái test được độc lập, và model không bao giờ phải ráp một cuộc gọi mười sáu field từ đầu.

## 02 — Mặc định idempotent

Model sẽ retry. Có khi vì lần gọi đầu time-out, có khi vì nó parse error response của bạn và kết luận rằng cuộc gọi chưa thành công, có khi vì lý do không logfile nào giải thích nổi. Nếu tool của bạn là `orders.charge` và một lần retry tính phí hai lần, bạn có sẵn một sự cố thanh toán đang chờ đợi.

Cách sửa giống hệt cách bạn sửa một webhook handler: một idempotency key được cung cấp (hoặc tự sinh) cho mỗi thao tác logic, và mẫu `check-and-insert` trong database. Lần gọi đầu chạy thật. Lần gọi thứ hai thấy có sẵn row và trả về kết quả đã cache. Model nhận được cùng câu trả lời cho cả hai lần và không biết — cũng không cần biết — đó là một lần retry.

Với tool đọc thì việc này gần như free, vì đọc tự nhiên đã idempotent. Kỷ luật này quan trọng nhất với các tool ghi mà chúng ta sẽ gặp ở bài 12, nhưng hình thành nó từ bây giờ nghĩa là bạn không phải gắn ngược lại sau này.

Một phác hoạ cụ thể về sự khác biệt (xem đầy đủ ở bài 13):

```ts
// ❌ Not idempotent — every retry charges again
async function chargeOrder({ orderId, amountCents }) {
  return await stripe.charges.create({ amount: amountCents, ... }); // → new Charge each call (retry = double-charge)
}

// ✅ Idempotent — second call returns the first call's result
async function chargeOrder({ orderId, amountCents, requestKey }) {
  const existing = await db.charges.findUnique({ where: { requestKey } }); // → Charge | null
  if (existing) return existing;                 // → fast-path: retry hits here, returns first result
  const charge = await stripe.charges.create({   // → Stripe Charge (only on first call)
    amount: amountCents,
    idempotency_key: requestKey,                 // → Stripe also dedups server-side via this header
  });
  return await db.charges.create({ data: { ...charge, requestKey } }); // → persisted Charge row
}
// → second call with same requestKey returns the first Charge; customer charged exactly once
```

Cùng chữ ký cộng thêm một tham số `requestKey`; khách hàng không còn nguy cơ bị tính phí hai lần khi model retry.

**`requestKey` đến từ đâu?**

Nó *không phải* là một UUID mới mà model tự sinh mỗi lần — như thế thì hỏng cả mẫu, vì lần retry sẽ tạo ra key khác và miss luôn dedup. Có ba chỗ nó có thể đến từ, theo thứ tự bạn sẽ dùng tới:

- **Orchestrator (tầng runtime nằm trên tool).** Một wrapper nhỏ nhìn vào `(toolName, args, turnId)` và sinh ra một key ổn định — thường là một UUID đúc một lần cho mỗi thao tác logic và dùng lại qua mọi lần retry. Đây là mặc định; thân tool chỉ nhận `requestKey` như một tham số.
- **Tool tự suy ra từ input.** Khi các args tự nhiên đã định danh thao tác (`charge:${orderId}:${amountCents}`), tool có thể hash chúng thành một key. Rẻ và đủ tốt cho hầu hết các write tool.
- **Model, thông qua schema.** Schema khai báo `requestKey: z.string().uuid()` để orchestrator có thể điền vào trước khi gọi. Model *thấy* field nhưng không chọn giá trị của nó — wrapper ghi key vào args trước khi tool chạy.

Mẫu đầy đủ, bao gồm biến thể an toàn với race-condition `INSERT … ON CONFLICT`, là chủ đề của bài 13. Tạm thời, mọi tool ghi bạn thiết kế nên có một slot `requestKey`; coi nó như chiếc dây an toàn để retry không phải trả giá gì cả.

## 03 — Description CHÍNH LÀ prompt

Thứ có đòn bẩy cao nhất bạn sẽ viết trong codebase AI của mình là chuỗi description trên mỗi tool. Nó được gửi cho model ở *mọi* turn. Nó là cái mà model dùng để quyết định *có* gọi tool hay không, *khi nào* gọi, và *với arguments nào*. Description mơ hồ bảo đảm cho một model bối rối.

Phiên bản tệ: `"Search customers."` Năm ký tự tín hiệu. Model phải đoán field nào tìm được, trả về cái gì, giới hạn ra sao.

Phiên bản tốt: `"Search customers by lapsed-days threshold, tag membership, or email domain. Returns up to 25 rows ordered by last_order_at descending. Use customers.list for unfiltered listing."` Ba câu, ba thứ giờ model biết mà không phải đoán. Để ý dòng tham chiếu chéo ở cuối — chỉ vào một tool anh em giúp model chọn đúng cái.

**Cùng một tool, ba phiên bản description**

```ts
// ❌ Vague — the model guesses what's searchable and what comes back
description: 'Search customers.';                // → 17 chars; model invents fields and shape

// ⚠ Better, but missing the boundary
description: 'Search customers by tag, email, or lapsed days.'; // → fields named, but no return shape, no sibling hint

// ✅ Tells the model what it filters on, what it returns,
//    how many, and which sibling to use when this isn't right
description: [
  'Search customers by lapsed-days threshold, tag membership, or email domain.', // → what it filters on
  'Returns up to 25 rows ordered by last_order_at descending.',                  // → what it returns + cap
  'Use customers.list for unfiltered listing or customers.byId for a single lookup.', // → sibling pointers
].join(' ');                                     // → "Search customers by ... 25 rows ... Use customers.list ..."
```

Ví dụ thứ hai cho một write tool, nơi mức độ rủi ro cao hơn và description đóng thêm vai trò lan can bảo vệ:

```ts
// ❌ Tells the model nothing about side effects
description: 'Send an email.';                   // → 14 chars; model assumes it actually sends — dangerous

// ✅ States the side effect, the safety net, and the boundary
description: [
  'Draft an email to a single customer and save it as a DRAFT.',  // → what it does (saves a draft)
  'Does NOT send — a human reviews drafts before they go out.',   // → safety net (no surprise sends)
  'For bulk drafts use email_drafts.create_for_segment.',         // → sibling pointer for the bulk case
].join(' ');                                     // → "Draft an email ... Does NOT send ... For bulk ..."
```

Mẫu chung cho mọi description tốt: *nó làm gì · nó trả về gì / side effect là gì · chọn tool anh em nào nếu cái này không đúng*. Ba câu, đòn bẩy lớn.

<Callout type="tip" title="Đọc description của bạn ra tiếng">
Cách test nhanh nhất cho description của tool: đọc to lên như đang brief cho đồng nghiệp. Nếu một câu khiến bạn vấp, model cũng sẽ vấp. Nếu bạn dùng jargon mà bình thường bạn vẫn phải giải thích, model không có được lời giải thích đó. Nếu bạn viết "etc." hay "vân vân", model không biết bạn vừa giấu đi cái gì. Description của tool là technical writing, và cùng những bản năng làm nên một README tốt sẽ làm nên một description tốt.
</Callout>

## 04 — Error có kiểu rõ ràng

Khi có chuyện gì đó hỏng bên trong một tool, model nhận error message bị throw ra nguyên văn dưới dạng một `ToolMessage` và quyết định làm gì tiếp theo. Quyết định đó sẽ bị ảnh hưởng bởi nội dung error của bạn.

`Error: ENOENT` không nói cho model điều gì hữu ích. Nó sẽ retry, hoặc bỏ cuộc, hoặc bịa ra rằng khách hàng không tồn tại.

`Error: Customer with id 7041 not found. Use customers.search to look up customers by name or email.` nói cho model biết chính xác chuyện gì đã xảy ra và chính xác phải làm gì thay vào đó. Thường thì model sẽ *làm theo gợi ý trong error message của bạn* và sửa cuộc gọi ở turn tiếp theo — nghĩa là một error được viết tốt trở thành một con đường tự phục hồi mà bạn không cần viết một dòng scaffolding nào.

Shape hoạt động: `<chuyện gì hỏng>. <vì sao>. <nên làm gì tiếp theo>.` Coi error message là một phần của prompt — bởi vì đó chính là cái nó là.

## 05 — Bao nhiêu tool là đủ?

Không có con số ma thuật nào, nhưng có các failure mode rõ ràng ở cả hai đầu. Quá ít thì model phải tự ráp các primitive nó không hiểu; quá nhiều thì nó lẫn lộn các tool anh em nghe na ná. Hầu hết agent production sống trong cái mà chúng ta sẽ gọi là <Term def="Một ẩn dụ mượn từ thể thao — vùng dễ chịu nơi model chọn tool tự tin và mọi description nằm gọn trong context mà không tranh giành nhau.">vùng ngọt</Term> — các dải dưới đây là cái thực tế xảy ra với các agent nhỏ và vừa:

| Số lượng tool | Bạn sẽ thấy gì |
| --- | --- |
| **5–15** | Vùng ngọt. Model chọn tự tin. Description nằm gọn trong context. |
| **15–30** | Vẫn ổn nếu tên gọn ghẽ và tool gom cụm theo domain (`customers.*`, `promotions.*`, `orders.*`). |
| **30–50** | Bắt đầu đau. Model chọn một anh em gần kề thay vì tool đúng, đặc biệt khi hai description có giao thoa. |
| **50+** | Dừng việc thêm. Đổi chiến lược. |

**Đơn vị là *ý định*, không phải *endpoint*.** Nếu backend của bạn có 200 REST route, bạn không expose 200 tool. Bạn expose 10–20 thứ mà người dùng thực sự sẽ yêu cầu agent làm. Một tool không prompt thực tế nào kích hoạt nó chỉ là token bạn trả tiền ở mọi turn — mọi turn — trong suốt vòng đời của agent.

Khi hệ thống thật sự lớn, ba lối thoát theo thứ tự độ phức tạp:

- **Scope per session.** Bind chỉ những tool liên quan đến tác vụ hiện tại. Agent *support* có 8 tool; agent *billing* có 8 tool khác. Cùng codebase, gọi `.bindTools()` khác nhau lúc khởi tạo.
- **Tool retrieval.** Index các tool description trong một vector store, retrieve top 10 cho mỗi message người dùng, bind đúng số đó cho turn đó. Hữu ích khi vượt ~50 tool và scoping không đủ.
- **Sub-agents.** Một router agent uỷ thác cho một `BillingAgent` mà các tool nội bộ của nó bị giấu khỏi parent. Đây là mẫu LangGraph ở Phần IV — mỗi agent ở yên trong vùng ngọt 5–15 của riêng nó.

Chọn một chiến lược, chọn một prompt, và xem những tool nào thực sự được bind cho turn đó. Baseline bind-all cho thấy cái *không* nên làm.

Một vài điều đáng để ý khi bạn lật qua lại giữa các chiến lược:

- **Bind tất cả 50** gửi mọi định nghĩa tool ở mọi turn. Model phải phân biệt giữa các anh em gần kề và prompt rất nặng.
- **Scope per session** là một thay đổi một dòng lúc khởi tạo (`.bindTools(billingTools)`). Model thấy mười lựa chọn sạch sẽ thay vì năm mươi cái nhiễu. Không có hạ tầng mới.
- **Tool retrieval** chấm điểm mọi tool so với prompt của người dùng và bind top 10. Để ý domain *liền kề* (ví dụ tool support khi task là billing) cũng được điểm trung bình — đó là tình huống thực tế nơi retrieval cứu lại một classifier hơi lệch.
- **Sub-agents** trả một chi phí cố định nhỏ (năm delegate tool của router) nhưng sub-agent sau đó chạy trong vùng ngọt 5–15 của riêng nó. Dùng khi một task có thể trải nhiều domain mà bạn vẫn muốn mỗi domain gọn gàng.

**Triển khai từng cái thế nào**

Bản phác LangChain chạy được cho mọi chiến lược trong demo. Điểm xuất phát chung là một catalog tool tổ chức theo domain.

```ts
// Shared setup — same for every strategy
import { ChatAnthropic } from '@langchain/anthropic'; // → ChatAnthropic class imported
import { tool } from '@langchain/core/tools';         // → tool() factory imported

const billingTools  = [refundsCreate, chargesCreate, /* ... 10 total */];      // → StructuredTool[10]
const supportTools  = [ticketsCreate, ticketsAssign,  /* ... 10 total */];     // → StructuredTool[10]
const scheduleTools = [calendarCreate, availabilityCheck, /* ... 10 total */]; // → StructuredTool[10]
const invTools      = [productsList, stockAdjust,      /* ... 10 total */];    // → StructuredTool[10]
const reportTools   = [reportDaily, exportsCsv,        /* ... 10 total */];    // → StructuredTool[10]
const ALL_TOOLS = [...billingTools, ...supportTools, ...scheduleTools, ...invTools, ...reportTools]; // → StructuredTool[50]

const baseModel = new ChatAnthropic({ model: 'claude-sonnet-4', temperature: 0 }); // → ChatAnthropic instance, no tools yet
```

*Bind cả 50 — baseline (đừng làm thế khi vượt ~15 tool)*

const agent = baseModel.bindTools(ALL_TOOLS);    // → Runnable bound with all 50 tools

// Every user message now sends ~3,000 tokens of tool definitions before the
// prompt even starts. The model has to discriminate between near-siblings

await agent.invoke([new HumanMessage(userMessage)]); // → AIMessage; ~3,000 tool-def tokens billed every turn

*Scope per session — một dòng thêm vào lúc khởi tạo*

```ts
const TOOLS_BY_DOMAIN = {                        // → Record<Domain, StructuredTool[]>
  billing:  billingTools,                        // → 10 tools
  support:  supportTools,                        // → 10 tools
  schedule: scheduleTools,                       // → 10 tools
  inv:      invTools,                            // → 10 tools
  report:   reportTools,                         // → 10 tools
} as const;
type Domain = keyof typeof TOOLS_BY_DOMAIN;      // → 'billing' | 'support' | 'schedule' | 'inv' | 'report'

function agentForSession(session: { domain: Domain }) {
  return baseModel.bindTools(TOOLS_BY_DOMAIN[session.domain]); // → Runnable bound with that domain's 10 tools
}
// → call agentForSession({ domain: 'billing' }) → agent that sees 10 tools, not 50
```

Chi tiết quan trọng: **`domain` *không* được suy ra từ prompt của người dùng.** Scope-per-session hoạt động chính vì một thứ khác — bản thân session — đã nói cho bạn biết bạn đang ở domain nào. Bốn tín hiệu, theo thứ tự ưu tiên:

```ts
// 1. Route / URL — the request hit /admin/billing/...
function pathToDomain(pathname: string): Domain {
  if (pathname.startsWith('/admin/billing'))  return 'billing';   // → 'billing'  for /admin/billing/...
  if (pathname.startsWith('/admin/support'))  return 'support';   // → 'support'  for /admin/support/...
  if (pathname.startsWith('/admin/schedule')) return 'schedule';  // → 'schedule' for /admin/schedule/...
  if (pathname.startsWith('/admin/inventory')) return 'inv';      // → 'inv'      for /admin/inventory/...
  if (pathname.startsWith('/admin/reports'))  return 'report';    // → 'report'   for /admin/reports/...
  throw new Error(`No domain mapping for ${pathname}`);           // → throws on an unknown route
}

// 2. User role — a "Billing Specialist" never sees support tools, even if they ask
const ROLE_TO_DOMAIN: Record<UserRole, Domain> = {  // → Record<UserRole, Domain> lookup table
  billing_specialist: 'billing',                    // → role → 'billing'
  support_agent:      'support',                    // → role → 'support'
  ops_manager:        'schedule',                   // → role → 'schedule'
  inventory_clerk:    'inv',                        // → role → 'inv'
  analyst:            'report',                     // → role → 'report'
};

// 3. Workspace / tenant config — a finance tenant binds finance tools
const domainFromTenant = (tenant: Tenant): Domain => tenant.assistantDomain; // → reads pre-configured Domain off the tenant row

// 4. Explicit UI affordance — user picks "What can I help with? [Billing | Support | …]"
const domainFromPicker = (selected: Domain): Domain => selected; // → identity — UI already gave us a Domain
```

Ráp lại ở biên request:

```ts
// app/api/chat/route.ts
export async function POST(req: NextRequest) {
  const { messages, tenantId } = await req.json();    // → { messages: ChatMessage[], tenantId: string }
  const user = await getCurrentUser(req);             // → User { role, ... }

  // Pick the domain from session signals. Try them in priority order.
  const domain: Domain =
    pathToDomain(req.nextUrl.pathname) ??             // → 'billing' if URL says so, else fall through
    ROLE_TO_DOMAIN[user.role]          ??             // → role-based fallback
    domainFromTenant(await getTenant(tenantId));      // → tenant-config fallback

  const agent = agentForSession({ domain });          // → Runnable with this domain's 10 tools
  return agent.stream(messages);  // → ReadableStream of AIMessage chunks; model sees 10 tools, not 50
}
```

Không có classifier call, không embedding lookup, không round-trip phụ. Route handler đã biết domain ngay vào thời điểm nó dựng agent.

**Khi nào scope-per-session *không* phải là chiến lược đúng.** Nếu cùng một ô chat phải xử lý bất cứ thứ gì người dùng gõ vào — phút này billing, phút sau scheduling, không có gợi ý UI nào giữa chúng — thì các tín hiệu session không nói cho bạn biết domain được. Bạn đã rời khỏi vùng *scope* và bước vào vùng *retrieval* (đọc prompt) hoặc *sub-agents* (cho một router LLM đọc prompt), cả hai sẽ xuất hiện bên dưới.

**Nếu bạn thực sự buộc phải phân loại prompt**

Đôi khi không có tín hiệu session — một ô chat duy nhất, người dùng kiểu ẩn danh, không có ngữ cảnh panel. Bạn vẫn có thể scope-per-session, nhưng cần một classifier `prompt → domain` đứng trước agent factory. Ba cách, chọn theo độ trễ, độ chính xác, và chi phí.

*Cách A — heuristic theo keyword (nhanh nhất, dễ vỡ)*

Không round-trip. Một regex cho mỗi domain, đếm vote rồi quyết. Đủ tốt làm prototype hoặc fallback.

```ts
const DOMAIN_KEYWORDS: Record<Domain, RegExp> = {  // → Record<Domain, RegExp> table
  billing:  /\b(refund|charge|invoice|subscription|payment|dispute)\b/i, // → matches billing nouns/verbs
  support:  /\b(ticket|bug|complaint|escalat|csat|customer|issue)\b/i,   // → matches support vocab
  schedule: /\b(meeting|calendar|availability|book|invite|time off)\b/i, // → matches calendar vocab
  inv:      /\b(product|stock|inventory|supplier|sku|warehouse)\b/i,     // → matches inventory vocab
  report:   /\b(report|chart|kpi|export|dashboard|trend)\b/i,            // → matches reporting vocab
};

function domainFromPrompt(message: string): Domain | null {
  const matches = (Object.entries(DOMAIN_KEYWORDS) as [Domain, RegExp][]) // → [['billing', /.../], ...]
    .map(([d, re]) => [d, (message.match(re) || []).length] as const)    // → [['billing', 2], ['support', 0], ...]
    .filter(([, n]) => n > 0)                                            // → keep domains with ≥1 hit
    .sort(([, a], [, b]) => b - a);                                      // → highest-count first
  return matches[0]?.[0] ?? null;                                        // → top domain or null
}

domainFromPrompt('Refund customer 7041\'s last charge of $48');
// → 'billing'  (matched: refund, charge)
```

*Cách B — LLM classifier giá rẻ (thêm một round-trip)*

Một model nhỏ và nhanh phân loại message trước khi agent chính chạy. ~50–150 ms, rất chính xác, dễ test.

```ts
import { ChatAnthropic } from '@langchain/anthropic'; // → ChatAnthropic class imported
import { z } from 'zod';                              // → Zod schema builder imported

// Use a cheap fast model — NOT the same model running the agent.
const classifier = new ChatAnthropic({ model: 'claude-haiku-4-5', temperature: 0 }) // → Haiku instance, deterministic
  .withStructuredOutput(                              // → Runnable that returns the typed object directly
    z.object({
      domain: z.enum(['billing', 'support', 'schedule', 'inv', 'report']), // → 5-way enum
      confidence: z.number().min(0).max(1),           // → 0..1 score
    }),
  );

async function domainFromPromptLLM(message: string): Promise<Domain> {
  const { domain } = await classifier.invoke([        // → { domain: Domain, confidence: number }
    {
      role: 'system',
      content: [
        'Classify the user task into one domain.',
        'billing  — refunds, charges, invoices, subscriptions, payments, disputes',
        'support  — tickets, customer issues, escalations, CSAT',
        'schedule — meetings, calendar, availability, time off',
        'inv      — products, stock, suppliers, purchase orders',
        'report   — analytics, dashboards, exports, KPIs',
      ].join('\n'),                                   // → one system string with the domain rubric
    },
    { role: 'user', content: message },               // → the actual user prompt
  ]);
  return domain;                                      // → 'billing' | 'support' | 'schedule' | 'inv' | 'report'
}

// Wired into the route handler:
export async function POST(req: NextRequest) {
  const { messages } = await req.json();              // → ChatMessage[] from the client
  const lastUser = messages.findLast((m) => m.role === 'user').content; // → newest user message text

  const domain = await domainFromPromptLLM(lastUser); // → 'billing' (≈80ms classifier round-trip)
  const agent = agentForSession({ domain });          // → Runnable with that domain's 10 tools
  return agent.stream(messages);                      // → streamed AIMessage chunks
}
```

Hai round-trip mỗi turn (classifier + agent chính), nhưng agent chính giờ chỉ thấy 10 tool của nó — thường là tiết kiệm token thuần.

*Cách C — embedding similarity (không cần LLM call)*

Tính sẵn một embedding cho mỗi domain (mô tả nó xử lý gì). Embed message của người dùng lúc request và chọn cái gần nhất. ~30–80 ms, không tốn phí LLM mỗi turn.

```ts
import { OpenAIEmbeddings } from '@langchain/openai'; // → OpenAIEmbeddings class imported

const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-3-small' }); // → embedder instance, 1536 dims

const DOMAIN_DESCRIPTIONS: Record<Domain, string> = {
  billing:  'Refunds, charges, invoices, subscriptions, payment methods, disputes.', // → corpus for billing centroid
  support:  'Support tickets, complaints, bug reports, escalations, CSAT.',          // → corpus for support centroid
  schedule: 'Meetings, calendar events, availability, time-off requests.',           // → corpus for schedule centroid
  inv:      'Products, stock, suppliers, purchase orders, SKU lookups.',             // → corpus for inv centroid
  report:   'Analytics reports, dashboards, KPI charts, CSV exports.',               // → corpus for report centroid
};

// Pre-computed once at boot.
const DOMAIN_VECTORS: Promise<Record<Domain, number[]>> = (async () => {
  const entries = Object.entries(DOMAIN_DESCRIPTIONS) as [Domain, string][]; // → [['billing', '...'], ...]
  const vectors = await embeddings.embedDocuments(entries.map(([, d]) => d)); // → number[][] (5 × 1536)
  return Object.fromEntries(entries.map(([d], i) => [d, vectors[i]])) as Record<Domain, number[]>; // → { billing: number[1536], ... }
})();

function cosine(a: number[], b: number[]): number {
  let dot = 0, na = 0, nb = 0;                       // → running sums
  for (let i = 0; i < a.length; i++) { dot += a[i] * b[i]; na += a[i] * a[i]; nb += b[i] * b[i]; } // → dot, ||a||², ||b||²
  return dot / (Math.sqrt(na) * Math.sqrt(nb));      // → cosine similarity in [-1, 1]
}

async function domainFromPromptEmbedding(message: string): Promise<Domain> {
  const [vec, vectors] = await Promise.all([embeddings.embedQuery(message), DOMAIN_VECTORS]); // → [number[1536], Record<Domain,number[]>]
  const ranked = (Object.entries(vectors) as [Domain, number[]][])
    .map(([d, v]) => [d, cosine(vec, v)] as const)   // → [['billing', 0.83], ['support', 0.41], ...]
    .sort(([, a], [, b]) => b - a);                  // → highest similarity first
  return ranked[0][0];                               // → top Domain (e.g. 'billing')
}
```

**Chọn classifier theo độ trễ, độ chính xác, chi phí**

| Cách tiếp cận | Độ trễ | Độ chính xác | Chi phí | Dùng khi |
| --- | --- | --- | --- | --- |
| Heuristic keyword | ~0 ms | Dễ vỡ (bỏ sót paraphrase) | Miễn phí | Prototype, hoặc fallback khi các đường khác fail |
| LLM classifier rẻ | ~50–150 ms | Rất cao | $/turn | Mặc định cho production; bạn vốn đã có nhà cung cấp LLM |
| Embedding similarity | ~30–80 ms | Cao | ≈1/30 chi phí LLM call | Khi bạn cũng cần similarity cho tool retrieval — tái sử dụng store |

**Ý quan trọng hơn**

Một khi bạn đang phân loại prompt để chọn toolset, bạn đã thực ra đang xây một *router* nhỏ. Đến điểm đó, chiến lược **sub-agents** bên dưới chính là cùng ý tưởng viết cho gọn gàng hơn: thay vì một classifier call riêng, router *chính là* một LLM agent với các tool là `delegate.<domain>(task)`. Một round-trip, một model, logic routing nằm trong system prompt và các description của delegate thay vì trong code TS của bạn.

Lộ trình di trú mà hầu hết team đi qua:

```text
session signal       →   pure scope-per-session (no prompt classification)
↓ no session signal
keyword fallback     →   scope-per-session + regex classifier   (cheap to start)
↓ keywords miss too often
cheap LLM classifier →   scope-per-session + LLM classifier     (still classical)
↓ prompts span domains in one turn
sub-agents (router)  →   no separate classifier; one LLM picks the delegate
```

*Tool retrieval — vector index các description, top-K mỗi turn*

```ts
import { MemoryVectorStore } from 'langchain/vectorstores/memory'; // → in-process vector store class
import { OpenAIEmbeddings } from '@langchain/openai';              // → OpenAIEmbeddings class
import { Document } from '@langchain/core/documents';              // → Document wrapper

// Indexed once at boot. Re-index when tool descriptions change.
const vectorStore = await MemoryVectorStore.fromDocuments(  // → MemoryVectorStore with 50 indexed tool docs
  ALL_TOOLS.map((t) =>
    new Document({
      pageContent: `${t.name}: ${t.description}`,           // → "billing.refunds.create: Issue a refund ..."
      metadata: { name: t.name },                           // → { name: 'billing.refunds.create' }
    }),
  ),
  new OpenAIEmbeddings({ model: 'text-embedding-3-small' }), // → embedder used for indexing + queries
);

async function agentForMessage(userMessage: string) {
  const hits = await vectorStore.similaritySearch(userMessage, 10); // → Document[10] sorted by similarity
  const names = new Set(hits.map((h) => h.metadata.name as string)); // → Set of 10 tool names
  const subset = ALL_TOOLS.filter((t) => names.has(t.name));         // → StructuredTool[10]
  return baseModel.bindTools(subset);                                // → Runnable bound with the top-10 tools
}

// Per turn:
const agent = await agentForMessage(latestUserMessage); // → Runnable customised to this message
await agent.invoke(messages);                           // → AIMessage; only ~600 tokens of tool defs sent
```

Vector store là phần mới. `MemoryVectorStore` ổn cho vài trăm tool; với hàng nghìn thì dùng Chroma, pgvector, hoặc bất kỳ store production nào — hình dạng lời gọi giống hệt nhau.

*Sub-agents — một router với các delegate tool*

```ts
import { createReactAgent } from '@langchain/langgraph/prebuilt'; // → prebuilt ReAct agent factory
import { z } from 'zod';                                          // → Zod schema builder

// Each sub-agent stays in its own 5–15 sweet spot.
const billingAgent  = createReactAgent({ llm: baseModel, tools: billingTools,  systemPrompt: 'You handle billing only.' });   // → CompiledStateGraph (10 tools)
const supportAgent  = createReactAgent({ llm: baseModel, tools: supportTools,  systemPrompt: 'You handle support only.' });   // → CompiledStateGraph (10 tools)
const scheduleAgent = createReactAgent({ llm: baseModel, tools: scheduleTools, systemPrompt: 'You handle scheduling only.' }); // → CompiledStateGraph (10 tools)
// ... one per domain

// One delegate tool per sub-agent. The router's whole catalog is these five.
const delegateBilling = tool(                          // → StructuredTool { name: 'router.delegate.billing' }
  async ({ task }) => {
    const result = await billingAgent.invoke({ messages: [new HumanMessage(task)] }); // → { messages: BaseMessage[] }
    return String(result.messages.at(-1)?.content ?? ''); // → final assistant message text
  },
  {
    name: 'router.delegate.billing',                   // → identifier the router uses
    description:
      'Delegate billing tasks — refunds, charges, invoices, subscriptions, payment methods, disputes — to the billing specialist.', // → router's only signal
    schema: z.object({ task: z.string() }),            // → router passes a single freeform task string
  },
);
// ... delegateSupport, delegateSchedule, delegateInv, delegateReport

const router = createReactAgent({                      // → CompiledStateGraph — the router itself
  llm: baseModel,
  tools: [delegateBilling, delegateSupport, delegateSchedule, delegateInv, delegateReport], // → 5 delegate tools, that's it
  systemPrompt: 'You are a router. Pick the right specialist and forward the task.',
});

// Top-level call:
await router.invoke({ messages: [new HumanMessage(userMessage)] }); // → final assistant message; sub-agent did the work
```

`description` của delegate là cái router đọc để quyết định. Hãy dành thời gian thật cho năm chuỗi đó — chúng là toàn bộ UX của lớp routing.

**Chọn theo tổng số tool, không phải theo thẩm mỹ**

| Số tool | Dùng |
| --- | --- |
| 5–15 | Bind hết. Đơn giản là tốt nhất. |
| 15–30 | Scope per session nếu các session tự nhiên tách theo domain. |
| 30–80 | Tool retrieval. Vector index là một dependency mới và một chút độ trễ thêm mỗi turn. |
| 80+ hoặc các task trải domain thật sự | Sub-agents. Description của các delegate trở thành UX mới. |

Câu hỏi quyết định cho bất kỳ tool nào bạn đang muốn thêm: *"liệu một prompt mới từ người dùng có bao giờ kích hoạt nó không, và liệu model có chọn nó thay vì năm anh em gần kề chỉ từ name và description không?"* Nếu không, đổi tên, gộp, hoặc xoá nó đi.

## 06 — Sai lầm thường gặp

Một checklist ngắn và phũ phàng. Nếu tool của bạn dính vào bất kỳ cái nào trong số này, refactor trước khi ship:

- **Tool nhận 12 tham số.** Model sẽ điền đúng ba cái, điền sai hai cái, và quên bảy cái. Tách thành các tool nhỏ hơn.
- **Tool ghi mà không trả về xác nhận.** Mọi tool ghi nên trả về cái nó vừa ghi — row, ID, timestamp. Trả `void` để lại cho model không có cách nào xác nhận cuộc gọi đã xong và cũng không có cách nào tham chiếu lại cái nó vừa làm.
- **Tool trả về văn bản tự do thay vì dữ liệu có cấu trúc.** *"I sent the email to ana@example.com at 14:32"* là một đoạn văn; model phải re-parse nó. Trả về `{ sentTo, sentAt }` và model có thể trích thẳng các giá trị.
- **Tool gọi các tool khác theo kiểu đệ quy.** Model là người dàn xếp chuỗi tool. Tool thì không. Một tool mà bên trong gọi ba tool khác đang giấu công việc khỏi model và làm round-trip không debug được.
- **Tool phụ thuộc vào global state.** *"Tool này dùng khách hàng đã được set ở lần gọi trước."* Không. Truyền customer ID tường minh mỗi lần. State ngầm qua các lần gọi là cách bạn xây những feature chạy ổn ở dev và âm thầm fail ở production.

**Cặp refactor — mỗi anti-pattern trông như thế nào, và cách sửa**

*Ghi âm thầm — trả về cái vừa ghi*

```ts
// ❌ Returns void; the model can't reference the row it just created
export const createDraft = tool(
  async ({ customerId, body }) => {
    await db.emailDrafts.create({ data: { customerId, body } }); // → row created (id discarded)
    return 'OK';                                                 // → 'OK' — model has no draftId to reference
  },
  { name: 'email_drafts.create', /* ... */ },
);

// ✅ Returns the row; the next turn can quote draftId, sentAt, etc.
export const createDraft = tool(
  async ({ customerId, body }) => {
    const draft = await db.emailDrafts.create({ data: { customerId, body } }); // → EmailDraft row
    return JSON.stringify({
      draftId: draft.id,                                         // → 'draft_01HX...'
      customerId: draft.customerId,                              // → 'cus_7041'
      createdAt: draft.createdAt.toISOString(),                  // → '2026-05-09T13:42:11.000Z'
      status: 'draft',                                           // → literal 'draft' so the model knows it didn't send
    });
    // → '{"draftId":"draft_01HX...","customerId":"cus_7041","createdAt":"2026-05-09T...","status":"draft"}'
  },
  { name: 'email_drafts.create', /* ... */ },
);
```

*Trả về tự do — đưa cho model JSON, không phải văn xuôi*

```ts
// ❌ The model has to parse English to find the values
return `Sent the email to ${email} at ${new Date().toISOString()}.`;
// → 'Sent the email to ana@example.com at 2026-05-09T13:42:11.000Z.'

// ✅ The model quotes structured fields directly
return JSON.stringify({ sentTo: email, sentAt: new Date().toISOString() });
// → '{"sentTo":"ana@example.com","sentAt":"2026-05-09T13:42:11.000Z"}'
```

*Tool đệ quy — đưa các bước lộ ra cho model thay vào đó*

```ts
// ❌ One tool hides three operations; the model can't reason about partial failure
export const createPromoAndDraftEmail = tool(
  async ({ discount, lapsedDays }) => {
    const promo = await createPromotion({ discount, lapsedDays });   // → Promotion
    const customers = await findLapsedCustomers({ lapsedDays });     // → Customer[] (could be 0 or 5,000)
    const drafts = await Promise.all(customers.map((c) => createDraft({ customerId: c.id, /* ... */ }))); // → Draft[] (any one can fail mid-flight)
    return JSON.stringify({ promo, draftCount: drafts.length });     // → '{"promo":{...},"draftCount":423}' — model can't see which customer failed
  },
  { /* ... */ },
);

// ✅ Three narrow tools the model orchestrates itself; each step is observable
export const createPromotion = tool(/* ... */);    // → StructuredTool 'promotions.create_for_lapsed_customers'
export const customersLapsed = tool(/* ... */);    // → StructuredTool 'customers.lapsed'
export const createDraft = tool(/* ... */);        // → StructuredTool 'email_drafts.create'
// → model picks them in sequence; each step's result lands in the transcript and is debuggable
```

*Global state ngầm — truyền ID mỗi lần*

```ts
// ❌ Relies on whatever was set on the last call
let CURRENT_CUSTOMER: Customer | null = null;        // → mutable module-level state, shared across requests in serverless

export const setCustomer = tool(/* sets CURRENT_CUSTOMER */); // → StructuredTool — side-effects the global
export const draftEmail = tool(
  async ({ body }) => {
    if (!CURRENT_CUSTOMER) throw new Error('No customer selected'); // → throws when caller forgot setCustomer first
    return await createDraft({ customerId: CURRENT_CUSTOMER.id, body }); // → Draft for whichever customer happened to be set
  },
  /* ... */
);

// ✅ Pure: every call carries its own context
export const draftEmail = tool(
  async ({ customerId, body }) => createDraft({ customerId, body }), // → Draft, no globals read
  {
    schema: z.object({
      customerId: z.string().uuid(),                 // → UUID enforced; can't reach this with a bad id
      body: z.string().min(20),                      // → at least 20 chars of body
    }),
    /* ... */
  },
);
// → call is reproducible from args alone; works the same in dev, prod, and serverless cold starts
```

Một bài tập hữu ích sau khi viết xong một tool: đọc lại nó và tự hỏi *"liệu một model chưa từng thấy code của tôi có gọi đúng được nó chỉ bằng name, description, và schema không?"* Nếu câu trả lời là không, bạn còn việc phải làm.

<Recap>
- Tool hẹp thắng tool rộng. Tên nên mô tả tình huống, không chỉ động từ.
- Mặc định idempotent. Model sẽ retry, thường là âm thầm. Hãy xây để chuẩn bị cho điều đó.
- Description của tool là một phần của prompt. Hãy dành thời gian thật cho nó; tham chiếu chéo các tool anh em.
- Error có kiểu rõ ràng là con đường phục hồi. *Cái gì hỏng. Vì sao. Nên làm gì thay thế.* Ba câu, đòn bẩy lớn.
- Bao nhiêu tool? 5–15 là vùng ngọt, 30+ bắt đầu đau, 50+ cần scoping, retrieval, hoặc sub-agents. Đếm theo ý định, không theo endpoint.
- Năm anti-pattern cần tránh: tool 12 tham số, ghi âm thầm, trả về văn xuôi tự do, tool đệ quy, global state ngầm.
</Recap>
