## How the pieces fit

**Embeddings 101 (L25)** is the bedrock. An embedding is `f(text) → vector`, cosine is the default metric, and the model lineup that matters in 2026 is small: `text-embedding-3-small` for default, `bge-m3` for serious Vietnamese recall. Pick one, store the model name, never mix.

**Chunking strategies (L26)** is the highest-leverage tuning you'll do all year. Section-aware first (markdown has `##` headings — use them), recursive inside large sections, 400 tokens with 60 overlap as the safe default. Capture metadata *at chunk time* — `source, heading, locale, updated_at, expires_at, hash, chunk_idx` — because adding fields later means re-embedding everything.

**Index types (L27)** is where you stop tuning prematurely. Under 50k vectors, flat is correct — perfect recall, zero ops. 50k to 5M, HNSW with defaults (`M=16, ef_construction=64`); tune `ef_search` per query if needed. Past 50M, you're in vector-DB territory.

**pgvector vs Qdrant (L28)** is the architectural fork. pgvector wins by default — one Postgres, one backup, ACID across vectors and orders. Qdrant wins for selective payload filters at scale, multi-tenant, built-in replication. You can start on pgvector and migrate to Qdrant later; it's not a one-way door.

**Qdrant hands-on (L29)** is the *if you do switch* lesson. Docker compose, a collection with payload indexes, deterministic IDs that make upsert idempotent, the payload-filter syntax that justifies the migration, and snapshots that don't pretend to be backups.

**Eval (L30)** is the metric that turns everything else into a decision. 20 rows, recall@k as the headline, MRR when ordering matters, latency on the same graph, nightly cron, alert on regression. Rerankers earn their latency only when `recall@10` is high but `recall@5` is low.

## The decision matrix (one table for Part VII)

| If your situation is … | Then pick … | Why |
|---|---|---|
| Postgres-shop, &lt;10M vectors | pgvector + HNSW defaults + `halfvec(1536)` if storage hurts | One database; ACID; ops surface stays flat |
| &lt;50k vectors, just getting started | pgvector + flat (no index) | Perfect recall, zero tuning, fine until you scale |
| Hot multi-tenant with strict payload filters | Qdrant + payload indexes | Filter inside HNSW traversal beats SQL post-filter at scale |
| Mostly English with some Vietnamese | `text-embedding-3-small` | Cheap, hosted, good-enough Vietnamese |
| Vietnamese-first, recall is bottleneck | `bge-m3` self-hosted via TEI | Best Vietnamese embedding, free if you run it |
| Need top-tier precision on ambiguous queries | Hybrid (L33) + Cohere `rerank-v3-multilingual` | Reranker reorders when recall has the chunks but rank is wrong |
| Three tuning knobs to choose between | Chunk size → embedding model → reranker (in that order) | Cheaper, simpler changes first; eval at each step |

## Pre-flight checklist before Part VII

If any of these aren't true, go back to the lesson before starting the concierge build:

- [ ] You can articulate the difference between flat, IVF, and HNSW — and which you'd pick for a 200k-vector corpus.
- [ ] You have a chunking strategy with a target token size and an overlap percentage you can defend.
- [ ] You know which embedding model you're going to use *and* you've stored the model name as a column in your schema design.
- [ ] You've decided pgvector vs Qdrant for the concierge, and you have a migration trigger (a metric crossing a threshold) — not a vibe.
- [ ] You have an `eval/retrieval.jsonl` with at least 10 rows ready to grow.
- [ ] You can name the three retrieval metrics (recall@k, MRR, nDCG) and which one you read first.

## Các mảnh ghép vào nhau

**Embedding 101 (L25)** là nền đá. Embedding là `f(text) → vector`, cosine là metric mặc định, và danh sách model quan trọng năm 2026 nhỏ: `text-embedding-3-small` cho mặc định, `bge-m3` cho recall tiếng Việt nghiêm túc. Chọn một, lưu tên model, không bao giờ mix.

**Chiến lược chunking (L26)** là núm tune có đòn bẩy cao nhất bạn làm cả năm. Section-aware trước (markdown có `##` heading — dùng), recursive bên trong section lớn, 400 token với overlap 60 là mặc định an toàn. Bắt metadata *lúc chunk* — `source, heading, locale, updated_at, expires_at, hash, chunk_idx` — vì thêm field sau = re-embed mọi thứ.

**Kiểu index (L27)** là chỗ bạn dừng tune sớm. Dưới 50k vector, flat đúng — recall hoàn hảo, zero ops. 50k đến 5M, HNSW mặc định (`M=16, ef_construction=64`); tune `ef_search` mỗi query nếu cần. Qua 50M, bạn vào lãnh thổ vector DB.

**pgvector vs Qdrant (L28)** là ngã rẽ kiến trúc. pgvector thắng mặc định — một Postgres, một backup, ACID xuyên vector và orders. Qdrant thắng cho payload filter chọn lọc ở scale, multi-tenant, replication built-in. Bạn có thể bắt đầu pgvector và migrate Qdrant sau; không phải cửa một chiều.

**Qdrant hands-on (L29)** là bài *nếu bạn chuyển*. Docker compose, collection với payload index, ID deterministic làm upsert idempotent, cú pháp payload-filter biện minh cho migration, và snapshot không giả vờ làm backup.

**Eval (L30)** là metric biến mọi thứ khác thành quyết định. 20 dòng, recall@k là headline, MRR khi thứ tự quan trọng, latency cùng biểu đồ, cron đêm, alert regression. Reranker trả công latency chỉ khi `recall@10` cao nhưng `recall@5` thấp.
