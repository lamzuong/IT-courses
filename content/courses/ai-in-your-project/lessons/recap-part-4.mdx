Part IV is where the architecture clicks together. Up to now you've had a model, a list of tools, and a loop somewhere in your code that calls one then the other. That loop has obvious limits — multi-turn state is fragile, branching on read-vs-write means tangled `if` statements, and pausing the whole thing for a human approval is something you'd have to invent from scratch. LangGraph is a small library that gives you a state machine instead of a loop. State, nodes, edges, conditional routing, `interrupt()`. By the end of this part you have a complete *AI Operator* — a graph that proposes mutations, asks for permission, executes the approved ones, and tells the truth about what it did.

## How the pieces fit

**Loop limits (L15)** is the diagnosis. The naïve `bindTools` loop from Part III works for two-turn conversations and gets fragile fast. Trimming history, branching on tool type, recovering from errors — every one of those becomes a bigger `if` block until the file is unreadable. LangGraph is the answer to that pain.

**Graph basics (L16)** is the vocabulary. *State* is a typed object that flows through the graph. *Nodes* are functions that read and write state. *Edges* connect nodes. *Conditional edges* pick the next node based on state. That's it; everything else is composition.

**First graph (L17)** rebuilds the Part III loop as a graph: an `agent` node that calls the model, a `tools` node that runs tool calls, edges that route between them. Same behaviour, but the structure is now visible and inspectable.

**Conditional routing (L18)** is where the agent learns to branch. After a tool call, the graph asks: *was this a read or a write?* Reads route back to `agent` automatically. Writes route to a `await_confirmation` node and pause. Two paths from one decision point — it's the architectural shape of every safe AI Operator.

**`interrupt()` (L19)** is LangGraph's killer feature. A node calls `interrupt({ tool, args })`, the graph pauses, your UI renders an approval card, the user clicks *yes* or *no*, the graph resumes from exactly where it stopped. State, message history, partial work — all preserved. There is no equivalent in a plain loop.

**End to end (L20)** is the payoff. *"Create a 15% promo for customers who haven't ordered in 60 days, draft them a re-engagement email."* The graph plans, calls read tools to find the cohort, asks for permission on the writes, executes the approved ones, returns a transcript that says exactly what happened. Every prior lesson's piece slots into one place.

<Recap>
- **State + nodes + edges + conditional routing.** The whole LangGraph mental model. Memorise those four words.
- **Why graphs over loops.** Loops branch with `if`; graphs branch with edges. The second scales when you have three or more decisions per turn.
- **Conditional routing on tool type.** Reads run automatically; writes route to `await_confirmation`. That single rule is the safety stance from L01 expressed in code.
- **`interrupt()` is pause-and-resume, not async/await.** The graph state is checkpointed; resuming days later from a different process still works (with a Postgres checkpointer — L24).
- **The lapsed-customer flow is the test.** If you can build it end-to-end with confirmation gates and a clean audit log, you've understood Part IV.
</Recap>
