## How the pieces fit

**Loop limits (L15)** is the diagnosis. The naïve `bindTools` loop from Part III works for two-turn conversations and gets fragile fast. Trimming history, branching on tool type, recovering from errors — every one of those becomes a bigger `if` block until the file is unreadable. LangGraph is the answer to that pain.

**Graph basics (L16)** is the vocabulary. *State* is a typed object that flows through the graph. *Nodes* are functions that read and write state. *Edges* connect nodes. *Conditional edges* pick the next node based on state. That's it; everything else is composition.

**First graph (L17)** rebuilds the Part III loop as a graph: an `agent` node that calls the model, a `tools` node that runs tool calls, edges that route between them. Same behaviour, but the structure is now visible and inspectable.

**Conditional routing (L18)** is where the agent learns to branch. After a tool call, the graph asks: *was this a read or a write?* Reads route back to `agent` automatically. Writes route to a `await_confirmation` node and pause. Two paths from one decision point — it's the architectural shape of every safe AI Operator.

**`interrupt()` (L19)** is LangGraph's killer feature. A node calls `interrupt({ tool, args })`, the graph pauses, your UI renders an approval card, the user clicks *yes* or *no*, the graph resumes from exactly where it stopped. State, message history, partial work — all preserved. There is no equivalent in a plain loop.

**End to end (L20)** is the payoff. *"Create a 15% promo for customers who haven't ordered in 60 days, draft them a re-engagement email."* The graph plans, calls read tools to find the cohort, asks for permission on the writes, executes the approved ones, returns a transcript that says exactly what happened. Every prior lesson's piece slots into one place.

## The four words, in detail

If you only remember four words from Part IV, make it **state + node + edge + conditional routing**. Everything LangGraph offers — checkpoints, interrupts, streaming, multi-agent — sits on top of these four primitives. The rest of this section is a one-paragraph reminder of what each one is and what makes it pull its weight.

**State** is a typed object that flows through the graph. One field is almost always `messages` (the transcript); other fields are added as the graph grows — `pendingTool` for the confirmation handshake (L18), `approved` for the human's verdict (L19). Each field declares a *reducer* — a tiny merge function — so a node can return only the slice it wants to change and the runtime knows how to fold that slice into the whole. Append-merge for messages, replace for everything else by default. *Why this matters:* nodes stay small and local because the merging logic lives in one place.

**A node** is a plain async function from state to a partial state update. No class hierarchy, no special interface. `agentNode` calls the model and returns `{ messages: [reply] }`; `toolsNode` runs the pending tool calls and returns `{ messages: results }`. A node that has nothing to say can return `{}` and the state passes through unchanged. *Why this matters:* every node is trivially testable — give it a state, assert on what it returns. No mock graph required.

**An edge** is a declaration of *what comes next*. Two flavours. An *unconditional edge* — `addEdge('tools', 'agent')` — says "after tools, always go back to agent." It's the bottom half of every loop you've written, but expressed once and visible in a diagram instead of buried in a closing brace. The edge is data, not control flow; the runtime reads it. *Why this matters:* the graph's topology is something you can draw, lint, and reason about — not something you have to read code to reconstruct.

<Callout type="note" title="Café analogy — why an edge is *data*, not control flow">
**The whole question is: where does the rule "after step A, do step B" live?** It can only live in one of two places. Two cafés, same menu, different answer — and the answer is the entire distinction.

**Café A — the rule lives *inside the cashier's head*.** At training, she's told: *"after you ring up the order, walk the ticket to the barista yourself."* She carries that rule with her. She does the next step. If you walk in and ask *"what's the flow of an order?"* — there's nothing to point at. You have to interview every worker and stitch their answers together. If the cashier quits mid-shift while holding a ticket, the ticket leaves with her: nobody else knew where it was going.

> That's a `while` loop. `await toolsNode(...)` is the cashier walking the ticket over. The route lives in the running function. Kill the process, lose the ticket.

**Café B — the rule lives on a *sheet of paper pinned to the wall*.** The cashier is told one thing only: *"stamp the ticket and drop it on the rail."* That's it. She has no idea what comes next, and she doesn't need to. On the wall next to the rail is a sheet:

```
cashier  →  barista
barista  →  runner
runner   →  done
```

A small motor on the rail reads that sheet and slides each ticket to the next station. The cashier *cannot* walk a ticket over — she has no idea where it goes. She can only stamp and drop.

> That's LangGraph. `graph.addEdge('cashier', 'barista')` is **one line on that wall sheet** — a piece of data, sitting at rest. The runtime is the rail motor. Nodes are workers who only stamp and drop.

**The shift is small but the consequences are huge.** Because the rule is a sheet of paper, not a habit in someone's head:

| Question | Café A (control flow) | Café B (edges-as-data) |
|---|---|---|
| Draw me the order flow. | Interview every worker. | Photograph the sheet. *(That's `drawMermaid()`.)* |
| Cashier collapses holding a ticket. | Ticket is lost. | Ticket is still on the rail. A new cashier walks in, stamps the next ticket. *(That's `interrupt()` + checkpointer — L19/L24.)* |
| Add a "manager approval" step. | Retrain every worker on the new route. | Pin one new line on the sheet. Workers don't change. |
| Pause shop overnight, resume tomorrow. | Impossible — workers go home with tickets in hand. | Trivial — the rail still has every ticket exactly where it was. |

**The one-line takeaway:**

> In **control flow**, *"the next step"* is a **verb the worker does**.
> In **edges-as-data**, *"the next step"* is a **noun pinned to the wall**.

LangGraph's whole bag of tricks — drawing the graph, pausing for human approval, resuming days later from another process, retrying just one node — all of it falls out of that one shift: moving the route from inside the worker to outside, onto paper the runtime can read.
</Callout>

**Conditional routing** is the same idea, but the next node is chosen by a small router function. `addConditionalEdges('agent', router, { tools: 'tools', await: 'await_confirmation', __end__: END })` says "after agent, ask the router which one." The router reads state — typically the last message — and returns a key from the map. This is where read-vs-write branching lives (L18): if the latest `tool_call` is a read tool, route back to `tools`; if it's a write, route to `await_confirmation` and pause. *Why this matters:* every multi-step plan eventually needs `if/else`. Conditional edges keep that `if/else` in one named place instead of scattering it across the loop.

<Callout type="note" title="Café analogy — the whole graph in one kitchen">
- **State** is the order ticket pinned to the rail. Every station can read it; no station rewrites the whole thing, they just stamp their part (`{ status: 'paid' }`, `{ items: [...] }`).
- **Nodes** are stations — cashier, barista, kitchen. Each one reads the ticket, does its job, stamps the slice it owns, and hands it back to the rail.
- **Edges** are the rail itself. *"After the cashier, the ticket goes to the barista."* The rail decides where the ticket travels — the stations don't shout at each other across the room.
- **Conditional routing** is the rail's *fork*: hot drink or cold drink? Eat-in or takeaway? A simple check at the fork sends the ticket down the correct branch. The fork is one named point on the rail, not a question every station has to ask itself.
</Callout>

Put together: state moves along the rail, nodes stamp it as it passes, edges decide where it goes next, and conditional routing is the fork in the rail. That's the whole machine — and the same shape scales from a two-node demo to the lapsed-customer workflow in L20.

## Demo — how the four pieces run and link together

Here is the whole graph as actual TypeScript, then a step-by-step trace of two inputs through it. Watch how each node returns only a slice of state (the reducer concatenates), and watch the router pick the branch.

```ts
// ① STATE — the shared object every node reads and writes
type State = {
  messages: string[];        // append-merged
  needsApproval: boolean;    // replace-merged
};

// ② NODES — pure async functions: read state, return a partial slice
async function agentNode(state: State): Promise<Partial<State>> {
  const last = state.messages.at(-1) ?? '';
  if (last.includes('list')) {
    return { messages: ['call: listProducts()'] };                       // a read
  }
  if (last.includes('delete')) {
    return { messages: ['call: deleteProduct(42)'], needsApproval: true }; // a write
  }
  return { messages: ['done.'] };                                        // nothing more to do
}

async function toolsNode(_state: State): Promise<Partial<State>> {
  return { messages: ['(read tool ran · 3 rows)'] };                     // simulated tool result
}

async function awaitNode(_state: State): Promise<Partial<State>> {
  return { messages: ['paused — needs human approval'] };                // in real code: interrupt()
}

// Build the graph
const graph = new StateGraph<State>({
  channels: {
    messages: { reducer: (a, b) => a.concat(b), default: () => [] },     // append-merge
    needsApproval: { reducer: (_a, b) => b, default: () => false },      // replace-merge
  },
});

graph.addNode('agent', agentNode);
graph.addNode('tools', toolsNode);
graph.addNode('await', awaitNode);

graph.setEntryPoint('agent');

// ③ EDGE — unconditional. After tools, always loop back to agent.
graph.addEdge('tools', 'agent');

// ④ CONDITIONAL ROUTING — the fork after agent
graph.addConditionalEdges(
  'agent',
  (state) => {                                                           // the router
    if (state.needsApproval) return 'await';
    const last = state.messages.at(-1) ?? '';
    if (last.startsWith('call:')) return 'tools';
    return '__end__';
  },
  { tools: 'tools', await: 'await', __end__: END },                      // router key → next node
);

const app = graph.compile();
await app.invoke({ messages: ['list products'] });
```

**What the two traces show together.** *Same code, same nodes, same edges* — only the input differs. The four primitives never had to know about each other:

- **STATE** quietly grew via reducers; no node ever touched another node's data.
- **NODES** were each three or four lines; none of them said *"next, call X"*.
- **EDGES** owned the "what comes next" logic — `tools → agent` was a one-line registration, never executed by user code.
- **CONDITIONAL ROUTING** is the only place where the read vs. write *decision* lives. Move that one router function to a different graph and you've shipped a different agent.

That's the link: state moves, nodes mutate slices, edges and the router decide where state goes next — and the runtime, not your code, is the thing actually moving it.

<Recap>
- **State + nodes + edges + conditional routing.** The whole LangGraph mental model. Memorise those four words.
- **Why graphs over loops.** Loops branch with `if`; graphs branch with edges. The second scales when you have three or more decisions per turn.
- **Conditional routing on tool type.** Reads run automatically; writes route to `await_confirmation`. That single rule is the safety stance from L01 expressed in code.
- **`interrupt()` is pause-and-resume, not async/await.** The graph state is checkpointed; resuming days later from a different process still works (with a Postgres checkpointer — L24).
- **The lapsed-customer flow is the test.** If you can build it end-to-end with confirmation gates and a clean audit log, you've understood Part IV.
</Recap>

## Các mảnh ghép vào nhau ra sao

**Giới hạn của vòng lặp (L15)** là phần chẩn đoán. Vòng `bindTools` ngây thơ ở Phần III chỉ chạy ổn cho hội thoại hai lượt và rất nhanh trở nên dễ vỡ. Cắt bớt history, rẽ nhánh theo loại tool, hồi phục từ lỗi — mỗi cái lại biến thành một khối `if` lớn hơn cho đến khi file không còn đọc được. LangGraph là câu trả lời cho nỗi đau đó.

**Graph cơ bản (L16)** là phần từ vựng. *State* là một object có kiểu chảy xuyên suốt graph. *Node* là các hàm đọc và ghi state. *Edge* nối các node. *Conditional edge* chọn node tiếp theo dựa trên state. Hết. Mọi thứ khác chỉ là cách ráp lại.

**Graph đầu tiên (L17)** dựng lại vòng lặp Phần III dưới dạng graph: một node `agent` gọi model, một node `tools` chạy các tool call, các edge định tuyến giữa chúng. Hành vi y hệt, nhưng cấu trúc giờ đã hiển hiện và có thể quan sát.

**Conditional routing (L18)** là nơi agent học cách rẽ nhánh. Sau một tool call, graph hỏi: *đây là đọc hay ghi?* Đọc thì auto quay về `agent`. Ghi thì đi tới node `await_confirmation` và tạm dừng. Hai con đường từ một điểm quyết định — đó là hình dạng kiến trúc của mọi AI Operator an toàn.

**`interrupt()` (L19)** là tính năng "sát thủ" của LangGraph. Một node gọi `interrupt({ tool, args })`, graph tạm dừng, UI của bạn render một thẻ duyệt, người dùng bấm *có* hoặc *không*, graph resume lại từ đúng chỗ đã dừng. State, lịch sử message, công việc còn dang dở — tất cả được giữ nguyên. Vòng lặp thường không có gì tương đương.

**End to end (L20)** là phần thưởng. *"Tạo voucher 15% cho khách hàng không đặt hàng trong 60 ngày, soạn email mời họ quay lại."* Graph lên kế hoạch, gọi tool đọc để tìm tập khách, xin phép cho các thao tác ghi, thực thi cái nào được duyệt, trả về một transcript kể lại chính xác chuyện đã xảy ra. Mảnh của mọi bài trước đều có một chỗ để cắm vào.

## Bốn từ khoá, chi tiết hơn

Nếu Phần IV chỉ còn lại bốn từ trong đầu bạn, hãy để đó là **state + node + edge + conditional routing**. Mọi thứ LangGraph cung cấp — checkpoint, interrupt, streaming, multi-agent — đều dựng trên bốn nguyên thuỷ này. Phần dưới đây nhắc lại mỗi từ trong một đoạn, kèm lý do *vì sao* nó đáng giá chừng đó "đất diễn".

**State** là một object có kiểu chảy xuyên suốt graph. Gần như lúc nào cũng có một field tên `messages` (transcript hội thoại); các field khác được thêm khi graph lớn lên — `pendingTool` cho cú bắt tay xác nhận (L18), `approved` cho phán quyết của con người (L19). Mỗi field khai báo một *reducer* — một hàm merge nhỏ xíu — để một node chỉ cần trả lại đúng lát cắt nó muốn đổi, và runtime biết cách gấp lát đó vào tổng thể. Mặc định: append cho `messages`, replace cho phần còn lại. *Vì sao điều này quan trọng:* các node giữ được tính nhỏ và cục bộ, vì logic merge sống ở một chỗ duy nhất.

**Node** là một hàm async thuần từ state ra một partial state update. Không class hierarchy, không interface đặc biệt. `agentNode` gọi model và trả `{ messages: [reply] }`; `toolsNode` chạy các tool call đang chờ và trả `{ messages: results }`. Một node không có gì để nói có thể trả `{}` và state đi qua nguyên vẹn. *Vì sao điều này quan trọng:* mỗi node test cực dễ — đưa cho nó một state, assert vào cái nó trả về. Không cần mock graph.

**Edge** là một khai báo về *bước kế tiếp*. Hai loại. *Edge vô điều kiện* — `addEdge('tools', 'agent')` — nói "sau `tools`, luôn quay về `agent`." Nó là nửa dưới của mọi vòng lặp bạn từng viết, nhưng được phát biểu một lần và *nhìn thấy được* trên sơ đồ, thay vì chôn trong dấu ngoặc đóng. Edge là dữ liệu, không phải control flow; runtime đọc nó. *Vì sao điều này quan trọng:* topology của graph là thứ bạn có thể vẽ, lint, và lập luận về — không phải thứ bạn phải đọc code mới tái dựng được.

<Callout type="note" title="Phép so sánh quán cà phê — vì sao edge là *dữ liệu*, không phải control flow">
**Toàn bộ câu hỏi gói gọn trong một câu: cái quy tắc "sau bước A, làm bước B" sống ở đâu?** Nó chỉ có thể sống ở một trong hai chỗ. Hai quán, cùng menu, hai câu trả lời khác nhau — và câu trả lời ấy chính là toàn bộ sự khác biệt.

**Quán A — quy tắc sống *trong đầu cô thu ngân*.** Lúc training, cô được dặn: *"thu xong tiền thì tự bưng ticket qua cho pha chế."* Quy tắc nằm trong đầu cô. Cô là người đi bước kế tiếp. Bạn bước vào quán hỏi *"đơn hàng chạy qua quán này theo trình tự nào?"* — chẳng có gì để chỉ. Bạn phải phỏng vấn từng nhân viên rồi tự ghép câu trả lời lại. Nếu cô thu ngân nghỉ giữa ca trong khi đang cầm một ticket, ticket biến mất theo cô — không ai khác biết nó đang đi đâu.

> Đó chính là một vòng `while`. `await toolsNode(...)` là cô thu ngân bưng ticket đi. Tuyến đường sống trong function đang chạy. Kill process là mất ticket.

**Quán B — quy tắc sống trên *một tờ giấy dán trên tường*.** Cô thu ngân chỉ được dặn đúng một điều: *"đóng dấu lên ticket, rồi thả vào ray."* Hết. Cô không biết bước kế tiếp là gì, và cũng không cần biết. Trên tường cạnh ray dán một tờ giấy:

```
thu ngân  →  pha chế
pha chế   →  runner
runner    →  xong
```

Một mô-tơ nhỏ trên ray đọc tờ giấy đó và đẩy mỗi ticket tới trạm kế tiếp. Cô thu ngân *không thể* bưng ticket đi — cô không biết nó đi đâu. Cô chỉ có thể đóng dấu và thả.

> Đó chính là LangGraph. `graph.addEdge('cashier', 'barista')` là **một dòng trên tờ giấy đó** — một mẩu dữ liệu, nằm yên trên tường. Runtime là cái mô-tơ ray. Node là nhân viên — chỉ đóng dấu và thả.

**Một thay đổi nhỏ, nhưng hệ quả thì cực lớn.** Vì quy tắc là tờ giấy chứ không phải thói quen trong đầu ai:

| Câu hỏi | Quán A (control flow) | Quán B (edge-là-dữ-liệu) |
|---|---|---|
| Vẽ sơ đồ luồng đơn hàng cho tôi. | Phải phỏng vấn từng nhân viên. | Chụp ảnh tờ giấy. *(Chính là `drawMermaid()`.)* |
| Cô thu ngân ngã bệnh, tay đang cầm ticket. | Mất ticket. | Ticket vẫn trên ray. Thu ngân mới bước vào, đóng dấu ticket tiếp theo. *(Chính là `interrupt()` + checkpointer — L19/L24.)* |
| Thêm bước "quản lý duyệt". | Đào tạo lại từng nhân viên về tuyến mới. | Dán thêm một dòng vào tờ giấy. Nhân viên không đổi gì. |
| Đóng cửa qua đêm, mai mở tiếp. | Bất khả — nhân viên cầm ticket về nhà. | Bình thường — ray vẫn giữ từng ticket đúng vị trí cũ. |

**Câu chốt dễ nhớ:**

> Trong **control flow**, *"bước kế tiếp"* là một **động từ nhân viên làm**.
> Trong **edge-là-dữ-liệu**, *"bước kế tiếp"* là một **danh từ dán trên tường**.

Mọi "phép thuật" của LangGraph — vẽ được sơ đồ graph, tạm dừng chờ con người duyệt, vài ngày sau resume từ một process khác, retry lại đúng một node — tất cả đều rơi ra từ một dịch chuyển duy nhất: chuyển tuyến đường *ra khỏi* nhân viên, *lên* tờ giấy mà runtime có thể đọc.
</Callout>

**Conditional routing** vẫn cùng ý tưởng, nhưng node kế tiếp do một hàm router nhỏ chọn. `addConditionalEdges('agent', router, { tools: 'tools', await: 'await_confirmation', __end__: END })` nói "sau `agent`, hỏi router xem đi đâu." Router đọc state — thường là message cuối — và trả về một khoá trong map. Đây là chỗ nhánh đọc-vs-ghi sinh sống (L18): nếu `tool_call` mới nhất là tool đọc, đi tới `tools`; nếu là tool ghi, đi tới `await_confirmation` và tạm dừng. *Vì sao điều này quan trọng:* mọi kế hoạch nhiều bước rồi sẽ cần `if/else`. Conditional edge giữ cái `if/else` đó ở một chỗ có tên, thay vì rải khắp vòng lặp.

<Callout type="note" title="Phép so sánh quán cà phê — cả graph gọn trong một bếp">
- **State** là tờ ticket order kẹp trên ray. Mọi trạm đều đọc được; không trạm nào viết lại cả tờ — họ chỉ đóng dấu phần của mình (`{ status: 'paid' }`, `{ items: [...] }`).
- **Node** là các trạm — thu ngân, pha chế, bếp. Mỗi trạm đọc ticket, làm việc của mình, đóng dấu lát mình phụ trách, rồi trả ticket về ray.
- **Edge** chính là cái ray. *"Sau thu ngân, ticket đi tới pha chế."* Cái ray quyết định ticket di chuyển đi đâu — các trạm không hét qua phòng cho nhau nghe.
- **Conditional routing** là *chỗ rẽ* trên ray: đồ nóng hay đồ lạnh? Ngồi lại hay mang đi? Một câu kiểm tra đơn giản ở chỗ rẽ đưa ticket xuống đúng nhánh. Chỗ rẽ là *một điểm có tên* trên ray, không phải câu hỏi mà trạm nào cũng phải tự đặt cho mình.
</Callout>

Ghép lại: state chạy dọc theo ray, node đóng dấu khi nó đi qua, edge quyết định nó đi đâu tiếp, và conditional routing là chỗ rẽ trên ray. Toàn bộ cỗ máy chỉ có vậy — và cùng hình dạng đó scale từ demo hai-node cho tới workflow khách hàng ngủ quên ở L20.

## Demo — bốn mảnh chạy và liên kết với nhau ra sao

Dưới đây là toàn bộ graph dưới dạng TypeScript thật, kèm trace từng bước cho hai input chạy xuyên qua nó. Để ý mỗi node chỉ trả về một *lát cắt* state (reducer sẽ ghép lại), và để ý router chọn nhánh.

```ts
// ① STATE — object chung mà mọi node đều đọc và ghi
type State = {
  messages: string[];        // append-merge
  needsApproval: boolean;    // replace-merge
};

// ② NODES — hàm async thuần: đọc state, trả về một partial slice
async function agentNode(state: State): Promise<Partial<State>> {
  const last = state.messages.at(-1) ?? '';
  if (last.includes('list')) {
    return { messages: ['call: listProducts()'] };                       // tool đọc
  }
  if (last.includes('delete')) {
    return { messages: ['call: deleteProduct(42)'], needsApproval: true }; // tool ghi
  }
  return { messages: ['done.'] };                                        // không còn gì để làm
}

async function toolsNode(_state: State): Promise<Partial<State>> {
  return { messages: ['(read tool ran · 3 rows)'] };                     // kết quả tool giả lập
}

async function awaitNode(_state: State): Promise<Partial<State>> {
  return { messages: ['paused — needs human approval'] };                // code thật: interrupt()
}

// Dựng graph
const graph = new StateGraph<State>({
  channels: {
    messages: { reducer: (a, b) => a.concat(b), default: () => [] },     // append-merge
    needsApproval: { reducer: (_a, b) => b, default: () => false },      // replace-merge
  },
});

graph.addNode('agent', agentNode);
graph.addNode('tools', toolsNode);
graph.addNode('await', awaitNode);

graph.setEntryPoint('agent');

// ③ EDGE — vô điều kiện. Sau tools, luôn quay lại agent.
graph.addEdge('tools', 'agent');

// ④ CONDITIONAL ROUTING — chỗ rẽ sau agent
graph.addConditionalEdges(
  'agent',
  (state) => {                                                           // router
    if (state.needsApproval) return 'await';
    const last = state.messages.at(-1) ?? '';
    if (last.startsWith('call:')) return 'tools';
    return '__end__';
  },
  { tools: 'tools', await: 'await', __end__: END },                      // khoá router → node kế tiếp
);

const app = graph.compile();
await app.invoke({ messages: ['list products'] });
```

**Hai trace nói gì khi gộp lại.** *Cùng code, cùng node, cùng edge* — chỉ input khác. Bốn nguyên thuỷ không bao giờ phải biết về nhau:

- **STATE** lặng lẽ lớn lên qua reducer; không node nào chạm vào dữ liệu của node khác.
- **NODES** mỗi cái chỉ ba bốn dòng; không node nào nói *"tiếp theo, gọi X"*.
- **EDGES** ôm trọn logic "bước kế tiếp" — `tools → agent` là một dòng đăng ký, không bao giờ được user code thực thi.
- **CONDITIONAL ROUTING** là chỗ *duy nhất* mà quyết định đọc-vs-ghi sống. Đem riêng hàm router đó qua graph khác là bạn đã ship một agent khác.

Đó là sợi dây liên kết: state di chuyển, node mutate slice, edge và router quyết state đi đâu — và runtime, không phải code của bạn, là cái thực sự đẩy nó đi.

<Recap>
- **State + node + edge + conditional routing.** Toàn bộ mô hình tinh thần của LangGraph. Học thuộc bốn từ này.
- **Vì sao chọn graph thay vì loop.** Loop rẽ nhánh bằng `if`; graph rẽ nhánh bằng edge. Cái thứ hai scale được khi bạn có ba quyết định trở lên mỗi lượt.
- **Conditional routing theo loại tool.** Đọc thì chạy tự động; ghi thì đi tới `await_confirmation`. Một quy tắc duy nhất đó chính là quan điểm an toàn từ L01 được viết bằng code.
- **`interrupt()` là pause-and-resume, không phải async/await.** State của graph được checkpoint; resume nhiều ngày sau từ một process khác vẫn chạy được (cần Postgres checkpointer — L24).
- **Flow khách hàng ngủ quên là phép thử.** Nếu bạn dựng được nó end-to-end với cổng xác nhận và một audit log sạch, là bạn đã hiểu Phần IV.
</Recap>
