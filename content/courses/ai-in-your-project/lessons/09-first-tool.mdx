A tool is a typed function the model can call. You define one with three things — a name, a Zod input schema, and a function that runs when the model decides to invoke it. The model never executes your code itself; it emits a structured request that says *"please call `customers.search` with these arguments"*, your runtime executes it, and the result goes back into the conversation as another message. The whole loop runs server-side, and your TypeScript is in charge from start to finish.

That separation is the entire point. Up until now the model has been a paragraph generator with read-only access to whatever you stuffed into the prompt. With tools, it gets verbs. It can search, fetch, count, and (once we get to writes in lesson 12) act. The prompt explains the world; the tools let the model touch it.

## 01 — Defining a tool with `tool()`

LangChain ships a thin wrapper that turns a function into something the model can call. The whole shape is one call:

```ts
import { tool } from '@langchain/core/tools';   // → tool() factory imported
import { z } from 'zod';                          // → Zod schema builder imported
import { db } from '@/lib/db';                   // → Prisma-like client imported

export const searchCustomers = tool(             // → StructuredTool { name: 'customers.search' }
  async ({ filter }) => {                        // → handler called only when model picks this tool
    const rows = await db.customers.findMany({   // → Customer[] from DB
      where: {
        lastOrderAt: filter.lapsedDays           // → branch: only when lapsedDays is set
          ? { lt: daysAgo(filter.lapsedDays) }   // → e.g. { lt: 2026-03-10T00:00:00Z } if lapsedDays=60
          : undefined,                           // → undefined → Prisma skips this filter
        tags: filter.tagIn ? { hasSome: filter.tagIn } : undefined, // → { hasSome: ['vip'] } or undefined
      },
      take: 25,                                  // → cap result set at 25 rows
    });
    return JSON.stringify(rows);                 // → '[{"id":"c_01","email":"..."}, ...]' (string for model)
  },
  {
    name: 'customers.search',                    // → identifier the model uses in tool_calls
    description: 'Search customers by lapsed-days threshold or by tag. Returns up to 25 rows.', // → injected into prompt every turn
    schema: z.object({                           // → ZodObject — enforced before handler runs
      filter: z.object({
        lapsedDays: z.number().int().positive().optional(), // → integer > 0 or omitted
        tagIn: z.array(z.string()).optional(),   // → string[] or omitted
      }),
    }),
  },
);
// → searchCustomers is now a tool the model can call by name 'customers.search'
```

Four things to notice. The `name` is the identifier the model uses when calling it — keep it dotted and namespaced (`customers.search`, not `searchCustomers`) so a directory of tools reads like an API surface. The `description` is what the model actually reads to decide *whether* to call this tool; we'll come back to it. The `schema` is a Zod object that constrains the arguments — the model is forced to fit its call into this shape, so your function body never has to defend against malformed input. The function itself is plain TypeScript; it returns a string (or anything stringifiable) which becomes the tool's reply.

## 02 — Binding tools to a model

Defining a tool doesn't expose it. You bind a list of tools to a model and the bound model is what you invoke:

```ts
import { ChatOpenAI } from '@langchain/openai';  // → ChatOpenAI class imported

const model = new ChatOpenAI({ model: 'gpt-4o-mini' }) // → ChatOpenAI instance (not yet bound)
  .bindTools([searchCustomers, listProducts, getOrderTotals]); // → Runnable bound with 3 tools

const response = await model.invoke([            // → AIMessage { content: '', tool_calls: [...] }
  { role: 'system', content: 'You are an operator assistant for a small e-commerce shop.' }, // → SystemMessage
  { role: 'user', content: 'Who hasn\'t ordered in the last two months?' },                  // → HumanMessage
]);
// → response.tool_calls = [{ id: 'call_abc', name: 'customers.search', args: { filter: { lapsedDays: 60 } } }]
```

The response will not be a finished answer this time. It'll come back with a `tool_calls` array — the model has decided that `customers.search` is the right next step, and it's asking your runtime to run it. Your job is to execute each call, append the result as a `ToolMessage`, and invoke the model again. The second call is where it sees the result and writes the prose answer.

Most of that loop is mechanical, and LangGraph (which we get to in Part IV) automates it. For this lesson, the picture you want in your head is just: *model emits tool calls, runtime executes them, runtime appends results, model speaks again.*

<Callout type="note" title="Tool descriptions are part of the prompt">
The `description` field is not documentation. It is sent to the model on every turn, alongside the schema, and it's how the model decides whether your tool is the right one to use. Write it as if a junior engineer were reading it for the first time: "Search customers by lapsed-days threshold or by tag. Returns up to 25 rows ordered by last-order date descending." Be specific about what it returns and what it doesn't.
</Callout>

## 04 — The tool-call protocol

The shape the model emits is small and well-specified. After your bound model runs, the response object contains a `tool_calls` array that looks like this:

```ts
{
  tool_calls: [                                  // → array — model can request several at once
    {
      id: 'call_abc123',                         // → unique per call; pair with ToolMessage.tool_call_id
      name: 'customers.search',                  // → matches a bound tool's name
      args: { filter: { lapsedDays: 60 } },      // → already Zod-validated against the tool's schema
    },
  ],
}
// → this is what response.tool_calls looks like after model.invoke(...)
```

Three fields. The `id` is so you can route the result back to the right call when the model asks for several at once (we'll see that in lesson 10). The `name` matches a tool you bound. The `args` are already parsed and Zod-validated against your schema — if the model produced something invalid, the whole turn fails and you handle the retry, rather than your tool body having to cope with junk.

Your runtime takes that array, runs each tool, and pushes a `ToolMessage` onto the conversation for each one with the matching `tool_call_id`. Then you call the model a second time, and it sees the results and continues. Two model invocations, one tool execution, one turn — that's the unit. Everything in this part of the course builds on it.

## 05 — Why Zod (and what you lose without it)

The Zod schema isn't decoration. It does two jobs at once: it tells the *model* what shape to emit, and it tells your *function* what it can trust. Drop the schema and you take both jobs back manually — and the result is messier than people expect on the first try.

**With Zod (what you've been writing)**

```ts
export const listCustomers = tool(               // → StructuredTool { name: 'customers.list' }
  async ({ cursor, limit }) => {                 // → cursor: string|null|undefined, limit: number≤50 — Zod-guaranteed
    const rows = await db.customers.findMany({ take: limit ?? 25, cursor }); // → Customer[] of size ≤ limit
    return JSON.stringify(rows);                 // → '[{"id":"c_01",...}]' string handed back to model
  },
  {
    name: 'customers.list',                      // → identifier the model uses
    description: 'List customers, 25 per page. Pass nextCursor to continue.', // → injected into prompt
    schema: z.object({                           // → ZodObject — invalid args throw before handler runs
      cursor: z.string().nullable().optional(),  // → string | null | undefined
      limit: z.number().int().min(1).max(50).optional(), // → integer in [1,50] or undefined
    }),
  },
);
// → listCustomers is now callable by the model as 'customers.list'
```

What the model emits, parsed and validated for you:

```json
{ "name": "customers.list", "args": { "cursor": "abc123", "limit": 25 } }
```

If the model tries `limit: 9999`, Zod throws *before* your DB call runs and the error becomes a recovery hint the model reads on the next turn. Your function body sees only valid input.

**Without any schema — `DynamicTool`**

```ts
import { DynamicTool } from '@langchain/core/tools'; // → DynamicTool class imported (no schema)

export const listCustomers = new DynamicTool({   // → DynamicTool instance — model sees only the description
  name: 'customers.list',                        // → identifier the model uses
  description: 'List customers, 25 per page. Pass cursor and limit.', // → only hint the model has about shape
  func: async (input: string) => {               // → input is one raw string, format unknown
    // input is a single raw string — whatever the model put there.
    // You parse it. You validate it. You apologise to your future self.
    const parsed = JSON.parse(input);            // → object on a good run; throws SyntaxError on a bad one
    const cursor = typeof parsed.cursor === 'string' ? parsed.cursor : null; // → string | null
    const limit = Number.isInteger(parsed.limit) && parsed.limit <= 50 ? parsed.limit : 25; // → 1..50 (default 25)
    const rows = await db.customers.findMany({ take: limit, cursor }); // → Customer[]
    return JSON.stringify(rows);                 // → '[{"id":"c_01",...}]'
  },
});
// → listCustomers works, but every turn you might get a different input format
```

Two consequences. The model has *no schema* to learn from — it guesses the format from the description alone, so one turn you get `'{"cursor":"abc","limit":25}'` and the next turn `'cursor=abc, limit=25'`. And nothing is validated: junk strings hit your DB code unless you write the checks yourself, every field, every time.

**With JSON Schema instead of Zod — `DynamicStructuredTool`**

```ts
import { DynamicStructuredTool } from '@langchain/core/tools'; // → DynamicStructuredTool class imported

export const listCustomers = new DynamicStructuredTool({ // → DynamicStructuredTool instance
  name: 'customers.list',                        // → identifier the model uses
  description: 'List customers, 25 per page.',  // → injected into prompt every turn
  schema: {                                      // → raw JSON Schema — provider enforces, but no TS narrowing
    type: 'object',
    properties: {
      cursor: { type: 'string' },                // → constrains cursor to a string
      limit: { type: 'integer', maximum: 50 },   // → constrains limit to integer ≤ 50
    },
  },
  func: async ({ cursor, limit }) => {           // → typed object, but TS infers loose `any` for fields
    const rows = await db.customers.findMany({ take: limit ?? 25, cursor }); // → Customer[]
    return JSON.stringify(rows);                 // → '[{"id":"c_01",...}]'
  },
});
// → listCustomers behaves like the Zod version at runtime, but TS won't catch typos
```

You get the typed object back — the LLM provider enforces the JSON Schema on its side — but you lose Zod's TypeScript narrowing and richer constraints (`email()`, `regex()`, `refine()`, custom error messages). The model behaves the same; your code is less safe.

**Side by side**

| Approach | Model emits | Function receives | Validation |
| --- | --- | --- | --- |
| Zod schema (`tool()`) | typed JSON args | typed object — `{ cursor, limit }` | runtime + TS types |
| JSON Schema (`DynamicStructuredTool`) | typed JSON args | typed object | runtime, weaker |
| No schema (`DynamicTool`) | a single string | `string` you parse yourself | none |

**Concretely: the same prompt, three responses**

User asks *"List the next 25 customers after cursor abc123."*

```ts
// With Zod — function receives:
{ cursor: 'abc123', limit: 25 }

// With JSON Schema — function receives:
{ cursor: 'abc123', limit: 25 }

// Without schema (DynamicTool) — function receives the raw string. On a good run:
'{"cursor":"abc123","limit":25}'
// On a bad run, same prompt, next turn:
'cursor: abc123, limit: 25'
// Or:
'List 25 customers after abc123'
```

That last column is why Zod isn't optional in this course. Every lesson from here forward — pagination cursors, filter objects, idempotency keys, structured tool errors — assumes the inputs your function sees have already been validated. Skip Zod and you spend the rest of Part III writing parsers instead of features.

- A tool is a typed function: name, Zod schema, and async function. `tool()` from `@langchain/core/tools` is the wrapper.
- `model.bindTools([...])` exposes a list of tools to a model. The bound model emits `tool_calls` instead of a final answer when it wants to use one.

- The protocol is small: `{ id, name, args }`. Run the call, return a `ToolMessage` with the matching `tool_call_id`, invoke again.
- Two model calls + one tool execution = one turn. That's the loop the rest of Part III is built on.
