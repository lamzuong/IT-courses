The difference between "the AI returned some text I have to parse with regex" and "the AI returned a typed object I can use directly" is one method call. That's the whole gap structured output closes. You hand the model a schema, the model is constrained to fit its answer into that schema, and your TypeScript code receives a value of the type you declared. No parsing, no defensive checks, no `try { JSON.parse(content) } catch`.

This lesson covers the API, why a Zod schema is the right place to draw the type boundary, what changes under the hood, and the gotchas around strictness. By the end, every place in your app that previously parsed AI output by hand will collapse into a one-liner.

## 01 — `withStructuredOutput`

### The problem it solves

Without structured output, your code has to guess what shape the model came back with and parse it defensively. The same prompt can return any of these on different days:

```ts
const reply = await model.invoke(
  'Extract the name and email from: "Hi, I\'m Phuoc <phuoc@example.com>"'
);

// reply.content is a STRING. It could be any of:
//   '{"name":"Phuoc","email":"phuoc@example.com"}'              ← clean JSON ✅
//   'Sure! Here is the JSON:\n```json\n{"name":"Phuoc"...'      ← JSON in a code fence
//   'Name: Phuoc, Email: phuoc@example.com'                     ← prose
//   'Hi! I\'d be happy to help. The name is Phuoc and the…'     ← chatty prose
//   '{"name":"Phuoc","email":null,"extra":"unexpected field"}'  ← extra junk
//
// You now write 50 lines of regex + JSON.parse + try/catch + fallback
// extraction. Every model update breaks half of it.
```

This is the actual day-to-day pain of "we tried adding AI and the parsing kept exploding." Structured output is the layer that makes it stop.

### The fix, in three steps

`withStructuredOutput(schema)` flips it around: you hand the model a Zod schema, the model is *forced* to return exactly that shape, and your code receives a typed object — no parsing, no try/catch.

```ts
import { z } from 'zod';

// Step 1 — declare the shape with Zod.
const Customer = z.object({
  name: z.string(),
  email: z.string().email().nullable(),
  segment: z.enum(['active', 'dormant', 'lapsed']),
});

// Step 2 — wrap the model. `extractor` is a new model bound to the schema.
const extractor = model.withStructuredOutput(Customer);

// Step 3 — invoke. You get a real TypeScript object back.
const result = await extractor.invoke(
  'Extract name, email, and segment from: "Hi Phuoc <phuoc@example.com>, you\'re a VIP active customer."'
);
// result is { name: 'Phuoc', email: 'phuoc@example.com', segment: 'active' }
// TypeScript knows it. Autocomplete works. No JSON.parse anywhere.
```

Read each step out loud and that's the entire mental model:

1. **Declare** the shape — what fields, what types, what enums are allowed.
2. **Wrap** the model — `model.withStructuredOutput(schema)` returns a new model that's been told to fit its answer into the schema.
3. **Invoke** — you get back a real TypeScript object, already validated against the schema.

If the model produces something that *doesn't* fit (wrong type, missing required field, an enum value that isn't in the list), the call throws — not a silent half-broken object. Errors loud and early, exactly where you want them.

### Three more examples — the patterns you'll use most

**Extraction** — pull structured fields out of free text:

```ts
const Order = z.object({
  orderNumber: z.string(),
  totalCents: z.number().int().positive(),
  itemCount: z.number().int().positive(),
  shipped: z.boolean(),
});

const extract = model.withStructuredOutput(Order);
const order = await extract.invoke(
  'Email body: "Your order #4287 totaling $89.50 (3 items) has shipped today."'
);
// → { orderNumber: '4287', totalCents: 8950, itemCount: 3, shipped: true }
```

The model did the unit conversion ($89.50 → 8950 cents) because the schema said `totalCents`, not `total`. Field *names* are part of the contract too.

**Classification** — sort an input into a fixed set of buckets:

```ts
const Triage = z.object({
  category: z.enum(['billing', 'shipping', 'product', 'account', 'other']),
  urgency: z.enum(['low', 'medium', 'high']),
  needsHuman: z.boolean(),
  summary: z.string().max(120),
});

const triage = model.withStructuredOutput(Triage);
const ticket = await triage.invoke(
  'Customer wrote: "I was charged twice for the same order and you haven\'t responded in 3 days!!"'
);
// → {
//     category: 'billing',
//     urgency: 'high',
//     needsHuman: true,
//     summary: 'Duplicate charge with no response — customer escalating.'
//   }
```

The enum is a guarantee: `category` will *never* come back as `'shipping_issue'` or `'BILLING'` or `'I think this is billing'`. Five values, exactly. Your downstream `switch (ticket.category)` becomes an exhaustive type-checked branch.

**Tool input shaping** — turn a vague natural-language request into the exact arguments your function expects:

```ts
const SearchArgs = z.object({
  lapsedDays: z.number().int().min(1).max(3650).optional(),
  tagIn: z.array(z.string()).optional(),
  limit: z.number().int().min(1).max(100).default(25),
});

const planner = model.withStructuredOutput(SearchArgs);
const args = await planner.invoke(
  'Plan a customer search for: "VIPs who haven\'t ordered in two months — top 10 only"'
);
// → { lapsedDays: 60, tagIn: ['vip'], limit: 10 }

// args is now safe to hand straight to your function:
const customers = await db.customers.search(args);
```

The model translated *"two months"* → `60`, *"VIPs"* → `tagIn: ['vip']`, *"top 10"* → `limit: 10`. Your search function never sees a free-form sentence — it sees the shape it was built for. This is the foundation of everything in Part III.

### A note on streaming

There's no streaming version of structured output (yet) for most providers — a half-built JSON object isn't useful to anyone (`{ "name": "Pho` is not valid JSON). The whole response has to land before you can validate it. Use **streaming** for chat (where each token is readable on its own); use **structured output** for extraction, classification, and tool inputs (where only the complete object is useful).

## 02 — Zod as the source of truth

A Zod schema does double duty in a way that's worth pausing on:

- It validates the model's output (via `withStructuredOutput`).
- It validates inputs from your API routes (via `schema.parse(req.body)`).
- It generates the TypeScript type (via `z.infer`).
- It can be turned into a JSON Schema for documentation, OpenAPI specs, or other tools.

One file, four jobs. The schema becomes the single place where the shape of `Customer` lives in your app. Change a field there and everything that touches it — the API, the database adapter, the AI extractor, your component props — gets a type error or a runtime error in the right place.

Compare to the alternative: a TypeScript `type` declaration plus a hand-written validator plus a separate prompt that says "return JSON with these fields." Three places to change, three opportunities for them to drift apart, and the AI prompt was always the one that lagged behind.

A concrete drift story you've probably lived. Day 1, three files agree:

```ts
// types.ts — what the rest of the app sees
type Customer = { name: string; email: string; segment: 'active' | 'lapsed' };

// validator.ts — what your API trusts
function isCustomer(x: any): x is Customer {
  return typeof x.name === 'string' && typeof x.email === 'string';
}

// prompt.ts — what you ask the AI
const SYSTEM = 'Return JSON with keys: name, email, segment.';
```

Day 30, product asks to add a `vip` segment and a phone number. You patch `types.ts` and `validator.ts`, ship, get paged at midnight: the AI is still returning the old three segments and no phone. The prompt file got missed — it always does, because it doesn't fail to compile.

The Zod version collapses all three into one declaration:

```ts
// customer.ts — the only file
export const Customer = z.object({
  name: z.string(),
  email: z.string().email(),
  phone: z.string().nullable(),
  segment: z.enum(['active', 'lapsed', 'vip']),
});
export type Customer = z.infer<typeof Customer>;

// usage
model.withStructuredOutput(Customer);   // AI bound to schema
Customer.parse(req.body);               // API bound to schema
```

Add `'vip'` to the enum and the AI, the API, and every TypeScript caller move in lockstep. The midnight page never happens.

The shape of the output is fixed at the moment you declare the schema. The model can't add a field, can't omit a required one, can't put a string where you asked for a number. `email` came back `null` because the source text didn't contain one — the schema permits that with `.nullable()`. `segment` came back `'lapsed'` rather than something the model invented, because the enum constrained it to exactly three values.

That last property is what makes structured output more than a parsing convenience. The model is *prevented* from hallucinating new categories. If your downstream code does `if (segment === 'churned')` you'll get a TypeScript error before you ship — because `'churned'` is not in the enum. Errors caught early are cheap.

## 04 — Strictness gotchas

Five surprises worth knowing before you hit them in production. Each one bites the same way: the schema looked fine in the editor, the demo passed once, and then the variant you didn't anticipate showed up.

### `.nullable()` vs `.optional()` — they're not the same

```ts
// Schema A — nullable: the key is ALWAYS present, value is the type or null
const A = z.object({ email: z.string().email().nullable() });
// Result shapes: { email: 'phuoc@example.com' }   OR   { email: null }
// TypeScript:    email: string | null

// Schema B — optional: the key MIGHT be missing entirely
const B = z.object({ email: z.string().email().optional() });
// Result shapes: { email: 'phuoc@…' }   OR   { email: undefined }   OR   { } ← key missing!
// TypeScript:    email: string | undefined
```

What breaks if you pick the wrong one:

```ts
// With .optional(), this caller silently explodes when the key is missing:
const customer = await extract.invoke(text);
sendEmail(customer.email.toLowerCase());
//                ^^^^^^ TypeError: Cannot read properties of undefined

// With .nullable(), TypeScript forces you to handle null explicitly:
if (customer.email !== null) sendEmail(customer.email.toLowerCase());
```

Rule of thumb: pick `.nullable()` unless the field is truly *omittable*. Models fill in `null` reliably; they leave fields out unpredictably.

### Enums — never use `z.string()` for a fixed set of values

```ts
// ❌ z.string() — model invents variations on every run
const Bad = z.object({ status: z.string() });
// Run the same prompt 100 times against the same input, you get:
//   'active' (47×), 'Active' (12×), 'ACTIVE' (3×),
//   'lapsed' (28×), 'lapsed-customer' (3×),
//   'churned' (4×), 'inactive' (2×), 'dormant' (1×)
// Your downstream switch statement now needs 8+ cases — and you'll
// miss one when a model update adds a 9th.

// ✅ z.enum([...]) — exactly four allowed values, ever
const Good = z.object({
  status: z.enum(['active', 'lapsed', 'churned', 'inactive']),
});
// Run 100 times, you get exactly those four strings — never a 9th.
// switch (s.status) is exhaustive: TypeScript fails compilation if
// you ever forget a case.
```

The enum constraint is *enforced by the model itself* — it physically cannot emit `'CHURNED'` or `'lapsed-customer'`, because those aren't in the schema it was given.

### Refinements — `.refine()` runs *after* the model returns

```ts
const Sku = z.string().refine(
  (s) => /^[A-Z]+-\d{4}$/.test(s),
  { message: 'SKU must look like "WIDGET-0042" — uppercase letters, dash, four digits.' },
);

const Product = z.object({ sku: Sku, priceCents: z.number().int() });
const extract = model.withStructuredOutput(Product);

// ✅ Happy path — model emits a valid SKU
await extract.invoke('Extract: "WIDGET-0042 at $19.99"');
// → { sku: 'WIDGET-0042', priceCents: 1999 }

// ❌ Sad path — model lowercases the SKU
await extract.invoke('Extract: "the widget-0042 costs $19.99"');
// 💥 Throws: SKU must look like "WIDGET-0042" — uppercase letters, dash, four digits.
// You handle this with a retry. The model didn't know the constraint
// at generation time — refinements only fire after parsing.
```

Use refinements for *safety nets* (preventing a malformed value from reaching your DB), not for *guidance* (telling the model what good looks like). For guidance, lean on `z.enum()`, `.regex()` on the base type, or a few-shot example in the prompt.

### Nested objects — flat almost always wins

```ts
// ❌ Three levels deep — smaller models drop fields silently
const Bad = z.object({
  customer: z.object({
    name: z.string(),
    contact: z.object({
      email: z.string(),
      phone: z.object({ country: z.string(), number: z.string() }),
    }),
  }),
});
// What a 3B-param model often returns:
// { customer: { name: 'Phuoc', contact: { email: 'phuoc@…' } } }
//                                          ^ phone object missing entirely

// ✅ Flat — every field has one home and the model fills all of them
const Good = z.object({
  customerName: z.string(),
  customerEmail: z.string(),
  phoneCountry: z.string(),
  phoneNumber: z.string(),
});
// Reliably filled by 3B and 8B models alike.
```

If the structure *really* matters (e.g. a tree of arbitrary depth), nest it — but expect to retry more often, and prefer a larger model.

### Required fields the model "forgets"

```ts
const Customer = z.object({
  name: z.string(),
  email: z.string(),
  segment: z.enum(['active', 'lapsed']),  // required, but a 3B model
                                           // omits this on ~5% of calls
});

const extract = model.withStructuredOutput(Customer);

// 1-in-20 calls throws:
//   ZodError: segment — Required

// A four-line retry pattern catches it cleanly:
async function extractWithRetry(input: string, max = 3) {
  for (let i = 0; i < max; i++) {
    try { return await extract.invoke(input); }
    catch (e) { if (i === max - 1) throw e; }
  }
  throw new Error('unreachable');
}

const result = await extractWithRetry(text);
// On a small model, this turns "5% silent corruption" into "0.0125% loud failure"
// (5% × 5% × 5%) — and the loud failure has a real error message you can act on.
```

The validation catching the omission is the win. Without `withStructuredOutput`, a missing field would silently propagate as `undefined` and break something three function calls away.

A rule of thumb: design your schemas tight on day one. It's easier to relax a constraint later than to add one after a thousand stored records were saved without it.

<Callout type="note" title="How structured output works under the hood">
On most providers, `withStructuredOutput` is implemented by passing your schema as a single tool the model is forced to call. The "response" is the tool's arguments. Some providers (notably OpenAI's JSON mode and Gemini's response schema) have a dedicated parameter, but the API surface LangChain gives you is the same. You don't need to know which path your provider takes — the contract is the same: you get a typed object or a thrown error.
</Callout>

<Recap>
- `model.withStructuredOutput(schema)` turns AI output into a typed object. No parsing, no retries.
- Use Zod once: for AI output, API input validation, TypeScript types, and JSON Schema export. One source of truth.
- Enums prevent the model from inventing categories. `nullable()` and `optional()` are not the same — choose carefully.
- Tight schemas on day one. Loosening is cheap; tightening after data is stored is not.
- Under the hood it's usually a forced tool call, but you can ignore that and trust the API.
</Recap>
