Every tool call has three possible outcomes — success, recoverable failure, unrecoverable failure — and the model only knows what you tell it. There's no exception trace it can read, no debugger it can attach. The string your tool function returns is the *entire* universe of information the model has about what just happened. If that string is *"Internal server error"*, the model is going to retry the same broken args and you'll get the same useless string back. If the string is *"discountValue must be ≤ 100 for percentage discounts; you sent 150"*, the model fixes it on the next turn and moves on.

The discipline this lesson is teaching is small but unfamiliar: **error messages are now part of your prompt.** Every byte of an error string is read by the model and used to decide what to do next. Write them with the same care you write tool descriptions.

## Validation errors — recoverable, almost free

Zod is doing a lot of work for you here. When the model sends `discountValue: 150` to a tool that's typed `z.number().int().min(1).max(100)`, the schema rejects the call before your function body ever runs. The error you return — verbatim from `error.format()` or paraphrased into one sentence — flows back as a `ToolMessage` and the model self-corrects on the next loop iteration. You did not write retry logic. You wrote a schema. The retry came for free.

This is the most common error class in practice and the cheapest to handle. Every tool in your system gets the same treatment: tight Zod schemas, clear violation messages, no manual error parsing. *"Field X must be Y, you sent Z"* is the template. The model can read that.

A full round-trip on a validation slip:

```
turn 1 → model emits:  createPromo({ code: "save15", discountValue: 150 })
         Zod rejects:  ["code: must match /^[A-Z0-9_-]+$/, you sent 'save15'",
                        "discountValue: must be ≤ 100, you sent 150"]

turn 2 → model retries: createPromo({ code: "SAVE15", discountValue: 15 })
         Zod accepts;   tool runs; returns the new row.
```

You wrote zero retry logic. The model read the violation message and fixed both fields on its own.

## Business-rule errors — recoverable with a hint

Some failures are not type violations — the args parse fine, but the operation can't proceed. *"Promo code WELCOME10 already exists."* *"Customer 999 has been deleted."* *"That email address has unsubscribed."* These are business rules; Zod can't catch them, and the database can't either until the actual write fires.

The pattern is the same shape: catch the error in your function body, return a structured failure with a useful message. The trick is naming the constraint *and* hinting at the fix.

```ts
catch (e) {
  if (isUniqueConstraint(e, 'code')) {            // → true if DB threw on a UNIQUE on `code`
    return JSON.stringify({
      ok: false,
      error: `promo code "${args.code}" already exists. Pick a different code (e.g. "${args.code}-V2").`
    });
    // → '{"ok":false,"error":"promo code \\"WELCOME10\\" already exists. Pick a different code (e.g. \\"WELCOME10-V2\\")."}'
  }
  throw e; // unrecoverable, let it propagate     // → orchestrator handles unknown errors
}
```

The model reads *"pick a different code (e.g. WELCOME10-V2)"* and the next attempt is `WELCOME10-V2`. No retry loop in your code, no special handling — the prompt itself handled it.

## Unrecoverable errors — fail loud, fail useful

The third category is failures the model *cannot* fix by retrying with different args. Your database is down. Stripe returns a 503. The model sent perfectly valid args to a perfectly valid tool, and the infrastructure underneath failed.

The right response is to mark the error as transient and let the orchestrator (not the model) decide the retry policy. A standard shape: `{ ok: false, error: 'TransientError: upstream timeout. Retry safe.', retriable: true }`. The model sees the message but doesn't change its args; the orchestrator catches the `retriable: true` flag and retries with exponential backoff. After three failures, the orchestrator gives up and surfaces the failure to the user — *"I couldn't reach the database, please try again in a minute."*

The principle: **the model handles arg-level errors, the orchestrator handles infrastructure errors, and the user handles the rest.** Never push infrastructure decisions into the model's context — it doesn't know your retry budget and shouldn't have to.

A full handler with the three categories side by side:

```ts
async function createPromoTool(args) {
  try {
    const parsed = promoSchema.parse(args);              // category 1: validation  // → typed Promo or throws ZodError
    const row = await db.insert(promotions).values(parsed).returning(); // → newly inserted Promotion row
    return { ok: true, data: row };                       // → { ok: true, data: { id, code, ... } } on success
  } catch (e) {
    if (e instanceof z.ZodError) {
      return { ok: false, error: zodToHint(e) };          // → { ok: false, error: 'discountValue must be ≤ 100, you sent 150' }
    }
    if (isUniqueConstraint(e, 'code')) {                   // category 2: business rule
      return { ok: false, error: `code "${args.code}" exists. Try "${args.code}-V2".` }; // → hint with fix
    }
    if (isTimeoutOr5xx(e)) {                               // category 3: infra
      return { ok: false, error: 'TransientError: DB timeout. Retry safe.', retriable: true }; // → orchestrator retries
    }
    throw e;                                               // → unknown error — orchestrator/log catches it
  }
}
```

One function, one envelope, three failure modes — each routed to whoever can actually fix it.

<Callout type="tip" title="Wrap every tool function in try/catch and return a result envelope">
Throw inside a LangChain tool and the error becomes a noisy, framework-mediated string the model has to parse around — *"Error: ZodError: ..."* with a stack trace appended. Wrap your function body in `try/catch` and return `{ ok: true, data }` or `{ ok: false, error }`. The model can read either branch. Your tests can read either branch. Your traces can read either branch. The single envelope shape pays for itself the first day.
</Callout>

## The error-as-prompt principle

The summary lesson: error messages are read by the model, not your team. You'll have a different audience for your stack traces (the logs you ship to Sentry) and your tool errors (the strings you return to the model). Don't conflate them.

A good tool error is short, names the constraint, identifies the offending value, and — when possible — suggests the fix. *"discountValue 150 exceeds the 100 max for pct discounts; try 15."* A bad one is *"Validation failed."* The work to write the good one is small. The leverage is enormous: the model reads it, fixes itself, and your user never sees the failure.

<Recap>
- Three outcomes per tool call: success, recoverable, unrecoverable. Decide which is which inside the tool, not outside it.
- Validation errors: tight Zod schemas + verbatim violation messages. The model self-corrects for free.
- Business-rule errors: catch in the function body, return structured failure with a hint at the fix.
- Infrastructure errors: mark transient, let the orchestrator retry with backoff, never make the model guess.
- Wrap every tool in `try/catch` and return `{ ok, data | error }`. One envelope, one model-readable shape, no stack traces.
- Treat error strings as part of the prompt. Short, specific, and instructive — the model reads them and acts on them.
</Recap>
