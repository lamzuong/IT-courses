Every tool call has three possible outcomes — success, recoverable failure, unrecoverable failure — and the model only knows what you tell it. There's no exception trace it can read, no debugger it can attach. The string your tool function returns is the *entire* universe of information the model has about what just happened. If that string is *"Internal server error"*, the model is going to retry the same broken args and you'll get the same useless string back. If the string is *"discountValue must be ≤ 100 for percentage discounts; you sent 150"*, the model fixes it on the next turn and moves on.

The discipline this lesson is teaching is small but unfamiliar: **error messages are now part of your prompt.** Every byte of an error string is read by the model and used to decide what to do next. Write them with the same care you write tool descriptions.

## Validation errors — recoverable, almost free

Zod is doing a lot of work for you here. When the model sends `discountValue: 150` to a tool that's typed `z.number().int().min(1).max(100)`, the schema rejects the call before your function body ever runs. The error you return — verbatim from `error.format()` or paraphrased into one sentence — flows back as a `ToolMessage` and the model self-corrects on the next loop iteration. You did not write retry logic. You wrote a schema. The retry came for free.

This is the most common error class in practice and the cheapest to handle. Every tool in your system gets the same treatment: tight Zod schemas, clear violation messages, no manual error parsing. *"Field X must be Y, you sent Z"* is the template. The model can read that.

A full round-trip on a validation slip:

```
turn 1 → model emits:  createPromo({ code: "save15", discountValue: 150 })
         Zod rejects:  ["code: must match /^[A-Z0-9_-]+$/, you sent 'save15'",
                        "discountValue: must be ≤ 100, you sent 150"]

turn 2 → model retries: createPromo({ code: "SAVE15", discountValue: 15 })
         Zod accepts;   tool runs; returns the new row.
```

You wrote zero retry logic. The model read the violation message and fixed both fields on its own.

## Business-rule errors — recoverable with a hint

Some failures are not type violations — the args parse fine, but the operation can't proceed. *"Promo code WELCOME10 already exists."* *"Customer 999 has been deleted."* *"That email address has unsubscribed."* These are business rules; Zod can't catch them, and the database can't either until the actual write fires.

The pattern is the same shape: catch the error in your function body, return a structured failure with a useful message. The trick is naming the constraint *and* hinting at the fix.

```ts
catch (e) {
  if (isUniqueConstraint(e, 'code')) {            // → true if DB threw on a UNIQUE on `code`
    return JSON.stringify({
      ok: false,
      error: `promo code "${args.code}" already exists. Pick a different code (e.g. "${args.code}-V2").`
    });
    // → '{"ok":false,"error":"promo code \\"WELCOME10\\" already exists. Pick a different code (e.g. \\"WELCOME10-V2\\")."}'
  }
  throw e; // unrecoverable, let it propagate     // → orchestrator handles unknown errors
}
```

The model reads *"pick a different code (e.g. WELCOME10-V2)"* and the next attempt is `WELCOME10-V2`. No retry loop in your code, no special handling — the prompt itself handled it.

## Unrecoverable errors — fail loud, fail useful

The third category is failures the model *cannot* fix by retrying with different args. Your database is down. Stripe returns a 503. The model sent perfectly valid args to a perfectly valid tool, and the infrastructure underneath failed.

The right response is to mark the error as transient and let the orchestrator (not the model) decide the retry policy. A standard shape: `{ ok: false, error: 'TransientError: upstream timeout. Retry safe.', retriable: true }`. The model sees the message but doesn't change its args; the orchestrator catches the `retriable: true` flag and retries with exponential backoff. After three failures, the orchestrator gives up and surfaces the failure to the user — *"I couldn't reach the database, please try again in a minute."*

The principle: **the model handles arg-level errors, the orchestrator handles infrastructure errors, and the user handles the rest.** Never push infrastructure decisions into the model's context — it doesn't know your retry budget and shouldn't have to.

A full handler with the three categories side by side:

```ts
async function createPromoTool(args) {
  try {
    const parsed = promoSchema.parse(args);              // category 1: validation  // → typed Promo or throws ZodError
    const row = await db.insert(promotions).values(parsed).returning(); // → newly inserted Promotion row
    return { ok: true, data: row };                       // → { ok: true, data: { id, code, ... } } on success
  } catch (e) {
    if (e instanceof z.ZodError) {
      return { ok: false, error: zodToHint(e) };          // → { ok: false, error: 'discountValue must be ≤ 100, you sent 150' }
    }
    if (isUniqueConstraint(e, 'code')) {                   // category 2: business rule
      return { ok: false, error: `code "${args.code}" exists. Try "${args.code}-V2".` }; // → hint with fix
    }
    if (isTimeoutOr5xx(e)) {                               // category 3: infra
      return { ok: false, error: 'TransientError: DB timeout. Retry safe.', retriable: true }; // → orchestrator retries
    }
    throw e;                                               // → unknown error — orchestrator/log catches it
  }
}
```

One function, one envelope, three failure modes — each routed to whoever can actually fix it.

<Callout type="tip" title="Wrap every tool function in try/catch and return a result envelope">
Throw inside a LangChain tool and the error becomes a noisy, framework-mediated string the model has to parse around — *"Error: ZodError: ..."* with a stack trace appended. Wrap your function body in `try/catch` and return `{ ok: true, data }` or `{ ok: false, error }`. The model can read either branch. Your tests can read either branch. Your traces can read either branch. The single envelope shape pays for itself the first day.
</Callout>

## The error-as-prompt principle

The summary lesson: error messages are read by the model, not your team. You'll have a different audience for your stack traces (the logs you ship to Sentry) and your tool errors (the strings you return to the model). Don't conflate them.

<Callout type="note" title="What is a stack trace?">
A **stack trace** is the list of function calls the runtime was inside *at the moment* an exception was thrown. Read it bottom-up and you see how execution got from your route handler down to the line that exploded. Read it top-down and you see what was running when things broke. It is the single most useful piece of debugging information your runtime gives you for free — and it is written for humans, not for an LLM.

A real one from a Node app, calling our `createPromo` tool:

```text
Error: P2002: Unique constraint failed on the fields: (`code`)
    at PromoTable.create (/app/src/db/promos.ts:42:18)
    at Object.createPromo (/app/src/tools/create-promo.ts:31:10)
    at runTool (/app/src/agent/runtime.ts:88:24)
    at processTicksAndRejections (node:internal/process/task_queues:95:5)
    at async POST (/app/src/app/api/agent/route.ts:54:9)
```

What each piece is:

- **`Error: P2002: Unique constraint failed...`** — the error class and message. This is the only line a human reads first; everything below is *how we got here*.
- The **stack frames** — each `at … (path:line:col)` is a function still on the call stack. The top one is where the throw happened (`PromoTable.create` at `promos.ts:42`). Each frame down is its caller. The bottom frame is your entry point (the route handler).
- **`async POST`** — frames marked `async` come from a different turn of the event loop; Node stitches them together so you don't lose context across `await`.

Why you keep this *out* of the string you return to the model:

- The model can't act on file paths it doesn't have. Telling it "the bug is at `promos.ts:42`" is noise — it's not in your repo.
- Stack traces are long. Every token of stack trace is a token you're paying for and a token of context the model has to scan past to find the actual problem.
- They leak internals. Function names, file structure, module versions — fine for Sentry, not for a string you might one day surface to a user.

So: stack trace → Sentry / Datadog / your logs (with the full `Error` object, request ID, user ID, etc.). The string you `return` from the tool is the one-line, model-readable form — `"promo code 'LAPSED15' already exists; try a different code"`. Same underlying failure, two different audiences, two different shapes.
</Callout>

A good tool error is short, names the constraint, identifies the offending value, and — when possible — suggests the fix. *"discountValue 150 exceeds the 100 max for pct discounts; try 15."* A bad one is *"Validation failed."* The work to write the good one is small. The leverage is enormous: the model reads it, fixes itself, and your user never sees the failure.

<Recap>
- Three outcomes per tool call: success, recoverable, unrecoverable. Decide which is which inside the tool, not outside it.
- Validation errors: tight Zod schemas + verbatim violation messages. The model self-corrects for free.
- Business-rule errors: catch in the function body, return structured failure with a hint at the fix.
- Infrastructure errors: mark transient, let the orchestrator retry with backoff, never make the model guess.
- Wrap every tool in `try/catch` and return `{ ok, data | error }`. One envelope, one model-readable shape, no stack traces.
- Treat error strings as part of the prompt. Short, specific, and instructive — the model reads them and acts on them.
</Recap>

## Lỗi validation — phục hồi được, gần như free

Zod đang làm rất nhiều việc cho bạn ở đây. Khi mô hình gửi `discountValue: 150` cho một tool có schema `z.number().int().min(1).max(100)`, schema từ chối lời gọi đó trước cả khi thân hàm của bạn chạy. Lỗi bạn trả về — nguyên văn từ `error.format()` hoặc diễn lại thành một câu — chảy ngược về dưới dạng `ToolMessage` và mô hình tự sửa ở vòng loop kế tiếp. Bạn không viết một dòng retry nào. Bạn viết một cái schema. Retry đến free.

Đây là loại lỗi phổ biến nhất trong thực tế và rẻ nhất để xử lý. Mọi tool trong hệ thống của bạn đều được đối xử như nhau: schema Zod chặt, message vi phạm rõ ràng, không cần parse lỗi thủ công. *"Trường X phải là Y, bạn gửi Z"* là cái template. Mô hình đọc được điều đó.

Một round-trip đầy đủ trên một lỗi validation:

```
turn 1 → model emits:  createPromo({ code: "save15", discountValue: 150 })
         Zod rejects:  ["code: must match /^[A-Z0-9_-]+$/, you sent 'save15'",
                        "discountValue: must be ≤ 100, you sent 150"]

turn 2 → model retries: createPromo({ code: "SAVE15", discountValue: 15 })
         Zod accepts;   tool runs; returns the new row.
```

Bạn viết không một dòng retry. Mô hình đọc message vi phạm và tự sửa cả hai trường.

## Lỗi business-rule — phục hồi được, kèm một gợi ý

Một số lỗi không phải vi phạm kiểu — args parse ổn, nhưng thao tác không tiến hành được. *"Promo code WELCOME10 đã tồn tại."* *"Khách hàng 999 đã bị xoá."* *"Email đó đã unsubscribe."* Đây là business rule; Zod không bắt được, và database cũng không bắt được cho đến khi lệnh ghi thật sự nổ ra.

Mẫu vẫn cùng hình dạng: bắt error trong thân hàm, trả về một failure có cấu trúc với message hữu ích. Mẹo là *gọi tên ràng buộc* **và** *gợi ý cách sửa*.

```ts
catch (e) {
  if (isUniqueConstraint(e, 'code')) {            // → true if DB threw on a UNIQUE on `code`
    return JSON.stringify({
      ok: false,
      error: `promo code "${args.code}" already exists. Pick a different code (e.g. "${args.code}-V2").`
    });
    // → '{"ok":false,"error":"promo code \\"WELCOME10\\" already exists. Pick a different code (e.g. \\"WELCOME10-V2\\")."}'
  }
  throw e; // unrecoverable, let it propagate     // → orchestrator handles unknown errors
}
```

Mô hình đọc *"pick a different code (e.g. WELCOME10-V2)"* và lần thử tiếp theo là `WELCOME10-V2`. Không có retry loop trong code của bạn, không có xử lý đặc biệt — chính prompt đã xử lý xong.

## Lỗi không phục hồi được — hỏng to, hỏng hữu ích

Loại thứ ba là những lỗi mà mô hình *không thể* sửa bằng cách retry với args khác. Database của bạn down. Stripe trả 503. Mô hình gửi args hoàn toàn hợp lệ cho một tool hoàn toàn hợp lệ, và lớp infrastructure phía dưới hỏng.

Phản ứng đúng là đánh dấu lỗi này là tạm thời và để orchestrator (không phải mô hình) quyết định chính sách retry. Một hình dạng chuẩn: `{ ok: false, error: 'TransientError: upstream timeout. Retry safe.', retriable: true }`. Mô hình thấy message nhưng không đổi args; orchestrator bắt cờ `retriable: true` và retry với exponential backoff. Sau ba lần thất bại, orchestrator bỏ cuộc và surface lỗi lên user — *"Tôi không liên hệ được database, vui lòng thử lại trong một phút."*

Nguyên lý: **mô hình xử lý lỗi tầng args, orchestrator xử lý lỗi infrastructure, còn người dùng xử lý phần còn lại.** Đừng bao giờ đẩy quyết định về infrastructure vào context của mô hình — nó không biết retry budget của bạn, và cũng không nên cần biết.

Một handler đầy đủ với ba loại đặt cạnh nhau:

```ts
async function createPromoTool(args) {
  try {
    const parsed = promoSchema.parse(args);              // category 1: validation  // → typed Promo or throws ZodError
    const row = await db.insert(promotions).values(parsed).returning(); // → newly inserted Promotion row
    return { ok: true, data: row };                       // → { ok: true, data: { id, code, ... } } on success
  } catch (e) {
    if (e instanceof z.ZodError) {
      return { ok: false, error: zodToHint(e) };          // → { ok: false, error: 'discountValue must be ≤ 100, you sent 150' }
    }
    if (isUniqueConstraint(e, 'code')) {                   // category 2: business rule
      return { ok: false, error: `code "${args.code}" exists. Try "${args.code}-V2".` }; // → hint with fix
    }
    if (isTimeoutOr5xx(e)) {                               // category 3: infra
      return { ok: false, error: 'TransientError: DB timeout. Retry safe.', retriable: true }; // → orchestrator retries
    }
    throw e;                                               // → unknown error — orchestrator/log catches it
  }
}
```

Một hàm, một envelope, ba chế độ hỏng — mỗi cái được định tuyến tới đúng kẻ thực sự có thể sửa nó.

<Callout type="tip" title="Bọc mọi hàm tool trong try/catch và trả về một envelope kết quả">
Ném exception bên trong một LangChain tool và lỗi sẽ trở thành một string "ồn ào" do framework trung gian, mà mô hình phải tự parse — *"Error: ZodError: ..."* kèm theo stack trace dài. Bọc thân hàm trong `try/catch` và trả về `{ ok: true, data }` hoặc `{ ok: false, error }`. Mô hình đọc được cả hai nhánh. Test của bạn đọc được cả hai nhánh. Trace của bạn đọc được cả hai nhánh. Một hình dạng envelope duy nhất trả lại phần đầu tư của nó ngay trong ngày đầu tiên.
</Callout>

## Nguyên lý "error là một phần của prompt"

Bài học tóm gọn: thông báo lỗi được đọc bởi mô hình, không phải đội của bạn. Bạn sẽ có hai khán giả khác nhau cho stack trace (log bạn ship lên Sentry) và cho tool error (string bạn trả về cho mô hình). Đừng trộn chúng vào nhau.

<Callout type="note" title="Stack trace là gì?">
**Stack trace** là danh sách các hàm mà runtime đang ở *bên trong* tại khoảnh khắc một exception được throw. Đọc từ dưới lên, bạn thấy đường đi của thực thi từ route handler xuống đến dòng phát nổ. Đọc từ trên xuống, bạn thấy cái gì đang chạy lúc mọi thứ vỡ. Nó là mẩu thông tin debug hữu ích nhất mà runtime cho bạn miễn phí — và nó được viết cho *con người*, không phải cho LLM.

Một stack trace thật từ một app Node, khi gọi tool `createPromo` của chúng ta:

```text
Error: P2002: Unique constraint failed on the fields: (`code`)
    at PromoTable.create (/app/src/db/promos.ts:42:18)
    at Object.createPromo (/app/src/tools/create-promo.ts:31:10)
    at runTool (/app/src/agent/runtime.ts:88:24)
    at processTicksAndRejections (node:internal/process/task_queues:95:5)
    at async POST (/app/src/app/api/agent/route.ts:54:9)
```

Từng phần là gì:

- **`Error: P2002: Unique constraint failed...`** — class lỗi và message. Đây là dòng duy nhất mà con người đọc trước tiên; tất cả phía dưới là *làm sao tới được đây*.
- Các **stack frame** — mỗi `at … (path:line:col)` là một hàm vẫn còn trên call stack. Frame trên cùng là chỗ throw xảy ra (`PromoTable.create` tại `promos.ts:42`). Mỗi frame xuống dưới là caller của nó. Frame dưới cùng là entry point của bạn (route handler).
- **`async POST`** — các frame có chữ `async` đến từ một lượt khác của event loop; Node ráp chúng lại với nhau để bạn không mất context qua `await`.

Vì sao bạn giữ nó *bên ngoài* chuỗi trả về cho mô hình:

- Mô hình không thể hành động trên những file path nó không có. Nói với nó "bug ở `promos.ts:42`" là nhiễu — file đó không nằm trong repo của nó.
- Stack trace rất dài. Mỗi token stack trace là một token bạn phải trả tiền và một token context mà mô hình phải lướt qua để tìm vấn đề thật sự.
- Nó leak nội thất. Tên hàm, cấu trúc module, version — ổn cho Sentry, không ổn cho chuỗi mà một ngày nào đó có thể hiện ra cho user.

Vậy nên: stack trace → Sentry / Datadog / log của bạn (kèm đầy đủ object `Error`, request ID, user ID, v.v.). Chuỗi bạn `return` từ tool là dạng một-dòng, đọc-được-bởi-mô-hình — `"promo code 'LAPSED15' đã tồn tại; thử mã khác"`. Cùng một lỗi gốc, hai khán giả khác nhau, hai hình dạng khác nhau.
</Callout>

Một tool error tốt thì ngắn, gọi tên ràng buộc, chỉ ra giá trị vi phạm, và — khi có thể — gợi ý cách sửa. *"discountValue 150 vượt mức tối đa 100 với pct discount; thử 15."* Một cái tệ là *"Validation failed."* Công sức viết cái tốt rất nhỏ. Đòn bẩy thì cực lớn: mô hình đọc nó, tự sửa, và người dùng của bạn không bao giờ thấy lỗi.

- Ba kết cục cho mỗi lời gọi tool: thành công, phục hồi được, không phục hồi được. Quyết định cái nào là cái nào *bên trong* tool, không phải bên ngoài.
- Lỗi validation: schema Zod chặt + message vi phạm nguyên văn. Mô hình tự sửa, free.
- Lỗi business-rule: bắt trong thân hàm, trả về failure có cấu trúc kèm một gợi ý cách sửa.
- Lỗi infrastructure: đánh dấu transient, để orchestrator retry với backoff, đừng bắt mô hình đoán.
- Bọc mọi tool trong `try/catch` và trả về `{ ok, data | error }`. Một envelope, một hình dạng mô-hình-đọc-được, không stack trace.
- Coi error string là một phần của prompt. Ngắn, cụ thể, mang tính hướng dẫn — mô hình đọc nó và hành động theo nó.
</Recap>
