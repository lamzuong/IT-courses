This lesson covers the LangGraph `interrupt()` primitive — what it does, where it goes, and how you resume the graph with the human's verdict. ## 01 — `interrupt()` from `@langchain/langgraph`

`interrupt()` is a function call you make from inside a node. When the runtime hits it, three things happen: the graph pauses on that node, the value you passed to `interrupt()` is surfaced to your application as the "what is the graph waiting for?" payload, and the rest of the node — and the rest of the graph — does not run. Not "is queued and runs later". Does not run. Control returns to your caller (`graph.invoke(...)` or `graph.stream(...)`), which finishes with a payload describing the interrupt and a `thread_id` you'll use to resume.

```ts
import { interrupt } from '@langchain/langgraph';

async function awaitConfirmationNode(state: GraphState): Promise<Partial<GraphState>> {
  const last = state.messages.at(-1);
  const writeCalls = (last?.tool_calls ?? []).filter((c) => isWriteTool(c.name));

  // The runtime suspends the graph here. The first arg is the payload your
  // UI receives; the return value is whatever your application passes back.
  const decision = interrupt({
    kind: 'confirm_writes',
    calls: writeCalls.map((c) => ({ id: c.id, name: c.name, args: c.args })),
  });

  // Execution only reaches this line when the graph is RESUMED.
  if (decision.approve === false) {
    return { messages: [new AIMessage('Cancelled — no changes committed.')] };
  }
  return {}; // pass-through; the next edge runs write_tools
}
```

Two things are happening in those eight lines. First, you've made the *intent* — "I'm about to write" — a piece of data your UI can render. The payload is structured: tool names, arg objects, anything else you want to show in the confirmation card. Second, you've made the *resumption* a function return: when the graph is resumed, `interrupt()` returns whatever your application supplied as the resume value. That's how the user's decision flows back into the node's local scope.

## 02 — Where to put it

Inside the `await_confirmation` node, before the write happens. That sentence sounds like one rule but is actually three.

The first rule is *inside a node*. `interrupt()` is not a top-level construct; it has to be called from within a node body so the runtime can capture the suspension. The `await_confirmation` node from lesson 18 is the natural home — that's the node the router sends every write through.

The second rule is *before the write happens*. If you call `interrupt()` after the database mutation, you've already done the thing you were asking permission for. The order has to be: classify → interrupt → wait for resume → mutate. The `write_tools` node, which actually executes the calls, comes *after* `await_confirmation` for exactly this reason.

The third rule is *one interrupt per pause*. Don't loop over the write calls and call `interrupt()` for each — bundle them into one payload and let the user approve or reject the whole batch. Two consecutive interrupts are two graph pauses, two thread states, two round-trips to your UI; almost never what you want.

## 03 — Resuming with a decision

When the user clicks Approve in your UI, you call `graph.invoke` again on the same thread. The new arguments tell LangGraph two things: continue from where you paused, and here is the value to return from `interrupt()`.

```ts
import { Command } from '@langchain/langgraph';

// First call — runs until the interrupt fires.
const first = await graph.invoke(
  { messages: [new HumanMessage(prompt)] },
  { configurable: { thread_id: 'op-2026-05-01-001' } },
);
// first.__interrupt__ === [{ value: { kind: 'confirm_writes', ... } }]

// User clicks Approve in the UI. Resume the graph with their decision:
const final = await graph.invoke(
  new Command({ resume: { approve: true } }),
  { configurable: { thread_id: 'op-2026-05-01-001' } },
);
```

`Command({ resume })` is the API for "here is what `interrupt()` should return on the resumed run." The `thread_id` is what makes this *the same graph state* — without a checkpointer keying state by `thread_id`, there's nowhere to resume *from*.

The same shape works for rejection. Instead of `{ approve: true }`, send `{ approve: false }` and let the node return a graceful cancellation message. The graph still terminates cleanly; nothing wrote.

<Callout type="tip">
  Interrupts only work if the graph state survives between the pause and the resume. That means a checkpointer — `MemorySaver` for development, a SQLite or Postgres backed checkpointer for production. The next lesson covers them in detail; for now, just know that without one your half-completed plan vanishes the moment the user closes the tab.
</Callout>

## 05 — Why this changes everything

Confirmation is now a graph primitive, not application logic. That sentence does a lot of work. Before, every write tool had to remember to ask first; every UI had to invent its own approval UI; every agent loop had to track which decisions were pending. After, there is one node where pauses live, one payload shape every approval card consumes, one resume API every UI calls. Adding a new write tool is *only* a registry entry — the gate already covers it. Adding a new approval UI is *only* a card component — the data is already structured. Adding an audit log is *only* a `write_tools` node hook — every committed mutation passed through the same point of control.

The other thing that changes: rejection is cheap. A user who would otherwise have to mentally veto an AI agent's decisions before they happened — by reading dense logs, by undoing accidental writes — now gets a clean two-button card before each one. That changes their relationship with the system from "watching a fast process to make sure nothing breaks" to "approving each step at the speed they want." Some users want one click per step. Some build auto-approval rules on top. Both are now configuration, not code.

When you ship this in production, the audit log writes "approved by user@example.com at 14:03:17" next to every mutation, and the regulator's question — *who decided to issue these refunds?* — has a five-character answer. That answer is what makes AI Operators legal.

- `interrupt(payload)` from `@langchain/langgraph` pauses the graph from inside a node and surfaces the payload to your application.
- Put the interrupt in `await_confirmation`, before any mutation. Bundle the whole batch into one payload — one pause per decision.
- Resume with `graph.invoke(new Command({ resume: decision }), { configurable: { thread_id } })`. The `thread_id` is what makes it the same graph state.
- Resume value is whatever you want — `{ approve: true }`, an edited args object, a rejection reason. Whatever shape your UI sends, your node receives.
- A checkpointer is required for interrupts to survive across requests or page reloads. (Next lesson.)
- Reject is a graph terminator: nothing wrote. That property is what makes the confirmation gate trustworthy enough to let an AI loose on a production database.
