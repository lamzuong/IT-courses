## 01 — `interrupt()` from `@langchain/langgraph`

`interrupt()` is a function call you make from inside a node. When the runtime hits it, three things happen: the graph pauses on that node, the value you passed to `interrupt()` is surfaced to your application as the "what is the graph waiting for?" payload, and the rest of the node — and the rest of the graph — does not run. Not "is queued and runs later". Does not run. Control returns to your caller (`graph.invoke(...)` or `graph.stream(...)`), which finishes with a payload describing the interrupt and a `thread_id` you'll use to resume.

```ts
import { interrupt } from '@langchain/langgraph'; // → suspension primitive

async function awaitConfirmationNode(state: GraphState): Promise<Partial<GraphState>> {
  const last = state.messages.at(-1);             // → newest message (an AIMessage with write tool_calls)
  const writeCalls = (last?.tool_calls ?? []).filter((c) => isWriteTool(c.name)); // → ToolCall[] only writes

  // The runtime suspends the graph here. The first arg is the payload your
  // UI receives; the return value is whatever your application passes back.
  const decision = interrupt({                    // → graph PAUSES; resumes with whatever Command.resume passes
    kind: 'confirm_writes',
    calls: writeCalls.map((c) => ({ id: c.id, name: c.name, args: c.args })), // → payload your UI renders as a card
  });
  // → on resume: decision === whatever the app sent back, e.g. { approve: true }

  // Execution only reaches this line when the graph is RESUMED.
  if (decision.approve === false) {
    return { messages: [new AIMessage('Cancelled — no changes committed.')] }; // → cancellation appended to transcript
  }
  return {}; // pass-through; the next edge runs write_tools // → empty partial = no state change
}
```

Two things are happening in those eight lines. First, you've made the *intent* — "I'm about to write" — a piece of data your UI can render. The payload is structured: tool names, arg objects, anything else you want to show in the confirmation card. Second, you've made the *resumption* a function return: when the graph is resumed, `interrupt()` returns whatever your application supplied as the resume value. That's how the user's decision flows back into the node's local scope.

## 02 — Where to put it

Inside the `await_confirmation` node, before the write happens. That sentence sounds like one rule but is actually three.

The first rule is *inside a node*. `interrupt()` is not a top-level construct; it has to be called from within a node body so the runtime can capture the suspension. The `await_confirmation` node from lesson 18 is the natural home — that's the node the router sends every write through.

The second rule is *before the write happens*. If you call `interrupt()` after the database mutation, you've already done the thing you were asking permission for. The order has to be: classify → interrupt → wait for resume → mutate. The `write_tools` node, which actually executes the calls, comes *after* `await_confirmation` for exactly this reason.

The third rule is *one interrupt per pause*. Don't loop over the write calls and call `interrupt()` for each — bundle them into one payload and let the user approve or reject the whole batch. Two consecutive interrupts are two graph pauses, two thread states, two round-trips to your UI; almost never what you want.

## 03 — Resuming with a decision

When the user clicks Approve in your UI, you call `graph.invoke` again on the same thread. The new arguments tell LangGraph two things: continue from where you paused, and here is the value to return from `interrupt()`.

```ts
import { Command } from '@langchain/langgraph'; // → resume command class

// First call — runs until the interrupt fires.
const first = await graph.invoke(
  { messages: [new HumanMessage(prompt)] },        // → seed transcript
  { configurable: { thread_id: 'op-2026-05-01-001' } }, // → keys this conversation in the checkpointer
);
// first.__interrupt__ === [{ value: { kind: 'confirm_writes', ... } }]   // → payload your UI renders

// User clicks Approve in the UI. Resume the graph with their decision:
const final = await graph.invoke(
  new Command({ resume: { approve: true } }),      // → returned to the awaiting interrupt() call
  { configurable: { thread_id: 'op-2026-05-01-001' } }, // → same thread, continues from the pause
);
// → final: completed GraphState (writes have committed; transcript ends with assistant summary)
```

`Command({ resume })` is the API for "here is what `interrupt()` should return on the resumed run." The `thread_id` is what makes this *the same graph state* — without a checkpointer keying state by `thread_id`, there's nowhere to resume *from*.

The same shape works for rejection. Instead of `{ approve: true }`, send `{ approve: false }` and let the node return a graceful cancellation message. The graph still terminates cleanly; nothing wrote.

<Callout type="tip">
  Interrupts only work if the graph state survives between the pause and the resume. That means a checkpointer — `MemorySaver` for development, a SQLite or Postgres backed checkpointer for production. The next lesson covers them in detail; for now, just know that without one your half-completed plan vanishes the moment the user closes the tab.
</Callout>

## 05 — Why this changes everything

Confirmation is now a graph primitive, not application logic. That sentence does a lot of work. Before, every write tool had to remember to ask first; every UI had to invent its own approval UI; every agent loop had to track which decisions were pending. After, there is one node where pauses live, one payload shape every approval card consumes, one resume API every UI calls. Adding a new write tool is *only* a registry entry — the gate already covers it. Adding a new approval UI is *only* a card component — the data is already structured. Adding an audit log is *only* a `write_tools` node hook — every committed mutation passed through the same point of control.

The other thing that changes: rejection is cheap. A user who would otherwise have to mentally veto an AI agent's decisions before they happened — by reading dense logs, by undoing accidental writes — now gets a clean two-button card before each one. That changes their relationship with the system from "watching a fast process to make sure nothing breaks" to "approving each step at the speed they want." Some users want one click per step. Some build auto-approval rules on top. Both are now configuration, not code.

When you ship this in production, the audit log writes "approved by user@example.com at 14:03:17" next to every mutation, and the regulator's question — *who decided to issue these refunds?* — has a five-character answer. That answer is what makes AI Operators legal.

## 01 — `interrupt()` từ `@langchain/langgraph`

`interrupt()` là một lời gọi hàm bạn thực hiện từ bên trong một node. Khi runtime gặp nó, ba việc xảy ra: graph dừng ở node đó, giá trị bạn truyền vào `interrupt()` được đẩy lên ứng dụng dưới dạng payload "graph đang đợi cái gì?", và phần còn lại của node — cùng phần còn lại của graph — *không* chạy. Không phải "được xếp hàng để chạy sau". *Không chạy*. Quyền điều khiển quay về caller của bạn (`graph.invoke(...)` hoặc `graph.stream(...)`), kết thúc với một payload mô tả interrupt và một `thread_id` bạn sẽ dùng để resume.

```ts
import { interrupt } from '@langchain/langgraph'; // → suspension primitive

async function awaitConfirmationNode(state: GraphState): Promise<Partial<GraphState>> {
  const last = state.messages.at(-1);             // → newest message (an AIMessage with write tool_calls)
  const writeCalls = (last?.tool_calls ?? []).filter((c) => isWriteTool(c.name)); // → ToolCall[] only writes

  // The runtime suspends the graph here. The first arg is the payload your
  // UI receives; the return value is whatever your application passes back.
  const decision = interrupt({                    // → graph PAUSES; resumes with whatever Command.resume passes
    kind: 'confirm_writes',
    calls: writeCalls.map((c) => ({ id: c.id, name: c.name, args: c.args })), // → payload your UI renders as a card
  });
  // → on resume: decision === whatever the app sent back, e.g. { approve: true }

  // Execution only reaches this line when the graph is RESUMED.
  if (decision.approve === false) {
    return { messages: [new AIMessage('Cancelled — no changes committed.')] }; // → cancellation appended to transcript
  }
  return {}; // pass-through; the next edge runs write_tools // → empty partial = no state change
}
```

Có hai chuyện đang diễn ra trong tám dòng đó. Thứ nhất, bạn đã biến *ý định* — "tôi sắp ghi" — thành một mảnh dữ liệu mà UI của bạn có thể render. Payload có cấu trúc: tên tool, object args, mọi thứ khác bạn muốn hiển thị trong card xác nhận. Thứ hai, bạn đã biến *việc tiếp tục* thành một giá trị trả về của hàm: khi graph được resume, `interrupt()` trả về đúng cái giá trị resume mà ứng dụng cung cấp. Đó là cách quyết định của người dùng chảy ngược lại vào phạm vi cục bộ của node.

## 02 — Đặt nó ở đâu

Bên trong node `await_confirmation`, trước khi thao tác ghi xảy ra. Câu này nghe như một luật, nhưng thực ra là ba.

Luật thứ nhất là *bên trong một node*. `interrupt()` không phải một cấu trúc top-level; nó phải được gọi từ trong thân một node để runtime có thể bắt được việc dừng. Node `await_confirmation` ở bài 18 là nhà tự nhiên của nó — đó là node mà router đẩy mọi write call qua.

Luật thứ hai là *trước khi thao tác ghi xảy ra*. Nếu bạn gọi `interrupt()` sau khi mutate database thì bạn đã làm xong việc mà bạn đang xin phép. Thứ tự phải là: phân loại → interrupt → đợi resume → mutate. Node `write_tools`, cái thực sự chạy các call, đứng *sau* `await_confirmation` đúng vì lý do này.

Luật thứ ba là *mỗi lần dừng chỉ một interrupt*. Đừng loop qua các write call và gọi `interrupt()` cho mỗi cái — gói chúng thành một payload và để người dùng duyệt hoặc từ chối cả batch. Hai interrupt liên tiếp là hai lần dừng graph, hai thread state, hai vòng round-trip tới UI; gần như không bao giờ là cái bạn muốn.

## 03 — Tiếp tục với một quyết định

Khi người dùng bấm Approve trong UI, bạn gọi `graph.invoke` lần nữa trên cùng thread. Các tham số mới nói với LangGraph hai điều: tiếp tục từ chỗ bạn đã dừng, và đây là giá trị cần trả về từ `interrupt()`.

```ts
import { Command } from '@langchain/langgraph'; // → resume command class

// First call — runs until the interrupt fires.
const first = await graph.invoke(
  { messages: [new HumanMessage(prompt)] },        // → seed transcript
  { configurable: { thread_id: 'op-2026-05-01-001' } }, // → keys this conversation in the checkpointer
);
// first.__interrupt__ === [{ value: { kind: 'confirm_writes', ... } }]   // → payload your UI renders

// User clicks Approve in the UI. Resume the graph with their decision:
const final = await graph.invoke(
  new Command({ resume: { approve: true } }),      // → returned to the awaiting interrupt() call
  { configurable: { thread_id: 'op-2026-05-01-001' } }, // → same thread, continues from the pause
);
// → final: completed GraphState (writes have committed; transcript ends with assistant summary)
```

`Command({ resume })` là API cho "đây là cái mà `interrupt()` nên trả về ở lần chạy tiếp tục." `thread_id` chính là thứ làm cho cái này *cùng một graph state* — không có checkpointer khoá state theo `thread_id`, không có chỗ nào để *tiếp tục từ đó*.

Cùng hình dạng đó dùng được cho việc từ chối. Thay vì `{ approve: true }`, gửi `{ approve: false }` và để node trả về một message huỷ nhẹ nhàng. Graph vẫn kết thúc gọn ghẽ; không có gì được ghi.

<Callout type="tip">
  Interrupt chỉ hoạt động nếu state của graph sống sót giữa lúc dừng và lúc resume. Nghĩa là cần một checkpointer — `MemorySaver` cho development, một checkpointer dùng SQLite hay Postgres cho production. Bài kế tiếp sẽ nói chi tiết; bây giờ, chỉ cần biết rằng nếu không có nó thì kế hoạch đang dở dang của bạn biến mất ngay khoảnh khắc người dùng đóng tab.
</Callout>

## 05 — Vì sao điều này thay đổi mọi thứ

Confirmation giờ là một primitive của graph, không phải logic ứng dụng. Câu đó gánh rất nhiều ý nghĩa. Trước đây, mỗi write tool phải nhớ hỏi trước; mỗi UI phải tự nghĩ ra UI duyệt; mỗi vòng agent phải tự theo dõi quyết định nào đang chờ. Sau này, có một node nơi các lần dừng sống ở đó, một hình dạng payload mà mọi card duyệt đều tiêu thụ, một API resume mà mọi UI đều gọi. Thêm một write tool mới *chỉ là* thêm một mục trong registry — cổng đã lo phần còn lại. Thêm một UI duyệt mới *chỉ là* một component card — dữ liệu đã có cấu trúc sẵn. Thêm một audit log *chỉ là* một hook ở node `write_tools` — mọi mutation đã commit đều đi qua cùng một điểm điều khiển.

Điều khác cũng thay đổi: từ chối trở nên rẻ. Một người dùng đáng lẽ phải ngồi tự veto quyết định của AI agent trong đầu trước khi chúng xảy ra — bằng cách đọc log dày đặc, bằng cách undo các thao tác ghi nhầm — giờ nhận được một card hai nút sạch sẽ trước mỗi lần. Điều đó thay đổi mối quan hệ của họ với hệ thống từ "canh một tiến trình chạy nhanh để chắc không có gì hỏng" sang "duyệt từng bước với tốc độ họ muốn." Có người muốn một click mỗi bước. Có người dựng luật auto-approve ở trên. Cả hai giờ là cấu hình, không phải code.

Khi bạn đem cái này lên production, audit log ghi "approved by user@example.com at 14:03:17" cạnh mỗi mutation, và câu hỏi của cơ quan quản lý — *ai đã quyết định hoàn các khoản tiền này?* — có một câu trả lời năm ký tự. Câu trả lời đó là thứ làm cho AI Operator hợp pháp.
