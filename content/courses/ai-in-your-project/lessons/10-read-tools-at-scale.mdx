One tool that returns 25 customers is fine. One tool that returns 25 million is a denial-of-service attack against your model's context window — and your bill. Once tools graduate from your local seed file to a production database, the design problem shifts from *"can I expose this query to the model"* to *"can I expose this query in a way that doesn't melt down the moment a row count gets interesting."*

This lesson is the read-side playbook. Three patterns will cover most of what you ever need: pagination so the model can ask for more without seeing everything, filtering so the model expresses intent and you translate it into a real query, and aggregation so the model can ask for *summaries* instead of raw rows. Each pattern shrinks the data flowing into context by an order of magnitude or more.

## 01 — Pagination

The simplest mistake is the most common: a tool that returns *all* matching rows. The right shape is cursor-based — the tool returns a fixed page plus an opaque cursor the model can hand back to ask for the next page:

```ts
export const listCustomers = tool(                // → StructuredTool { name: 'customers.list' }
  async ({ cursor, limit = 25 }) => {             // → cursor: string|null|undefined, limit defaults to 25
    const rows = await db.customers.findMany({    // → Customer[] of size up to limit+1
      take: limit + 1,                            // → fetch one probe row to detect "has more"
      cursor: cursor ? { id: parseInt(cursor, 10) } : undefined, // → resume from cursor or start at top
      orderBy: { id: 'asc' },                     // → stable ordering for cursor walking
    });
    const nextCursor = rows.length > limit ? String(rows[limit].id) : null; // → '101' or null on last page
    return JSON.stringify({ rows: rows.slice(0, limit), nextCursor });      // → '{"rows":[...25 rows...],"nextCursor":"101"}'
  },
  {
    name: 'customers.list',                       // → identifier the model uses
    description: 'List customers, 25 per page. Pass nextCursor from the previous page to continue.',
    schema: z.object({
      cursor: z.string().nullable().optional(),   // → string | null | undefined
      limit: z.number().int().min(1).max(50).optional(), // → integer in [1, 50] or undefined
    }),
  },
);
```

A few things to notice. The `limit` is bounded — `min(1).max(50)` means even a confused model can't ask for ten thousand rows. The cursor here is a stringified ID for teaching clarity; in production you'd encode it (e.g. `Buffer.from(JSON.stringify({id})).toString('base64url')`) so the model can't synthesize one by guessing the scheme — it just hands back whatever string the previous response gave it. The shape `{ rows, nextCursor }` is the convention the model learns once and applies across every paginated tool you build.

Models are good at multi-step reasoning when the protocol is consistent. If half your tools paginate with `cursor` and half with `page`, you'll see the model send `page: 2` to a cursor tool and stop. Pick one shape and apply it everywhere.

### What `cursor` actually does

`cursor` is not a Zod feature — it's a field name we picked for pagination. Zod is only describing its *shape*; the meaning comes from how the tool body uses it.

The Zod part is saying three things about the field:

- `z.string()` — must be a string when present.
- `.nullable()` — `null` is also allowed (model can explicitly send "no cursor").
- `.optional()` — the model can omit it on the first call.

Zod doesn't know what a cursor is. The pagination behaviour is what your function does with the value:

```text
Turn 1 — first page, no cursor yet
  model emits:   { args: { limit: 25 } }
  tool returns:  { rows: [...25 customers...], nextCursor: "eyJpZCI6MTAwfQ" }

Turn 2 — model wants more, hands the cursor back
  model emits:   { args: { cursor: "eyJpZCI6MTAwfQ", limit: 25 } }
  tool returns:  { rows: [...next 25...], nextCursor: "eyJpZCI6MTI1fQ" }

Turn 3 — last page
  model emits:   { args: { cursor: "eyJpZCI6MTI1fQ", limit: 25 } }
  tool returns:  { rows: [...final 12...], nextCursor: null }
```

The model never *invents* a cursor — it just hands back whatever string the previous response gave it. That is why the tool body branches on presence:

```ts
cursor: cursor ? { id: parseInt(cursor, 10) } : undefined, // → { id: 100 } when cursor present; undefined on first call
```

If a cursor is present, start from that ID. If not, start from the beginning. When `nextCursor` comes back as `null`, the model knows there are no more pages.

**Why a cursor and not `page: 2`?**

- **Stable under inserts.** Offset-based pagination (`OFFSET 25`) shifts every row when a new one is inserted between calls — you skip a customer. A cursor points at a specific row's ID, so concurrent inserts don't move the window.
- **No total-count required.** The DB doesn't have to count all matching rows; it just walks forward from the cursor.
- **Opaque and encodable.** You can pack multiple sort fields into one base64 string so the model can't fabricate a valid one by guessing the scheme.

**A worked example: the offset-shift bug**

Say there are 50 customers in the DB, ordered by `id`, and the model is paginating 25 per page.

```text
T₀ — initial state, 50 rows ordered by id
  ids: [1, 2, 3, ..., 25, 26, 27, ..., 50]
```

*With `page: 2` (offset-based)*

```ts
// Tool body — bad
const rows = await db.customers.findMany({
  skip: (page - 1) * 25,                          // → SQL OFFSET; shifts when rows insert/delete
  take: 25,                                       // → SQL LIMIT
  orderBy: { id: 'asc' },                         // → 1, 2, 3, ...
});

// Turn 1: page = 1
//   SQL: SELECT ... ORDER BY id ASC LIMIT 25 OFFSET 0
//   returns ids [1..25]  ✅

// Between turns: a new signup. id 51 inserted, but actually
// the new row gets id 26 because the schema reuses gaps.
// (Or imagine the order is by created_at and the new row
// sorts into the middle — same effect.)

// Turn 2: page = 2
//   SQL: SELECT ... ORDER BY id ASC LIMIT 25 OFFSET 25
//   returns ids [27..51]  ❌  id 26 was silently skipped.
```

*With `cursor` (id-based)*

```ts
// Tool body — good
const PAGE_SIZE = 25;                             // → page size constant

const rows = await db.customers.findMany({        // → Customer[] of size up to 26
  take: PAGE_SIZE + 1, // 25 + 1 probe row to detect "more pages"   // → SQL LIMIT 26
  cursor: cursor ? { id: parseInt(cursor, 10) } : undefined,        // → resume key, or undefined on first call
  skip: cursor ? 1 : 0, // skip the cursor row itself               // → don't redeliver the cursor row
  orderBy: { id: 'asc' },                         // → stable ordering for cursor walks
});

const hasMore = rows.length > PAGE_SIZE;          // → true if probe row came back, else false
const pageRows = rows.slice(0, PAGE_SIZE);              // → first 25 rows (drop the probe)
const nextCursor = hasMore ? String(pageRows[PAGE_SIZE - 1].id) : null; // → '25' on first page, null at end

return JSON.stringify({ rows: pageRows, nextCursor });  // → '{"rows":[...25 rows...],"nextCursor":"25"}'
```

The query asks for **26** rows — `PAGE_SIZE + 1`. The tool only **returns 25** to the model. That extra row is a probe: if it came back, there's at least one more page, so we set `nextCursor` to the last id we're sending. If it didn't come back, we've hit the end and `nextCursor` is `null`. Cheaper than a separate `COUNT(*)` query and uses the index we're already walking.

Watching it run on a 50-row table:

```text
Turn 1: cursor = null
  DB returns 26 rows (probe came back) → ids [1..26]
  hasMore  = true
  pageRows = rows.slice(0, 25) = [1..25]
  nextCursor = "25"   (last id in pageRows — not 26)
  Tool returns: { rows: [1..25], nextCursor: "25" }

Between turns: same insert as before, new row sorts as id 26.

Turn 2: cursor = "25"
  SQL: SELECT ... WHERE id > 25 ORDER BY id ASC LIMIT 26
  DB returns 26 rows → ids [26..51]   ← id 26 is the inserted row, included naturally
  hasMore  = true if there were more, false if these are the last
  pageRows = [26..50]
  nextCursor = "50" (or null if there were exactly 26 left)
```

The cursor is *"the last row I handed back"*, not *"the row I haven't shown yet"*. That's why on turn 2 the query is `WHERE id > 25` — start strictly *after* what the caller has already seen.

The offset version skipped the new row because *position 26 in the ordered list moved* between turns. The cursor version anchored on `id > 25` instead of "row #26", so concurrent writes can't shift the window. Same bug also bites you with `created_at DESC` feeds, search results, audit logs — anywhere new rows appear during pagination, which is everywhere.

The role split, in one line: *Zod validates what shape the cursor must be. Your tool function decides what the cursor means.*

## 02 — Filtering

The second pattern is letting the model say *what* it wants instead of getting everything. The temptation is to expose a tool that takes a SQL string. Don't. The model does not need to learn SQL; it needs an object that describes intent.

```ts
const filterSchema = z.object({                   // → ZodObject describing the allowed filter fields
  lapsedDays: z.number().int().positive().max(3650).optional(), // → integer 1..3650 or undefined
  tagIn: z.array(z.string().max(40)).max(10).optional(),        // → up to 10 tag strings (each ≤40 chars)
  emailDomain: z.string().max(80).optional(),     // → string ≤80 chars or undefined
  createdAfter: z.string().datetime().optional(), // → ISO 8601 string or undefined
});

export const searchCustomers = tool(              // → StructuredTool { name: 'customers.search' }
  async ({ filter }) => {
    const where: Prisma.CustomerWhereInput = {};  // → start with empty where clause
    if (filter.lapsedDays) where.lastOrderAt = { lt: daysAgo(filter.lapsedDays) }; // → { lt: Date 60d ago }
    if (filter.tagIn) where.tags = { hasSome: filter.tagIn };                       // → { hasSome: ['vip'] }
    if (filter.emailDomain) where.email = { endsWith: '@' + filter.emailDomain };   // → { endsWith: '@gmail.com' }
    if (filter.createdAfter) where.createdAt = { gte: new Date(filter.createdAfter) }; // → { gte: Date }
    const rows = await db.customers.findMany({ where, take: 25 }); // → Customer[] of size ≤ 25
    return JSON.stringify({ rows });              // → '{"rows":[{"id":"c_01",...}, ...]}'
  },
  {
    name: 'customers.search',                     // → identifier the model uses
    description: 'Search customers by lapsed-days, tags, email domain, or creation date.',
    schema: z.object({ filter: filterSchema }),   // → outer schema wrapping the filter object
  },
);
```

The Zod schema does two jobs at once. It tells the model what filters are available (the schema is part of the tool's exposed surface) and it locks down what your function will ever receive. The body of the function is a switch from validated-intent to parameterized query — you're the translator, not the model.

<Callout type="warning" title="Never interpolate the model's filter into raw SQL">
The single largest class of AI security incidents in the last two years has been someone exposing a tool that takes a SQL fragment, a JSONPath expression, or a free-form `where` string. The model will eventually emit `'; DROP TABLE customers; --` because someone in a chat asked it nicely. Always parse with Zod, always translate to a parameterized query, always treat the model's input as adversarial. *You* write the SQL; the model only describes what it wants.
</Callout>

## 03 — Aggregation

The third pattern is the one that saves the most tokens: when the model wants to know *about* the data rather than the data itself, expose a tool that pre-computes the answer.

```ts
export const totalsBySegment = tool(              // → StructuredTool { name: 'orders.totals_by_segment' }
  async ({ segment }) => {
    const cohort = await db.customers.findMany({  // → Customer[] (id-only projection)
      where: segment === 'lapsed-pre-q4'          // → branch on segment
        ? { lastOrderAt: { lt: daysAgo(60) } }    // → 'lapsed' cohort: last order > 60d ago
        : { lastOrderAt: { gte: daysAgo(30) } },  // → 'active' cohort: ordered in last 30d
      select: { id: true },                       // → only need ids for the next step
    });
    const ids = cohort.map((c) => c.id);          // → ['c_01', 'c_02', ...]
    const orders = await db.orders.findMany({ where: { customerId: { in: ids } } }); // → Order[] for those customers
    const total = orders.reduce((s, o) => s + o.totalCents, 0); // → 318420 (sum of cents)
    return JSON.stringify({
      customerCount: ids.length,                  // → 47
      orderCount: orders.length,                  // → 53
      totalRevenueCents: total,                   // → 318420
      avgOrderValueCents: orders.length ? Math.round(total / orders.length) : 0, // → 6008
    });
    // → '{"customerCount":47,"orderCount":53,"totalRevenueCents":318420,"avgOrderValueCents":6008}'
  },
  {
    name: 'orders.totals_by_segment',
    description: 'Return revenue, order count, and average order value for a named customer segment.',
    schema: z.object({ segment: z.enum(['lapsed-pre-q4', 'active-30d']) }), // → only 2 allowed segments
  },
);
```

A model asking *"how much have lapsed customers spent?"* shouldn't need to see every order. It needs four numbers. Pre-compute the four numbers, return them, and the synthesis paragraph the model writes will be just as good — and the round-trip will use 200 tokens instead of 200,000.

A useful heuristic: if a model ever asks a tool for raw rows just to count them, you're missing an aggregation tool. Add `count_by_tag`, `revenue_by_month`, `top_products_for_customer`. Each one is fifteen lines of TypeScript and saves a context window.

<Recap>
- Pagination: cursor-based, bounded `limit`, return `{ rows, nextCursor }`. Pick one shape and use it everywhere.
- Filtering: a typed `filter` object, Zod-validated, translated to a parameterized query in your function body.
- Aggregation: pre-compute summaries the model can ask for. Beats raw rows on cost, latency, and answer quality.
- Never interpolate model output into SQL, JSONPath, or any other code surface. Parse, validate, translate.
- Good tool descriptions + good schemas = the model orchestrates multi-tool sequences correctly without you scripting them.
</Recap>

## 01 — Pagination

Sai lầm đơn giản nhất lại là phổ biến nhất: một tool trả về *tất cả* các dòng khớp. Hình dạng đúng là cursor-based — tool trả về một trang có kích thước cố định cộng thêm một cursor "đục" (opaque) mà model có thể gửi lại để xin trang kế tiếp:

```ts
export const listCustomers = tool(                // → StructuredTool { name: 'customers.list' }
  async ({ cursor, limit = 25 }) => {             // → cursor: string|null|undefined, limit defaults to 25
    const rows = await db.customers.findMany({    // → Customer[] of size up to limit+1
      take: limit + 1,                            // → fetch one probe row to detect "has more"
      cursor: cursor ? { id: parseInt(cursor, 10) } : undefined, // → resume from cursor or start at top
      orderBy: { id: 'asc' },                     // → stable ordering for cursor walking
    });
    const nextCursor = rows.length > limit ? String(rows[limit].id) : null; // → '101' or null on last page
    return JSON.stringify({ rows: rows.slice(0, limit), nextCursor });      // → '{"rows":[...25 rows...],"nextCursor":"101"}'
  },
  {
    name: 'customers.list',                       // → identifier the model uses
    description: 'List customers, 25 per page. Pass nextCursor from the previous page to continue.',
    schema: z.object({
      cursor: z.string().nullable().optional(),   // → string | null | undefined
      limit: z.number().int().min(1).max(50).optional(), // → integer in [1, 50] or undefined
    }),
  },
);
```

Vài chỗ đáng để ý. `limit` bị chặn — `min(1).max(50)` nghĩa là một model có hoang mang cách mấy cũng không xin được mười nghìn dòng. Cursor ở đây là một ID dạng chuỗi để cho dễ dạy; trong production bạn sẽ encode nó (ví dụ `Buffer.from(JSON.stringify({id})).toString('base64url')`) để model không thể tự "đoán format" rồi bịa ra một cursor — model chỉ gửi lại đúng chuỗi mà response trước đã đưa cho nó. Hình dạng `{ rows, nextCursor }` là quy ước model học một lần rồi áp dụng cho mọi tool có phân trang mà bạn xây.

Model rất giỏi suy luận nhiều bước khi giao thức nhất quán. Nếu một nửa tool của bạn phân trang bằng `cursor` còn một nửa bằng `page`, bạn sẽ thấy model gửi `page: 2` vào tool dùng cursor rồi đứng hình. Chọn một hình dạng và áp dụng nó cho tất cả.

### `cursor` thực ra làm gì

`cursor` không phải là feature của Zod — đó chỉ là một tên field chúng ta đặt cho việc phân trang. Zod chỉ đang mô tả *shape* của nó; ý nghĩa đến từ cách thân hàm tool sử dụng nó.

Phần Zod đang nói ba điều về field này:

- `z.string()` — nếu có thì phải là string.
- `.nullable()` — `null` cũng được phép (model có thể chủ động gửi "không có cursor").
- `.optional()` — model có thể bỏ qua field này ở lần gọi đầu tiên.

Zod không hiểu cursor là gì. Hành vi phân trang là do cái mà hàm của bạn làm với giá trị đó:

```text
Turn 1 — first page, no cursor yet
  model emits:   { args: { limit: 25 } }
  tool returns:  { rows: [...25 customers...], nextCursor: "eyJpZCI6MTAwfQ" }

Turn 2 — model wants more, hands the cursor back
  model emits:   { args: { cursor: "eyJpZCI6MTAwfQ", limit: 25 } }
  tool returns:  { rows: [...next 25...], nextCursor: "eyJpZCI6MTI1fQ" }

Turn 3 — last page
  model emits:   { args: { cursor: "eyJpZCI6MTI1fQ", limit: 25 } }
  tool returns:  { rows: [...final 12...], nextCursor: null }
```

Model không bao giờ *bịa* ra cursor — nó chỉ gửi lại đúng chuỗi mà response trước đã trả về. Đó là lý do thân tool rẽ nhánh dựa vào việc có hay không có cursor:

```ts
cursor: cursor ? { id: parseInt(cursor, 10) } : undefined, // → { id: 100 } when cursor present; undefined on first call
```

Có cursor thì bắt đầu từ ID đó. Không có thì bắt đầu từ đầu. Khi `nextCursor` trả về là `null`, model biết là không còn trang nào nữa.

**Vì sao dùng cursor thay vì `page: 2`?**

- **Ổn định khi có insert.** Pagination kiểu offset (`OFFSET 25`) sẽ làm mọi dòng dịch chỗ khi có dòng mới chèn vào giữa các lần gọi — bạn sẽ bỏ sót một khách hàng. Cursor trỏ vào ID của một dòng cụ thể, nên các insert chạy song song không làm dịch cửa sổ.
- **Không cần đếm tổng.** DB không phải đếm tất cả dòng khớp; nó chỉ đi tiếp từ cursor.
- **Đục và mã hoá được.** Bạn có thể đóng gói nhiều field sort vào một chuỗi base64 để model không thể tự nghĩ ra một cursor hợp lệ bằng cách đoán format.

**Ví dụ chi tiết: bug offset-shift**

Giả sử có 50 khách hàng trong DB, sắp xếp theo `id`, model phân trang 25 dòng mỗi trang.

```text
T₀ — initial state, 50 rows ordered by id
  ids: [1, 2, 3, ..., 25, 26, 27, ..., 50]
```

*Với `page: 2` (offset-based)*

```ts
// Tool body — bad
const rows = await db.customers.findMany({
  skip: (page - 1) * 25,                          // → SQL OFFSET; shifts when rows insert/delete
  take: 25,                                       // → SQL LIMIT
  orderBy: { id: 'asc' },                         // → 1, 2, 3, ...
});

// Turn 1: page = 1
//   SQL: SELECT ... ORDER BY id ASC LIMIT 25 OFFSET 0
//   returns ids [1..25]  ✅

// Between turns: a new signup. id 51 inserted, but actually
// the new row gets id 26 because the schema reuses gaps.
// (Or imagine the order is by created_at and the new row
// sorts into the middle — same effect.)

// Turn 2: page = 2
//   SQL: SELECT ... ORDER BY id ASC LIMIT 25 OFFSET 25
//   returns ids [27..51]  ❌  id 26 was silently skipped.
```

*Với `cursor` (id-based)*

```ts
// Tool body — good
const PAGE_SIZE = 25;                             // → page size constant

const rows = await db.customers.findMany({        // → Customer[] of size up to 26
  take: PAGE_SIZE + 1, // 25 + 1 probe row to detect "more pages"   // → SQL LIMIT 26
  cursor: cursor ? { id: parseInt(cursor, 10) } : undefined,        // → resume key, or undefined on first call
  skip: cursor ? 1 : 0, // skip the cursor row itself               // → don't redeliver the cursor row
  orderBy: { id: 'asc' },                         // → stable ordering for cursor walks
});

const hasMore = rows.length > PAGE_SIZE;          // → true if probe row came back, else false
const pageRows = rows.slice(0, PAGE_SIZE);              // → first 25 rows (drop the probe)
const nextCursor = hasMore ? String(pageRows[PAGE_SIZE - 1].id) : null; // → '25' on first page, null at end

return JSON.stringify({ rows: pageRows, nextCursor });  // → '{"rows":[...25 rows...],"nextCursor":"25"}'
```

Query xin **26** dòng — `PAGE_SIZE + 1`. Tool chỉ **trả về 25** dòng cho model. Dòng dư ra là một "probe": nếu nó về tới, nghĩa là còn ít nhất một trang nữa, nên ta đặt `nextCursor` là id cuối cùng mà ta đang gửi đi. Nếu không về, ta đã đến cuối và `nextCursor` là `null`. Rẻ hơn nhiều so với query `COUNT(*)` riêng và dùng luôn index mà ta đang duyệt.

Xem nó chạy trên bảng 50 dòng:

```text
Turn 1: cursor = null
  DB returns 26 rows (probe came back) → ids [1..26]
  hasMore  = true
  pageRows = rows.slice(0, 25) = [1..25]
  nextCursor = "25"   (last id in pageRows — not 26)
  Tool returns: { rows: [1..25], nextCursor: "25" }

Between turns: same insert as before, new row sorts as id 26.

Turn 2: cursor = "25"
  SQL: SELECT ... WHERE id > 25 ORDER BY id ASC LIMIT 26
  DB returns 26 rows → ids [26..51]   ← id 26 is the inserted row, included naturally
  hasMore  = true if there were more, false if these are the last
  pageRows = [26..50]
  nextCursor = "50" (or null if there were exactly 26 left)
```

Cursor là *"dòng cuối cùng tôi đã trả về"*, không phải *"dòng tôi chưa cho xem"*. Đó là lý do ở turn 2 query là `WHERE id > 25` — bắt đầu từ vị trí *ngay sau* những gì caller đã thấy.

Phiên bản offset bỏ sót dòng mới vì *vị trí thứ 26 trong danh sách đã được sắp xếp đã bị dịch chuyển* giữa các turn. Phiên bản cursor neo vào `id > 25` thay vì "dòng số 26", nên các thao tác ghi song song không làm dịch cửa sổ. Bug tương tự cũng cắn bạn ở các feed `created_at DESC`, kết quả tìm kiếm, audit log — bất cứ chỗ nào có dòng mới xuất hiện trong lúc đang phân trang, mà chỗ nào cũng vậy cả.

Tóm vai trò trong một dòng: *Zod xác thực shape mà cursor phải có. Hàm tool của bạn quyết định cursor có ý nghĩa gì.*

## 02 — Filtering

Mẫu thứ hai là để model nói ra *cái gì* nó muốn, thay vì lấy hết về. Cám dỗ thường gặp là expose một tool nhận vào một chuỗi SQL. Đừng. Model không cần học SQL; nó cần một object mô tả ý định.

```ts
const filterSchema = z.object({                   // → ZodObject describing the allowed filter fields
  lapsedDays: z.number().int().positive().max(3650).optional(), // → integer 1..3650 or undefined
  tagIn: z.array(z.string().max(40)).max(10).optional(),        // → up to 10 tag strings (each ≤40 chars)
  emailDomain: z.string().max(80).optional(),     // → string ≤80 chars or undefined
  createdAfter: z.string().datetime().optional(), // → ISO 8601 string or undefined
});

export const searchCustomers = tool(              // → StructuredTool { name: 'customers.search' }
  async ({ filter }) => {
    const where: Prisma.CustomerWhereInput = {};  // → start with empty where clause
    if (filter.lapsedDays) where.lastOrderAt = { lt: daysAgo(filter.lapsedDays) }; // → { lt: Date 60d ago }
    if (filter.tagIn) where.tags = { hasSome: filter.tagIn };                       // → { hasSome: ['vip'] }
    if (filter.emailDomain) where.email = { endsWith: '@' + filter.emailDomain };   // → { endsWith: '@gmail.com' }
    if (filter.createdAfter) where.createdAt = { gte: new Date(filter.createdAfter) }; // → { gte: Date }
    const rows = await db.customers.findMany({ where, take: 25 }); // → Customer[] of size ≤ 25
    return JSON.stringify({ rows });              // → '{"rows":[{"id":"c_01",...}, ...]}'
  },
  {
    name: 'customers.search',                     // → identifier the model uses
    description: 'Search customers by lapsed-days, tags, email domain, or creation date.',
    schema: z.object({ filter: filterSchema }),   // → outer schema wrapping the filter object
  },
);
```

Zod schema làm hai việc cùng lúc. Nó nói cho model biết những filter nào đang sẵn có (schema là một phần của bề mặt expose ra của tool) và nó khoá chặt những gì hàm của bạn từng có thể nhận. Thân hàm là cái cầu chuyển từ *"ý định đã được xác thực"* sang *"query có tham số hoá"* — bạn là người dịch, không phải model.

<Callout type="warning" title="Đừng bao giờ nội suy filter của model vào SQL thô">
Hai năm gần đây, loại sự cố bảo mật AI lớn nhất chính là chuyện ai đó expose ra một tool nhận vào một mảnh SQL, một biểu thức JSONPath, hoặc một chuỗi `where` tự do. Sớm muộn gì model cũng sẽ phát ra `'; DROP TABLE customers; --` chỉ vì có người trong chat hỏi tử tế. Luôn parse bằng Zod, luôn dịch sang query có tham số hoá, luôn coi input từ model là thù địch. *Bạn* viết SQL; model chỉ mô tả nó muốn gì.
</Callout>

## 03 — Aggregation

Mẫu thứ ba là mẫu tiết kiệm nhiều token nhất: khi model muốn biết *về* dữ liệu chứ không phải bản thân dữ liệu, hãy expose một tool tính sẵn câu trả lời.

```ts
export const totalsBySegment = tool(              // → StructuredTool { name: 'orders.totals_by_segment' }
  async ({ segment }) => {
    const cohort = await db.customers.findMany({  // → Customer[] (id-only projection)
      where: segment === 'lapsed-pre-q4'          // → branch on segment
        ? { lastOrderAt: { lt: daysAgo(60) } }    // → 'lapsed' cohort: last order > 60d ago
        : { lastOrderAt: { gte: daysAgo(30) } },  // → 'active' cohort: ordered in last 30d
      select: { id: true },                       // → only need ids for the next step
    });
    const ids = cohort.map((c) => c.id);          // → ['c_01', 'c_02', ...]
    const orders = await db.orders.findMany({ where: { customerId: { in: ids } } }); // → Order[] for those customers
    const total = orders.reduce((s, o) => s + o.totalCents, 0); // → 318420 (sum of cents)
    return JSON.stringify({
      customerCount: ids.length,                  // → 47
      orderCount: orders.length,                  // → 53
      totalRevenueCents: total,                   // → 318420
      avgOrderValueCents: orders.length ? Math.round(total / orders.length) : 0, // → 6008
    });
    // → '{"customerCount":47,"orderCount":53,"totalRevenueCents":318420,"avgOrderValueCents":6008}'
  },
  {
    name: 'orders.totals_by_segment',
    description: 'Return revenue, order count, and average order value for a named customer segment.',
    schema: z.object({ segment: z.enum(['lapsed-pre-q4', 'active-30d']) }), // → only 2 allowed segments
  },
);
```

Một model hỏi *"các khách hàng lapsed đã tiêu bao nhiêu?"* không cần thấy mọi đơn hàng. Nó cần bốn con số. Tính sẵn bốn con số đó, trả về, và đoạn văn tổng hợp model viết ra cũng chất lượng y vậy — còn round-trip dùng 200 token thay vì 200.000.

Một heuristic hữu ích: nếu model từng xin tool các dòng dữ liệu thô chỉ để đếm chúng, bạn đang thiếu một tool aggregation. Thêm `count_by_tag`, `revenue_by_month`, `top_products_for_customer`. Mỗi cái chỉ mười lăm dòng TypeScript và tiết kiệm cả một context window.

<Recap>
- Pagination: dùng cursor, `limit` có chặn, trả về `{ rows, nextCursor }`. Chọn một hình dạng và xài ở mọi nơi.
- Filtering: một object `filter` có kiểu, được Zod xác thực, được hàm của bạn dịch sang query có tham số hoá.
- Aggregation: tính sẵn các bản tóm tắt để model gọi. Thắng dữ liệu thô về chi phí, độ trễ, và chất lượng câu trả lời.
- Đừng bao giờ nội suy output của model vào SQL, JSONPath, hay bất cứ bề mặt code nào khác. Parse, validate, dịch.
- Description tốt + schema tốt = model tự sắp xếp được chuỗi gọi nhiều tool đúng cách mà bạn không cần viết kịch bản.
</Recap>
