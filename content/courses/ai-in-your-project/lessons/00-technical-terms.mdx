## Models & inference

- **LLM (Large Language Model)** — the AI itself. Takes text in, produces text out. Examples: GPT-4, Claude, Llama, Mistral. In this course we run a small one locally via Ollama.
- **Parameter count / model size** — the number of learned weights in the neural network, written with a `B` for billions. `qwen2.5:3b` has 3 billion parameters; `llama3.1:8b` has 8 billion. More parameters generally means a smarter, more reliable model — but also more RAM, more disk, and slower per-token speed.

  Lesson 03 has you pull a 3B or 8B model. If you hit reliability issues on the 3B, the fix is often "swap the model name to the 8B" — same code, much better behaviour.

  _A concrete example._ Same input (`"Hi, our customer Phuoc <phuoc@savyu.com> hasn't ordered in 3 months — mark as lapsed."`), same Zod schema (`{ name, email, segment: enum['active','lapsed'] }`), same `withStructuredOutput` call — different model sizes, run 100 times each:

  | Model              | Correct    | Failed    | Latency  | $ per run |
  | ------------------ | ---------- | --------- | -------- | --------- |
  | `qwen2.5:3b` (3B)  | 78 / 100   | 22 / 100  | ~3.0 s   | free      |
  | `llama3.1:8b` (8B) | 97 / 100   | 3 / 100   | ~5.5 s   | free      |
  | `claude-sonnet`    | 100 / 100  | 0 / 100   | ~1.4 s   | ~$0.001   |

  The 22 failures on the 3B model break down as: 14× capitalisation drift (`"Lapsed"` instead of `"lapsed"`, fails the enum), 5× missing the `segment` field entirely, 3× hallucinating `segment: "inactive"` (not in the enum). All three failure modes are exactly the gotchas from lesson 08 §04.

  _The takeaway:_ you can run the **whole course** on a 3B model — it's slow and you'll hit retries — but if a lesson's structured-output demo flakes on you, change `'qwen2.5:3b'` to `'llama3.1:8b'` in the same `ChatOllama({ model: ... })` line and the failures usually go away. Save the hosted frontier model for production (lesson 37).

- **Token** — the unit a model reads and writes in. Roughly ¾ of an English word. _"Hello, world"_ is about 3 tokens. Pricing, context limits, and speed are all measured in tokens.
- **Context window** — the maximum number of tokens the model can "see" at once (your prompt + its reply). When you exceed it, older messages get cut. Bigger windows aren't free — they cost more and slow down.

- **Inference** — one round-trip to the model: send tokens in, get tokens out. _"Slow inference"_ means the model is taking a long time to reply.

  _A concrete example._ One inference call, timed:

  ```ts
  console.time('inference');
  const reply = await model.invoke('What is the capital of France?');
  console.timeEnd('inference');
  // → inference: 1247ms

  // reply.content: "The capital of France is Paris."
  ```

  Three rules of thumb fall straight out of that breakdown:

  - **Output length dominates.** Doubling the output roughly doubles inference time. That's why a "short answer" instruction in your prompt isn't just a style preference — it's a real speed-up.
  - **Input length matters far less.** Adding 100 tokens of system prompt costs ~10 ms of prefill. Adding 100 tokens to the *output* costs ~5 seconds at 50 ms/token.
  - **Model size is a multiplier on every output token.** A 3B model might emit at ~30 ms/token; an 8B at ~50 ms; a 14B at ~80 ms. Same 24-token reply, very different wall-clock times. (See the model-size chart further up.)
- **Temperature** — a knob from 0 to ~2 that controls randomness. `0` = same answer every time (good for tools and structured output). `0.7` = creative (good for drafting copy).
- **Streaming** — receiving the reply token-by-token as the model produces it, instead of waiting for the whole thing. Lesson 4 wires this up.

- **Provider** — the service that hosts the model. **Ollama** runs models on your laptop. **OpenAI**, **Anthropic**, **Google** host them in the cloud. LangChain lets you swap providers without rewriting your app (lesson 37).

## Prompts & messages

- **Prompt** — the text you send to the model. Casual usage; in code it's usually structured as messages.
- **System prompt** — a special "instructions to the AI" message at the top. Defines role, tone, hard rules. _"You are a helpful CMS assistant. Never invent customer IDs."_ Lesson 5.

<Demo title="Toggle the system prompt — same question, very different answers">
  <SystemPromptToggleDemo />
</Demo>

- **User message / assistant message** — the back-and-forth turns of a conversation. The chat history is just an array of these.
- **Few-shot examples** — including 1–5 worked examples in the prompt to show the model the pattern you want. _"Here's how I'd phrase a re-engagement email; now do one for this customer."_
- **Prompt engineering** — the skill of writing prompts that get reliable output. Less black-magic than the internet implies. Lesson 6.
- **Hallucination** — when the model confidently makes up a fact. The standard failure mode. Tools and structured output reduce it; they don't eliminate it.

## Tools & agents

- **Tool** (a.k.a. **function call**) — a TypeScript function the model is allowed to call, with a typed schema describing its arguments. The model says _"I want to call `searchCustomers({ tag: 'lapsed' })`"_; your code runs it; you hand the result back. Lesson 9.
- **Tool call** — one specific request from the model to invoke a tool, including the arguments it chose.
- **Read tool / write tool** — our own naming. Read tools fetch data (safe to auto-run). Write tools change the database or send messages (need confirmation).
- **Agent** — an LLM in a loop with tools. It reads, decides, calls a tool, reads the result, decides again, eventually replies.
- **ReAct loop** — _Reason + Act_. The classic agent pattern: model thinks → calls a tool → reads result → thinks again → repeat until done. Lessons 15 and 17.
- **Schema** (Zod) — a TypeScript-first way to declare _"this tool takes an object with `customerId: string` and `discount: number between 0 and 100"_. Both validates input and tells the model what's expected. Lessons 8 and 9.
- **Structured output** — forcing the model to reply with JSON that matches a schema, instead of free text. Lesson 8.
- **Idempotent** — calling the same operation twice has the same effect as calling it once. _"Mark order #42 as cancelled"_ is idempotent; _"Send an email"_ is not. Lesson 13.

## LangChain & LangGraph

- **LangChain** — a library that gives you one common interface across every model provider, plus helpers for prompts, tools, parsing, etc.
- **LangGraph** — built on top of LangChain. Lets you express your agent as a _graph_ of steps with state, branching, and pauses. The reason this course gets human-in-the-loop without writing a state machine by hand. Lesson 16+.
- **Node** — one step in the graph. Usually a function: _"call the model"_, _"run a tool"_, _"ask the user for confirmation"_.
- **Edge** — the connection from one node to the next. Plain edges always go to the same place; **conditional edges** branch based on state (_"if the next tool is a write, route to the confirmation node; otherwise auto-run"_).
- **State** — a typed object that flows through every node in the graph. Each node reads it and returns updates. This is where messages, pending tool calls, and confirmation status live.
- **StateGraph** — the LangGraph class you use to wire nodes and edges together.
- **Interrupt** — LangGraph's killer feature. A node calls `interrupt(...)`, the graph pauses and persists, your UI shows the user what's pending, the user clicks Approve, the graph resumes. Lesson 19.
- **Checkpointer** — the storage backend (in-memory, SQLite, Postgres) that lets a paused graph survive a server restart so the user can confirm later.

## Production & safety

- **Prompt injection** — an attack where the model reads attacker-controlled text (a customer note, a scraped page) and treats it as instructions. Mitigated with input sanitisation, output validation, and never giving the AI ambient authority. Lesson 21.
- **Guardrails** — checks around the model: input filters, output validators, rate limits. Belt-and-braces.
- **Eval** (evaluation) — a regression test for an AI. Run a fixed set of inputs, check the outputs against expected behaviour. Lesson 22.
- **Trace / observability** — logging every model call, tool call, and graph step so you can debug _"why did the agent do that?"_ days later. Lesson 23.
- **Token budget / cost cap** — a per-request or per-day ceiling on token usage so a runaway loop can't drain your credit card.
- **Rate limit** — provider-imposed cap on requests per minute. You'll hit it eventually; lesson 14 covers retries with backoff.
- **HITL (Human-in-the-loop)** — the design pattern where the AI proposes and the human approves before any irreversible action. The whole point of Part IV.

## Models & inference

- **LLM (Large Language Model)** — chính là "AI". Nhận chữ vào, sinh chữ ra. Ví dụ: GPT-4, Claude, Llama, Mistral. Trong khoá học này, ta chạy một model nhỏ ngay trên máy bạn qua Ollama.
- **Parameter count / model size (số tham số / kích cỡ model)** — số trọng số đã học được trong mạng neural, viết kèm chữ `B` nghĩa là tỷ. `qwen2.5:3b` có 3 tỷ tham số; `llama3.1:8b` có 8 tỷ. - **Token** — đơn vị mà model đọc và viết. Khoảng ¾ một từ tiếng Anh. _"Hello, world"_ tầm 3 token. Giá tiền, giới hạn context, và tốc độ đều được đo bằng token.
- **Context window** — số token tối đa mà model có thể "nhìn thấy" cùng lúc (prompt của bạn + phần trả lời của nó). Khi bạn vượt giới hạn, các message cũ sẽ bị cắt. Window lớn hơn không miễn phí — đắt tiền hơn và chậm hơn.

- **Inference** — một vòng đi-về tới model: gửi token vào, nhận token ra. _"Inference chậm"_ nghĩa là model đang trả lời lâu.

  ```ts
  // → inference: 1247ms

  // reply.content: "The capital of France is Paris."
  ```

  - **Độ dài output là chính.** Output dài gấp đôi thì thời gian inference cũng tăng gấp đôi. Đó là lý do vì sao chỉ thị "trả lời ngắn gọn" trong prompt không chỉ là sở thích về phong cách — nó là một mẹo tăng tốc thật.
  - **Độ dài input ít quan trọng hơn nhiều.** Thêm 100 token vào system prompt chỉ tốn ~10 ms ở giai đoạn prefill. Thêm 100 token vào *output* sẽ tốn ~5 giây với tốc độ 50 ms/token.
  - **Kích cỡ model là một hệ số nhân lên từng output token.** Model 3B có thể sinh ~30 ms/token; 8B ~50 ms; 14B ~80 ms. Cùng một câu trả lời 24 token, nhưng thời gian thực rất khác nhau. (Xem lại bảng kích cỡ model ở trên.)

## Prompt & message

- **Prompt** — đoạn văn bản bạn gửi tới model. Cách nói thông thường; trong code thì nó thường được cấu trúc thành các message.
- **System prompt** — một message đặc biệt kiểu "chỉ thị cho AI" đặt ở trên cùng. Định nghĩa vai trò, giọng điệu, các quy tắc cứng. _"You are a helpful CMS assistant. Never invent customer IDs."_ Bài 5.

- **User message / assistant message** — các lượt qua lại trong một cuộc hội thoại. Lịch sử chat chỉ là một mảng gồm những thứ này.
- **Few-shot examples** — đưa vào prompt 1–5 ví dụ đã giải sẵn để chỉ cho model pattern bạn muốn. _"Đây là cách tôi viết một email mời quay lại; giờ hãy viết một cái cho khách hàng này."_
- **Prompt engineering** — kỹ năng viết prompt để có output đáng tin cậy. Không thần bí như trên mạng hay đồn. Bài 6.
- **Hallucination (ảo giác)** — khi model tự tin bịa ra một sự thật. Kiểu lỗi tiêu biểu nhất. Tool và structured output làm giảm chứ không loại bỏ được.

## Tool & agent

- **Tool** (còn gọi là **function call**) — một hàm TypeScript mà model được phép gọi, kèm theo schema có kiểu rõ ràng mô tả các tham số. Model nói _"Tôi muốn gọi `searchCustomers({ tag: 'lapsed' })`"_; code của bạn chạy nó; bạn đưa kết quả ngược lại cho model. Bài 9.
- **Tool call** — một yêu cầu cụ thể từ model để gọi một tool, kèm theo các tham số nó đã chọn.
- **Read tool / write tool** — cách đặt tên riêng của khoá học. Read tool dùng để lấy dữ liệu (an toàn để tự động chạy). Write tool thay đổi database hoặc gửi tin nhắn (cần xác nhận).
- **Agent** — một LLM chạy trong một vòng lặp với các tool. Nó đọc, quyết định, gọi tool, đọc kết quả, lại quyết định, cuối cùng trả lời.
- **ReAct loop** — _Reason + Act_ (Suy luận + Hành động). Pattern agent kinh điển: model suy nghĩ → gọi tool → đọc kết quả → suy nghĩ tiếp → lặp lại cho đến khi xong. Bài 15 và 17.
- **Schema** (Zod) — cách khai báo theo kiểu "TypeScript trước" rằng _"tool này nhận một object có `customerId: string` và `discount: number trong khoảng 0 đến 100"_. Vừa validate input vừa cho model biết kỳ vọng đầu vào là gì. Bài 8 và 9.
- **Structured output** — ép model trả về JSON khớp với một schema, thay vì văn bản tự do. Bài 8.
- **Idempotent** — gọi cùng một thao tác hai lần thì hiệu ứng giống như gọi một lần. _"Đánh dấu order #42 là cancelled"_ là idempotent; _"Gửi một email"_ thì không. Bài 13.

## LangChain & LangGraph

- **LangChain** — một thư viện cung cấp một giao diện chung cho mọi provider model, kèm các tiện ích cho prompt, tool, parsing, v.v.
- **LangGraph** — xây trên LangChain. Cho phép bạn diễn đạt agent của mình dưới dạng một _graph_ gồm các bước có state, có rẽ nhánh, và có thể tạm dừng. Là lý do khoá học này có được pattern human-in-the-loop mà không phải tự tay viết một state machine. Từ bài 16 trở đi.
- **Node** — một bước trong graph. Thường là một hàm: _"gọi model"_, _"chạy một tool"_, _"hỏi user để xác nhận"_.
- **Edge** — kết nối từ node này sang node kế tiếp. Edge thường thì luôn đi đến cùng một đích; **conditional edge** (edge có điều kiện) thì rẽ nhánh dựa trên state (_"nếu tool tiếp theo là write thì đi tới node xác nhận; còn không thì tự động chạy"_).
- **State** — một object có kiểu rõ ràng, chảy xuyên qua mọi node trong graph. Mỗi node đọc nó và trả về các update. Đây là chỗ chứa message, các tool call đang chờ, và trạng thái xác nhận.
- **StateGraph** — class trong LangGraph mà bạn dùng để ráp node và edge lại với nhau.
- **Interrupt** — tính năng "đinh" của LangGraph. Một node gọi `interrupt(...)`, graph tạm dừng và lưu lại trạng thái, UI của bạn hiển thị cho user biết cái gì đang chờ, user bấm Approve, graph chạy tiếp. Bài 19.
- **Checkpointer** — backend lưu trữ (in-memory, SQLite, Postgres) giúp một graph đang tạm dừng sống sót qua lúc server khởi động lại, để user có thể xác nhận sau.

## Production & an toàn

- **Prompt injection** — một kiểu tấn công, trong đó model đọc văn bản do kẻ tấn công kiểm soát (ghi chú của khách hàng, một trang web được scrape về) và coi đó như là chỉ thị. Giảm thiểu bằng việc lọc input, validate output, và tuyệt đối không trao cho AI "ambient authority" (quyền hạn ngầm). Bài 21.
- **Guardrails** — các lớp kiểm tra bao quanh model: bộ lọc input, bộ validate output, rate limit. Kiểu "thắt lưng cộng đai".
- **Eval** (đánh giá) — bài test regression cho AI. Chạy một bộ input cố định, đối chiếu output với hành vi mong đợi. Bài 22.
- **Trace / observability** — log lại mọi cuộc gọi model, mọi tool call, mọi bước trong graph để mấy ngày sau bạn còn debug được _"vì sao agent lại làm vậy?"_. Bài 23.
- **Token budget / cost cap** — trần token theo từng request hoặc theo ngày để một vòng lặp "vỡ trận" không hút sạch hạn mức thẻ tín dụng của bạn.
- **Rate limit** — giới hạn do provider áp lên số request mỗi phút. Sớm muộn cũng đụng phải; bài 14 nói về retry kèm backoff.
- **HITL (Human-in-the-loop)** — pattern thiết kế trong đó AI đề xuất còn con người duyệt trước khi bất kỳ hành động không thể đảo ngược nào diễn ra. Đây chính là tinh thần của cả Phần IV.
