Remember the `while` loop from lesson 14? The agent thought, called a tool, thought again, called another tool, and stopped when it was done. It worked. But every new requirement — *"confirm before writes", "log every action", "route differently when the user is logged out"* — meant wedging another `if` branch into the same loop. After three or four additions, the function gets too dense to read.

In this lesson we rebuild that exact same loop as a **graph**. Same tools, same prompts, same model, same final behaviour. What changes is the *layout*: instead of one growing function, we have a small diagram made of named boxes (called **nodes**) connected by labeled arrows (called **edges**). Every step sits in a specific spot on the page; every decision is a labeled arrow you can point at.

Picture a small café with two stations:

- The **head barista** (`agent` node) reads the order and decides what to do next.
- The **prep helper** (`tools` node) executes whatever the barista asks for — steam milk, brew espresso, grab syrup — and hands the result back.

After every barista step, there's exactly one question: *does the barista need more prep, or is the drink ready to serve?* If more prep is needed, the order routes to the helper and comes back. If the drink is ready, it goes out and the run ends. That's the entire pattern — called **ReAct** (Reason, Act, Reason, Act, ... until the model decides it's done). For the rest of Part IV we'll bolt new behaviours onto this café: a "confirm before writing" gate, an audit log, persistence across restarts. Each new behaviour is a new node and a new arrow — never a new `if` inside a tangled function.

## 01 — What flows through the graph (the *state*)

As the barista and prep helper hand work back and forth, *something* travels with the order — the running transcript of who said what. In a `while` loop that "something" would just be a local variable. In a graph it gets promoted to a first-class object called the **state**.

For our basic ReAct graph, the state is comically simple: a single list of messages. Every node reads it; every node appends to it. That's the whole thing.

```ts
import { StateGraph, END } from '@langchain/langgraph';      // → graph builder + END sentinel
import { HumanMessage, type BaseMessage } from '@langchain/core/messages'; // → message types

type GraphState = { messages: BaseMessage[] };    // → state shape: just a transcript

const graph = new StateGraph<GraphState>({        // → empty StateGraph<GraphState>
  channels: {
    messages: { reducer: (a, b) => a.concat(b), default: () => [] }, // → append-merge with [] default
  },
});
```

Four words in that code recur all lesson, so it's worth grounding them now:

- **StateGraph** — the builder you give nodes and edges to. You'll call `.addNode()`, `.addEdge()`, then `.compile()` to turn it into something runnable.
- **Channel** (the `messages` key inside `channels`) — one named slot in the state. A graph can have several channels side by side (later we'll add `plan`, `pendingTool`, `audit`). This lesson uses exactly one: `messages`.
- **Reducer** — the function that answers *"when a node returns a new value for this channel, how do I merge it with the existing value?"*

  The subtle part — a node *never* returns the whole state. It only returns the slice it's contributing. The reducer is a two-argument function: `a` is the existing channel value, `b` is whatever the node just returned, and it must combine them into the new value. Walking through `(a, b) => a.concat(b)` with real data:

  ```ts
  // Before agentNode runs:
  state.messages = [HumanMessage("Who hasn't ordered in 60 days?")]   // ← this is `a`

  // agentNode returns its contribution:
  { messages: [AIMessage(tool_calls: [...])] }                         // ← this is `b`

  // Reducer runs: a.concat(b)  →  new state.messages:
  [HumanMessage("Who hasn't ordered..."), AIMessage(tool_calls: [...])]
  ```

  Why it matters: flip `a.concat(b)` to `(a, b) => b` and every node would overwrite the whole transcript with whatever it just returned. The customer's `HumanMessage` would disappear the moment the agent posted its reply, and on round 2 the model would walk in blind — no idea what the customer asked. *Append-merge* — stacking the new chunk onto the end of the old list — is the right default for a transcript that grows turn by turn. Other channels can need other reducers: later lessons add a `plan` channel where each new plan replaces the old one instead of stacking on top. For a conversation, stacking is what you want.
- **END** (imported alongside `StateGraph`) — a built-in finish line, not a node you write code for. Think of it as the slot the barista routes the order to when the drink is ready: the runtime sees `END` and shuts the run down instead of sending the order to another station.

For lesson 17, a transcript is all we need to faithfully reproduce the lesson 14 loop. Later lessons will grow the state to include a plan, a `pendingTool`, an audit trail — each as its own new channel.

## 02 — The agent node

The agent node is one model call. Bind the tools, invoke against the current message list, append whatever the model returns. The model decides on its own whether to emit `tool_calls`; the node doesn't care.

```ts
async function agentNode(state: GraphState): Promise<Partial<GraphState>> {
  const reply = await model.bindTools(allTools).invoke(state.messages); // → AIMessage (may have tool_calls)
  return { messages: [reply] };                   // → reducer appends to state.messages
}
```

If the reply has tool calls, the router will send us to the tools node next. If it doesn't, the router will send us to `END`. Either way, the agent node's job is done.

#### Example — what `agentNode` does on round 1

The runtime has just started. State coming in is the seed transcript — one `HumanMessage`:

```ts
state.messages = [
  HumanMessage("Who hasn't ordered in 60 days?")
]
```

Inside the node, `model.bindTools(allTools).invoke([HumanMessage(...)])` runs. The model sees `customers.search` among its bound tools and decides it needs to query the database. It returns:

```ts
AIMessage({
  content: "",                                    // ← no prose yet — the model is "thinking via a tool"
  tool_calls: [
    {
      id: "call_a3f1",                            // ← unique id assigned by the model
      name: "customers.search",
      args: { lastOrderBefore: "2026-03-17", limit: 100 }
    }
  ]
})
```

`agentNode` returns `{ messages: [aiMessage] }`. The reducer on the `messages` channel appends it. State after the node:

```ts
[
  HumanMessage("Who hasn't ordered in 60 days?"),
  AIMessage(tool_calls: [customers.search])      // ← state.messages.at(-1)
]
```

The conditional edge router (section 04) will read `state.messages.at(-1)`, see `tool_calls.length === 1`, and route the runtime to `tools`. If instead the model had returned an `AIMessage` with no `tool_calls` and a finished prose answer, the router would have routed to `END` and we'd be done in a single round.

## 03 — The tools node

The tools node looks at the most recent message — which the agent just appended — and runs each tool call it finds. Each result is a `ToolMessage` keyed back to the original call's `id`, so the model can match results to requests on the next turn.

```ts
import { ToolMessage } from '@langchain/core/messages'; // → ToolMessage class for tool results

async function toolsNode(state: GraphState): Promise<Partial<GraphState>> {
  const last = state.messages.at(-1);             // → last AIMessage from agent node
  const calls = last?.tool_calls ?? [];           // → ToolCall[] (may be empty if router misfired)
  const out: ToolMessage[] = [];                  // → accumulator for results
  for (const call of calls) {
    const tool = TOOL_REGISTRY[call.name];        // → StructuredTool registered at boot
    const result = await tool.invoke(call.args);  // → string result of the tool execution
    out.push(new ToolMessage({ tool_call_id: call.id, content: result })); // → matches result back to call.id
  }
  return { messages: out };                       // → ToolMessage[] appended to transcript
}
```

`TOOL_REGISTRY` is just a `Record<string, Tool>` you build once at startup from the same tools you bound to the model. The reducer on `messages` appends each `ToolMessage` to the transcript automatically.

#### Example — what `toolsNode` does after the agent decides

Picking up exactly where the agent-node example left off. State coming in:

```ts
state.messages = [
  HumanMessage("Who hasn't ordered in 60 days?"),
  AIMessage({                                     // ← state.messages.at(-1)
    content: "",
    tool_calls: [
      { id: "call_a3f1", name: "customers.search", args: { lastOrderBefore: "2026-03-17", limit: 100 } }
    ]
  })
]
```

Inside the node, four things happen in order:

1. **Find the trigger.** `state.messages.at(-1)` → the `AIMessage` above.
2. **Pull the calls.** `last.tool_calls` → an array of one `ToolCall`.
3. **Execute each call against the registry.** For this single call: `TOOL_REGISTRY['customers.search'].invoke({ lastOrderBefore: '2026-03-17', limit: 100 })`. The tool returns a string (typical pattern — most tools return JSON-as-string):

   ```json
   [
     {"id":"c_42","email":"linh@example.com","lastOrderAt":"2025-12-01"},
     {"id":"c_57","email":"trang@example.com","lastOrderAt":"2025-11-22"}
   ]
   ```

4. **Wrap each result in a `ToolMessage`, keyed by `tool_call_id`.** The id is what lets the model match results back to its original requests on the next round.

   ```ts
   new ToolMessage({
     tool_call_id: "call_a3f1",                   // ← MUST match the call's id
     content: '[{"id":"c_42",...},{"id":"c_57",...}]'
   })
   ```

`toolsNode` returns `{ messages: [toolMessage] }`. The reducer appends. State after the node:

```ts
[
  HumanMessage("Who hasn't ordered in 60 days?"),
  AIMessage(tool_calls: [customers.search]),
  ToolMessage(tool_call_id: "call_a3f1", content: "[{...},{...}]")  // ← .at(-1)
]
```

The unconditional edge `tools → agent` fires next. The runtime loops back to `agentNode`. On round 2 the model now sees the customer list at the end of the transcript and writes a prose answer (no more `tool_calls`); the router routes to `END`; the run terminates.

<Callout type="tip" title="Why the `tool_call_id` matters">
If the agent had emitted **two** tool calls in one `AIMessage` (parallel reads — say `customers.search` AND `orders.totalThisWeek`), `toolsNode` would loop twice and emit two `ToolMessage`s. Each carries the `tool_call_id` of the originating call. On the next round, the model sees both results in the transcript and pairs them up by id — *"`call_a3f1` was the customers query, this is its result; `call_b7e2` was the orders query, that's its result."* Drop the id and the model has to guess from positional order — which works until it doesn't (a tool errors out and shifts the array).
</Callout>

## 04 — The one decision: loop or stop?

A graph this small has exactly one decision to make: *after the barista speaks, do we send the order back to the prep helper, or is the drink ready to serve?* That decision lives in a small function called a **router** (in LangGraph terminology, the function attached to a *conditional edge*). It looks at the current state and returns either the name of the next node, or the special value `END`.

```ts
import { isAIMessage } from '@langchain/core/messages'; // → type guard for AIMessage

function shouldContinue(state: GraphState): 'tools' | typeof END {
  const last = state.messages.at(-1);             // → last message (AIMessage if agent just ran)
  if (last && isAIMessage(last) && (last.tool_calls?.length ?? 0) > 0) return 'tools'; // → loop back if pending calls
  return END;                                     // → otherwise stop
}

graph
  .addNode('agent', agentNode)                    // → registers 'agent'
  .addNode('tools', toolsNode)                    // → registers 'tools'
  .setEntryPoint('agent')                         // → graph starts here
  .addConditionalEdges('agent', shouldContinue, { tools: 'tools', [END]: END }) // → branch on shouldContinue
  .addEdge('tools', 'agent');                     // → after tools, always loop back

const app = graph.compile();                      // → CompiledStateGraph — runnable

const final = await app.invoke({                  // → final GraphState
  messages: [new HumanMessage('Who hasn’t ordered in 60 days?')], // → seed transcript
});
// → final.messages.at(-1) is the AIMessage with the prose answer
```

Read the wiring like a recipe:

1. `addNode('agent', agentNode)` — register the barista station.
2. `addNode('tools', toolsNode)` — register the prep helper station.
3. `setEntryPoint('agent')` — every run begins at the barista.
4. `addConditionalEdges('agent', shouldContinue, { tools: 'tools', [END]: END })` — after the barista, the router decides where to go. If `shouldContinue` returns `'tools'`, head to the prep station; if it returns `END`, stop.
5. `addEdge('tools', 'agent')` — after the prep station, *always* return to the barista. No decision needed; the barista is the one who reads the result and decides what's next.
6. `graph.compile()` — freeze the structure into a runnable.
7. `app.invoke({ messages: [...] })` — feed in the seed transcript and let the graph run until something returns `END`.

That's the entire ReAct pattern in nine lines of edge declarations. The behaviour you saw in lesson 14 — model thinks, calls tools, loops, eventually answers — is now visible *as a shape* on the page instead of folded inside a `while` body.

Cái thay đổi là *cách bố trí*: thay vì một function cứ phình ra, ta có một sơ đồ nhỏ gồm các hộp có tên (gọi là **node**) nối với nhau bằng các mũi tên có nhãn (gọi là **edge**). Mỗi bước nằm ở một vị trí cụ thể trên trang giấy; mỗi quyết định là một mũi tên bạn có thể chỉ tay vào.

Hình dung một quán cà phê nhỏ với hai trạm:

- **Barista chính** (`agent` node) đọc order rồi quyết định bước tiếp theo.
- **Phụ pha chế** (`tools` node) làm bất cứ việc gì barista yêu cầu — đánh sữa, pha espresso, lấy syrup — rồi trả kết quả lại.

Sau mỗi lượt barista, đúng một câu hỏi: *barista có cần phụ pha chế làm thêm gì không, hay ly đã sẵn sàng để giao?* Nếu cần thêm, order chạy qua phụ pha chế rồi quay về. Nếu xong, ly được giao và run kết thúc. Đó là toàn bộ mô hình — gọi là **ReAct** (Reason, Act, Reason, Act, ... cho tới khi model quyết định xong). Trong phần còn lại của Phần IV, ta sẽ lắp thêm nhiều thứ vào "quán" này: một cổng "xác nhận trước khi ghi", một sổ audit, persistence khi restart. Mỗi thứ lắp thêm là một node mới + một mũi tên mới — chứ không phải một nhánh `if` mới trong một function rối rắm.

## 01 — Cái gì trôi qua graph (đó gọi là *state*)

Khi barista và phụ pha chế đưa qua đưa lại công việc, *có thứ gì đó* đi cùng với order — chính là transcript hội thoại ai vừa nói gì. Trong một vòng `while` thì "thứ đó" chỉ là một biến cục bộ. Trong một graph thì nó được "lên hạng" thành một đối tượng chính danh tên là **state** (trạng thái).

Với graph ReAct cơ bản, state đơn giản đến mức buồn cười: một danh sách messages duy nhất. Mỗi node đọc danh sách; mỗi node append vào danh sách. Toàn bộ là thế.

```ts
import { StateGraph, END } from '@langchain/langgraph';      // → graph builder + END sentinel
import { HumanMessage, type BaseMessage } from '@langchain/core/messages'; // → message types

type GraphState = { messages: BaseMessage[] };    // → state shape: just a transcript

const graph = new StateGraph<GraphState>({        // → empty StateGraph<GraphState>
  channels: {
    messages: { reducer: (a, b) => a.concat(b), default: () => [] }, // → append-merge with [] default
  },
});
```

Bốn từ trong đoạn code đó sẽ tái xuất suốt bài, nên cần định nghĩa ngay bây giờ:

- **StateGraph** — bộ "builder" để bạn lắp node và edge vào. Bạn sẽ gọi `.addNode()`, `.addEdge()`, rồi `.compile()` để biến nó thành một thứ chạy được.
- **Channel** (cái `messages` bên trong `channels`) — một "ngăn có tên" trong state. Một graph có thể có nhiều channel cùng lúc (các bài sau ta sẽ thêm `plan`, `pendingTool`, `audit`). Bài này có đúng một: `messages`.
- **Reducer** — hàm trả lời câu hỏi *"khi một node return giá trị mới cho channel này, ta gộp với giá trị cũ ra sao?"*

  Điểm tinh tế ở đây — mỗi node *không* trả về toàn bộ state mới. Nó chỉ trả về phần *nó đóng góp thêm*. Reducer là một hàm nhận hai đối số: `a` là giá trị hiện có trong channel, `b` là cái node vừa return, và nó phải gộp hai cái thành giá trị mới. Đi qua `(a, b) => a.concat(b)` với dữ liệu thật:

  ```ts
  // Trước khi agentNode chạy:
  state.messages = [HumanMessage("Khách nào chưa đặt hàng 60 ngày qua?")]   // ← đây là `a`

  // agentNode return phần đóng góp của nó:
  { messages: [AIMessage(tool_calls: [...])] }                                // ← đây là `b`

  // Reducer chạy: a.concat(b)  →  state.messages mới:
  [HumanMessage("Khách nào chưa đặt hàng..."), AIMessage(tool_calls: [...])]
  ```

  Vì sao quan trọng: đổi `a.concat(b)` thành `(a, b) => b` thì mỗi node sẽ ghi đè toàn bộ transcript bằng đúng cái nó vừa return. `HumanMessage` của khách sẽ biến mất ngay khi agent trả lời lượt đầu, và ở vòng 2 model bước vào trong trạng thái mù — không còn biết khách đã hỏi gì. *Append-merge* — nối phần mới vào cuối danh sách cũ — mới là default đúng cho một transcript lớn dần qua từng lượt. Channel khác có thể cần reducer khác: các bài sau ta thêm channel `plan` mà mỗi plan mới sẽ thay plan cũ chứ không xếp chồng. Với hội thoại, xếp chồng mới là cái bạn cần.
- **END** (import cùng với `StateGraph`) — một vạch đích có sẵn, không phải node bạn phải tự viết code. Hình dung nó là chỗ barista đưa order tới khi ly đã sẵn sàng để giao: runtime thấy `END` và đóng run lại, thay vì gửi order sang một trạm khác.

Với bài 17, một transcript là đủ để tái hiện trung thành vòng lặp ở bài 14. Các bài sau ta sẽ mở rộng state để có thêm plan, `pendingTool`, audit trail — mỗi cái là một channel mới riêng.

## 02 — Agent node

Agent node là một lần gọi model. Bind tool vào, invoke với danh sách message hiện tại, append bất cứ thứ gì model trả về. Model tự quyết định có phát ra `tool_calls` hay không; node không quan tâm.

```ts
async function agentNode(state: GraphState): Promise<Partial<GraphState>> {
  const reply = await model.bindTools(allTools).invoke(state.messages); // → AIMessage (may have tool_calls)
  return { messages: [reply] };                   // → reducer appends to state.messages
}
```

Nếu reply có tool call, router sẽ đưa ta tới tools node. Nếu không, router sẽ đưa ta tới `END`. Dù sao, công việc của agent node đã xong.

#### Ví dụ — `agentNode` làm gì ở vòng 1

Runtime vừa bắt đầu. State đi vào là transcript hạt giống — đúng một `HumanMessage`:

```ts
state.messages = [
  HumanMessage("Khách hàng nào chưa đặt hàng 60 ngày qua?")
]
```

Bên trong node, `model.bindTools(allTools).invoke([HumanMessage(...)])` chạy. Model thấy có `customers.search` trong danh sách tool đã bind và quyết định cần truy vấn DB. Nó trả về:

```ts
AIMessage({
  content: "",                                    // ← chưa có prose — model đang "nghĩ qua tool"
  tool_calls: [
    {
      id: "call_a3f1",                            // ← id duy nhất do model tạo
      name: "customers.search",
      args: { lastOrderBefore: "2026-03-17", limit: 100 }
    }
  ]
})
```

`agentNode` return `{ messages: [aiMessage] }`. Reducer của channel `messages` append nó vào. State sau khi node chạy xong:

```ts
[
  HumanMessage("Khách hàng nào chưa đặt hàng 60 ngày qua?"),
  AIMessage(tool_calls: [customers.search])      // ← state.messages.at(-1)
]
```

Router của edge có điều kiện (mục 04) sẽ đọc `state.messages.at(-1)`, thấy `tool_calls.length === 1`, và đẩy runtime sang `tools`. Nếu thay vào đó model trả về `AIMessage` không có `tool_calls` và đã viết câu trả lời prose hoàn chỉnh, router sẽ đẩy thẳng tới `END` — xong trong đúng một vòng.

## 03 — Tools node

Tools node nhìn vào message mới nhất — cái mà agent vừa append — và chạy mỗi tool call nó tìm thấy. Mỗi kết quả là một `ToolMessage` được khoá lại theo `id` của call gốc, để model có thể đối chiếu kết quả với request ở lượt sau.

```ts
import { ToolMessage } from '@langchain/core/messages'; // → ToolMessage class for tool results

async function toolsNode(state: GraphState): Promise<Partial<GraphState>> {
  const last = state.messages.at(-1);             // → last AIMessage from agent node
  const calls = last?.tool_calls ?? [];           // → ToolCall[] (may be empty if router misfired)
  const out: ToolMessage[] = [];                  // → accumulator for results
  for (const call of calls) {
    const tool = TOOL_REGISTRY[call.name];        // → StructuredTool registered at boot
    const result = await tool.invoke(call.args);  // → string result of the tool execution
    out.push(new ToolMessage({ tool_call_id: call.id, content: result })); // → matches result back to call.id
  }
  return { messages: out };                       // → ToolMessage[] appended to transcript
}
```

`TOOL_REGISTRY` chỉ là một `Record<string, Tool>` bạn build một lần lúc khởi động từ chính những tool bạn đã bind vào model. Reducer trên `messages` tự động append từng `ToolMessage` vào transcript.

#### Ví dụ — `toolsNode` làm gì sau khi agent quyết định

Bắt đúng nơi ví dụ agent node bỏ lại. State đi vào:

```ts
state.messages = [
  HumanMessage("Khách hàng nào chưa đặt hàng 60 ngày qua?"),
  AIMessage({                                     // ← state.messages.at(-1)
    content: "",
    tool_calls: [
      { id: "call_a3f1", name: "customers.search", args: { lastOrderBefore: "2026-03-17", limit: 100 } }
    ]
  })
]
```

Bên trong node, bốn việc xảy ra theo thứ tự:

1. **Tìm cái "kích hoạt".** `state.messages.at(-1)` → `AIMessage` ở trên.
2. **Rút danh sách call ra.** `last.tool_calls` → mảng một `ToolCall`.
3. **Thực thi từng call qua registry.** Với call duy nhất này: `TOOL_REGISTRY['customers.search'].invoke({ lastOrderBefore: '2026-03-17', limit: 100 })`. Tool trả về một string (pattern phổ biến — hầu hết tool trả JSON-stringify):

   ```json
   [
     {"id":"c_42","email":"linh@example.com","lastOrderAt":"2025-12-01"},
     {"id":"c_57","email":"trang@example.com","lastOrderAt":"2025-11-22"}
   ]
   ```

4. **Bọc mỗi kết quả vào một `ToolMessage`, keyed theo `tool_call_id`.** Cái id này chính là thứ giúp model đối chiếu kết quả với request ban đầu ở vòng kế tiếp.

   ```ts
   new ToolMessage({
     tool_call_id: "call_a3f1",                   // ← BẮT BUỘC trùng id của call
     content: '[{"id":"c_42",...},{"id":"c_57",...}]'
   })
   ```

`toolsNode` return `{ messages: [toolMessage] }`. Reducer append. State sau khi node chạy xong:

```ts
[
  HumanMessage("Khách hàng nào chưa đặt hàng 60 ngày qua?"),
  AIMessage(tool_calls: [customers.search]),
  ToolMessage(tool_call_id: "call_a3f1", content: "[{...},{...}]")  // ← .at(-1)
]
```

Edge vô điều kiện `tools → agent` fire tiếp theo. Runtime quay lại `agentNode`. Ở vòng 2, model giờ đã thấy danh sách khách hàng ở cuối transcript và viết câu trả lời prose (không còn `tool_calls` nữa); router đẩy runtime tới `END`; run kết thúc.

<Callout type="tip" title="Vì sao `tool_call_id` quan trọng?">
Nếu agent emit **hai** tool call trong cùng một `AIMessage` (đọc song song — ví dụ `customers.search` VÀ `orders.totalThisWeek`), `toolsNode` sẽ lặp hai lần và emit hai `ToolMessage`. Mỗi cái mang `tool_call_id` của call gốc. Ở vòng kế, model thấy cả hai kết quả trong transcript và ghép theo id — *"`call_a3f1` là query khách hàng, đây là kết quả của nó; `call_b7e2` là query orders, đó là kết quả của nó."* Bỏ id đi và model phải đoán theo thứ tự xuất hiện — vẫn chạy đúng cho tới khi không đúng (một tool lỗi và xáo mảng).
</Callout>

## 04 — Quyết định duy nhất: lặp tiếp hay dừng?

Một graph nhỏ như vậy chỉ có đúng một quyết định cần đưa ra: *sau khi barista nói xong, ta gửi order quay lại phụ pha chế, hay ly đã sẵn sàng để giao?* Quyết định đó nằm trong một function nhỏ gọi là **router** (trong từ vựng LangGraph, đó là function gắn vào một *conditional edge*). Nó nhìn vào state hiện tại rồi return tên của node tiếp theo, hoặc giá trị đặc biệt `END`.

```ts
import { isAIMessage } from '@langchain/core/messages'; // → type guard for AIMessage

function shouldContinue(state: GraphState): 'tools' | typeof END {
  const last = state.messages.at(-1);             // → last message (AIMessage if agent just ran)
  if (last && isAIMessage(last) && (last.tool_calls?.length ?? 0) > 0) return 'tools'; // → loop back if pending calls
  return END;                                     // → otherwise stop
}

graph
  .addNode('agent', agentNode)                    // → registers 'agent'
  .addNode('tools', toolsNode)                    // → registers 'tools'
  .setEntryPoint('agent')                         // → graph starts here
  .addConditionalEdges('agent', shouldContinue, { tools: 'tools', [END]: END }) // → branch on shouldContinue
  .addEdge('tools', 'agent');                     // → after tools, always loop back

const app = graph.compile();                      // → CompiledStateGraph — runnable

const final = await app.invoke({                  // → final GraphState
  messages: [new HumanMessage('Who hasn’t ordered in 60 days?')], // → seed transcript
});
// → final.messages.at(-1) is the AIMessage with the prose answer
```

Đọc phần wiring như đọc một công thức:

1. `addNode('agent', agentNode)` — đăng ký trạm barista.
2. `addNode('tools', toolsNode)` — đăng ký trạm phụ pha chế.
3. `setEntryPoint('agent')` — mọi lần chạy đều bắt đầu ở barista.
4. `addConditionalEdges('agent', shouldContinue, { tools: 'tools', [END]: END })` — sau barista, router quyết định đi đâu. Nếu `shouldContinue` return `'tools'`, đi tới trạm phụ pha chế; nếu return `END`, dừng.
5. `addEdge('tools', 'agent')` — sau trạm phụ pha chế, *luôn luôn* quay về barista. Không cần quyết định gì cả; barista là người sẽ đọc kết quả và quyết bước tiếp theo.
6. `graph.compile()` — đóng băng cấu trúc thành một thứ có thể chạy được.
7. `app.invoke({ messages: [...] })` — đưa transcript hạt giống vào và để graph chạy tới khi có cái gì đó return `END`.

Đó là toàn bộ mô hình ReAct trong chín dòng khai báo edge. Hành vi bạn thấy ở bài 14 — model suy nghĩ, gọi tool, lặp, cuối cùng trả lời — giờ đã hiện ra *như một hình dạng* trên trang giấy, thay vì bị gấp lại bên trong thân một vòng `while`.

## Bạn đang có gì lúc này

Về mặt kiến trúc, bạn có đúng thứ bài 14 đã có — cộng thêm chỗ để lớn lên. Bài kế tiếp tách tools node thành hai. Bài sau nữa thêm một interrupt trước các thao tác ghi, đó là điểm nhấn của toàn bộ khoá học. Bài sau nữa ráp workflow khách hàng lapsed từ đầu tới cuối, có xác nhận và audit log, trên cùng cái xương sống graph này.

<Recap>
- State cho một ReAct agent cơ bản chỉ là `{ messages }` — một transcript lớn dần theo từng node.
- Agent node gọi model với các tool đã bind và append reply. Toàn bộ công việc của nó là thế.
- Tools node nhìn message cuối, chạy từng `tool_call`, và append một `ToolMessage` tương ứng.
- Router quyết định lặp hay dừng bằng cách kiểm tra message cuối có tool call không. Một edge có điều kiện, hai đích đến.
- Compile và invoke. Graph tái hiện vòng lặp bài 14 với các bước được đặt tên và chỗ để đặt mọi mối lo trong tương lai.
- Đòn bẩy của việc chuyển từ `while` sang graph là: hành vi mới trở thành node mới — không phải nhánh mới bên trong một hàm cứ to dần.
</Recap>
