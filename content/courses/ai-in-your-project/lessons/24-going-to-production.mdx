You've shipped an AI Operator that runs locally on Ollama. It plans, it asks for confirmation, it commits to a real database, it audits, it tests. Now ship it for real. The gap between your laptop and a production URL is smaller than it looks — it's an import, an env var, a deploy command, a couple of guard rails — but every step has a place where a small mistake quietly costs money. This lesson walks the gap.

The promise is small and concrete: by the end of this page, you've swapped the model provider, configured the secrets, deployed to Vercel, capped the budget, and added a rate limiter. That is the floor for a production AI feature. Everything else — multi-tenant policies, audit retention, per-customer fine-tuning — is a layer on top.

## 01 — Provider swap

Ollama was the right choice for development: free, local, fast iteration, no rate-limit surprises while you're writing tests. It is the wrong choice for production: it doesn't ship to Vercel, it would crush a small VPS, and the open-weights models lag the hosted frontier. Swap to a hosted provider with one import:

```ts
// Before: dev on your laptop
import { ChatOllama } from '@langchain/ollama';   // → local Ollama adapter
const llm = new ChatOllama({ model: 'llama3.1:8b', temperature: 0 }); // → ChatOllama instance hitting localhost

// After: production
import { ChatAnthropic } from '@langchain/anthropic'; // → hosted Anthropic adapter
const llm = new ChatAnthropic({
  model: 'claude-sonnet-4',                       // → hosted frontier model
  apiKey: process.env.ANTHROPIC_API_KEY,          // → secret loaded from env
  temperature: 0,                                 // → deterministic for tool calling
});
// → llm: same BaseChatModel interface — every downstream call site is unchanged
```

The interface is the same. Your graph code does not change. Your tools do not change. Your tests, if you wrote them with `FakeListChatModel`, also do not change. This is the payoff for sticking to the LangChain abstraction the whole way through — the swap is a one-line edit and a re-deploy.

(`@langchain/openai` and `@langchain/google-genai` are the same shape. Pick on price and capability; switching providers later is another one-line edit.)

## 02 — Env vars

```bash
# .env.local for development
ANTHROPIC_API_KEY=sk-ant-api03-...                # → loaded into process.env by Next.js / dotenv

# Vercel: project settings → environment variables
# Add ANTHROPIC_API_KEY for the Production environment.   # → injected at build/runtime
# Optionally add LANGSMITH_API_KEY and LANGSMITH_TRACING=true.  # → flips on tracing
```

Two rules. Never commit a real key to git — your editor probably has a hook that complains; trust it. Never reuse the same key across dev and prod — rotate dev keys liberally and keep the prod key tightly scoped. Most providers let you set per-key rate limits in their dashboard; a dev key with a $5/day cap stays out of trouble even if you accidentally `console.log` it on a Tuesday.

## 03 — Deploy

```bash
vercel deploy --prod                              # → builds and promotes the deployment to the production URL
```

That's the command. The interesting parts happen around it. Vercel's Functions are stateless; your in-memory `MemorySaver` checkpointer (lesson 20) does not survive between requests. Switch it to a Postgres-backed checkpointer before the first deploy, or your interrupts will lose state every time the function cold-starts:

```ts
import { PostgresSaver } from '@langchain/langgraph-checkpoint-postgres'; // → durable graph checkpointer

const checkpointer = PostgresSaver.fromConnString(process.env.POSTGRES_URL!); // → PostgresSaver instance
await checkpointer.setup();                       // → ensures checkpoint tables exist (idempotent)

const graph = builder.compile({ checkpointer });  // → CompiledStateGraph that persists state to Postgres
// → interrupts now survive Vercel Function cold starts and tab closes
```

Your audit log table from lesson 12 lives in the same Postgres. Reuse the connection. One database is enough for a small app.

## 04 — Rate limits, three flavours

Three layers, each catching a different kind of accident.

**Provider rate limits** are not yours to set; they are the API's own. Anthropic, OpenAI, and Google all return 429 on overrun. Wrap your LLM calls in a retry-with-backoff (lesson 14) and you'll handle most transient cases without the user noticing.

**Per-user rate limits** are yours. A simple sliding-window counter — say, 50 requests per user per hour — keeps a single curious user from accidentally setting the entire month's budget on fire. Vercel KV or Upstash Redis is enough; a few lines of middleware in front of the agent handler.

**Global circuit-breakers** are the kill-switch from lesson 23. A daily cap, checked on every request, that pauses the whole agent if today's spend exceeds the cap. The user gets a friendly *"AI is paused for the day"* card instead of a 500.

All three together is the floor. The first time one of them fires in production, you will be glad it did.

## 05 — Cost monitoring

You wrote the callback handler in lesson 23. Wire it on every model in production. Build the three queries (yesterday's spend, top spenders, daily trend) into a small admin page or a scheduled email; you don't need a fancy dashboard, you need numbers in front of your face often enough that surprises become impossible.

<Callout type="tip">
  For your first production deploy, set the daily budget cap to **$10/day** and walk away. If you hit it, your prompt or your loop has a bug — find the bug, then raise the cap. The cap is not what limits your business; it is what protects you while you find the limits of the code.
</Callout>

## 04 — Rate limit, ba hương vị

Ba lớp, mỗi lớp bắt một kiểu tai nạn khác nhau.

**Rate limit từ nhà cung cấp** không phải do bạn đặt; là của API. Anthropic, OpenAI và Google đều trả 429 khi vượt mức. Bọc lệnh gọi LLM của bạn bằng retry-with-backoff (bài 14) là bạn đã xử lý được hầu hết các ca tạm thời mà user không hề hay biết.

**Rate limit theo user** là của bạn. Một bộ đếm sliding window đơn giản — ví dụ 50 request/user/giờ — sẽ ngăn một user tò mò vô tình đốt cháy ngân sách cả tháng. Vercel KV hoặc Upstash Redis là đủ; vài dòng middleware đặt trước handler agent.

**Circuit-breaker toàn cục** là kill-switch từ bài 23. Một cap theo ngày, kiểm tra mỗi request, tạm dừng toàn bộ agent nếu chi tiêu hôm nay vượt cap. User nhận được một thẻ thân thiện *"AI tạm nghỉ trong ngày"* thay vì lỗi 500.

Cả ba cùng nhau mới là cái sàn. Lần đầu tiên một trong số chúng kích hoạt trong production, bạn sẽ thấy may là có nó.
