The model will retry. It will retry because a network call hiccupped and the orchestrator re-emitted the tool call. It will retry because a streaming connection dropped halfway through. It will retry because, on rare and unsettling days, it second-guessed itself and called the same tool twice in the same turn for reasons no log will ever explain. The first time you see your *"create promo"* tool fire two and a half times for one user request, you will understand idempotency in your bones. Better to understand it now.

The fix is not making the retries stop — they're going to happen, sometimes legitimately. The fix is making retries **safe**: writing your tools so that a retry of the same logical operation produces the same outcome as the first call did. One row, not two. One charge, not two. One email, not two.

## The simplest idempotency key

The pattern is borrowed wholesale from webhook design and payment APIs. Every logical operation gets a **request key** — a UUID generated by the orchestrator before the tool call — and the tool stores that key alongside the row it writes. Before inserting, the tool looks up the key. Found? Return the existing row. Not found? Insert the new one and remember the key.

```ts
export const createPromo = tool(                  // → StructuredTool { name: 'promotions.create' }
  async ({ code, discountPct, requestKey }) => {
    const existing = await db.promotions.findUnique({ where: { requestKey } }); // → Promotion | null
    if (existing) {
      return JSON.stringify({ ok: true, deduped: true, promo: existing });      // → '{"ok":true,"deduped":true,"promo":{...}}' on retry
    }
    const promo = await db.promotions.create({    // → newly inserted Promotion row
      data: { code, discountPct, requestKey, createdAt: new Date() },
    });
    return JSON.stringify({ ok: true, deduped: false, promo }); // → '{"ok":true,"deduped":false,"promo":{...}}' on first call
  },
  {
    name: 'promotions.create',                    // → identifier the model uses
    description: 'Create a discount promo. Pass a stable requestKey to make retries safe.',
    schema: z.object({
      code: z.string().min(3).max(40),            // → string of 3..40 chars
      discountPct: z.number().int().min(1).max(75), // → integer 1..75
      requestKey: z.string().uuid(),              // → UUID — orchestrator fills this before invocation
    }),
  },
);
```

The `requestKey` is part of the tool's schema, which means the call site has to provide it. The crucial detail: **the model does not pick the value on its own.** Three patterns for where the key comes from, in rough order of how often you'll reach for them.

### Pattern A — the orchestrator supplies it (default)

A small wrapper around `agent.invoke` (or your LangGraph `tools` node) intercepts every tool call. For each *logical operation* it mints a UUID once, stores it on the in-flight turn, and re-uses the same UUID across every retry the model emits. The tool body just receives `requestKey` as a normal argument.

```ts
async function withIdempotency(toolName: string, rawArgs: unknown, exec: (args: any) => Promise<unknown>) {
  const args: any = rawArgs;                      // → freshly-validated tool args from the model
  if (!args.requestKey) {
    // One key per (turnId + toolName + stableHash(args)). Same args → same key,
    // even if the model fires the tool three times in the same turn.
    args.requestKey = sha256(`${currentTurnId()}:${toolName}:${stableStringify(args)}`); // → deterministic 64-char hex
  }
  return exec(args);                              // → result of the underlying tool execution
}
```

The model can retry as much as it wants; the wrapper produces the same key for the same args, the dedup table catches duplicates, the user never sees a double-charge. This is the default and what `createPromo` above assumes.

### Pattern B — the tool derives it from inputs

When the args themselves are enough to identify *one* legitimate operation, the tool can build the key in the function body. No wrapper needed.

```ts
async function chargeOrder({ orderId, amountCents }) {
  const requestKey = `charge:${orderId}:${amountCents}`; // → 'charge:7041:2500' — derived from inputs
  const existing = await db.charges.findUnique({ where: { requestKey } }); // → Charge | null
  if (existing) return existing;                  // → return previous Charge on retry
  return await db.charges.create({ data: { orderId, amountCents, requestKey } }); // → new Charge on first call
}
```

This is fine when there is no realistic case where you want two of the same charge in quick succession. *"Charge order 7041 for $25"* should produce one row no matter how many times the model emits the call.

### Pattern C — the model picks the value (anti-pattern)

```ts
schema: z.object({
  // ❌ The model can fabricate a fresh UUID on every retry, defeating dedup.
  requestKey: z.string().uuid(),                  // → fresh UUID per turn → every retry misses dedup
}),
```

If the model is the one choosing the UUID and treats every retry as "a new attempt," each retry gets a fresh key, every key misses the dedup, and you write three rows for one user request. Avoid. The schema may *expose* the field (LangChain's `tool()` has no separate channel for non-model args), but the orchestrator must fill it before the tool runs — the model is allowed to see the slot, never to pick the value.

**The rule of thumb.** If a retry should produce *the same row*, the key must be stable across retries. That stability comes from the orchestrator (Pattern A) or the args themselves (Pattern B). Anything that lets the model pick a fresh UUID on retry breaks the pattern.

## Idempotency at the database level

The `findUnique`-then-`create` pattern in the snippet above has a race window: two retries arriving in the same millisecond can both miss the `findUnique` and both insert. For low-volume tools you'll never see it; for high-volume tools you will, and the fix is to push the dedup down into the database itself.

A unique index on `requestKey` plus `INSERT … ON CONFLICT DO NOTHING` does the job. The first call inserts the row. The second call's insert is rejected by the unique constraint and the wrapper re-reads the existing row and returns it. You get the same external behavior — one row, two safe responses — without the race. In Postgres it's `ON CONFLICT (request_key) DO NOTHING RETURNING id`. In SQLite (which this course uses for the final project) it's `INSERT OR IGNORE`. Drizzle, Prisma, and Kysely all expose the conflict clause directly.

Concretely, here's the schema and the rewritten `chargeOrder` from Pattern B above:

```sql
-- migration: add the dedup column and the index that closes the race
ALTER TABLE charges ADD COLUMN request_key TEXT NOT NULL;
CREATE UNIQUE INDEX charges_request_key_uq ON charges(request_key);
```

```ts
// Postgres — raw SQL via your driver of choice
async function chargeOrder({ orderId, amountCents }) {
  const requestKey = `charge:${orderId}:${amountCents}`;        // → derived from inputs (Pattern B)

  const inserted = await db.query<{ id: number }>(
    `INSERT INTO charges (order_id, amount_cents, request_key)
     VALUES ($1, $2, $3)
     ON CONFLICT (request_key) DO NOTHING
     RETURNING id`,                                             // → returns one row if we won, zero if a sibling won
    [orderId, amountCents, requestKey],
  );

  if (inserted.rows[0]) return inserted.rows[0];                // → we inserted; this is the first call
  return await db.charges.findUnique({ where: { requestKey } }); // → the conflicting row is the canonical one
}
```

Two retries arriving in the same millisecond now both run the same `INSERT`. The unique index guarantees exactly one of them is allowed in; the other gets zero rows back from `RETURNING`, falls through to the `findUnique`, and reads what the winner just wrote. Same external behavior as Pattern B's `findUnique`-then-`create`, but the race window is gone — the database is the arbiter, not the application code.

The same thing in Drizzle (Postgres):

```ts
const inserted = await db
  .insert(charges)
  .values({ orderId, amountCents, requestKey })
  .onConflictDoNothing({ target: charges.requestKey })          // → maps to ON CONFLICT (request_key) DO NOTHING
  .returning({ id: charges.id });
```

And in SQLite — the runtime for this course's final project — the conflict clause is spelled differently but does the same thing:

```sql
INSERT OR IGNORE INTO charges (order_id, amount_cents, request_key)
VALUES (?, ?, ?);
-- → no RETURNING in older SQLite; do a follow-up SELECT keyed by request_key.
```

Prisma exposes it as `upsert` with an empty `update`, Kysely as `.onConflict(oc => oc.column('request_key').doNothing())`. The shape is the same in all four: one statement, the database enforces uniqueness, and the wrapper falls back to a keyed read when it loses the race.

## Out-of-band idempotency

Some write tools don't have a natural orchestrator-supplied key — particularly writes that look like *"send email to this list."* The model didn't generate a `requestKey` for the message; the orchestrator might not have one either, especially if the same tool is being called from multiple entry points (a chat session, a scheduled job, an admin click). The fix is to generate the dedup key **inside the tool** from the operation's content.

A reasonable shape for an email tool: `(customer_id, subject_hash, hour_bucket)`. The `hour_bucket` is `floor(now / 3600s)` — same email to the same customer within the same hour collapses to one. Different hours, two emails. This is what *real* mailing systems do; it's how you don't accidentally double-blast a list when a job runs twice.

Generated dedup keys feel less rigorous than orchestrator-supplied ones — they're a heuristic about *what counts as the same operation* — but they catch the failure modes that matter. *"Two retries within 200ms of each other"* is the dangerous case, and any reasonable hash of the args will catch it.

<Recap>
- The model will retry; build your write tools so retries are safe to call.
- Pass a `requestKey` per logical operation. The first call inserts, retries see the key and return the existing row.
- The orchestrator (or a derivation of the args) supplies the key — never the model picking a fresh UUID on retry.
- Push dedup into the database where you can — unique indexes plus `ON CONFLICT DO NOTHING` close the race window.
- For tools without an obvious orchestrator key, generate one from the args (customer + subject hash + hour bucket).
- Idempotency is the discipline that makes confirmation gates possible — the gate dedupes pending UI, the dedup key dedupes committed rows. Both layers, every write.
</Recap>

## Idempotency key đơn giản nhất

Mẫu này vay nguyên từ thiết kế webhook và payment API. Mỗi thao tác logic nhận một **request key** — một UUID do orchestrator sinh ra trước khi gọi tool — và tool lưu key đó cùng với dòng nó ghi xuống. Trước khi insert, tool tra key. Tìm thấy? Trả về dòng đã có. Không thấy? Insert dòng mới và nhớ key.

```ts
export const createPromo = tool(                  // → StructuredTool { name: 'promotions.create' }
  async ({ code, discountPct, requestKey }) => {
    const existing = await db.promotions.findUnique({ where: { requestKey } }); // → Promotion | null
    if (existing) {
      return JSON.stringify({ ok: true, deduped: true, promo: existing });      // → '{"ok":true,"deduped":true,"promo":{...}}' on retry
    }
    const promo = await db.promotions.create({    // → newly inserted Promotion row
      data: { code, discountPct, requestKey, createdAt: new Date() },
    });
    return JSON.stringify({ ok: true, deduped: false, promo }); // → '{"ok":true,"deduped":false,"promo":{...}}' on first call
  },
  {
    name: 'promotions.create',                    // → identifier the model uses
    description: 'Create a discount promo. Pass a stable requestKey to make retries safe.',
    schema: z.object({
      code: z.string().min(3).max(40),            // → string of 3..40 chars
      discountPct: z.number().int().min(1).max(75), // → integer 1..75
      requestKey: z.string().uuid(),              // → UUID — orchestrator fills this before invocation
    }),
  },
);
```

`requestKey` là một phần của schema của tool, nghĩa là phía gọi tool phải cung cấp nó. Chi tiết then chốt: **mô hình không tự chọn giá trị này.** Ba mẫu cho câu hỏi "key đến từ đâu", xếp đại khái theo tần suất bạn sẽ dùng.

### Mẫu A — orchestrator cấp key (mặc định)

Một wrapper nhỏ quanh `agent.invoke` (hoặc node `tools` của LangGraph) chặn mọi lời gọi tool. Với mỗi *thao tác logic*, nó mint một UUID đúng một lần, lưu vào lượt đang chạy, và tái sử dụng cùng UUID đó cho mọi lần retry mà mô hình phát ra. Thân hàm tool chỉ việc nhận `requestKey` như một tham số bình thường.

```ts
async function withIdempotency(toolName: string, rawArgs: unknown, exec: (args: any) => Promise<unknown>) {
  const args: any = rawArgs;                      // → freshly-validated tool args from the model
  if (!args.requestKey) {
    // One key per (turnId + toolName + stableHash(args)). Same args → same key,
    // even if the model fires the tool three times in the same turn.
    args.requestKey = sha256(`${currentTurnId()}:${toolName}:${stableStringify(args)}`); // → deterministic 64-char hex
  }
  return exec(args);                              // → result of the underlying tool execution
}
```

Mô hình muốn retry bao nhiêu lần tuỳ thích; wrapper tạo cùng một key cho cùng một bộ args, bảng dedup bắt các bản trùng, người dùng không bao giờ thấy bị trừ tiền hai lần. Đây là mặc định, và cũng là điều mà `createPromo` ở trên giả định.

### Mẫu B — tool tự suy ra key từ input

Khi bản thân các args đã đủ để xác định *một* thao tác hợp lệ, tool có thể tự dựng key trong thân hàm. Không cần wrapper.

```ts
async function chargeOrder({ orderId, amountCents }) {
  const requestKey = `charge:${orderId}:${amountCents}`; // → 'charge:7041:2500' — derived from inputs
  const existing = await db.charges.findUnique({ where: { requestKey } }); // → Charge | null
  if (existing) return existing;                  // → return previous Charge on retry
  return await db.charges.create({ data: { orderId, amountCents, requestKey } }); // → new Charge on first call
}
```

Cách này phù hợp khi thực tế không có trường hợp nào bạn muốn có *hai* lần charge giống nhau cách nhau vài giây. *"Trừ tiền đơn 7041 25 đô"* nên tạo ra đúng một dòng, bất kể mô hình phát lời gọi đó bao nhiêu lần.

### Mẫu C — mô hình tự chọn giá trị (anti-pattern)

```ts
schema: z.object({
  // ❌ The model can fabricate a fresh UUID on every retry, defeating dedup.
  requestKey: z.string().uuid(),                  // → fresh UUID per turn → every retry misses dedup
}),
```

Nếu chính mô hình là kẻ chọn UUID và coi mỗi lần retry là "một lần thử mới", thì mỗi retry có một key mới, mọi key đều miss dedup, và bạn ghi ba dòng cho một yêu cầu duy nhất của user. Tránh. Schema có thể *để lộ* trường này ra (hàm `tool()` của LangChain không có kênh riêng cho args "không cho mô hình thấy"), nhưng orchestrator phải điền nó trước khi tool chạy — mô hình được phép nhìn thấy ô trống, không bao giờ được tự điền giá trị.

**Quy tắc ngón cái.** Nếu một lần retry phải tạo ra *cùng một dòng*, thì key phải ổn định qua các lần retry. Sự ổn định đó đến từ orchestrator (Mẫu A) hoặc từ chính args (Mẫu B). Bất cứ thứ gì cho phép mô hình tự chọn UUID mới khi retry đều phá vỡ mẫu này.

## Idempotency ở tầng database

Mẫu `findUnique` rồi `create` trong snippet ở trên có một cửa sổ race: hai lần retry rơi cùng một mili-giây có thể cả hai cùng miss `findUnique` và cùng insert. Với tool ít traffic bạn sẽ không bao giờ thấy; với tool nhiều traffic bạn sẽ thấy, và cách sửa là đẩy dedup xuống thẳng database.

Một unique index trên `requestKey` cộng với `INSERT … ON CONFLICT DO NOTHING` là đủ. Lần gọi đầu tiên insert dòng. Insert của lần thứ hai bị unique constraint từ chối, wrapper đọc lại dòng đã có và trả về. Bạn vẫn có hành vi đối ngoại y hệt — một dòng, hai response an toàn — mà không còn race. Trong Postgres là `ON CONFLICT (request_key) DO NOTHING RETURNING id`. Trong SQLite (cái mà khoá học này dùng cho dự án cuối) là `INSERT OR IGNORE`. Drizzle, Prisma và Kysely đều expose mệnh đề conflict trực tiếp.

Cụ thể, đây là schema và `chargeOrder` từ Pattern B ở trên viết lại:

```sql
-- migration: thêm cột dedup và index đóng cửa sổ race
ALTER TABLE charges ADD COLUMN request_key TEXT NOT NULL;
CREATE UNIQUE INDEX charges_request_key_uq ON charges(request_key);
```

```ts
// Postgres — SQL thô qua driver bạn dùng
async function chargeOrder({ orderId, amountCents }) {
  const requestKey = `charge:${orderId}:${amountCents}`;        // → derive từ inputs (Pattern B)

  const inserted = await db.query<{ id: number }>(
    `INSERT INTO charges (order_id, amount_cents, request_key)
     VALUES ($1, $2, $3)
     ON CONFLICT (request_key) DO NOTHING
     RETURNING id`,                                             // → trả về 1 dòng nếu ta thắng, 0 dòng nếu thằng kia thắng
    [orderId, amountCents, requestKey],
  );

  if (inserted.rows[0]) return inserted.rows[0];                // → ta là người insert; đây là lần gọi đầu tiên
  return await db.charges.findUnique({ where: { requestKey } }); // → dòng "xung đột" chính là dòng chuẩn
}
```

Hai lần retry rơi cùng một mili-giây bây giờ cùng chạy một `INSERT`. Unique index bảo đảm đúng một trong hai được cho vào; cái kia nhận 0 dòng từ `RETURNING`, rơi xuống `findUnique`, đọc đúng cái mà người thắng vừa ghi. Hành vi đối ngoại giống y Pattern B với `findUnique` rồi `create`, nhưng cửa sổ race biến mất — database là người phân xử, không phải code app.

Cùng kiểu đó trong Drizzle (Postgres):

```ts
const inserted = await db
  .insert(charges)
  .values({ orderId, amountCents, requestKey })
  .onConflictDoNothing({ target: charges.requestKey })          // → map ra ON CONFLICT (request_key) DO NOTHING
  .returning({ id: charges.id });
```

Và trong SQLite — runtime cho dự án cuối của khoá học — mệnh đề conflict viết khác nhưng làm cùng một việc:

```sql
INSERT OR IGNORE INTO charges (order_id, amount_cents, request_key)
VALUES (?, ?, ?);
-- → SQLite phiên bản cũ không có RETURNING; làm thêm một SELECT theo request_key.
```

Prisma expose nó dưới dạng `upsert` với phần `update` để rỗng, Kysely là `.onConflict(oc => oc.column('request_key').doNothing())`. Hình dạng giống nhau ở cả bốn: một câu lệnh, database enforce uniqueness, wrapper fallback sang một SELECT theo key khi nó thua race.

## Idempotency "ngoài luồng"

Một số write tool không có sẵn key tự nhiên từ orchestrator — đặc biệt là những thao tác kiểu *"gửi email cho danh sách này."* Mô hình không sinh `requestKey` cho message; orchestrator có thể cũng không có, nhất là khi cùng một tool được gọi từ nhiều entry point khác nhau (một phiên chat, một job định kỳ, một cú click của admin). Cách sửa là sinh key dedup **bên trong tool** từ nội dung của thao tác.

Một hình dạng hợp lý cho email tool: `(customer_id, subject_hash, hour_bucket)`. `hour_bucket` là `floor(now / 3600s)` — cùng email gửi cho cùng khách trong cùng một giờ sẽ gộp thành một. Khác giờ, hai email. Đây chính là cách các hệ thống gửi mail *thật* hoạt động; đó là cách bạn không vô tình "double-blast" một danh sách khi một job chạy hai lần.

Key dedup tự sinh có cảm giác kém chặt chẽ hơn key do orchestrator cấp — chúng là một heuristic về *điều gì được tính là cùng một thao tác* — nhưng chúng bắt được các chế độ hỏng quan trọng. *"Hai lần retry cách nhau trong vòng 200ms"* là trường hợp nguy hiểm, và bất kỳ hash hợp lý nào của args cũng bắt được nó.

<Recap>
- Mô hình sẽ retry; hãy xây write tool sao cho retry là an toàn.
- Truyền `requestKey` cho mỗi thao tác logic. Lần đầu insert, các retry thấy key và trả về dòng đã có.
- Orchestrator (hoặc một derivation từ args) cấp key — không bao giờ để mô hình chọn UUID mới khi retry.
- Đẩy dedup xuống database ở mọi nơi có thể — unique index cộng với `ON CONFLICT DO NOTHING` đóng cửa sổ race.
- Với những tool không có key tự nhiên từ orchestrator, sinh key từ args (customer + subject hash + hour bucket).
- Idempotency là kỷ luật giúp cổng xác nhận khả thi — cổng dedup phần UI đang chờ, key dedup dedup các dòng đã commit. Cả hai lớp, cho mọi thao tác ghi.
</Recap>
