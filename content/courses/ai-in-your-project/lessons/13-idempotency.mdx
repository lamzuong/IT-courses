The model will retry. It will retry because a network call hiccupped and the orchestrator re-emitted the tool call. It will retry because a streaming connection dropped halfway through. It will retry because, on rare and unsettling days, it second-guessed itself and called the same tool twice in the same turn for reasons no log will ever explain. The first time you see your *"create promo"* tool fire two and a half times for one user request, you will understand idempotency in your bones. Better to understand it now.

The fix is not making the retries stop — they're going to happen, sometimes legitimately. The fix is making retries **safe**: writing your tools so that a retry of the same logical operation produces the same outcome as the first call did. One row, not two. One charge, not two. One email, not two.

## The simplest idempotency key

The pattern is borrowed wholesale from webhook design and payment APIs. Every logical operation gets a **request key** — a UUID generated by the orchestrator before the tool call — and the tool stores that key alongside the row it writes. Before inserting, the tool looks up the key. Found? Return the existing row. Not found? Insert the new one and remember the key.

```ts
export const createPromo = tool(
  async ({ code, discountPct, requestKey }) => {
    const existing = await db.promotions.findUnique({ where: { requestKey } });
    if (existing) {
      return JSON.stringify({ ok: true, deduped: true, promo: existing });
    }
    const promo = await db.promotions.create({
      data: { code, discountPct, requestKey, createdAt: new Date() },
    });
    return JSON.stringify({ ok: true, deduped: false, promo });
  },
  {
    name: 'promotions.create',
    description: 'Create a discount promo. Pass a stable requestKey to make retries safe.',
    schema: z.object({
      code: z.string().min(3).max(40),
      discountPct: z.number().int().min(1).max(75),
      requestKey: z.string().uuid(),
    }),
  },
);
```

The `requestKey` is part of the tool's schema, which means the model has to provide it. In practice your orchestrator generates it once per logical operation and passes it through every retry of the same call. The model never sees a fresh UUID for a retry — only for a genuinely new operation.

## Idempotency at the database level

The `findUnique`-then-`create` pattern in the snippet above has a race window: two retries arriving in the same millisecond can both miss the `findUnique` and both insert. For low-volume tools you'll never see it; for high-volume tools you will, and the fix is to push the dedup down into the database itself.

A unique index on `requestKey` plus `INSERT … ON CONFLICT DO NOTHING` does the job. The first call inserts the row. The second call's insert is rejected by the unique constraint and the wrapper re-reads the existing row and returns it. You get the same external behavior — one row, two safe responses — without the race. In Postgres it's `ON CONFLICT (request_key) DO NOTHING RETURNING id`. In SQLite (which this course uses for the final project) it's `INSERT OR IGNORE`. Drizzle, Prisma, and Kysely all expose the conflict clause directly.

## Out-of-band idempotency

Some write tools don't have a natural orchestrator-supplied key — particularly writes that look like *"send email to this list."* The model didn't generate a `requestKey` for the message; the orchestrator might not have one either, especially if the same tool is being called from multiple entry points (a chat session, a scheduled job, an admin click). The fix is to generate the dedup key **inside the tool** from the operation's content.

A reasonable shape for an email tool: `(customer_id, subject_hash, hour_bucket)`. The `hour_bucket` is `floor(now / 3600s)` — same email to the same customer within the same hour collapses to one. Different hours, two emails. This is what *real* mailing systems do; it's how you don't accidentally double-blast a list when a job runs twice.

Generated dedup keys feel less rigorous than orchestrator-supplied ones — they're a heuristic about *what counts as the same operation* — but they catch the failure modes that matter. *"Two retries within 200ms of each other"* is the dangerous case, and any reasonable hash of the args will catch it.

<Recap>
- The model will retry; build your write tools so retries are safe to call.
- Pass a `requestKey` per logical operation. The first call inserts, retries see the key and return the existing row.
- Push dedup into the database where you can — unique indexes plus `ON CONFLICT DO NOTHING` close the race window.
- For tools without an obvious orchestrator key, generate one from the args (customer + subject hash + hour bucket).
- Idempotency is the discipline that makes confirmation gates possible — the gate dedupes pending UI, the dedup key dedupes committed rows. Both layers, every write.
</Recap>
