By the end of lesson 14 you have something that genuinely works: a tool-calling agent that loops, recovers from errors, and produces a sensible final answer. For a single question with one or two tool calls, this loop is roughly all you need. The trouble starts the moment your prompts get even slightly more ambitious — *"create a promo, find the customers, draft them an email"* — and the abstractions you've leaned on so far quietly start to crack.

This lesson is the seam between Part III and Part IV. We're not going to write much code; we're going to look at the loop, name what it can't do, and notice that any honest fix would, step by step, reinvent something that already exists. That something is LangGraph, and the next lesson is its proper introduction. Today is the *why*.

## Where the bare loop works

The `while (lastMessage.tool_calls)` shape from lesson 14 is genuinely good at a narrow band of work. A user asks one question, the model decides whether to look something up, runs one or two reads, and writes the answer. The loop terminates because the model stops emitting tool calls; the orchestrator returns the final assistant message; the user reads it and the request ends. *"How many lapsed customers do we have?"* fits this shape perfectly. So does *"what was our top-selling SKU last week?"* or *"draft me a one-line subject line for a re-engagement email."*

These are *interactive Q&A* shapes. One turn from the user, a few internal turns from the model, one final reply. The loop is the right tool, the code is short, and you don't need anything fancier.

## Where it cracks

Now try *"Create a 15% promo for customers who haven't ordered in 60 days, and draft them a re-engagement email."* That's three steps minimum: create the promo, look up the recipients, draft the emails. Each step depends on the one before it. Each one is a write — the kind of action you really do not want firing without a human signing off. And each one can fail in a way that matters: maybe the promo code already exists, maybe two of the recipients have unsubscribed, maybe the draft tool times out on the third email and the first two have already been saved.

The bare loop has nothing to say about any of this. It can't pause before a write. It can't branch — *"if the segment is bigger than fifty people, ask the human first; otherwise just go."* It can't undo. If the second of three writes fails, you don't have any clean way to ask the model *"please continue from step three"*; you can only run the whole prompt again and hope it converges to the same plan. And there's nowhere natural to put a checkpoint, so if the user closes the tab mid-workflow, the half-completed plan is gone.

These aren't edge cases. They're the everyday shape of *"AI that does work in your product."*

## What you'd start writing

If you decided to fix this without help, you'd reach for the same tools every framework eventually does. You'd add a `state` object the model and the loop both read from. You'd split your single loop into named steps — `plan`, `confirm`, `execute`, `summarize` — and put logic between them: *"if the next step is a write, hand control back to the UI and wait for an approval"*; *"if a step failed, route to a recovery node instead of the next one."* You'd add a way to pause and resume, which means the state has to be serializable. You'd add a way to stream partial progress, which means each step has to emit events. You'd find yourself building a small graph: nodes are steps, edges are transitions, and the runtime walks the graph until something stops it.

By the time you've got all of that working — really working, with timeouts and retries and resumption tokens — you'll have spent a couple of weeks reinventing a state-machine library, badly. There's nothing wrong with that learning trajectory. There's also no reason to do it twice when someone has done it well already.

The trajectory in pseudocode, just so the inevitability is concrete:

```ts
// Day 1 — the lesson 14 loop
while (lastMessage.tool_calls?.length) {           // → keep looping while the model keeps requesting tools
  const results = await runToolCalls(lastMessage.tool_calls); // → ToolMessage[] from the executions
  messages.push(...results);                       // → append tool results to the transcript
  lastMessage = await model.invoke(messages);      // → next AIMessage (may have more tool_calls or be final)
}

// Day 7 — "we need to ask before writes"
while (lastMessage.tool_calls?.length) {
  const writes = lastMessage.tool_calls.filter(isWrite); // → ToolCall[] that mutate state
  if (writes.length) {
    const decision = await askUser(writes);   // ← but how do you save state if the user closes the tab?  // → { approve: boolean }
    if (!decision.approve) break;                  // → user said no — abort the loop
  }
  // ...

// Day 21 — pausable, with state, branching, retries, persistence
type State = { messages: BaseMessage[]; pending?: ToolCall[]; phase: 'plan' | 'confirm' | 'execute' };
const checkpoint = await loadCheckpoint(threadId); // → State | null
let state = checkpoint ?? { messages: [], phase: 'plan' }; // → resume or start fresh
while (state.phase !== 'done') {
  state = await STEP_FUNCTIONS[state.phase](state); // → next State
  await saveCheckpoint(threadId, state);           // → persist so the user can close the tab and come back
  if (state.phase === 'confirm') return { interrupt: state.pending }; // → return to UI; resume after Approve
}
```

By the end you've written nodes (`STEP_FUNCTIONS`), edges (the `phase` transitions), state, and checkpointing — the four primitives the next lesson introduces with names and tests.

## The pivot

LangGraph is, very precisely, the state machine you'd otherwise write yourself, well-engineered, with the API conventions LangChain users already know. State + nodes + edges; conditional routing; pauses, resumption, persistence; streaming; first-class checkpointing. It looks at the bare loop the same way you do — *"this is fine for one-shot Q&A and not enough for anything else"* — and gives you the next layer up without throwing away what you already know.

Everything you've built so far in Part III still applies. Tool definitions don't change. Zod schemas don't change. The system prompt and few-shot examples don't change. What changes is the orchestrator: instead of `while (toolCalls.length) { ... }`, you describe a graph and `compile()` it. The model still calls tools; the tools still return strings; you still validate and you still log. Only the runner is different — and from the next lesson on, dramatically more capable.

## Nơi cái loop trần làm tốt việc của nó

Hình dạng `while (lastMessage.tool_calls)` ở bài 14 thật sự giỏi ở một dải công việc hẹp. Người dùng hỏi một câu, mô hình quyết định xem có cần tra cứu gì không, chạy một hai cú đọc, rồi viết câu trả lời. Loop kết thúc vì mô hình thôi phát tool call; orchestrator trả về AIMessage cuối; người dùng đọc và yêu cầu kết thúc. *"Có bao nhiêu khách hàng lâu không quay lại?"* hợp tuyệt đối với hình dạng này. *"SKU bán chạy nhất tuần trước là gì?"* hay *"soạn cho tôi một subject line một dòng cho email re-engagement"* cũng vậy.

Đây là hình dạng *Q&A tương tác*. Một lượt từ user, vài lượt nội bộ từ mô hình, một câu trả lời cuối. Loop là tool đúng, code ngắn, bạn không cần gì cầu kỳ hơn.

## Nơi nó nứt

Giờ thử *"Tạo voucher giảm 15% cho khách hàng đã không đặt hàng trong 60 ngày, và soạn cho họ một email re-engagement."* Đó là tối thiểu ba bước: tạo voucher, tra danh sách người nhận, soạn email. Mỗi bước phụ thuộc bước trước. Mỗi bước là một thao tác ghi — loại hành động mà bạn rất *không* muốn nổ ra mà không có một con người ký duyệt. Và mỗi bước có thể hỏng theo cách có hậu quả: có thể mã voucher đã tồn tại, có thể hai người trong danh sách đã unsubscribe, có thể tool soạn nháp timeout ở email thứ ba trong khi hai email đầu đã được lưu.

Cái loop trần không có gì để nói về bất kỳ điều nào trong số đó. Nó không thể tạm dừng trước một thao tác ghi. Nó không thể branch — *"nếu segment lớn hơn năm mươi người thì hỏi người dùng trước; còn không thì cứ đi."* Nó không thể undo. Nếu cú ghi thứ hai trong ba cú thất bại, bạn không có cách sạch sẽ nào để bảo mô hình *"vui lòng tiếp tục từ bước ba"*; bạn chỉ có thể chạy lại toàn bộ prompt và hy vọng nó hội tụ về cùng một kế hoạch. Và không có chỗ tự nhiên nào để đặt checkpoint, nên nếu người dùng đóng tab giữa workflow, kế hoạch đang dở mất sạch.

Đây không phải edge case. Đây là hình dạng đời thường của *"AI làm việc trong sản phẩm của bạn."*

## Bạn sẽ bắt đầu viết những gì

Nếu bạn quyết định tự sửa mà không nhờ ai, bạn sẽ với tay tới đúng những tool mà mọi framework rốt cuộc đều dùng. Bạn sẽ thêm một object `state` mà cả mô hình lẫn loop cùng đọc. Bạn sẽ tách cái loop đơn của mình thành các bước có tên — `plan`, `confirm`, `execute`, `summarize` — và đặt logic ở giữa chúng: *"nếu bước kế tiếp là một thao tác ghi, hãy giao quyền điều khiển trở lại UI và chờ duyệt"*; *"nếu một bước hỏng, định tuyến tới node recovery thay vì node kế tiếp."* Bạn sẽ thêm cơ chế pause/resume, nghĩa là state phải serialize được. Bạn sẽ thêm cơ chế stream tiến độ, nghĩa là mỗi bước phải emit event. Cuối cùng bạn sẽ thấy mình đang xây một cái graph nhỏ: node là bước, edge là chuyển trạng thái, runtime đi qua graph cho đến khi có gì đó dừng nó lại.

Đến lúc bạn làm tất cả những thứ đó chạy được — chạy thật, có timeout có retry có token resumption — bạn đã mất vài tuần để tái phát minh một thư viện state-machine, một cách kém chất lượng. Quỹ đạo học đó không có gì sai. Nhưng cũng không có lý do gì để làm chuyện đó lần thứ hai khi có người đã làm tốt rồi.

Quỹ đạo đó bằng pseudo-code, để sự tất yếu trở nên cụ thể:


## Bước ngoặt

LangGraph, rất chính xác, chính là cái state machine mà bạn sẽ phải tự viết, được kỹ sư hoá tử tế, với các quy ước API mà người dùng LangChain đã quen. State + nodes + edges; conditional routing; pause, resume, persistence; streaming; checkpoint hạng nhất. Nó nhìn vào cái loop trần đúng theo cách bạn nhìn — *"cái này ổn cho Q&A một phát, không đủ cho bất cứ thứ gì khác"* — và đưa bạn lên lớp tiếp theo mà không vứt bỏ những gì bạn đã biết.

Tất cả những gì bạn đã xây ở Phần III vẫn áp dụng được. Định nghĩa tool không đổi. Schema Zod không đổi. System prompt và few-shot example không đổi. Cái thay đổi là orchestrator: thay vì `while (toolCalls.length) { ... }`, bạn mô tả một graph và `compile()` nó. Mô hình vẫn gọi tool; tool vẫn trả về string; bạn vẫn validate và vẫn log. Chỉ có cái runner là khác — và từ bài kế tiếp trở đi, mạnh mẽ hơn hẳn.
