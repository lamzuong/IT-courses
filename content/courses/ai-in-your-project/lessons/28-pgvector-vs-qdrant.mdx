## 01 — The five axes that decide it

Forget "which is faster" — they're both fast enough at any reasonable scale. The real questions:

**Ops surface.** Do you want one daemon to back up, monitor, restart, version-upgrade — or two? pgvector adds zero new ops; Qdrant adds a second system you have to learn.

**Payload filters at scale.** Can you filter the result set efficiently? *"Top 10 chunks where `locale = 'vi'` AND `category = 'returns'` AND `not expired`"*. pgvector does this with SQL but the planner can struggle when the filter is very selective on a large HNSW index. Qdrant filters *inside* the HNSW traversal — its killer feature.

**Replication and HA.** Do you need multi-node redundancy? pgvector inherits whatever Postgres replication you've set up (streaming / logical / Patroni). Qdrant has Raft-based replication built in — three nodes, automatic failover, no manual intervention.

**Transactional consistency.** Do your vectors need to commit/rollback alongside your `orders` table? pgvector is in the same transaction. Qdrant is its own world — eventually consistent, no cross-system ACID.

**Cost.** pgvector adds disk + RAM to your existing Postgres bill — usually marginal. Qdrant adds another machine (or another cluster), another monitoring contract, another on-call rotation.

## 02 — The pgvector-wins case

You should default to pgvector if any of these hold:

- **You already run Postgres.** You inherit backup, replication, monitoring, IAM. The marginal cost of pgvector is the disk for the embeddings and the RAM for the HNSW index.
- **Your corpus is under ~10M vectors.** Throughput and latency stay well under 10 ms p95 with HNSW defaults. No reason to add a second system.
- **You filter against `JOIN`ed business data.** *"Find chunks where `category = ?` AND `expires_at > now()`"* is one SQL statement against pgvector + the products table. With Qdrant you'd need to duplicate the filterable fields into Qdrant's payload, then keep them in sync.
- **You need transactional consistency.** If a chunk and the product it describes must commit together (or roll back together), they need to be in the same database.
- **Your team is small.** Two-person teams cannot afford to operate two databases. Five-person teams can. Ten-person teams must.

For a typical e-commerce concierge in Vietnam, **all five of these conditions hold**. pgvector is the right call. That's why L32 builds on pgvector.

## 03 — The Qdrant-wins case

You should plan a migration to Qdrant (or start there) if any of these hold:

- **Payload filters are very selective and frequent.** *"Show only documents from this customer's tenant, in this language, marked active"* on a 50M-vector index. pgvector's planner can degrade badly here; Qdrant's filterable HNSW shines.
- **Multi-tenancy with strict per-tenant isolation.** Qdrant's payload-indexed collections handle "millions of small tenants" gracefully — pgvector with a `tenant_id` filter starts to struggle past 10k tenants without partitioning.
- **You need built-in replication and sharding.** Qdrant clusters with Raft; you `kubectl scale` and it spreads. With pgvector you wire up Patroni or a managed Postgres HA.
- **Vector-only access patterns.** If your queries are *purely* vector + payload, never JOINing to relational data, you're paying the Postgres overhead for no benefit.
- **You're past 50M vectors.** HNSW indexes get fat. Qdrant's binary quantization and disk-backed indices give you more headroom than pgvector at this scale.

Qdrant is also a reasonable choice when your stack isn't Postgres-centric — a Mongo + Node shop with no existing Postgres is happier adding Qdrant than introducing Postgres just for pgvector.

## 04 — The full decision matrix

For context, here's how the broader landscape sits in 2026. Sources you should benchmark yourself before any migration:

| Solution | Type | Best at | Worst at | When to pick |
|---|---|---|---|---|
| **pgvector** | Postgres extension | Operational simplicity, JOINs, ACID | Filter-heavy 50M+ workloads | Default if you have Postgres |
| **Qdrant** | Self-hosted / cloud, Rust | Payload filters, replication, multi-tenant | Adds a daemon; eventual consistency | When pgvector hits its scaling wall |
| **Weaviate** | Self-hosted / cloud, Go | GraphQL API, hybrid built-in | Heavier ops; smaller community | When you want hybrid + reranking out of the box |
| **Milvus** | Self-hosted / cloud, distributed | Billion-scale, GPU acceleration | Operational complexity (etcd, Pulsar) | True billion-vector workloads |
| **Pinecone** | Hosted only | Zero ops, predictable latency | Vendor lock-in, per-pod minimums | When ops time is more expensive than license cost |
| **Chroma** | Embedded / hosted | Quickest local dev experience | Limited prod hardening | Prototypes, notebooks, RAG demos |
| **Elastic / OpenSearch** | Search engine + vector | Already have it for logs | Recall lower than vector-native | Don't add it for vectors alone; piggyback if you have it |
| **Redis (RediSearch)** | In-memory + vector | Sub-ms latency on hot data | RAM cost; vector second-class | Cache layer in front of pgvector |

The two **default-correct** answers in 2026: **pgvector** if you already have Postgres, **Qdrant** if you don't or you've outgrown pgvector. Anything else needs a specific reason.

## 05 — Migration: start small, move only if you must

If you're in doubt, start on pgvector. The migration to Qdrant is well-understood and the data is portable (export embeddings + payload as JSONL, re-upsert in Qdrant). The reverse — Qdrant → pgvector — is also doable but rarer.

A clean migration path that doesn't bet the farm:

1. **Phase 1 (months 0–6).** Everything on pgvector. Build the eval set (L30). Track recall@k and query p95 latency in a dashboard.
2. **Phase 2 (when a metric crosses a threshold).** Latency p95 > 200ms, recall@10 < 0.92 after tuning, or filter queries timing out. Spin up Qdrant in parallel.
3. **Phase 3.** Dual-write to both for a week. Compare recall on the same eval set. If Qdrant is meaningfully better and the ops cost is justified, cut over reads.
4. **Phase 4.** Keep pgvector for vectors that JOIN to products (small index, fast queries). Put the big static corpus in Qdrant. **Both can coexist; you don't have to pick one for the whole company.**

<Callout type="tip" title="The trigger is a graph, not a vibe">
"It feels slow" is not a migration trigger. "p95 latency on the FAQ retrieval crossed 300 ms last Tuesday and stayed there even after tuning ef_search" is. Build the dashboard first; let the data tell you when.
</Callout>

## 01 — Năm trục quyết định

Quên "cái nào nhanh hơn" — cả hai đủ nhanh ở mọi scale hợp lý. Câu hỏi thật:

**Ops surface.** Bạn muốn một daemon để backup, monitor, restart, version-upgrade — hay hai? pgvector thêm zero ops mới; Qdrant thêm hệ thống thứ hai phải học.

**Payload filter ở scale.** Lọc result set hiệu quả được không? *"Top 10 chunk nơi `locale = 'vi'` AND `category = 'returns'` AND chưa hết hạn"*. pgvector làm bằng SQL nhưng planner có thể vật lộn khi filter rất chọn lọc trên index HNSW lớn. Qdrant lọc *bên trong* traversal HNSW — killer feature của nó.

**Replication và HA.** Cần dự phòng đa node không? pgvector kế thừa replication Postgres bạn set (streaming / logical / Patroni). Qdrant có replication Raft built-in — ba node, failover tự động, không can thiệp tay.

**Nhất quán giao dịch.** Vector của bạn cần commit/rollback cùng bảng `orders` không? pgvector cùng giao dịch. Qdrant là thế giới riêng — eventually consistent, không ACID xuyên hệ thống.

**Chi phí.** pgvector thêm disk + RAM vào bill Postgres có sẵn — thường nhỏ. Qdrant thêm máy khác (hoặc cluster khác), hợp đồng monitoring khác, lịch on-call khác.

## 02 — Trường hợp pgvector thắng

Bạn nên mặc định pgvector nếu bất kỳ điều nào sau đúng:

- **Đã chạy Postgres.** Kế thừa backup, replication, monitoring, IAM. Chi phí biên của pgvector là disk cho embedding và RAM cho index HNSW.
- **Corpus dưới ~10M vector.** Throughput và latency thoải mái dưới 10 ms p95 với mặc định HNSW. Không lý do gì thêm hệ thống thứ hai.
- **Lọc dựa vào data business `JOIN`.** *"Tìm chunk nơi `category = ?` AND `expires_at > now()`"* là một SQL trên pgvector + bảng products. Với Qdrant phải duplicate field lọc được vào payload Qdrant, rồi giữ đồng bộ.
- **Cần nhất quán giao dịch.** Nếu chunk và sản phẩm nó mô tả phải commit cùng nhau (hoặc rollback cùng nhau), chúng phải ở cùng database.
- **Team nhỏ.** Team hai người không kham được vận hành hai database. Team năm có thể. Team mười phải.

Cho concierge e-commerce điển hình ở Việt Nam, **cả năm điều đều đúng**. pgvector là lựa chọn đúng. Đó là vì sao L32 dựng trên pgvector.

## 03 — Trường hợp Qdrant thắng

Bạn nên lên kế hoạch migration sang Qdrant (hoặc bắt đầu từ đó) nếu bất kỳ điều nào sau đúng:

- **Payload filter rất chọn lọc và thường xuyên.** *"Chỉ hiện document từ tenant này của khách, ngôn ngữ này, đánh dấu active"* trên index 50M vector. Planner pgvector có thể xuống cấp tệ ở đây; HNSW có filter của Qdrant toả sáng.
- **Multi-tenancy với cách ly per-tenant nghiêm.** Collection payload-indexed của Qdrant xử lý "triệu tenant nhỏ" duyên dáng — pgvector với filter `tenant_id` bắt đầu vật lộn qua 10k tenant không partition.
- **Cần replication và sharding built-in.** Cluster Qdrant với Raft; bạn `kubectl scale` và nó trải. Với pgvector bạn wire Patroni hoặc Postgres HA managed.
- **Pattern truy cập chỉ vector.** Nếu query *thuần* là vector + payload, không bao giờ JOIN sang data relational, bạn đang trả overhead Postgres không lợi ích.
- **Qua 50M vector.** Index HNSW phình ra. Quantization binary và index disk-backed của Qdrant cho bạn nhiều headroom hơn pgvector ở scale này.

Qdrant cũng là lựa chọn hợp lý khi stack không Postgres-centric — shop Mongo + Node không có Postgres sẵn vui hơn khi thêm Qdrant hơn là giới thiệu Postgres chỉ để dùng pgvector.

## 04 — Ma trận quyết định đầy đủ

Cho bối cảnh, đây là cảnh quan rộng hơn ngồi ở đâu năm 2026. Nguồn bạn nên benchmark trước bất kỳ migration nào:

| Giải pháp | Loại | Giỏi nhất | Tệ nhất | Khi chọn |
|---|---|---|---|---|
| **pgvector** | Extension Postgres | Đơn giản vận hành, JOIN, ACID | Workload filter-nặng 50M+ | Mặc định nếu có Postgres |
| **Qdrant** | Self-host / cloud, Rust | Payload filter, replication, multi-tenant | Thêm daemon; eventual consistency | Khi pgvector chạm tường scaling |
| **Weaviate** | Self-host / cloud, Go | API GraphQL, hybrid built-in | Ops nặng hơn; cộng đồng nhỏ hơn | Khi muốn hybrid + reranking out of box |
| **Milvus** | Self-host / cloud, phân tán | Scale tỷ, tăng tốc GPU | Phức tạp vận hành (etcd, Pulsar) | Workload tỷ vector thật |
| **Pinecone** | Chỉ hosted | Zero ops, latency dự đoán được | Vendor lock-in, minimum per-pod | Khi thời gian ops đắt hơn chi phí license |
| **Chroma** | Embedded / hosted | Trải nghiệm dev local nhanh nhất | Cứng cáp production hạn chế | Prototype, notebook, demo RAG |
| **Elastic / OpenSearch** | Search engine + vector | Đã có sẵn cho log | Recall thấp hơn vector-native | Đừng thêm chỉ cho vector; ghép thêm nếu đã có |
| **Redis (RediSearch)** | In-memory + vector | Latency dưới ms cho data hot | Chi phí RAM; vector hạng hai | Lớp cache trước pgvector |

Hai đáp án **default-đúng** năm 2026: **pgvector** nếu đã có Postgres, **Qdrant** nếu không hoặc đã vượt qua pgvector. Cái khác cần lý do cụ thể.

## 05 — Migration: bắt đầu nhỏ, chỉ chuyển khi bắt buộc

Nếu nghi ngờ, bắt đầu pgvector. Migration sang Qdrant hiểu rõ và data portable (export embedding + payload thành JSONL, re-upsert vào Qdrant). Ngược lại — Qdrant → pgvector — cũng làm được nhưng hiếm.

Đường migration sạch không đặt cược trang trại:

1. **Phase 1 (tháng 0–6).** Mọi thứ trên pgvector. Dựng eval set (L30). Theo dõi recall@k và query p95 latency trong dashboard.
2. **Phase 2 (khi metric vượt ngưỡng).** Latency p95 > 200ms, recall@10 < 0.92 sau tune, hoặc query filter timeout. Spin up Qdrant song song.
3. **Phase 3.** Dual-write cả hai một tuần. So recall trên cùng eval set. Nếu Qdrant tốt hơn có ý nghĩa và chi phí ops chính đáng, cut over read.
4. **Phase 4.** Giữ pgvector cho vector JOIN sang products (index nhỏ, query nhanh). Đặt corpus tĩnh lớn vào Qdrant. **Cả hai có thể cùng tồn tại; bạn không phải chọn một cho cả công ty.**

<Callout type="tip" title="Trigger là biểu đồ, không phải cảm giác">
"Có vẻ chậm" không phải trigger migration. "p95 latency trên retrieval FAQ vượt 300 ms thứ Ba tuần trước và ở đó kể cả sau khi tune ef_search" mới phải. Dựng dashboard trước; để data nói khi nào.
</Callout>

<Recap>
- Quyết định dựa trên năm trục: **ops surface, payload filter, replication, nhất quán giao dịch, chi phí** — không phải tốc độ HNSW thuần.
- **pgvector thắng mặc định** nếu có Postgres và dưới ~10M vector và cần JOIN / ACID. Đa số concierge sống ở đây.
- **Qdrant thắng** cho payload filter chọn lọc ở scale, multi-tenant, replication built-in, hoặc workload vector thuần qua 50M.
- Ngoài pgvector và Qdrant, phần còn lại là niche — Pinecone cho hosted zero-ops, Milvus cho tỷ-scale, Chroma cho prototype, Weaviate cho GraphQL+hybrid.
- Đường migration: bắt đầu pgvector, đo, và chỉ chuyển khi *metric biểu đồ* vượt ngưỡng. Cả hai có thể cùng tồn tại — không phải chọn một cho cả công ty.
</Recap>

Tiếp: Qdrant hands-on. Docker compose, một collection, upsert từ cùng `knowledge/` markdown, và payload filter làm Qdrant xứng công.
