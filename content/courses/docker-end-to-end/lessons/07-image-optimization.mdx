## 1. `.dockerignore` — thứ luôn quên đầu tiên

</VI>

<EN>

Every `docker build` starts by sending the *build context* (the folder you pass) from the CLI to the daemon. **Everything** in that folder, even if your Dockerfile never `COPY`s it, gets sent. Without a `.dockerignore`, your `node_modules` (300 MB), your `.git` folder (50 MB), your local logs, your `.env` files — all of it gets streamed to the daemon for every build.

The fix is one file in the same folder as your Dockerfile:

```text
# .dockerignore — looks like .gitignore, behaves like .gitignore
node_modules
.git
.next
dist
build
coverage
.env
.env.*
*.log
.vscode
.idea
.DS_Store
**/__pycache__
**/*.pyc
README.md      # if you really do not need it in the image
```

What it gives you:

- **Smaller, faster context transfer.** A 500 MB context becomes a 5 MB context.
- **Better cache reuse.** Without it, `COPY . .` invalidates because of every log file Webpack writes.
- **Security.** Your `.env` cannot accidentally end up baked into the image.

Run this once to see what is currently being shipped:

```bash
du -sh .                                                            # → total folder size on disk
docker build . 2>&1 | head -5                                       # → first line shows "transferring context: XX MB"
```

The two numbers should not be in the same ballpark. If they are, you do not have a `.dockerignore`.

## 2. Choosing a base image

</EN>

<VI>

Mỗi `docker build` bắt đầu bằng việc gửi *build context* (thư mục bạn truyền) từ CLI sang daemon. **Toàn bộ** thư mục đó, dù Dockerfile không `COPY` tới, đều được gửi. Không có `.dockerignore`, `node_modules` (300 MB), folder `.git` (50 MB), log local, file `.env` — tất cả stream sang daemon mỗi lần build.

Cách chữa là một file cùng thư mục với Dockerfile:

```text
# .dockerignore — nhìn giống .gitignore, cư xử giống .gitignore
node_modules
.git
.next
dist
build
coverage
.env
.env.*
*.log
.vscode
.idea
.DS_Store
**/__pycache__
**/*.pyc
README.md      # nếu thật sự không cần trong image
```

Nó cho bạn gì:

- **Context nhỏ hơn, gửi nhanh hơn.** Context 500 MB còn 5 MB.
- **Cache tái dùng tốt hơn.** Không có nó, `COPY . .` mất cache vì mỗi file log Webpack ghi ra.
- **Bảo mật.** `.env` của bạn không lỡ nướng vào image.

Chạy thử để biết hiện đang ship cái gì:

```bash
du -sh .                                                            # → tổng dung lượng thư mục
docker build . 2>&1 | head -5                                       # → dòng đầu hiện "transferring context: XX MB"
```

Hai con số đừng nên gần nhau. Nếu gần — bạn chưa có `.dockerignore`.

<EN>

A practical decision tree:

- **Use `*-alpine`** for most production runtime images. ~40 MB base. Watch out for native modules: if `npm install` complains about `python` or `gcc`, install them temporarily in the builder stage, not the runtime stage.
- **Use `*-slim`** (Debian-based) when alpine causes pain (TLS gotchas, native libs only shipping glibc wheels, e.g., some Python ML packages).
- **Use the full image** (`node:20`, `python:3.13`) only in *builder* stages. Never ship it.
- **Use `distroless`** when you want the smallest possible attack surface in production. Google's distroless images contain just the language runtime — no shell, no package manager, nothing else. Operationally a bit harder (you cannot `docker exec -it … sh`), but the security trade is real.

```dockerfile
FROM gcr.io/distroless/nodejs20-debian12                            # → no /bin/sh, no apt, no anything
COPY --from=builder /app/dist /app/dist
COPY --from=builder /app/node_modules /app/node_modules
WORKDIR /app
CMD ["dist/index.js"]                                                # → distroless: CMD args go directly to the runtime
```

## 3. BuildKit cache mounts

</EN>

<VI>

Cây quyết định thực dụng:

- **Dùng `*-alpine`** cho hầu hết image runtime production. Base ~40 MB. Cẩn thận native module: nếu `npm install` than về `python` hay `gcc`, cài tạm trong stage builder, không phải runtime.
- **Dùng `*-slim`** (Debian) khi alpine gây đau (gặp TLS, lib native chỉ phát hành wheel glibc, ví dụ vài package ML Python).
- **Dùng image full** (`node:20`, `python:3.13`) chỉ trong stage *builder*. Đừng bao giờ ship.
- **Dùng `distroless`** khi muốn bề mặt tấn công nhỏ nhất ở production. Image distroless của Google chỉ chứa runtime ngôn ngữ — không shell, không package manager, không gì khác. Vận hành hơi khó hơn (không `docker exec -it … sh` được), nhưng trade an toàn là thật.

```dockerfile
FROM gcr.io/distroless/nodejs20-debian12                            # → không có /bin/sh, không apt, không gì
COPY --from=builder /app/dist /app/dist
COPY --from=builder /app/node_modules /app/node_modules
WORKDIR /app
CMD ["dist/index.js"]                                                # → distroless: arg của CMD đi thẳng vào runtime
```

## 3. BuildKit cache mount

</VI>

<EN>

BuildKit is Docker's modern build engine, on by default since Docker 23. One of its best features is `--mount=type=cache`: a directory mounted at build time that **persists across builds**, but does **not** end up in the final image.

This is the magic that makes `npm install` fast on CI:

```dockerfile
# syntax=docker/dockerfile:1.7

FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev
```

What happens:

- BuildKit keeps `/root/.npm` (the npm cache) on the build host, indexed by a content key.
- Next build, that directory is mounted again into the build container at the same path. npm sees its cache; downloads are mostly skipped.
- The cache directory is **not copied into the resulting layer**. It only lives on the build host. Your image stays small.

The first line, `# syntax=docker/dockerfile:1.7`, opts into newer Dockerfile syntax for the cache-mount feature. Always include it when you use `--mount`.

Equivalent for pip:

```dockerfile
# syntax=docker/dockerfile:1.7
FROM python:3.13-slim
WORKDIR /app
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --no-cache-dir -r requirements.txt
```

Equivalent for apt:

```dockerfile
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt/lists \
    apt-get update && apt-get install -y curl
```

A CI build that used to download 300 packages every time now downloads only the new ones. Build times often drop 50–80%.

## 4. The pruning pass

</EN>

<VI>

BuildKit là engine build hiện đại của Docker, mặc định từ Docker 23 trở đi. Một trong những tính năng đáng giá nhất là `--mount=type=cache`: một thư mục mount lúc build, **bền vững qua nhiều lần build**, nhưng **không** đi vào image cuối.

Đây là phép thuật giúp `npm install` trên CI nhanh:

```dockerfile
# syntax=docker/dockerfile:1.7

FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev
```

Chuyện xảy ra:

- BuildKit giữ `/root/.npm` (cache của npm) trên build host, đánh index bằng content key.
- Lần build sau, thư mục đó mount lại vào container build cùng path. npm thấy cache; download phần lớn được bỏ qua.
- Thư mục cache **không** được copy vào layer kết quả. Nó chỉ sống trên build host. Image của bạn vẫn nhỏ.

Dòng đầu, `# syntax=docker/dockerfile:1.7`, bật cú pháp Dockerfile mới hơn cho tính năng cache-mount. Luôn để khi dùng `--mount`.

Tương đương cho pip:

```dockerfile
# syntax=docker/dockerfile:1.7
FROM python:3.13-slim
WORKDIR /app
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --no-cache-dir -r requirements.txt
```

Tương đương cho apt:

```dockerfile
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt/lists \
    apt-get update && apt-get install -y curl
```

Một build CI trước đây tải 300 package mỗi lần, giờ chỉ tải cái mới. Build thường giảm 50–80%.

## 4. Pruning pass

</VI>

<EN>

Even with all the above, junk creeps in. A pruning pass at the end of the builder stage removes:

```dockerfile
RUN apt-get clean \
 && rm -rf /var/lib/apt/lists/*                                     # → Debian: package lists, 30+ MB
RUN find . -name "*.test.ts" -delete \
 && find . -name "*.spec.ts" -delete                                # → strip test files
RUN npm prune --production                                          # → remove devDependencies left over
```

In multi-stage land, most of this is moot — the builder stage is thrown away. But if you keep a single stage (smaller projects, scripts, prototypes), the prune pass is the cleanup at the very end.

## What good looks like

Run this on a finished image:

```bash
docker images myapp                                                 # → SIZE column
docker history myapp                                                # → which layer holds what
docker run --rm -it myapp sh                                        # → poke around the filesystem; what is in there?
du -sh /app /usr/local /var                                         # → biggest folders inside the container
```

Good Node runtime targets: **&lt;150 MB total, &lt;50 MB writable layer**. Good Go runtime targets: **&lt;25 MB total**. Good Python web runtime: **&lt;200 MB total** (Python's stdlib is fat).

If you are far above these, the prune commands above plus a `.dockerignore` audit usually find the leak.

## Café analogy

</EN>

<VI>

Dù làm hết các bước trên, rác vẫn lén vào. Một pass dọn ở cuối stage builder bóc đi:

```dockerfile
RUN apt-get clean \
 && rm -rf /var/lib/apt/lists/*                                     # → Debian: list package, 30+ MB
RUN find . -name "*.test.ts" -delete \
 && find . -name "*.spec.ts" -delete                                # → bóc file test
RUN npm prune --production                                          # → bỏ devDependencies còn sót
```

Trong thế giới multi-stage, phần lớn việc trên không cần — stage builder bị vứt. Nhưng nếu giữ một stage duy nhất (project nhỏ, script, prototype), pass prune là dọn cuối cùng.

## "Tốt" trông ra sao

Chạy với image đã build xong:

```bash
docker images myapp                                                 # → cột SIZE
docker history myapp                                                # → layer nào ôm cái gì
docker run --rm -it myapp sh                                        # → vào filesystem ngó; có gì trong đó?
du -sh /app /usr/local /var                                         # → thư mục lớn nhất trong container
```

Mục tiêu cho Node runtime tốt: **&lt;150 MB tổng, &lt;50 MB layer writable**. Cho Go runtime: **&lt;25 MB tổng**. Cho Python web: **&lt;200 MB tổng** (stdlib Python "to mập").

Nếu bạn vượt xa các con số đó, lệnh prune phía trên + audit `.dockerignore` thường tìm ra chỗ rò.

<Recap>
- `.dockerignore` is the cheapest, highest-impact change in any Docker project. Add it before optimising anything else.
- Base image choice: alpine for most runtimes, slim for native-lib pain, distroless when security is paramount, full only in builder stages.
- BuildKit cache mounts (`RUN --mount=type=cache,…`) persist npm/pip/apt caches across builds without leaking into the final image. Add `# syntax=docker/dockerfile:1.7` at the top.
- Targets: Node &lt;150 MB, Go &lt;25 MB, Python web &lt;200 MB. If you are far above, run `docker history` and `du -sh` inside the container — the leak shows up. Next lesson: tags and registries — where these tiny, beautiful images actually go.
</Recap>
