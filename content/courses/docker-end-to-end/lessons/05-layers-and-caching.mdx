You will rebuild the same image a hundred times in a project's life. Five seconds versus fifty seconds matters. The difference between those two numbers is *cache hit rate*, and the cache hit rate is entirely a function of how you ordered the lines in your Dockerfile.

## How the cache decides

When Docker executes a Dockerfile, it walks instructions top to bottom. For each instruction, it asks one question:

> *"Have I built this exact instruction on top of this exact previous layer before? If yes, reuse the cached result. If no, run it fresh — and from this point onward, nothing further can be cached."*

That last clause is the cliff. The moment one line *misses* the cache, every line after it also misses. Even if line 10 is byte-identical to last build, it follows line 5 which changed, so its base is different, so the cache key is different, so Docker re-runs it.

That means **earlier lines are more important than later lines**. Put the stable stuff (base image, package manifest, dependency install) early. Put the volatile stuff (your source code) late.

## Cache quyết định ra sao

Khi Docker chạy Dockerfile, nó đi từ trên xuống. Mỗi instruction, nó hỏi đúng một câu:

> *"Tôi đã build chính instruction này bên trên chính layer trước đó này chưa? Có → dùng kết quả cache. Không → chạy mới — và từ điểm này trở đi, không gì còn cache được nữa."*

Mệnh đề cuối là vực sâu. Khoảnh khắc một dòng *miss* cache, mọi dòng sau cũng miss. Dù dòng 10 giống y lần build trước từng byte, nó vẫn theo sau dòng 5 đã đổi → base khác → cache key khác → Docker build lại.

Nghĩa là **dòng trước quan trọng hơn dòng sau**. Đặt thứ ổn định (base image, manifest dependency, cài deps) lên trên. Đặt thứ hay đổi (source code) xuống dưới.

## The pattern that wins

This is the single most important pattern in a Dockerfile. Memorise it.

```dockerfile
FROM node:20-alpine
WORKDIR /app

# ① Copy ONLY the dependency manifests first.
COPY package*.json ./

# ② Install deps. As long as package.json + lock don't change, this layer is cached.
RUN npm ci --omit=dev

# ③ Now copy the rest of the source. Code changes only bust this layer.
COPY . .

CMD ["node", "src/index.js"]
```

Why it works:

- Lines 1–4 only re-run when `package.json` or `package-lock.json` changes — i.e., when you add/remove a dependency. That happens maybe once a week.
- Line 5 (`COPY . .`) runs on every code change. That happens fifty times a day.
- The expensive thing (`npm ci`, often 30–60 seconds) sits *above* line 5, so it stays in cache.

Result: typical rebuild after a code change goes from ~60 s to ~2 s.

## What the build output is telling you

Run `docker build .` and look at every line. The keywords are everything:

```text
 => [internal] load build definition from Dockerfile           0.0s
 => => transferring dockerfile: 312B                            0.0s
 => [internal] load .dockerignore                              0.0s
 => => transferring context: 67B                                0.0s
 => [1/5] FROM docker.io/library/node:20-alpine                0.0s
 => CACHED [2/5] WORKDIR /app                                  0.0s    ← cached
 => CACHED [3/5] COPY package*.json ./                          0.0s    ← cached
 => CACHED [4/5] RUN npm ci --omit=dev                          0.0s    ← cached ⇐ the big win
 => [5/5] COPY . .                                              0.1s    ← rebuilt (no CACHED keyword)
 => exporting to image                                         0.2s
```

Two words to spot: `CACHED` (good) and the absence of it (rebuilt). If you see `CACHED [4/5] RUN npm ci` you have done it right.

## Pattern thắng cuộc

Đây là pattern quan trọng *nhất* trong một Dockerfile. Thuộc lòng.

```dockerfile
FROM node:20-alpine
WORKDIR /app

# ① Copy CHỈ các file manifest trước.
COPY package*.json ./

# ② Cài deps. Bao lâu package.json + lock không đổi, layer này còn cache.
RUN npm ci --omit=dev

# ③ Bây giờ mới copy phần source còn lại. Sửa code chỉ làm vỡ layer này.
COPY . .

CMD ["node", "src/index.js"]
```

Vì sao chạy:

- Dòng 1–4 chỉ build lại khi `package.json` hoặc `package-lock.json` đổi — tức khi bạn add/remove dependency. Việc đó cả tuần xảy ra một lần.
- Dòng 5 (`COPY . .`) chạy mỗi lần sửa code. Việc đó xảy ra 50 lần/ngày.
- Cái đắt tiền (`npm ci`, thường 30–60 giây) nằm *bên trên* dòng 5, nên còn cache.

Kết quả: rebuild sau khi sửa code thường từ ~60 s xuống ~2 s.

## Cố ý làm vỡ cache lúc nào

</VI>

<EN>

Cache hits are usually what you want, but not always. Two cases where you must bust it on purpose:

```bash
docker build --no-cache -t myapp .                                # → ignore the cache entirely
docker build --pull -t myapp .                                    # → re-pull base image even if locally cached
```

- `--no-cache` — *"start fresh."* Use when you suspect cache poisoning, or when a `RUN apt-get install` is silently picking up an old package list.
- `--pull` — *"make sure my base image is the latest tag."* `node:20-alpine` moves forward with security patches. Without `--pull`, you stay on the version you first downloaded forever.

A subtle one inside the Dockerfile itself:

```dockerfile
# Cache-busting "marker" instruction. Bumping the date forces everything below to rebuild.
ARG CACHE_BUST=2026-05-19
RUN echo "$CACHE_BUST" > /tmp/cachebust
```

Niche, but useful when you have a `RUN git clone …` or `RUN curl …` whose result you want to refresh on demand.

## Combine `RUN` instructions to keep image small

Each `RUN` becomes a layer, and layers carry their full filesystem diff. If you install `curl` in one layer and then remove it in the next, the first layer *still* contains curl on disk forever.

```dockerfile
# ❌ Bad: curl stays in the image even after the "removal"
RUN apk add --no-cache curl
RUN curl -L https://example.com/file.tar.gz -o /tmp/f.tar.gz
RUN tar -xz -C /usr/local -f /tmp/f.tar.gz && rm /tmp/f.tar.gz
RUN apk del curl

# ✅ Good: download, extract, remove the tool all in one layer
RUN apk add --no-cache --virtual .build-deps curl \
 && curl -L https://example.com/file.tar.gz -o /tmp/f.tar.gz \
 && tar -xz -C /usr/local -f /tmp/f.tar.gz \
 && rm /tmp/f.tar.gz \
 && apk del .build-deps
```

The `--virtual .build-deps` trick is Alpine-specific: it groups installed packages under a label so a single `apk del .build-deps` removes them all. Debian/Ubuntu equivalent: `apt-get install` + `apt-get remove --purge`.

Cache hit thường là cái bạn muốn, nhưng không phải lúc nào. Hai trường hợp phải cố ý làm vỡ:

```bash
docker build --no-cache -t myapp .                                # → bỏ cache hoàn toàn
docker build --pull -t myapp .                                    # → kéo lại base image dù đã có local
```

- `--no-cache` — *"làm lại từ đầu."* Dùng khi nghi cache nhiễm, hoặc khi một `RUN apt-get install` đang âm thầm xài danh sách package cũ.
- `--pull` — *"đảm bảo base image của tôi là tag mới nhất."* `node:20-alpine` vẫn được vá bảo mật. Không `--pull` là kẹt mãi ở bản đầu tiên bạn tải.

Một mẹo tinh tế trong chính Dockerfile:

```dockerfile
# Instruction "marker" làm vỡ cache. Đổi ngày là mọi dòng phía dưới build lại.
ARG CACHE_BUST=2026-05-19
RUN echo "$CACHE_BUST" > /tmp/cachebust
```

Ít dùng, nhưng hữu ích khi có `RUN git clone …` hay `RUN curl …` mà bạn muốn refresh on demand.

## Gộp các `RUN` để image nhỏ

Mỗi `RUN` là một layer, và layer giữ trọn diff filesystem. Cài `curl` ở một layer rồi xoá ở layer kế, layer đầu *vẫn* chứa curl trên đĩa, mãi mãi.

```dockerfile
# ❌ Sai: curl còn trong image dù đã "xoá"
RUN apk add --no-cache curl
RUN curl -L https://example.com/file.tar.gz -o /tmp/f.tar.gz
RUN tar -xz -C /usr/local -f /tmp/f.tar.gz && rm /tmp/f.tar.gz
RUN apk del curl

# ✅ Đúng: tải, giải nén, xoá tool trong cùng một layer
RUN apk add --no-cache --virtual .build-deps curl \
 && curl -L https://example.com/file.tar.gz -o /tmp/f.tar.gz \
 && tar -xz -C /usr/local -f /tmp/f.tar.gz \
 && rm /tmp/f.tar.gz \
 && apk del .build-deps
```

Mẹo `--virtual .build-deps` chỉ có ở Alpine: gom các package đã cài dưới một nhãn để một lệnh `apk del .build-deps` xoá hết. Trên Debian/Ubuntu: `apt-get install` + `apt-get remove --purge`.

## Three exercises you can do right now

1. Take the Dockerfile from lesson 4. Change one line in `src/index.js`. Run `docker build` and time it. Now move `COPY . .` to *before* `COPY package*.json ./` and `RUN npm ci`. Change `src/index.js` again. Time it again. You will see the difference in seconds, live.

2. In a fresh build, add a Python-only requirements file and a `pip install`. Verify that subsequent code changes still use the npm install cache. Mixed-language images are normal; the ordering principle holds across all package managers.

3. Add `RUN apk add --no-cache curl` somewhere in the middle of the file. Run `docker image history` after build and find that layer's size. Then move it to the very last `RUN` (still keep the install grouped with `&& apk del curl` in one line). Re-run history. See the difference in image size.

<Recap>
- Every Dockerfile instruction is a layer with a cache key. One miss invalidates everything below it.
- The golden order: **stable on top, volatile on bottom**. `COPY package*.json ./` then `RUN npm ci` *before* `COPY . .`. Code changes stop triggering a full reinstall.
- Read the build log for `CACHED` vs no-CACHED. Cache hits are visible.
- Combine related `RUN` commands with `&&` and clean up in the same layer; leftover packages stay on disk if you remove them in a later layer. Lesson 6 takes the next step — multi-stage builds let you ship an image that does not include your build tools at all.
</Recap>
