So far we have used images other people built. Now we build our own. The thing that turns "a folder with my code" into "an image I can `docker run`" is a single text file called the **Dockerfile**.

A Dockerfile is a recipe in the most literal sense: a list of instructions, top to bottom, each one applied to the result of the one before it. The daemon reads it, executes each instruction in a temporary container, and snapshots the result as a new layer. When it reaches the end, it bundles all those layers with a bit of metadata (default command, exposed ports, working directory) and you have an image.

## Dockerfile nhỏ nhất

</VI>

```dockerfile
# Dockerfile
FROM node:20-alpine                # → start from official Node 20 on Alpine Linux
WORKDIR /app                       # → cd /app inside the image (creates the dir if missing)
COPY package*.json ./              # → copy package.json + lockfile from host into /app
RUN npm ci --omit=dev              # → install only production deps; produces a new layer
COPY . .                           # → copy the rest of the source code
EXPOSE 3000                        # → document: this app listens on 3000 (does NOT publish it)
CMD ["node", "src/index.js"]       # → default command when the container starts
```

### 1. `FROM` — the starting point

`FROM` declares the base image you build *on top of*. Without it, you have nothing — there is no kernel, no filesystem, nothing. The base image is somebody else's published image; it has already done the work of *"a Linux box with the right runtime installed"*.

```dockerfile
FROM node:20-alpine                # → small, official, fast to pull
FROM python:3.13-slim              # → official Python
FROM scratch                       # → truly empty (Go binaries; advanced)
```

Pick the most specific tag you reasonably can. `node:20-alpine` is better than `node:latest` because `latest` rolls forward without warning. Pinning beats surprise.

### 2. `WORKDIR` — set the working directory

`WORKDIR /app` is equivalent to *"`mkdir -p /app && cd /app`, for this and every subsequent instruction"*. It also sets the default directory when the container starts. Use it; without it you end up with `COPY . /usr/src/app` repeated everywhere.

### 3. `COPY` — pull files from the host into the image

`COPY <src> <dest>`. Source paths are relative to the **build context** (the folder you pass to `docker build`). Destination paths are inside the image, anchored at `WORKDIR`.

```dockerfile
COPY package*.json ./              # → just dependency manifests; both 'package.json' and 'package-lock.json'
COPY src/ ./src/                   # → only the src directory
COPY . .                           # → everything in the build context (respects .dockerignore — see lesson 7)
```

A sibling instruction `ADD` exists. Use `COPY` 99% of the time. `ADD` has magic (auto-extract tar archives, fetch URLs) that surprises people; stick to `COPY`.

### 4. `RUN` — execute a command at build time

`RUN` runs a command *inside* a throwaway container based on the current state, and snapshots the result. This is how you install packages, compile code, generate files.

```dockerfile
RUN npm ci --omit=dev              # → install production npm deps
RUN apk add --no-cache curl        # → install curl using Alpine's package manager
RUN npm run build                  # → run a build script
```

Each `RUN` makes a new layer. Combining related commands with `&&` keeps the layer count and the image size sensible — we will see this more in lesson 5.

### 5. `ENV` — set environment variables

```dockerfile
ENV NODE_ENV=production            # → visible to the process at runtime
ENV PORT=3000
```

Anything set with `ENV` is baked into the image. The process inside the container will read it via `process.env.NODE_ENV` etc. Do not put secrets here — they end up in the image history forever (we cover the right way in lesson 11).

### 6. `EXPOSE` — document a port

`EXPOSE 3000` is **documentation only**. It does *not* publish the port to the host. Without `-p 3000:3000` (or a Compose equivalent), nothing on your laptop can reach the container's 3000. `EXPOSE` just tells humans and tools *"this app expects to be reached on port 3000."*

### 7. `CMD` — the default command

`CMD` is what the container runs when no command is passed to `docker run`. Two forms:

```dockerfile
CMD ["node", "src/index.js"]       # → exec form (preferred). PID 1 is node directly.
CMD node src/index.js              # → shell form. PID 1 is /bin/sh -c, which then runs node.
```

Use the **exec form** (the JSON array) for anything you care about. Why? Because in shell form, your app does not receive SIGTERM cleanly — the shell does. That breaks graceful shutdown. The exec form makes your process PID 1 in the container, which is what you almost always want.

### 1. `FROM` — điểm khởi đầu

`FROM` khai báo image gốc bạn build *bên trên*. Không có nó là tay không — không kernel, không filesystem, không gì. Image gốc là image do người khác đã publish; nó làm sẵn cái việc *"một máy Linux có sẵn runtime cần thiết"*.

```dockerfile
FROM node:20-alpine                # → nhỏ, official, kéo nhanh
FROM python:3.13-slim              # → Python official
FROM scratch                       # → rỗng thật sự (cho binary Go; nâng cao)
```

Chọn tag cụ thể nhất hợp lý. `node:20-alpine` tốt hơn `node:latest` vì `latest` âm thầm trượt tới phiên bản mới. Pin tốt hơn bị ngạc nhiên.

### 2. `WORKDIR` — đặt thư mục làm việc

`WORKDIR /app` tương đương *"`mkdir -p /app && cd /app`, cho instruction này và mọi instruction phía sau"*. Đồng thời đặt thư mục mặc định khi container start. Dùng nó; không dùng là viết `COPY . /usr/src/app` lặp đi lặp lại.

### 3. `COPY` — kéo file từ host vào image

`COPY <src> <dest>`. Source tương đối so với **build context** (thư mục bạn truyền cho `docker build`). Destination nằm trong image, neo theo `WORKDIR`.

```dockerfile
COPY package*.json ./              # → chỉ manifest dependencies; cả 'package.json' lẫn 'package-lock.json'
COPY src/ ./src/                   # → chỉ thư mục src
COPY . .                           # → toàn bộ build context (tuân theo .dockerignore — bài 7)
```

Có một instruction "anh em" là `ADD`. 99% trường hợp xài `COPY`. `ADD` có phép thuật (tự giải nén tar, fetch URL) hay làm người ta ngạc nhiên; bám lấy `COPY` cho lành.

### 4. `RUN` — chạy command lúc build

`RUN` chạy command *bên trong* một container tạm dựa trên trạng thái hiện tại, rồi snapshot kết quả. Đây là cách bạn cài package, compile code, sinh file.

```dockerfile
RUN npm ci --omit=dev              # → cài npm deps production
RUN apk add --no-cache curl        # → cài curl bằng package manager của Alpine
RUN npm run build                  # → chạy script build
```

Mỗi `RUN` tạo một layer mới. Gộp các command liên quan bằng `&&` giúp số layer và kích thước image hợp lý — bài 5 sẽ rõ hơn.

### 5. `ENV` — đặt biến môi trường

```dockerfile
ENV NODE_ENV=production            # → process thấy biến này lúc runtime
ENV PORT=3000
```

Mọi thứ đặt bằng `ENV` được nướng vào image. Process trong container đọc qua `process.env.NODE_ENV` v.v. Đừng nhét secret vào đây — nó nằm trong image history vĩnh viễn (bài 11 sẽ chỉ cách đúng).

### 6. `EXPOSE` — chỉ là tài liệu

`EXPOSE 3000` **chỉ là tài liệu**. Nó *không* publish port ra host. Không có `-p 3000:3000` (hoặc tương đương trong Compose) thì máy bạn không gọi tới được port 3000 trong container. `EXPOSE` chỉ nói với con người và tool *"app này muốn được gọi tới qua port 3000"*.

### 7. `CMD` — command mặc định

`CMD` là cái container chạy khi bạn không truyền command cho `docker run`. Có hai dạng:

```dockerfile
CMD ["node", "src/index.js"]       # → dạng exec (nên dùng). PID 1 là node thẳng.
CMD node src/index.js              # → dạng shell. PID 1 là /bin/sh -c, sau đó nó chạy node.
```

Dùng **dạng exec** (mảng JSON) cho mọi thứ bạn quan tâm. Vì sao? Vì dạng shell, app của bạn không nhận SIGTERM đàng hoàng — shell nhận. Hỏng luôn graceful shutdown. Dạng exec giúp process của bạn là PID 1 trong container, đó là cái bạn gần như luôn muốn.

### 8. `ENTRYPOINT` — phần không cấu hình được của command

`ENTRYPOINT` và `CMD` làm cả thế giới rối. Đây là quy tắc giải quyết:

</VI>

<Callout type="note" title="CMD vs ENTRYPOINT — the rule that fixes everything">
**EN —** Mental model: `ENTRYPOINT` is the **fixed binary**; `CMD` is the **default arguments**. At container start, Docker runs `ENTRYPOINT + CMD` concatenated. Anything you pass to `docker run <image> <args>` **replaces `CMD`**, but leaves `ENTRYPOINT` alone.

```dockerfile
ENTRYPOINT ["node"]
CMD ["src/index.js"]
```

- `docker run myapp` → runs `node src/index.js` (default).
- `docker run myapp src/cli.js` → runs `node src/cli.js` (overrode args).
- `docker run myapp` cannot accidentally run anything other than `node`. That is the safety `ENTRYPOINT` buys you.

For 90% of web apps, **only `CMD`** is fine. Use `ENTRYPOINT` when your image is a *tool* (a CLI, a database, anything you want to lock to one binary).

**VI —** Mô hình tư duy: `ENTRYPOINT` là **binary cố định**; `CMD` là **default arguments**. Lúc container start, Docker chạy `ENTRYPOINT + CMD` nối lại. Cái bạn truyền cho `docker run <image> <args>` **thay thế `CMD`**, nhưng `ENTRYPOINT` vẫn nguyên.

```dockerfile
ENTRYPOINT ["node"]
CMD ["src/index.js"]
```

- `docker run myapp` → chạy `node src/index.js` (mặc định).
- `docker run myapp src/cli.js` → chạy `node src/cli.js` (đè args).
- `docker run myapp` không lỡ tay chạy gì khác ngoài `node`. Đó là cái an toàn `ENTRYPOINT` mang lại.

90% web app, **chỉ cần `CMD`** là đủ. Dùng `ENTRYPOINT` khi image của bạn là *công cụ* (một CLI, một database, thứ bạn muốn khoá vào đúng một binary).
</Callout>

## A complete, runnable example

Make a folder, put four files in it, and you have a real Dockerised app.

`package.json`:

```json
{
  "name": "first-image",
  "version": "1.0.0",
  "main": "src/index.js",
  "scripts": { "start": "node src/index.js" },
  "dependencies": { "express": "^4.21.0" }
}
```

`src/index.js`:

```js
import express from 'express';
const app = express();
app.get('/', (_, res) => res.send('hello from a container'));
app.listen(3000, () => console.log('listening on 3000'));
```

`Dockerfile`:

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "src/index.js"]
```

`.dockerignore` (we will go deeper in lesson 7):

```
node_modules
.git
.env
```

Build and run:

```bash
docker build -t first-image:1.0 .                                  # → build, tag as 'first-image:1.0'. The '.' is the build context.
docker run --rm -p 3000:3000 first-image:1.0                       # → run, expose port 3000
curl http://localhost:3000                                          # → "hello from a container"
```

What just happened in seven steps:

1. The daemon read every file in `.` (the build context) and sent it to itself.
2. It pulled `node:20-alpine` from Docker Hub (only the first time).
3. It set `/app` as the working directory.
4. It copied your two manifest files; ran `npm ci`; snapshot the `node_modules` layer.
5. It copied the rest of your code.
6. It tagged the final layer stack as `first-image:1.0`.
7. `docker run` started a container from that image, mapped port 3000, and the Express server replied to your `curl`.

## Five mistakes new authors make

</EN>

<VI>

## Năm lỗi người mới hay mắc

</VI>

<EN>

1. **No `.dockerignore`** — your `node_modules` gets copied *and then npm install runs anyway*. Builds become slow and image becomes huge. Lesson 7 covers `.dockerignore` properly.
2. **Copying everything before installing deps** — every code change busts the dependency cache, so every build re-installs node_modules. Lesson 5 explains why ordering matters.
3. **Shell-form `CMD`** — `CMD node src/index.js` instead of `CMD ["node", "src/index.js"]`. Breaks SIGTERM. Always use exec form.
4. **Putting secrets in `ENV`** — they end up in the image. Anyone who pulls the image can read them via `docker image inspect`. Use runtime env (`-e` or `--env-file`) instead.
5. **Forgetting `EXPOSE` is documentation only** — and being surprised that `localhost:3000` does not work because no `-p` was passed.

</EN>

<VI>

1. **Không có `.dockerignore`** — `node_modules` bị copy vào *rồi npm install vẫn chạy*. Build chậm, image to. Bài 7 chỉ chi tiết `.dockerignore`.
2. **Copy tất cả trước khi cài deps** — chỉ cần sửa code là cache deps vỡ, build nào cũng cài lại node_modules. Bài 5 giải thích thứ tự rất quan trọng.
3. **`CMD` dạng shell** — `CMD node src/index.js` thay vì `CMD ["node", "src/index.js"]`. Hỏng SIGTERM. Luôn dùng exec form.
4. **Nhét secret vào `ENV`** — nó nằm trong image. Ai pull image về cũng đọc được qua `docker image inspect`. Dùng env lúc runtime (`-e` hoặc `--env-file`) thay vì vậy.
5. **Quên rằng `EXPOSE` chỉ là tài liệu** — rồi ngạc nhiên vì `localhost:3000` không vào được do quên `-p`.

</VI>

<Recap>
- A Dockerfile is a text recipe. Each instruction makes one image layer. `FROM, WORKDIR, COPY, RUN, ENV, EXPOSE, CMD, ENTRYPOINT` cover 95% of real files.
- `EXPOSE` documents a port; `-p` publishes it. They are not the same.
- `CMD` is overridable defaults; `ENTRYPOINT` is the fixed binary. For web apps you usually only need `CMD`. Always use the JSON-array (exec) form.
- `docker build -t name:tag .` builds the current folder. The `.` is the build context — everything in it (minus `.dockerignore`) gets shipped to the daemon. Lesson 5 explains why ordering the instructions changes rebuild speed by 10×.
</Recap>
