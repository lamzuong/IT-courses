## A complete GitHub Actions workflow

`.github/workflows/deploy.yml`:

```yaml
name: build-and-deploy

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_OWNER: ${{ github.repository_owner }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write                                                 # → needed to push to GHCR

    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3                           # → BuildKit-powered builder

      - uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # ── api image ────────────────────────────────────────────────
      - name: api — meta
        id: api_meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_OWNER }}/coffee-api
          tags: |
            type=sha,prefix=,format=short                             # → e.g. 'a1b2c3d'
            type=ref,event=branch                                     # → 'main'
            type=raw,value=latest,enable={{is_default_branch}}        # → also 'latest'

      - name: api — build & push
        uses: docker/build-push-action@v6
        with:
          context: ./api
          push: true
          tags: ${{ steps.api_meta.outputs.tags }}
          cache-from: type=gha
          cache-to:   type=gha,mode=max

      # ── web image ────────────────────────────────────────────────
      - name: web — meta
        id: web_meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_OWNER }}/coffee-web
          tags: |
            type=sha,prefix=,format=short
            type=ref,event=branch
            type=raw,value=latest,enable={{is_default_branch}}

      - name: web — build & push
        uses: docker/build-push-action@v6
        with:
          context: ./web
          push: true
          tags: ${{ steps.web_meta.outputs.tags }}
          cache-from: type=gha
          cache-to:   type=gha,mode=max

      - name: Image scan (fail on HIGH/CRITICAL)
        uses: docker/scout-action@v1
        with:
          command: cves
          image: ${{ env.REGISTRY }}/${{ env.IMAGE_OWNER }}/coffee-api:${{ github.sha }}
          only-severities: critical,high
          exit-code: true

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production                                           # → optional gate for manual approval
    steps:
      - name: SSH and pull-and-up
        uses: appleboy/ssh-action@v1.0.3
        with:
          host:     ${{ secrets.PROD_HOST }}
          username: ${{ secrets.PROD_USER }}
          key:      ${{ secrets.PROD_SSH_KEY }}
          script: |
            cd /srv/coffee-shop
            export TAG=${GITHUB_SHA:0:7}
            docker compose pull
            docker compose up -d
            docker image prune -af --filter "until=168h"
```

## How the cache makes CI fast

The two lines that matter:

```yaml
cache-from: type=gha
cache-to:   type=gha,mode=max
```

`type=gha` stores BuildKit layer cache in GitHub Actions cache. The next run pulls the cache before building; identical layers are reused without re-running their commands. The first build is slow (cache miss); every build after is *almost as fast as your laptop's rebuild*. Without these lines, every CI build is a `--no-cache` build.

## What `docker/metadata-action` is doing

Computing the right set of tags by hand is fiddly. `metadata-action` generates them from a small DSL:

- `type=sha,format=short` → `a1b2c3d` (the commit SHA, short form).
- `type=ref,event=branch` → the branch name (`main`).
- `type=raw,value=latest,enable={{is_default_branch}}` → also tag `latest`, but only on the default branch.

The image ends up with multiple tags pointing at the same digest — convenient for consumers who want to pin to a SHA *or* track `latest`.

## The deploy step, in detail

```yaml
- name: SSH and pull-and-up
  uses: appleboy/ssh-action@v1.0.3
  with:
    script: |
      cd /srv/coffee-shop
      export TAG=${GITHUB_SHA:0:7}
      docker compose pull
      docker compose up -d
      docker image prune -af --filter "until=168h"
```

On the server, your `docker-compose.yml` and `.env` already exist in `/srv/coffee-shop`. `.env` has `TAG=${TAG}` placeholder, or you set it directly per-deploy. Three lines do the work:

1. `docker compose pull` — fetch the new images by SHA.
2. `docker compose up -d` — replace each service whose image changed. Compose stops the old container with SIGTERM, waits the grace period, starts the new one, waits for health to pass.
3. `docker image prune` — clean up old image layers a week old.

No downtime if your healthcheck is right and your app handles SIGTERM (lesson 16).

## Vì sao cache làm CI nhanh

Hai dòng đáng giá:

```yaml
cache-from: type=gha
cache-to:   type=gha,mode=max
```

`type=gha` lưu cache layer BuildKit vào cache của GitHub Actions. Lần chạy kế kéo cache trước khi build; layer giống y được tái dùng, không phải chạy lại lệnh. Build đầu chậm (miss cache); mọi build sau *gần nhanh bằng rebuild trên laptop*. Không có hai dòng này, mỗi build CI là build `--no-cache`.

## `docker/metadata-action` đang làm gì

Tính tay đúng tập tag rất rối. `metadata-action` sinh từ một DSL nhỏ:

- `type=sha,format=short` → `a1b2c3d` (SHA commit, dạng ngắn).
- `type=ref,event=branch` → tên branch (`main`).
- `type=raw,value=latest,enable={{is_default_branch}}` → tag thêm `latest`, chỉ trên default branch.

Image cuối có nhiều tag trỏ tới cùng digest — tiện cho người dùng muốn pin theo SHA *hoặc* track `latest`.

## Bước deploy, chi tiết

```yaml
- name: SSH và pull-and-up
  uses: appleboy/ssh-action@v1.0.3
  with:
    script: |
      cd /srv/coffee-shop
      export TAG=${GITHUB_SHA:0:7}
      docker compose pull
      docker compose up -d
      docker image prune -af --filter "until=168h"
```

Trên server, `docker-compose.yml` và `.env` đã sẵn ở `/srv/coffee-shop`. `.env` có placeholder `TAG=${TAG}`, hoặc bạn set trực tiếp mỗi lần deploy. Ba dòng làm việc:

1. `docker compose pull` — fetch image mới theo SHA.
2. `docker compose up -d` — thay từng service có image đổi. Compose stop container cũ bằng SIGTERM, đợi grace period, start container mới, đợi health pass.
3. `docker image prune` — dọn layer image cũ hơn 1 tuần.

Không downtime nếu healthcheck đúng và app xử lý SIGTERM (bài 16).

## Hệ CI khác, cùng shape

</VI>

<EN>

The shape is universal. Translate the YAML, the pipeline is the same.

- **GitLab CI** — `docker:dind` service, same build/push commands, `docker login $CI_REGISTRY -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD`.
- **CircleCI** — `setup_remote_docker`, same commands, cache via CircleCI workspaces.
- **Buildkite/Jenkins** — same `docker buildx build --push` shape; the difference is auth and where to store the cache.

The action names change. The four steps do not.

## Local-equivalent commands

Want to do exactly what the workflow does, by hand? Two commands:

```bash
docker buildx build \
  --platform linux/amd64 \
  -t ghcr.io/myteam/coffee-api:$(git rev-parse --short HEAD) \
  -t ghcr.io/myteam/coffee-api:latest \
  --push \
  ./api

ssh prod.example.com \
  'cd /srv/coffee-shop && TAG='$(git rev-parse --short HEAD)' docker compose pull && docker compose up -d'
```

This is the *whole* CI/CD pipeline minus the YAML. Useful for emergency hotfixes when CI is sick.

Shape mang tính phổ quát. Dịch YAML thôi, pipeline giống nhau.

- **GitLab CI** — service `docker:dind`, cùng lệnh build/push, `docker login $CI_REGISTRY -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD`.

Tên action đổi. Bốn bước không đổi.

<Callout type="note" title="Café analogy">
**EN —** A CI/CD pipeline is *the chain's overnight delivery service*. The moment HQ approves a new recipe (`git push main`), the warehouse prints the recipe card with this morning's date (build), couriers it to every shop's mailbox by dawn (push), and the manager reads the new card and quietly swaps it onto the wall while the previous barista finishes her last drink (deploy). Customers walking in at 7 AM order from the new recipe without ever knowing the menu changed at 3 AM.

Khoảnh khắc HQ duyệt công thức mới (`git push main`), kho in thẻ công thức với ngày sáng nay (build), giao tới hộp thư mỗi cửa hàng trước bình minh (push), và quản lý đọc thẻ mới và lặng lẽ đổi lên tường trong khi barista cũ pha xong ly cuối (deploy). Khách 7 giờ sáng order theo công thức mới mà không biết menu đã đổi lúc 3 giờ sáng.

<Recap>
- Four steps: checkout → build → push → deploy. Every CI system implements the same steps; only the YAML changes.
- BuildKit + `cache-from/cache-to: type=gha` makes CI almost as fast as local rebuilds. The two lines are not optional.
- `docker/metadata-action` generates the tag set; `docker/build-push-action` does the build and push; `docker scout` blocks CVE regressions.
- The deploy step is `ssh → compose pull → compose up -d`. With healthchecks and graceful shutdown (lesson 16), there is no downtime. Lesson 19 looks past a single host — where Compose stops being enough.
</Recap>
