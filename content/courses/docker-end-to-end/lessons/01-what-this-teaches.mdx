Docker exists because of one sentence: *"it works on my machine."* That sentence has destroyed more weekends than any other in software. Containers are the answer to it — a way to package an application together with the exact filesystem, libraries, and runtime it needs, so that the box you tested on your laptop is bit-for-bit the same box that runs in production.

Everything else in this course — Dockerfiles, layers, volumes, networks, Compose, registries, healthchecks — is just refinement of that one idea: ship the environment with the code.

## Why not just use a VM?

That was the previous answer, and it still works. A virtual machine boots a whole operating system on top of your operating system. It is reliable and it is isolated. It is also slow to start (minutes), heavy on disk (gigabytes), and brutal on RAM. You cannot ship 30 services this way on a laptop.

A container is the same idea — *"give me an isolated environment"* — but it shares the host's kernel instead of booting its own. The result is wildly cheaper:

A container starts in milliseconds, weighs tens of megabytes, and lets you pack dozens of services on a single laptop without sweating. That is the entire trade-off.

## The recipe / dish metaphor

Two words you must never confuse:

- **Image** — the recipe. A read-only template. Built once, shared, versioned, pulled from a registry.
- **Container** — a dish that came out of the recipe. The running thing. Has state, logs, a PID, can be stopped, started, deleted.

One recipe ⇒ many dishes. From a single `node:20-alpine` image you can spin up a hundred containers, each isolated from the others, each with its own writable layer on top.

## The story we will containerise

Across this course we are going to build, step by step, a real-looking online ordering site for a small coffee shop. Here is what the finished thing has, so you know where every lesson is heading:

- A **Next.js** frontend the customer sees.
- A **Node/Express** API the frontend calls.
- A **Postgres** database where orders live.
- A **Redis** cache for session and cart state.
- An **Nginx** reverse proxy in front, so there is one domain and one port exposed to the outside.

Five services. By lesson 15 they are all running together with one `docker compose up`. By lesson 18 they auto-build and ship on every `git push`.

## Phép so sánh: công thức / dĩa món

Hai từ đừng bao giờ lẫn lộn:

- **Image** — công thức nấu ăn. Read-only, dùng làm khuôn. Build một lần, share, version, kéo về từ registry.
- **Container** — dĩa món ra từ công thức. Cái đang chạy thật. Có state, có log, có PID, có thể stop / start / xoá.

Một công thức ⇒ nhiều dĩa món. Từ image `node:20-alpine` bạn có thể bật một trăm container, mỗi cái cô lập riêng, mỗi cái có lớp ghi đè riêng bên trên.

## What you need to follow along

- A laptop with **Docker Desktop** installed (Mac, Windows, or Linux). We install it in lesson 2.
- About 4 GB of free RAM and 10 GB of free disk. Not negotiable — images and volumes eat space.
- Familiarity with the terminal. You should be comfortable with `cd`, `ls`, basic shell commands, and reading error output.
- *Optional but lovely:* a code editor with Docker syntax support (VS Code with the Docker extension is what most of us use).

You do not need to know Kubernetes. You do not need to have run a server before. You do not need to know what a kernel is. We explain everything before we use it.

## How the lessons are structured

Every lesson follows the same shape, and the shape is deliberate:

1. **Why** — what problem this concept solves. If you skip this you will be typing commands you cannot debug.
2. **What** — the new idea or syntax.
3. **Examples** — three runnable snippets, increasing in realism. The first is the smallest possible thing that works; the last is something that looks like real code.
4. **Diagram** — an SVG so you can see, not just read, what is happening.
5. **Recap** — four bullets you can come back to without re-reading the lesson.

You can read every lesson in roughly 15 minutes. You can do it hands-on in 30. Total course time, hands-on with the project: **two solid weekends**.

## A small promise

By lesson 20, you will be the person at work who actually knows how this stuff works. Not "I copied a Dockerfile from Stack Overflow" knows. *Knows.* Can write the Dockerfile from a blank file. Can debug why a container restarts five seconds after it boots. Can read a Compose file and predict the network topology without running it. That is what *end-to-end* means in the title.

## Cấu trúc một bài học

Mỗi bài đều theo cùng một khung, và khung này cố ý:

1. **Vì sao (Why)** — vấn đề mà khái niệm này giải. Bỏ qua phần này là gõ lệnh mà không debug được.
2. **Cái gì (What)** — ý tưởng hoặc cú pháp mới.
3. **Ví dụ (Examples)** — ba đoạn code chạy được, độ thật tăng dần. Cái đầu là phiên bản nhỏ nhất chạy được; cái cuối giống code thật.
4. **Diagram** — SVG để bạn *nhìn* thấy, không chỉ *đọc*, chuyện đang xảy ra.
5. **Recap** — 4 gạch đầu dòng có thể quay lại tra sau mà không cần đọc lại bài.

Mỗi bài đọc khoảng 15 phút, làm tay khoảng 30 phút. Toàn khoá làm tay cùng project: **hai cuối tuần đủ.**
