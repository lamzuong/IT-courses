## Image = recipe. Container = dish.

An **image** is a read-only stack of filesystem layers plus a tiny bit of metadata (what command to run, which port to expose, which env vars are set by default). You build it once, push it to a registry, pull it back anywhere. Nothing about it changes.

A **container** is what you get when the daemon takes an image and adds a thin **writable layer** on top. Anything the running process writes — log files, temp files, downloaded packages — lives in that writable layer. Two containers from the same image have their own writable layers; they cannot see each other's writes.

## Image = công thức. Container = dĩa món.

**Image** là một chồng layer filesystem read-only, kèm chút metadata (chạy command nào, mở port nào, env mặc định ra sao). Build một lần, push lên registry, pull về bất cứ đâu. Không có gì thay đổi.

**Container** là cái bạn có khi daemon lấy image đó và thêm một **layer writable** mỏng lên trên. Tiến trình chạy ghi gì — log, file tạm, package tải về — đều rơi vào layer writable đó. Hai container cùng image có hai layer writable riêng; không nhìn thấy thứ bên kia ghi.

This is also why your **data disappears when you remove a container**. The writable layer is part of the container; `docker rm` deletes it. Anything you cared about — database files, uploaded images — needs to live in a **volume** (lesson 9), which is outside the writable layer.

- **created** — container exists on disk but has never run. `docker create` produces this state. Useful in scripts where you want to configure mounts/networks before starting.
- **running** — your process is alive inside the container.
- **paused** — all processes inside are frozen via the Linux *cgroup freezer*. Memory is preserved; CPU is taken away. `docker pause` / `unpause`. Niche, but useful when debugging a race condition.
- **stopped (exited)** — process exited (normally or because you `docker stop`-ed it). Filesystem is still there. You can `start` it again or `rm` it.
- **removed** — gone. Writable layer deleted. Image untouched.

## Image storage: where does it actually live?

When you `docker pull nginx`, the daemon downloads the layers and stores them under its data root (on Linux: `/var/lib/docker`). On Mac/Windows the daemon lives inside the VM, but the principle is identical.

Each layer is content-addressed by a SHA-256 hash. *"If a layer with that hash already exists, do not download it again."* This is how pulling a second image based on the same `node:20-alpine` base is nearly instant — Docker already has those layers.

Three commands to look at this reality:

```bash
docker images                                                     # → list images (REPOSITORY  TAG  IMAGE ID  SIZE)
docker image inspect nginx                                        # → JSON: layers, env, cmd, config
docker image history nginx                                        # → one row per layer, with size and command that made it
```

Run that last one. It is a peek behind the curtain.

## Pulling vs running

</EN>

<VI>

- **created** — container đã có trên đĩa nhưng chưa từng chạy. `docker create` sinh ra trạng thái này. Hữu ích trong script muốn cấu hình mount/network trước khi start.
- **running** — process bên trong đang sống.
- **paused** — toàn bộ process bên trong bị đóng băng bằng *cgroup freezer* của Linux. Còn RAM, không còn CPU. `docker pause` / `unpause`. Ít dùng, nhưng có ích khi debug race condition.
- **stopped (exited)** — process đã thoát (do tự nó, hoặc do `docker stop`). Filesystem vẫn còn. Có thể `start` lại hoặc `rm`.
- **removed** — biến mất. Layer writable bị xoá. Image không bị động tới.

## Image nằm ở đâu trên đĩa?

Khi `docker pull nginx`, daemon tải các layer về và lưu trong data root (trên Linux: `/var/lib/docker`). Trên Mac/Windows, daemon ở trong VM, nhưng nguyên tắc giống nhau.

Mỗi layer được định danh bằng hash SHA-256 của nội dung. *"Nếu layer với hash đó đã có rồi, không tải lại."* Đây là lý do kéo image thứ hai cùng base `node:20-alpine` gần như tức thì — Docker đã có sẵn layer đó.

Ba lệnh nhìn thấy việc này:

```bash
docker images                                                     # → list image (REPOSITORY  TAG  IMAGE ID  SIZE)
docker image inspect nginx                                        # → JSON: layer, env, cmd, config
docker image history nginx                                        # → mỗi dòng là một layer, kèm size và command tạo ra nó
```

Chạy lệnh cuối. Đó là một lần ngó vào hậu trường.

## `pull` vs `run`

</VI>

<EN>

```bash
docker pull node:20-alpine                                        # → only downloads the image
docker run node:20-alpine node -e "console.log('hi')"             # → run a one-shot command in it
```

`pull` only ever downloads. `run` will pull-if-missing, then create, then start. 95% of the time you only need `run` — but knowing they are separable is what lets you write a CI step that *pre-pulls* images on a build machine while doing other work.

## Three runs, three lessons

```bash
# 1. One-shot — run, print, exit.
docker run --rm alpine echo "from inside the container"            # → 'from inside the container'

# 2. Interactive — open a shell, poke around, exit.
docker run --rm -it alpine sh                                      # → drops you in /# prompt
# inside: ls /, cat /etc/os-release, exit

# 3. Long-running — server in background, browser visit, then clean up.
docker run -d -p 8080:80 --name web nginx                          # → start nginx detached
curl http://localhost:8080                                          # → HTML of welcome page
docker stop web && docker rm web                                   # → clean up
```

Two new flags worth committing:

- `--rm` — **auto-remove on exit**. Without it, every one-shot container piles up in `docker ps -a`. With it, the container is gone the instant it exits. Use it for anything throwaway.
- `-it` — **interactive + TTY**. The two flags you need when you want to actually type into the container.

```bash
# 1. One-shot — chạy, in ra, thoát.
docker run --rm alpine echo "từ bên trong container"               # → in dòng đó ra

# 3. Chạy lâu — server chạy nền, mở trình duyệt, dọn dẹp.
docker run -d -p 8080:80 --name web nginx                          # → bật nginx chạy nền

docker stop web && docker rm web                                   # → dọn
```

Hai cờ mới nên thuộc:

- `--rm` — **tự xoá khi thoát**. Không có nó, mỗi container one-shot đều chất đống trong `docker ps -a`. Có nó, container biến mất ngay khi thoát. Dùng cho mọi thứ chạy xài rồi vứt.
- `-it` — **interactive + TTY**. Hai cờ phải có khi bạn muốn thật sự *gõ* vào container.

## What you can prove right now

Run these and notice the times. The point is to *see* sharing in action:

```bash
# Pull a base image once.
docker pull node:20-alpine                                         # → first time: ~40 MB downloaded

# Pull anything else built on the same base — see how fast.
docker pull keymetrics/pm2:latest-alpine                           # → only the new layers download

# See the layer reuse.
docker image history node:20-alpine
docker image history keymetrics/pm2:latest-alpine
```

The bottom layers will be byte-identical (same SHA-256). That is the layered filesystem paying you back.

## Dọn dẹp

<Recap>
- Image is a read-only stack of layers (built once, shared). Container = image + a per-instance writable layer. `docker rm` only deletes the writable layer; the image stays.
- Lifecycle: created → running → paused / stopped → removed. Each transition is a different command (`create`, `start`, `pause`, `stop`, `rm`).
- Layers are content-addressed by SHA-256, so two images sharing a base do not store it twice. `docker image history <name>` shows the layers.
- Day-one flags: `--rm` (auto-clean throwaway containers), `-it` (interactive shell), `-d` (background), `-p` (ports), `--name` (memorable name).
</Recap>
