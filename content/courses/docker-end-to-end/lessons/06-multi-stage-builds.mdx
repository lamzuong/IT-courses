- The single biggest optimisation a Dockerfile can give you: building in one image and shipping in another.
- The exact pattern for a Node app: from 1.2 GB single-stage image down to 90 MB multi-stage image.
- How `COPY --from=builder` actually works.
- Multi-stage for Next.js, for a Go binary, for Python. Same idea, three flavours.
</Callout>

You have a problem. The image you built in lesson 4 contains:

- The Alpine base (~10 MB) — needed at runtime.
- The Node.js runtime (~30 MB) — needed at runtime.
- Your source code (~3 MB) — needed at runtime.
- `node_modules` with `devDependencies` (~200 MB) — *only needed at build*.
- The TypeScript compiler, build tools, test runners (~400 MB) — *only needed at build*.
- npm's cache (~100 MB) — *not needed at all*.

You are shipping nearly a gigabyte of stuff to your servers that has no business being there. It is slow to push, slow to pull, and a security surface (every build tool is a potential attack target).

**Multi-stage builds** are the fix. One Dockerfile, two `FROM`s. The first stage is your *workshop* — it has the compilers, the dev deps, all the heavy tools. The second stage is your *shop window* — it has only the finished artifact and the bare minimum to run it. When the build finishes, only the final stage is kept. The workshop is thrown away.

Bạn có một vấn đề. Image build ở bài 4 chứa:

- Alpine base (~10 MB) — runtime cần.
- Runtime Node.js (~30 MB) — runtime cần.
- Source code của bạn (~3 MB) — runtime cần.
- `node_modules` kèm `devDependencies` (~200 MB) — *chỉ cần lúc build*.
- TypeScript compiler, build tool, test runner (~400 MB) — *chỉ cần lúc build*.
- Cache của npm (~100 MB) — *không cần luôn*.

Bạn đang ship gần 1 GB rác lên server, không có lý do gì để chúng ở đó. Push chậm, pull chậm, và là bề mặt bảo mật (mỗi build tool là một mục tiêu tấn công tiềm năng).

**Multi-stage build** là lời giải. Một Dockerfile, hai `FROM`. Stage đầu là *xưởng* — có compiler, dev deps, công cụ nặng. Stage hai là *tủ trưng bày* — chỉ có artifact đã hoàn thiện và mức tối thiểu để chạy. Khi build xong, chỉ stage cuối được giữ. Xưởng bị vứt đi.

## A real multi-stage Dockerfile for a TypeScript Node app

```dockerfile
# ── Stage 1 ── builder ────────────────────────────────────────────
FROM node:20 AS builder                                            # → full Node image; has python, gcc for native modules
WORKDIR /app

COPY package*.json ./
RUN npm ci                                                          # → install ALL deps (dev + prod) for the build

COPY tsconfig.json ./
COPY src ./src
RUN npm run build                                                   # → produces /app/dist

# ── Stage 2 ── runtime ────────────────────────────────────────────
FROM node:20-alpine AS runtime                                      # → tiny base; no compilers
WORKDIR /app

ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev                                               # → prod deps only — much smaller

COPY --from=builder /app/dist ./dist                                # → bring only the compiled output across

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Read it top to bottom:

1. `FROM node:20 AS builder` — first stage. The `AS builder` label is the handle other stages will use to refer back to it.
2. We install **all** dependencies, including dev, because we need the TypeScript compiler.
3. We compile: `npm run build` runs `tsc`, output goes to `/app/dist`.
4. `FROM node:20-alpine AS runtime` — second stage. **Completely fresh image.** It does *not* inherit anything from the builder.
5. We install **only production** deps. Smaller and faster.
6. `COPY --from=builder /app/dist ./dist` — *the* multi-stage line. Reach back into the builder stage's filesystem and copy that one folder over.
7. The final image contains: alpine + node + prod node_modules + dist. Nothing else.

Build and compare:

```bash
docker build -t myapp:multi .
docker images myapp                                                 # → look at the SIZE column. ~90 MB vs ~1.2 GB.
```

## Một Dockerfile multi-stage thật cho app Node + TypeScript

```dockerfile
# ── Stage 1 ── builder ────────────────────────────────────────────
FROM node:20 AS builder                                            # → image Node đầy đủ; có python, gcc cho native module
WORKDIR /app

COPY package*.json ./
RUN npm ci                                                          # → cài TOÀN BỘ deps (dev + prod) để build

COPY tsconfig.json ./
COPY src ./src
RUN npm run build                                                   # → sinh /app/dist

# ── Stage 2 ── runtime ────────────────────────────────────────────
FROM node:20-alpine AS runtime                                      # → base nhỏ; không có compiler
WORKDIR /app

ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev                                               # → chỉ prod deps — nhỏ hơn nhiều

COPY --from=builder /app/dist ./dist                                # → mang đúng folder output sang

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Đọc từ trên xuống:

1. `FROM node:20 AS builder` — stage đầu. Nhãn `AS builder` là tên các stage khác sẽ dùng để gọi ngược về nó.
2. Cài **tất cả** dependency, kể cả dev, vì cần TypeScript compiler.
3. Compile: `npm run build` chạy `tsc`, output ra `/app/dist`.
4. `FROM node:20-alpine AS runtime` — stage hai. **Image hoàn toàn mới.** *Không* kế thừa bất cứ gì từ builder.
5. Chỉ cài **prod** deps. Nhỏ hơn, nhanh hơn.
6. `COPY --from=builder /app/dist ./dist` — *dòng* multi-stage. Vươn vào filesystem của stage builder và copy đúng cái folder đó qua.
7. Image cuối chứa: alpine + node + prod node_modules + dist. Hết.

Build và so sánh:

```bash
docker build -t myapp:multi .
docker images myapp                                                 # → nhìn cột SIZE. ~90 MB so với ~1.2 GB.
```

## `--from` tham chiếu được những gì

</VI>

<EN>

```dockerfile
COPY --from=builder /app/dist ./dist                                # → another stage by name
COPY --from=0       /app/dist ./dist                                # → another stage by index (0 = first FROM)
COPY --from=nginx:alpine /etc/nginx/conf.d /default-conf            # → an external image, not in this file!
```

The third form is wildly useful. You can yank a config file out of the published `nginx` image without ever installing nginx in your build context. Same idea for grabbing tools (`busybox`, `curl-tool`, certificate bundles).

## Multi-stage for Next.js (the standalone output trick)

Next.js has a built-in optimisation called the **standalone output**. Set `output: 'standalone'` in `next.config.js` and `next build` produces a self-contained `.next/standalone/` folder with only the dependencies the app actually imports — sometimes 90% smaller than the full `node_modules`.

```dockerfile
# ── deps ───────────────────────────────────────────────────────────
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

# ── builder ────────────────────────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build                                                   # → produces .next/standalone

# ── runtime ────────────────────────────────────────────────────────
FROM node:20-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production

COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000
CMD ["node", "server.js"]                                           # → Next produces this entrypoint
```

Three stages this time:

- `deps` — install dependencies. Cached as long as `package.json` is unchanged.
- `builder` — reuse the deps layer, copy source, run `next build`.
- `runtime` — copy only the standalone server + static assets. Final image often <100 MB even for a sizeable Next app.

## Multi-stage cho Next.js (mẹo standalone output)

Next.js có một tối ưu tích hợp tên là **standalone output**. Đặt `output: 'standalone'` trong `next.config.js`, và `next build` sinh ra folder `.next/standalone/` tự đủ chỉ với những dependency app thật sự import — đôi khi nhỏ hơn `node_modules` đầy đủ 90%.

```dockerfile
# ── deps ───────────────────────────────────────────────────────────
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

# ── builder ────────────────────────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build                                                   # → sinh .next/standalone

# ── runtime ────────────────────────────────────────────────────────
FROM node:20-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production

COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000
CMD ["node", "server.js"]                                           # → Next sinh sẵn entrypoint này
```

Lần này 3 stage:

- `deps` — cài dependency. Còn cache bao lâu `package.json` không đổi.
- `builder` — tái dùng layer deps, copy source, chạy `next build`.
- `runtime` — copy chỉ server standalone + asset tĩnh. Image cuối thường <100 MB ngay cả với Next app cỡ vừa.

## Multi-stage cho binary Go (cú đẹp nhất)

</VI>

<EN>

This is where the technique really shines. Go compiles to a static binary. The "runtime" needs nothing — no language runtime, no libc, often not even a shell.

```dockerfile
FROM golang:1.23 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/server ./cmd/server   # → static binary, ~15 MB

FROM scratch                                                        # → literally empty. No OS.
COPY --from=builder /out/server /server
EXPOSE 8080
ENTRYPOINT ["/server"]
```

Final image: ~15 MB. There is *nothing* in it except your binary. No shell to exec into, no curl, no anything. Hard to attack a container that has no commands.

## Multi-stage for Python (deps in builder, copy site-packages over)

```dockerfile
FROM python:3.13-slim AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends gcc \
 && rm -rf /var/lib/apt/lists/*
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt           # → installs into /root/.local

FROM python:3.13-slim AS runtime
WORKDIR /app
COPY --from=builder /root/.local /root/.local                       # → packages, no gcc
COPY . .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "-m", "myapp"]
```

Same trick: builder has the compiler for native wheels, runtime gets only the resulting `site-packages` via `--user`'s tidy install location.

## Building only one stage

```bash
docker build --target builder -t myapp:dev .                        # → stop at the builder stage
docker run --rm -it myapp:dev sh                                    # → poke around the dev image (has TS, etc.)
```

`--target` lets you build only up to a named stage. Useful for a dev image that includes the toolchain, while the default build produces the slim runtime image. One Dockerfile, two outputs.

## Three common mistakes

1. **Copying `node_modules` from builder to runtime** — you copy a folder full of compiled native binaries that were built against the *builder* image's libc. Often works, sometimes does not (musl vs glibc). Safer: run `npm ci --omit=dev` in the runtime stage and only `COPY --from=builder` the *compiled output*.
2. **Forgetting `--omit=dev` in runtime** — defeats the whole point. The whole reason to have a runtime stage is to ditch dev deps.
3. **Using `AS` labels that collide with Docker keywords** — `AS production` is fine; `AS final` is fine; `AS RUN` will confuse things. Stick to lowercase descriptive names.

</EN>

## Build chỉ một stage

```bash
docker build --target builder -t myapp:dev .                        # → dừng ở stage builder
docker run --rm -it myapp:dev sh                                    # → vào image dev khám phá (có TS, v.v.)
```

`--target` cho phép build chỉ tới một stage nhất định. Hữu ích khi muốn image dev có toolchain, còn build mặc định cho image runtime gọn. Một Dockerfile, hai output.

## Ba lỗi hay gặp

1. **Copy `node_modules` từ builder sang runtime** — bạn đang copy folder chứa binary native build cho libc của image *builder*. Đôi khi chạy, đôi khi không (musl vs glibc). An toàn hơn: chạy `npm ci --omit=dev` trong stage runtime và chỉ `COPY --from=builder` *output đã compile*.
2. **Quên `--omit=dev` trong runtime** — phá toàn bộ ý nghĩa. Lý do duy nhất có stage runtime là vứt dev deps.
3. **Đặt nhãn `AS` trùng từ khoá Docker** — `AS production` được; `AS final` được; `AS RUN` thì rắc rối. Dùng chữ thường mô tả là an toàn.

</VI>

<Recap>
- Multi-stage = two (or more) `FROM` lines. Build in one, ship from another. The build stage is discarded.
- `COPY --from=<stage>` is the bridge — only the named files cross over. Everything else from the build stage is gone.
- For Node: builder has all deps + source, runtime has only prod deps + compiled output. Typical reduction: 1.2 GB → 90 MB.
- For Go: builder + `FROM scratch`. Final image is just your binary. Tens of MB. Lesson 7 takes the next step — `.dockerignore`, distroless images, BuildKit cache mounts.
</Recap>
