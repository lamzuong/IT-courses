Real apps need configuration: database URLs, API keys, feature flags, log levels. Docker gives you four places to put that configuration, and choosing the wrong place is how secrets end up in public images.

The decision is mostly about *when the value is known* and *who is allowed to see it*.

### 1. `ARG` — build-time variables

`ARG` declares a variable that is available **only during the build**. The dockerfile author passes it in with `--build-arg`.

```dockerfile
ARG NODE_VERSION=20                                                 # → default value
FROM node:${NODE_VERSION}-alpine

ARG BUILD_NUMBER                                                    # → no default; must be passed
RUN echo "built as $BUILD_NUMBER" > /etc/build-info
```

```bash
docker build --build-arg BUILD_NUMBER=1234 -t myapp .                # → set BUILD_NUMBER for this build
```

**Critical caveat:** `ARG` values are *visible in the image history*. `docker image history myapp` and `docker history --no-trunc myapp` will show them. **Never** use `ARG` for a secret.

### 2. `ENV` — image-baked environment variables

`ENV` sets a variable that is part of the image and visible to processes inside it via `process.env.X` / `os.environ['X']` / `$X`.

```dockerfile
ENV NODE_ENV=production
ENV PORT=3000
ENV LOG_LEVEL=info
```

Use for: **non-secret defaults** and runtime constants. `NODE_ENV=production` is the classic example.

Same critical caveat: `ENV` values are baked into the image. Anyone who pulls the image and runs `docker image inspect myapp` will see them in clear text. **Never** use `ENV` for a password, API key, or any other secret.

### 3. Runtime — `-e` and `--env-file`

The right place for *all* dynamic config and *most* secrets. The variables exist only for the lifetime of the container; they are not part of the image.

```bash
# One variable at a time.
docker run -d -e DATABASE_URL='postgres://app:secret@db:5432/coffee' myapp

# A whole file of variables — best for local dev.
docker run -d --env-file ./.env myapp

# Pull from your shell.
docker run -d -e API_KEY="$API_KEY" myapp                            # → forwards your shell's value
```

A `.env` file is just `KEY=value` per line, no quotes:

```text
# .env  (NEVER commit this)
DATABASE_URL=postgres://app:secret@db:5432/coffee
JWT_SECRET=long-random-string
REDIS_URL=redis://cache:6379
```

Add `.env` and `.env.*` to `.gitignore` and to `.dockerignore`. Keep a `.env.example` checked in with empty/placeholder values, so new developers know which variables to set.

### 4. BuildKit secrets — credentials for build steps

Sometimes you need a credential *during the build* (a private npm token to pull a packaged library, an AWS key to pull a model file). You cannot use `ARG`/`ENV` because they leak. BuildKit's `--secret` solves this: the secret is mounted into a `RUN` step as a file, and the mount **is not retained in the image**.

```dockerfile
# syntax=docker/dockerfile:1.7
FROM node:20-alpine
WORKDIR /app
COPY package*.json .npmrc.template ./
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) \
 && envsubst < .npmrc.template > .npmrc \
 && npm ci --omit=dev \
 && rm .npmrc
```

```bash
docker build --secret id=npm_token,src=$HOME/.npm-private-token -t myapp .
```

The token file is mounted at `/run/secrets/npm_token` *only while that RUN runs*. After it finishes, no trace in any layer. `docker image inspect myapp` will not reveal it.

## How they compose in Compose

</EN>

<VI>

### 1. `ARG` — biến lúc build

`ARG` khai báo biến **chỉ có lúc build**. Người viết Dockerfile truyền vào bằng `--build-arg`.

```dockerfile
ARG NODE_VERSION=20                                                 # → giá trị mặc định
FROM node:${NODE_VERSION}-alpine

ARG BUILD_NUMBER                                                    # → không mặc định; phải truyền
RUN echo "built as $BUILD_NUMBER" > /etc/build-info
```

```bash
docker build --build-arg BUILD_NUMBER=1234 -t myapp .                # → set BUILD_NUMBER cho build này
```

**Cảnh báo quan trọng:** Giá trị `ARG` *thấy được trong image history*. `docker image history myapp` và `docker history --no-trunc myapp` sẽ in ra. **Đừng bao giờ** dùng `ARG` cho secret.

### 2. `ENV` — biến môi trường nướng vào image

`ENV` đặt biến thuộc image, các process bên trong đọc qua `process.env.X` / `os.environ['X']` / `$X`.

```dockerfile
ENV NODE_ENV=production
ENV PORT=3000
ENV LOG_LEVEL=info
```

Dùng cho: **mặc định không phải secret** và hằng số runtime. `NODE_ENV=production` là ví dụ kinh điển.

Cùng cảnh báo: `ENV` được nướng vào image. Ai pull image rồi chạy `docker image inspect myapp` cũng thấy clear text. **Đừng bao giờ** đặt password, API key, hay secret nào khác vào `ENV`.

### 3. Runtime — `-e` và `--env-file`

Đúng chỗ cho *mọi* config động và *phần lớn* secret. Biến chỉ sống suốt đời container; không thuộc image.

```bash
# Từng biến một.
docker run -d -e DATABASE_URL='postgres://app:secret@db:5432/coffee' myapp

# Cả file biến — tốt cho dev local.
docker run -d --env-file ./.env myapp

# Lấy từ shell hiện tại.
docker run -d -e API_KEY="$API_KEY" myapp                            # → chuyển tiếp giá trị từ shell
```

File `.env` chỉ là `KEY=value` mỗi dòng, không quote:

```text
# .env  (TUYỆT ĐỐI không commit)
DATABASE_URL=postgres://app:secret@db:5432/coffee
JWT_SECRET=chuoi-random-rat-dai
REDIS_URL=redis://cache:6379
```

Thêm `.env` và `.env.*` vào `.gitignore` và `.dockerignore`. Giữ một file `.env.example` commit lên repo với giá trị trống / placeholder, để dev mới biết phải đặt biến nào.

### 4. BuildKit secret — credential cho bước build

Đôi khi bạn cần credential *trong lúc build* (npm token private để pull thư viện riêng, AWS key để tải file model). Không dùng được `ARG`/`ENV` vì chúng rò. `--secret` của BuildKit giải: secret được mount vào một bước `RUN` dưới dạng file, và mount đó **không được giữ trong image**.

```dockerfile
# syntax=docker/dockerfile:1.7
FROM node:20-alpine
WORKDIR /app
COPY package*.json .npmrc.template ./
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) \
 && envsubst < .npmrc.template > .npmrc \
 && npm ci --omit=dev \
 && rm .npmrc
```

```bash
docker build --secret id=npm_token,src=$HOME/.npm-private-token -t myapp .
```

File token được mount ở `/run/secrets/npm_token` *chỉ trong lúc RUN đó chạy*. Sau đó, không dấu vết trong bất cứ layer nào. `docker image inspect myapp` không thấy nó.

## Compose dùng các thứ này ra sao

</VI>

<EN>

Compose pulls all four together cleanly:

```yaml
services:
  api:
    build:
      context: ./api
      args:
        BUILD_NUMBER: ${BUILD_NUMBER}                                # → ARG, interpolated from your shell or .env
    image: coffee-api:${TAG:-latest}
    env_file: .env                                                  # → --env-file equivalent
    environment:                                                    # → -e equivalent; wins over env_file
      NODE_ENV: production
      DATABASE_URL: ${DATABASE_URL}                                  # → variable substitution from .env or shell
    secrets:
      - postgres_password                                            # → mounts /run/secrets/postgres_password

secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt                            # → file on the host, not in image
```

Two important details:

1. Compose **automatically reads `.env`** in the project directory and substitutes `${VAR}` placeholders into the YAML. That is how `${TAG:-latest}` and `${DATABASE_URL}` get filled in.
2. **`environment` overrides `env_file`** when both set the same key. Useful for one-off tweaks.

## The secret-baking trap

A classic mistake:

```dockerfile
# ❌ Never do this.
ENV DB_PASSWORD=hunter2
```

Then someone pulls the image and runs:

```bash
docker image inspect myapp --format '{{json .Config.Env}}'
# → ["NODE_ENV=production","DB_PASSWORD=hunter2"]      ← secret right there
```

Or this:

```dockerfile
# ❌ Equally bad.
ARG NPM_TOKEN
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > /root/.npmrc \
 && npm ci \
 && rm /root/.npmrc                                                  # → ARG value still in history!
```

Even though you removed `.npmrc`, the `ARG NPM_TOKEN=…` value remains in `docker history`. Use `--secret` instead.

## Variable precedence (in Compose)

When a variable comes from multiple sources, the *later* one wins. The order from low to high precedence:

1. Defaults in the Dockerfile (`ENV NODE_ENV=development`).
2. Variables in `env_file:`.
3. Inline `environment:` in the Compose file.
4. Shell variables (when running `docker compose up`).
5. Variables explicitly passed with `docker compose run -e VAR=value`.

So you can override anything when debugging without editing config files:

```bash
LOG_LEVEL=debug docker compose up                                    # → wins over env_file's LOG_LEVEL=info
```

## Sanity-check what your container actually sees

</EN>

<VI>

Compose ráp cả bốn cách rất gọn:

```yaml
services:
  api:
    build:
      context: ./api
      args:
        BUILD_NUMBER: ${BUILD_NUMBER}                                # → ARG, lấy từ shell hoặc .env
    image: coffee-api:${TAG:-latest}
    env_file: .env                                                  # → tương đương --env-file
    environment:                                                    # → tương đương -e; thắng env_file
      NODE_ENV: production
      DATABASE_URL: ${DATABASE_URL}                                  # → biến thay từ .env hoặc shell
    secrets:
      - postgres_password                                            # → mount /run/secrets/postgres_password

secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt                            # → file trên host, không vào image
```

Hai chi tiết quan trọng:

1. Compose **tự đọc `.env`** trong thư mục project và thay `${VAR}` trong YAML. Nhờ vậy `${TAG:-latest}` và `${DATABASE_URL}` được điền.
2. **`environment` thắng `env_file`** khi cùng đặt một key. Hữu ích để override nhanh.

## Bẫy nướng secret

Lỗi kinh điển:

```dockerfile
# ❌ Đừng bao giờ làm vậy.
ENV DB_PASSWORD=hunter2
```

Rồi ai đó pull image và chạy:

```bash
docker image inspect myapp --format '{{json .Config.Env}}'
# → ["NODE_ENV=production","DB_PASSWORD=hunter2"]      ← secret nằm ngay đó
```

Hoặc thế này:

```dockerfile
# ❌ Tệ tương đương.
ARG NPM_TOKEN
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > /root/.npmrc \
 && npm ci \
 && rm /root/.npmrc                                                  # → giá trị ARG vẫn nằm trong history!
```

Dù bạn xoá `.npmrc`, giá trị `ARG NPM_TOKEN=…` vẫn nằm trong `docker history`. Dùng `--secret` thay vì vậy.

## Thứ tự ưu tiên biến (trong Compose)

Khi biến đến từ nhiều nguồn, cái *sau* thắng. Thứ tự từ thấp → cao:

1. Mặc định trong Dockerfile (`ENV NODE_ENV=development`).
2. Biến trong `env_file:`.
3. `environment:` inline trong file Compose.
4. Biến shell (khi chạy `docker compose up`).
5. Biến truyền tường minh với `docker compose run -e VAR=value`.

Vì vậy có thể override bất cứ gì khi debug mà không sửa file config:

```bash
LOG_LEVEL=debug docker compose up                                    # → thắng LOG_LEVEL=info trong env_file
```

## Kiểm tra container thật sự thấy gì

</VI>

<EN>

When a service does not behave as expected, *first* check what env it actually sees:

```bash
docker exec api env                                                  # → dump all env vars inside the container
docker exec api printenv DATABASE_URL                                # → just one
docker compose config                                                # → print the fully-rendered Compose, with substitutions
```

`docker compose config` is the secret weapon. It shows you the YAML *after* `.env` interpolation, so you can verify the values about to be used without starting any container.

## Café analogy

</EN>

<VI>

Khi một service hành xử lạ, *đầu tiên* xem nó thấy env gì:

```bash
docker exec api env                                                  # → dump mọi env var trong container
docker exec api printenv DATABASE_URL                                # → chỉ một biến
```

`docker compose config` là vũ khí bí mật. Nó hiện YAML *sau* khi interpolate `.env`, để bạn kiểm tra giá trị sắp dùng mà không phải start container nào.

## Ví dụ quán cà phê

<Callout type="note" title="Café analogy">
**EN —** A coffee shop has four places it can write information. **`ARG`** is a *chalk note on the kitchen door for one shift* — gone when service ends, but visible to anyone walking past during the shift. **`ENV`** is *printed on the apron* — every barista who wears the apron sees it; do not put your bank PIN there. **`-e` at runtime** is the *headset the manager gives the barista for that shift* — heard only inside the shop, removed at the end. **BuildKit secrets** are *the supplier's password whispered for one delivery* — used once, nothing written down.

**VI —** Quán cà phê có bốn chỗ để ghi thông tin. **`ARG`** là *ghi chú phấn trên cửa bếp cho một ca* — hết ca thì xoá, nhưng ai đi ngang trong ca cũng đọc được. **`ENV`** là *in lên tạp dề* — barista nào mặc cũng thấy; đừng nhét mã PIN ngân hàng vào đó. **`-e` lúc runtime** là *tai nghe quản lý phát cho barista trong ca đó* — chỉ trong quán nghe, tan ca thì trả lại. **BuildKit secret** là *mật khẩu nhà cung cấp thì thầm cho một lần giao* — dùng một lần, không ghi lại.
</Callout>

<Recap>
- Four places: `ARG` (build, visible in history), `ENV` (image-baked, visible in inspect), runtime `-e`/`--env-file` (per-container, safe), BuildKit `--secret` (build, not retained).
- Anything secret → runtime env or BuildKit secret. Anything not secret → `ENV` defaults are fine.
- In Compose: `.env` is auto-interpolated, `env_file` loads bulk vars, `environment` overrides per-service. Use `docker compose config` to see the rendered result.
- The `.env` file lives outside git and outside the image. `.env.example` lives inside git. Lesson 12 closes Part 3 with logs, exec, and how to debug a container without rebuilding.
</Recap>
