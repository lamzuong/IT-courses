You can run a single container and have it talk to the world via `-p` ports. The interesting question starts when you want **two containers to talk to each other**. The answer is networks — and once you understand it, every Compose file becomes readable.

## The thing that surprises everyone

Spin up two containers and try this:

```bash
docker run -d --name api1 -p 4000:3000 mywebapp                      # → API on host port 4000
docker run -d --name api2 mywebapp                                   # → no port published

docker exec -it api2 sh
# inside api2:
curl http://localhost:4000                                            # → ❌ connection refused
```

Surprising? Each container has its own loopback interface. Inside `api2`, `localhost` means *"this container."* `api1` is a different container with a different `localhost`. They might as well be on different laptops.

To make them talk, both have to be **on the same Docker network**.

Chạy một container lẻ, gọi ra thế giới qua port `-p` thì dễ. Câu hỏi thú vị bắt đầu khi bạn muốn **hai container nói chuyện với nhau**. Lời đáp là *network* — hiểu xong, mọi file Compose đều đọc được.

## Cái khiến mọi người ngạc nhiên

Bật hai container, thử:

```bash
docker run -d --name api1 -p 4000:3000 mywebapp                      # → API trên host port 4000
docker run -d --name api2 mywebapp                                   # → không publish port

docker exec -it api2 sh
# bên trong api2:
curl http://localhost:4000                                            # → ❌ connection refused
```

Ngạc nhiên chưa? Mỗi container có một loopback riêng. Trong `api2`, `localhost` nghĩa là *"chính container này"*. `api1` là một container khác với `localhost` khác. Coi như nằm trên hai cái laptop khác nhau.

Để chúng nói chuyện, cả hai phải **ở chung một Docker network**.

### Bridge — the default, and almost always what you want

A **bridge network** is a private virtual switch the daemon manages. Containers attached to the same bridge can see each other by IP and — critically — **by name**. Traffic from the bridge to the outside goes through NAT, like your home router.

```bash
docker network create coffee-net                                     # → custom bridge network
docker run -d --network coffee-net --name db postgres:16-alpine
docker run -d --network coffee-net --name api node:20-alpine sleep infinity
docker exec -it api sh
# inside api:
ping db                                                               # → resolves and replies
curl http://db:5432                                                   # → talks to Postgres at hostname 'db'
```

**Why `db` works as a hostname:** Docker runs an embedded DNS resolver. On a user-defined bridge, every container's `--name` becomes a DNS name that other containers on the same network can resolve. This is the *foundation* of every Compose file you will ever write.

A subtlety: the **default** bridge network (`docker0`, used if you do not pass `--network`) does *not* do automatic DNS by name — only IP. That is why you should **create your own bridge networks** and stop using the default. The fix is one extra `docker network create` per project.

### Host — no isolation, raw performance

```bash
docker run --rm --network host nginx                                 # → nginx binds to host's port 80 directly
```

`--network host` removes the container's network namespace entirely. The container shares the host's network stack. Port 80 inside the container = port 80 on your host. No `-p` flag needed; no NAT overhead.

Use it for:

- Latency-sensitive workloads where the bridge's NAT is measurably hurting.
- Tools that need to see the host's network interfaces directly (intrusion detection, traffic capture).

**Caveats:** Linux only. On Mac/Windows, Docker Desktop's VM means `--network host` actually means *"host of the VM"* not *"your laptop"*, which is rarely what people want.

### None — no network at all

```bash
docker run --rm --network none alpine ip addr                        # → only `lo` is present
```

The container has no external network. Loopback only. Useful for sandboxing untrusted code, batch jobs that should not phone home, or tests of *"how does my app behave with no network?"*.

## Service discovery by name

This is the single most important pattern in multi-container apps. Read it twice.

> *"On a user-defined bridge network, every container's name is a DNS hostname that every other container on the same network can resolve."*

That is why a Compose file like this works:

```yaml
services:
  api:
    build: ./api
    environment:
      DATABASE_URL: postgres://app:secret@db:5432/coffee              # → 'db' is a hostname!
      REDIS_URL:    redis://cache:6379                                # → 'cache' is a hostname!
  db:
    image: postgres:16-alpine
    environment: { POSTGRES_PASSWORD: secret }
  cache:
    image: redis:7-alpine
```

`api` connects to `db:5432` and `cache:6379`. It is not magic — Compose creates a bridge network, attaches all three services to it, and each service name becomes a DNS name. Lesson 14 covers how Compose wires it under the hood.

## `-p` vs network membership: two different ideas

</EN>

### Bridge — mặc định, gần như luôn là cái bạn muốn

**Bridge network** là một switch ảo private mà daemon quản lý. Container gắn vào cùng bridge thấy được nhau qua IP và — quan trọng nhất — **qua tên**. Lưu lượng từ bridge ra ngoài đi qua NAT, giống router nhà bạn.

```bash
docker network create coffee-net                                     # → bridge network tự đặt
docker run -d --network coffee-net --name db postgres:16-alpine
docker run -d --network coffee-net --name api node:20-alpine sleep infinity
docker exec -it api sh
# bên trong api:
ping db                                                               # → resolve và phản hồi
curl http://db:5432                                                   # → gọi Postgres qua hostname 'db'
```

**Vì sao `db` chạy như hostname:** Docker chạy một DNS resolver tích hợp. Trên bridge tự tạo, mỗi `--name` của container thành một DNS name mà các container khác cùng network resolve được. Đây là *nền tảng* của mọi file Compose bạn sẽ viết.

Một chi tiết tinh tế: bridge **mặc định** (`docker0`, dùng nếu bạn không truyền `--network`) *không* tự DNS theo tên — chỉ theo IP. Vì vậy bạn nên **tự tạo bridge** và bỏ dùng cái mặc định. Sửa: thêm đúng một `docker network create` mỗi project.

### Host — không cô lập, perf max

```bash
docker run --rm --network host nginx                                 # → nginx bind thẳng port 80 của host
```

`--network host` gỡ bỏ namespace network của container. Container dùng chung network stack với host. Port 80 trong container = port 80 trên host. Không cần `-p`; không NAT overhead.

Dùng khi:

- Workload nhạy latency, NAT của bridge ảnh hưởng đo được.
- Công cụ cần thấy interface mạng của host trực tiếp (IDS, capture traffic).

**Cẩn thận:** Chỉ Linux. Trên Mac/Win, Docker Desktop có VM nên `--network host` thực ra là *"host của VM"* chứ không phải *"laptop của bạn"*, hiếm khi là cái bạn muốn.

### None — không network nào

```bash
docker run --rm --network none alpine ip addr                        # → chỉ có `lo`
```

Container không có network ngoài. Chỉ loopback. Dùng cho: sandbox code không tin được, batch job không nên "phone home", test *"app cư xử ra sao khi không mạng?"*.

## Service discovery theo tên

Đây là pattern *quan trọng nhất* trong app nhiều container. Đọc hai lần.

> *"Trên bridge tự tạo, tên mỗi container là một DNS hostname mà mọi container khác cùng network resolve được."*

Vì vậy một file Compose như này chạy được:

```yaml
services:
  api:
    build: ./api
    environment:
      DATABASE_URL: postgres://app:secret@db:5432/coffee              # → 'db' là hostname!
      REDIS_URL:    redis://cache:6379                                # → 'cache' là hostname!
  db:
    image: postgres:16-alpine
    environment: { POSTGRES_PASSWORD: secret }
  cache:
    image: redis:7-alpine
```

`api` kết nối tới `db:5432` và `cache:6379`. Không phép thuật — Compose tạo một bridge network, gắn cả ba service vào đó, và mỗi tên service thành một DNS name. Bài 14 chỉ Compose ráp việc này ra sao bên dưới.

## `-p` vs gia nhập network: hai ý khác nhau

</VI>

<EN>

These two are constantly confused. They are independent.

- **`--network <name>`** controls *which other containers can talk to me*. Internal-only routing.
- **`-p host:container`** controls *whether the host (and the outside world) can talk to me*. External publishing.

A typical production setup:

```bash
docker network create coffee-net

# Only the proxy publishes a port to the host.
docker run -d --network coffee-net -p 80:80 --name nginx nginx
# API and DB do NOT publish ports. They are only reachable from inside coffee-net.
docker run -d --network coffee-net --name api mywebapp
docker run -d --network coffee-net --name db  -e POSTGRES_PASSWORD=secret postgres:16-alpine
```

The API and DB are *unreachable* from your laptop. They can only be reached by other containers on `coffee-net`. The only public surface is nginx on port 80. That is exactly how the project in lesson 15 is wired.

## Inspecting and debugging networks

```bash
docker network ls                                                    # → list networks
docker network inspect coffee-net                                    # → JSON; lists every container, its IP, the subnet
docker network connect coffee-net api                                # → attach an existing container to a network
docker network disconnect coffee-net api                             # → detach it

# DNS sanity check from inside a container:
docker exec -it api getent hosts db                                  # → IP of 'db' on this network
docker exec -it api nslookup db                                      # → if 'dns-utils' is installed

# Port reachability from inside a container:
docker exec -it api wget -O- http://db:5432 --timeout=2              # → opens a TCP connection
```

`docker network inspect` is the X-ray. If a service cannot reach another, this command tells you (a) whether they share a network, (b) what IPs they have, (c) whether DNS is set up.

## Common confusions

1. **"I can reach the API from my laptop with `localhost:4000` but the web container can't reach `localhost:4000`"** — exactly. Inside the web container, `localhost` is the *web container*. Use the API service name and the *container* port: `http://api:3000`.
2. **"I bridged two containers but they cannot ping each other"** — you are probably on the *default* bridge, which lacks automatic DNS. Create a user-defined bridge with `docker network create` and attach both.
3. **"`-p 80:80` works on my Mac but `--network host` doesn't"** — Docker Desktop runs a Linux VM; `host` means the VM's host (i.e., the VM itself), not your Mac. Stick with bridge + `-p` on Desktop.

## Café analogy

</EN>

<VI>

Hai ý này rất hay bị lẫn. Chúng độc lập.

- **`--network <name>`** quyết định *container nào khác nói chuyện với tôi được*. Định tuyến nội bộ.
- **`-p host:container`** quyết định *host (và thế giới bên ngoài) gọi tới tôi được không*. Public ra ngoài.

Layout sản xuất điển hình:

```bash
docker network create coffee-net

# Chỉ proxy publish port ra host.
docker run -d --network coffee-net -p 80:80 --name nginx nginx
# API và DB KHÔNG publish port. Chỉ container trong coffee-net gọi được.
docker run -d --network coffee-net --name api mywebapp
docker run -d --network coffee-net --name db  -e POSTGRES_PASSWORD=secret postgres:16-alpine
```

API và DB *không gọi tới được* từ laptop bạn. Chỉ container khác trong `coffee-net` gọi được. Bề mặt public duy nhất là nginx trên port 80. Đó chính xác là cách project bài 15 được ráp.

## Inspect và debug network

```bash
docker network ls                                                    # → list network
docker network inspect coffee-net                                    # → JSON; liệt kê mọi container, IP, subnet
docker network connect coffee-net api                                # → gắn container đã có vào một network
docker network disconnect coffee-net api                             # → tháo ra

# Kiểm tra DNS từ trong container:
docker exec -it api getent hosts db                                  # → IP của 'db' trên network này
docker exec -it api nslookup db                                      # → nếu có 'dns-utils'

# Kiểm tra port có gọi được không từ trong container:
docker exec -it api wget -O- http://db:5432 --timeout=2              # → mở TCP connection
```

`docker network inspect` là kính X-quang. Nếu một service không gọi được service khác, lệnh này nói cho bạn (a) hai bên có share network không, (b) IP gì, (c) DNS đã set chưa.

## Nhầm lẫn thường gặp

1. **"Từ laptop tôi gọi API bằng `localhost:4000` được, nhưng container web không gọi `localhost:4000` được"** — đúng. Trong container web, `localhost` là *container web*. Dùng tên service API và port *container*: `http://api:3000`.
2. **"Tôi đã cho 2 container chung bridge mà chúng không ping nhau được"** — có lẽ bạn đang trên bridge *mặc định*, không có DNS tự động. Tạo bridge tự đặt bằng `docker network create` và gắn cả hai vào.
3. **"`-p 80:80` chạy trên Mac mà `--network host` không"** — Docker Desktop chạy VM Linux; `host` là host của VM (chính cái VM), không phải Mac của bạn. Trên Desktop, cứ bridge + `-p`.

<Callout type="note" title="Café analogy">
**EN —** Imagine the coffee shop has *intercoms* between stations. The barista has one; the kitchen has one; the cashier has one. They press a button labelled with a name (*"barista"*, *"kitchen"*) and talk to that station — they do not need to know which corner of the building it is in. That is DNS by name on a bridge network. The *front door* is a separate thing — only the cashier's counter touches the outside (publishes a port); customers do not walk straight into the kitchen. The `--network` flag adds you to the intercom system; the `-p` flag puts you at the front door.

**VI —** Hình dung quán có *intercom* nối các trạm. Barista có một cái; bếp có một cái; thu ngân có một cái. Họ bấm nút có ghi tên (*"barista"*, *"bếp"*) và nói chuyện với trạm đó — không cần biết nó nằm góc nào toà nhà. Đó là DNS theo tên trên bridge network. *Cửa trước* là chuyện khác — chỉ quầy thu ngân chạm tới bên ngoài (publish port); khách không đi thẳng vào bếp. Cờ `--network` thêm bạn vào hệ thống intercom; cờ `-p` đặt bạn ra cửa trước.

<Recap>
- Two containers can only talk if they share a network. `localhost` inside a container is *that container*, never another.
- Drivers: **bridge** (default for projects, has DNS by name on user-defined bridges), **host** (no isolation, Linux only), **none** (sandbox).
- DNS by service name (`db`, `cache`, `api`) is the foundation of every Compose file. Always create user-defined bridges; the default bridge lacks name resolution.
- `--network` = who can talk to me internally. `-p host:container` = who can talk to me externally. They are independent flags. Lesson 11 picks up the next runtime concern — env vars, config, secrets.
</Recap>
