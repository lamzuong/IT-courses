The final project is a real, multi-service coffee-shop platform — every concept from the 20 lessons wired into one running stack. You can clone it, type `docker compose up`, and have five services talking to each other in under two minutes.

## What it does

A small online ordering site for a coffee shop. The customer browses a menu (Next.js), places an order (Node/Express API), the order is stored (Postgres), session and cart state are cached (Redis), and the whole thing is reached through a single domain via a reverse proxy (Nginx). Five services, one network, one command to bring up.

## File layout

```
coffee-shop/
  docker-compose.yml             # the whole stack in one file
  .env.example                   # copy to .env, fill secrets
  nginx/
    Dockerfile
    nginx.conf                   # routes / → web, /api → api
  web/                           # Next.js
    Dockerfile                   # multi-stage: deps → build → runner
    package.json
    src/...
  api/                           # Node/Express
    Dockerfile                   # multi-stage: deps → build → runner
    src/index.ts
    src/db.ts                    # talks to postgres by service name 'db'
    src/cache.ts                 # talks to redis by service name 'cache'
  db/
    init.sql                     # schema + seed
  .github/workflows/deploy.yml   # build, push to GHCR, deploy
```

## What you'll demonstrate by finishing it

- **Multi-stage Dockerfiles** (lesson 6) for `web` and `api` — final images under 150 MB each.
- **Named volumes** (lesson 9) for Postgres and Redis — data survives `docker compose down`.
- **Internal-only networking** (lesson 10) — only Nginx is reachable from the host; everything else only talks on `coffee-net`.
- **Env-driven config** (lesson 11) — DB credentials, JWT secret, Redis URL all come from `.env`; nothing baked into the image.
- **Healthchecks** (lesson 16) — `api` waits for `db` to be healthy before starting; Nginx waits for `api`.
- **CI pipeline** (lesson 18) — push to `main` → GitHub Actions builds both images, tags with the commit SHA, pushes to GHCR.

## How long this takes

If you do the lessons alongside building it, two solid weekends. If you just clone the finished repo and study it, an afternoon to read everything and run it.

## Học xong project này bạn sẽ chứng minh được gì

- **Multi-stage Dockerfile** (bài 6) cho `web` và `api` — image cuối dưới 150 MB mỗi cái.
- **Named volume** (bài 9) cho Postgres và Redis — dữ liệu không mất khi `docker compose down`.
- **Network nội bộ** (bài 10) — chỉ Nginx ra ngoài host được; mấy thằng còn lại chỉ nói chuyện trong `coffee-net`.
- **Config qua env** (bài 11) — mật khẩu DB, JWT secret, Redis URL đều đọc từ `.env`; không nhét vào image.
- **Healthcheck** (bài 16) — `api` đợi `db` healthy mới start; Nginx đợi `api` healthy mới start.
- **Pipeline CI** (bài 18) — push lên `main` → GitHub Actions build 2 image, tag theo commit SHA, push lên GHCR.

<Callout type="note" title="Why this project / Tại sao chọn project này">
A single-service Dockerfile is a tutorial. A five-service Compose stack is the actual job. The moment you have services that depend on each other, you need networking, healthchecks, secrets, and proper volume handling — exactly the topics the last three parts of the course cover. By finishing this stack you will have used every command, file, and concept from the course in anger, on something that looks like real production work.
